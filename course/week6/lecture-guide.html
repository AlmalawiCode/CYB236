CH 14 Denial of Service (DoS)

  • Goal: Exhaust server/client resources or request unavailable resources
  • 3 Types: Regex DoS · Logical DoS · DDoS
  • ReDoS: Evil regex (e.g. /^((ab)*)+$/) causes exponential backtracking
    • Time doubles with every 2 chars added to malicious input
    • Safe input = fast; trailing bad char = catastrophic slowdown
  • Logical DoS: Abuse resource-heavy endpoints (SQL joins, DB writes, file backups)
    • Tip: time an endpoint ~100x — if it slows as data grows, it's vulnerable
  • DDoS: Botnet floods at network layer (UDP flooding)
    • Lifecycle: malware → infected devices → attacker commands → flood target
    • IoT = easy target (weak defaults, rare firmware updates)
  • Always test in dev, never production!

CH 15 Exploiting Third-Party Dependencies

  • Risk: Open-source software gets less rigorous review; one vulnerability can affect millions of apps
  • 4 Integration Methods & risks:
    • Git fork/branch – unreviewed upstream code pulled accidentally
    • Self-hosted installer – runs as admin → backdoor RCE risk
    • Copy/paste source – upstream patches never reach your copy
    • Package manager (npm) – one bad package cascades to millions
  • Real npm attacks:
    • left-pad (2016): unpublished package broke global builds
    • eslint-scope (2018): stolen creds → credential harvesting malware
    • event-stream (2018): hidden sub-dep stole Bitcoin wallets
  • CVE Databases: NVD (US DoC, CVSS scored) · Mitre CVE (US DHS, global IDs)
  • npm resolves recursive child deps — you may run thousands of packages you never chose

CH 26 Defending Against DoS

  • Logging first: Record every request + response time, CPU/RAM per endpoint
  • Regex DoS defense:
    • Scan with linting tools: safe-regex, vuln-regex-detector in CI/CD
    • Never allow user-supplied regex — always server-defined only
  • Logical DoS defense (classify endpoints High/Med/Low risk):
    • Rate limiting · Resource quotas · Request timeouts
    • Pagination · Response caching · Async job queues
  • DDoS defense (cannot prevent, only mitigate):
    • Bandwidth mgmt service – vendor absorbs traffic, drops malicious packets
    • Blackholing – route suspicious traffic to decoy servers
    • Traffic baselining – know normal patterns before aggressive filtering
  • Implement anti-DoS proactively — before an attack happens

CH 27 Securing Third-Party Dependencies

  • Dependency tree: Full graph including transitive (4th-party) deps
    • Command: npm ls --depth=3
    • Same library can have vulnerable + patched versions coexisting
  • Automated scanning:
    • npm audit / npm audit fix
    • snyk test / snyk monitor
    • Gate CI/CD pipeline — evaluate every new dep at intake
  • Isolation: Run risky 3rd-party code on its own server
    • Communicate via JSON over HTTP → limits blast radius if compromised
    • Trade-off: HTTP round-trip overhead
  • Secure version pinning (4 techniques):
    • Remove ^ caret → pin exact version in package.json
    • npm shrinkwrap → locks all sub-deps, commit the file
    • Reference Git SHAs → prevents silent re-publish attacks
    • Private npm mirror → CI/CD pulls only vetted packages
  • Defense-in-depth: no single control is enough — layer them
