💡 Introduction

This lab continues from Lab 04. You will explore three more critical vulnerability classes: SQL Injection (escaping a database query), Command Injection (escaping to the host operating system), and CSRF (forging a legitimate user's request from another origin).

All tasks use DVWA — the same environment from Lab 04 — with the Security Level set to Low.

⚠️

Authorized Use Only

All tasks are performed against DVWA running locally on your own Kali machine. These techniques must never be applied to external systems without explicit written permission.

🎯 Learning Objectives

By the end of this lab, you will be able to:

  • Observe how SQL injection escapes a query and returns unintended data
  • Explain the role of the truthy condition (OR 1=1) in a SQL injection attack
  • Observe how command injection appends OS commands to a legitimate server request
  • Distinguish between SQL injection (database) and command injection (operating system)
  • Identify a CSRF vulnerability by observing the GET-based password change and absence of a token
  • Explain why browsers automatically "authorize" requests triggered from other origins

📊 Task 1: SQL Injection

💡

Concept Recap — SQL Injection

SQL injection occurs when user input is concatenated directly into an SQL query instead of being parameterized. The attacker provides input that changes the logic of the query — escaping the intended lookup and running their own SQL commands, returning data the application never intended to expose.

DVWA Module: SQL Injection

Click SQL Injection in the left navigation. You will see a simple form with a single field: "User ID".

1A: Normal Lookup

  1. Type 1 in the User ID field and click Submit
  2. One user's details appear (First name, Surname)
  3. Try 2, then 3 — a different user appears each time
💡

What the Server Does Behind the Scenes

The server builds a query like this:
SELECT first_name, last_name FROM users WHERE user_id = '1';

Your input is placed directly inside the SQL string — without any sanitization. This is the flaw.

1B: Inject a Truthy Condition

  1. Clear the field and type: 1 OR 1=1
  2. Click Submit
  3. Instead of one user, the page now shows all users in the database
⚠️

What Happened to the Query?

Your input changed the query to:
SELECT first_name, last_name FROM users WHERE user_id = '1 OR 1=1';

Because 1=1 is always true, the WHERE clause matches every row. All users are returned — data the application should never expose to a regular user.

1C: Escape the Quote

  1. Clear the field and type: 1' OR '1'='1
  2. Click Submit
  3. Again, all users are returned — this time the opening ' closes the server's own quote, so everything after it becomes raw SQL
💡

Why the Single Quote Matters

The single quote at the start (1') closes the quote the server opened around your input. Everything after the closing quote is interpreted as SQL commands, not as a string value. This is the fundamental mechanism of SQL injection.

🖥 Task 2: Command Injection

💡

Concept Recap — Command Injection

Command injection occurs when user input is passed to an OS shell command without sanitization. The attacker appends extra OS commands using shell operators like ; or &&. These run on the server's operating system with the web server's privileges — making this the most dangerous injection type.

DVWA Module: Command Injection

Click Command Injection in the left navigation. You will see a form asking for an IP address to ping.

2A: Normal Ping

  1. Type 127.0.0.1 in the IP address field and click Submit
  2. The page displays the output of the ping command — round-trip times to your own machine
💡

What the Server Is Doing

The server runs a shell command like: ping -c 4 127.0.0.1
Your input is inserted directly into that command. The shell runs it and the output is displayed on the page. What happens if you add more commands after the IP?

2B: Append a Second Command

  1. In the IP field, type: 127.0.0.1; whoami
  2. Click Submit
  3. The page shows the ping output AND the result of whoami — the name of the OS account the web server runs under

2C: List the Server's Files

  1. Type: 127.0.0.1; ls /var/www/html
  2. Click Submit
  3. The ping output is followed by a listing of files in the web server's document root

2D: Reveal the Server's Hostname

  1. Type: 127.0.0.1; hostname
  2. Click Submit
  3. The server's hostname appears alongside the ping output
🔴

Why This Is the Most Dangerous Injection Type

SQL injection targets the database. Command injection targets the entire operating system. With OS access via a web form, an attacker can read any file the server process has access to, list directories, discover running processes, and much more.

🔓 Task 3: CSRF — The Forged Request

💡

Concept Recap — CSRF

Cross-Site Request Forgery exploits the fact that browsers automatically attach session cookies to every request — including requests triggered by a malicious page. An attacker crafts a URL that, when visited by a logged-in user, performs a privileged action on their behalf without their knowledge.

DVWA Module: CSRF

Click CSRF in the left navigation. You will see a form that allows the current user to change their password.

3A: Observe the Change-Password Form

  1. Look at the form — it has two fields: New Password and Confirm New Password
  2. Type test123 in both fields and click Change
  3. You see the success message: "Password Changed."
  4. Now look at the URL in the browser address bar — the new password values appear directly as query parameters
⚠️

First Problem: GET Request for a State Change

A password change is a state-changing operation. It must use a POST request — not GET. A GET request embeds all parameters in the URL, meaning any link, image, or embedded object on any webpage can silently trigger that same action when a logged-in user visits it.

3B: Forge a Request via the URL Bar

  1. Look at the current URL — it contains something like:
    ?password_new=test123&password_conf=test123&Change=Change
  2. Edit the URL directly in the address bar, replacing test123 with a different value (e.g., hacked)
  3. Press Enter
  4. The password changed again — by a URL visit alone, with no form interaction

What This Simulates:

Imagine this URL is embedded as an invisible <img src="..."> on a webpage. A logged-in DVWA user who visits that page would have their password changed automatically — without clicking anything, without seeing anything, without knowing anything happened.

3C: Check for a CSRF Token in the Page Source

  1. Right-click on the CSRF module page and select View Page Source
  2. Look at the <form> element
  3. There is no hidden token field — no csrf_token, no _token, no nonce
💡

What a CSRF Token Would Look Like

A properly protected form would include a hidden field:
<input type="hidden" name="csrf_token" value="x8f3Kpq2...">

The server validates this token on every submission. Because the attacker cannot know the token value, forged requests from other origins are rejected. This is the primary CSRF defense.

🔧 Troubleshooting

❌ DVWA doesn't load (connection refused)

Solution: sudo docker ps — if dvwa not listed, run sudo docker start dvwa. Go to http://localhost.

❌ SQL Injection — only one user appears regardless of input

Solution: Confirm Security Level is Low (DVWA Security menu). Medium and High add filters. Also try 1' OR '1'='1 (with single quotes) if the plain version doesn't work.

❌ Command injection shows no output after the ping

Solution: Use a semicolon (;), not a comma. Try && if semicolon fails: 127.0.0.1 && whoami. Security Level must be Low.

❌ CSRF — "Password Changed" message doesn't appear

Solution: Make sure you are still logged in. Session cookies may have expired after the database reset. Log out and log back in with admin / password, then navigate back to the CSRF module.

❌ DVWA container already exists when running docker run

Solution: Just start it: sudo docker start dvwa. docker run is for the very first time only.
