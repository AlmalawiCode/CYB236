Q2 What Firefox extension did you install to route browser traffic through Burp Suite? Why is it preferred over Firefox's built-in proxy settings? 3 pts
✓ Model Answer

The extension is FoxyProxy (FoxyProxy Standard or FoxyProxy Basic).

Advantages over Firefox's built-in proxy settings:

  • One-click toggle: Enable or disable the proxy from the toolbar icon instantly, instead of navigating to Settings → Network Settings each time.
  • Multiple profiles: Save different proxy configurations (e.g., Burp on 8080, ZAP on 8081) and switch between them instantly.
  • Visual indicator: The toolbar icon changes color to show whether traffic is being proxied, preventing accidental testing with the proxy off.

Q3 Explain how Burp Suite's intercepting proxy works. Describe the complete path of an HTTP request from Firefox, through FoxyProxy, through Burp Suite, to the target web server — and back. 8 pts
✓ Model Answer

The complete path of an HTTP request through the Burp Suite intercepting proxy:

  • Step 1: The user types a URL in Firefox and presses Enter. Firefox generates an HTTP request.
  • Step 2: FoxyProxy (browser extension) intercepts the outgoing request and redirects it to 127.0.0.1:8080 instead of sending it directly to the target web server.
  • Step 3: Burp Suite is listening on 127.0.0.1:8080. It receives the raw HTTP request and records it in the HTTP History.
  • Step 4: If Intercept is ON, Burp pauses the request and displays it to the user. The user can read, modify, or drop the request before clicking Forward. If Intercept is OFF, Burp forwards it automatically.
  • Step 5: Burp forwards the request (original or modified) to the actual target web server.
  • Step 6: The target server processes the request and sends its HTTP response back to Burp.
  • Step 7: Burp records the response in HTTP History and passes it back to Firefox.
  • Step 8: Firefox renders the page normally. From the user's perspective the page loads as usual, but Burp has a complete record of every raw request and response.

Key capabilities gained: Burp can record all traffic, pause and modify requests mid-flight, replay requests with Repeater, and build a Site Map of all discovered endpoints.

Grading Notes (8 pts)
  • 2 pts — Correctly describes FoxyProxy redirecting traffic to 127.0.0.1:8080.
  • 2 pts — Explains Burp receives, records, and optionally pauses (intercepts) the request.
  • 2 pts — Explains Burp forwards the request to the target server.
  • 2 pts — Explains the response flows back: server → Burp → Firefox.
  • Partial credit per section if the concept is stated but imprecisely.
  • 0 pts — Blank or fundamentally incorrect (e.g., "Burp sends traffic directly to the internet").
intercepting proxy 127.0.0.1:8080 HTTP History Intercept ON/OFF Forward modify request

Q4 What HTTP method did DVWA's login form use? Where do the credentials (username and password) appear in the raw request captured by Burp? 5 pts
✓ Model Answer

DVWA's login form uses the POST method.

The credentials appear in the request body (not the URL):

POST /login.php HTTP/1.1 Host: 10.0.2.15 Content-Type: application/x-www-form-urlencoded username=admin&password=password&Login=Login

POST is more appropriate than GET for login forms because POST sends credentials in the request body, which is not visible in the URL, browser history, or server access logs. GET would embed the username and password in the URL query string, exposing them in multiple places.

Q5 What is the name of the session cookie set by DVWA after login? Explain what an attacker could do if they steal this cookie value. 5 pts
✓ Model Answer

The session cookie is named PHPSESSID. It is a 32-character hexadecimal string generated by PHP to uniquely identify the user's session.

If an attacker steals this cookie:

  • Session hijacking: The attacker sets the stolen PHPSESSID value in their own browser (e.g., using DevTools → Application → Cookies). The server recognizes this cookie as belonging to a valid logged-in session.
  • Full account access: The attacker gains the same access as the victim — they can browse DVWA, access vulnerability modules, and perform any action the victim's session allows, all without knowing the victim's password.

This attack is called session hijacking.

Grading Notes
  • 2 pts — Correctly names PHPSESSID.
  • 3 pts — Explains session hijacking: attacker impersonates the user without needing the password.
  • Partial (1–2 pts): Correct cookie name but vague explanation ("they can hack the account").
  • 0 pts — Blank or wrong cookie name.
PHPSESSID session hijacking impersonation cookie theft
Q6 List at least two security headers that are missing from DVWA's HTTP responses. For each header, name the specific attack it protects against. 8 pts
✓ Model Answer

DVWA is intentionally insecure and is missing most security headers. Accept any two from the following:

Missing Header Attack It Protects Against
Content-Security-Policy Cross-Site Scripting (XSS) — restricts which scripts, styles, and resources the browser is allowed to load, blocking injected malicious scripts.
X-Frame-Options Clickjacking — prevents the page from being embedded in an <iframe> on a malicious site, stopping UI redress attacks.
X-Content-Type-Options MIME-sniffing attacks — prevents the browser from guessing a file's MIME type, which could cause a non-script file to be interpreted as executable JavaScript.
Strict-Transport-Security Man-in-the-Middle / protocol downgrade — forces the browser to use HTTPS only, preventing interception on plain HTTP connections.
Grading Notes (8 pts — 4 pts per header)

  • 2 pts per header — Correct header name but vague attack description ("it improves security").
  • 0 pts per header — Invented header name or completely wrong attack mapping.
  • Accept any two from the table above. Also accept X-XSS-Protection (deprecated but still discussed in the lab).
Content-Security-Policy X-Frame-Options X-Content-Type-Options Strict-Transport-Security clickjacking MIME sniffing
Q7 In Burp's Repeater, you changed id=1 to id=1'. What did the server respond with? Why is exposing database error messages a security problem, and what information does an attacker gain? 12 pts
✓ Model Answer

What the server responded with:

You have an error in your SQL syntax; check the manual that corresponds to your MySQL server version for the right syntax to use near ''1''' at line 1

Why this is a security problem:

  • Confirms the vulnerability: The SQL error proves that user input (id parameter) is injected directly into a SQL query without sanitization or parameterization. The attacker now knows SQL injection is possible.
  • Reveals the database engine: The error message explicitly mentions "MySQL server version," telling the attacker which database is in use. This lets them use MySQL-specific injection syntax.
  • Reveals query structure: The error message shows parts of the SQL query structure (e.g., the quoting pattern ''1'''), helping the attacker craft precise injection payloads.
  • Facilitates further exploitation: With this information, the attacker can extract data from other tables, read files from the server, or potentially gain remote code execution — all through refined SQL injection payloads.

Best practice: Production applications should display a generic error page (e.g., "Something went wrong") and log detailed errors server-side only. Database errors should never be shown to users.

Grading Notes (12 pts)
  • 3 pts — Correctly describes or quotes the MySQL syntax error.
  • 3 pts — Explains that the error confirms user input is injected into SQL (vulnerability confirmation).
  • 3 pts — Explains what information the attacker gains (database type, query structure).
  • 3 pts — Explains the broader impact (enables further SQL injection, data extraction).
  • Partial credit per criterion if concept is mentioned but not elaborated.
  • 0 pts — Blank or states "an error appeared" with no analysis.
SQL syntax error MySQL SQL injection no sanitization query structure information leakage

Part C
Intruder & Password Attacks — DVWA Brute Force
45 pts
Q8 Write the exact Intruder attack mode you selected. Explain why it was the correct choice when testing passwords with a known username. 4 pts
✓ Model Answer

The attack mode used was Sniper.

Why Sniper is correct: We have one payload list (the 15-password wordlist) and one payload position (the password field). The username is already known (admin) and stays fixed. Sniper cycles each wordlist entry through the single marked position one at a time — exactly what is needed for a single-parameter dictionary attack.

The other modes are for different scenarios: Battering Ram inserts the same payload into all positions simultaneously; Pitchfork iterates multiple lists in parallel (matched pairs); Cluster Bomb tests all combinations of multiple lists. None of these are needed when fuzzing a single parameter.

Grading Notes
  • 2 pts — Correctly names Sniper.
  • 2 pts — Explains why: one position, one list, username fixed.
  • 1 pt — Names Sniper but no explanation.
  • 0 pts — Blank or names a wrong mode.
Sniper one position one wordlist fixed username
Q9 All 15 Intruder responses returned HTTP status code 200 OK. Explain how you identified which response contained the correct password. What was the password? 6 pts
✓ Model Answer

All 15 responses returned HTTP 200 OK — DVWA at Low security does not use a different status code for success vs. failure. Therefore, the status code could not distinguish them.

The correct password was identified using the response length (Length column):

  • 14 failed login attempts all returned the same response length (approximately ~4719 bytes) — the page containing the error message "Username and/or password incorrect."
  • 1 successful login returned a noticeably different (larger) length (approximately ~4779 bytes) — the page containing the welcome message "Welcome to the password protected area admin."

The correct password was password (the 3rd entry in the wordlist).

Q10 What happened when you switched DVWA to Medium security and re-ran the Intruder attack? What additional defense appeared at High security? How did each affect the attack? 8 pts
✓ Model Answer

Medium Security — Rate Limiting (server-side delay):

  • DVWA Medium adds a 2-second server-side sleep before responding to each brute force login attempt.
  • The attack still completes and the correct password can still be found, but each request takes 2+ seconds instead of milliseconds.
  • For 15 passwords: ~30 seconds. For rockyou.txt (14 million passwords) at 2 sec/guess: over 324 days — making large-scale dictionary attacks impractical.

High Security — CSRF Token:

  • DVWA High adds a user_token parameter (CSRF token) to every brute force request. This is a random value generated fresh with each page load and valid for only one request.
  • The Intruder attack completely fails: the token captured in the original request is consumed by the first attempt, and all 14 remaining attempts are rejected because they reuse an already-consumed token.
  • The correct password cannot be identified even though it is in the wordlist, because the server rejects every request after the first one.
Grading Notes (8 pts)
  • 2 pts — Correctly describes Medium: server-side delay / sleep (mentions 2 seconds).
  • 2 pts — Explains the impact on large wordlists (impractical timing).
  • 2 pts — Correctly describes High: CSRF token (user_token) added.
  • 2 pts — Explains the impact: token is single-use, Intruder reuses it, all but one request fail.
  • Partial credit: Mentions "it got slower" (1 pt for Medium); mentions "something extra appeared" (1 pt for High).
2-second delay rate limiting user_token CSRF token single-use attack fails
Q11 Complete the comparison table: Dictionary Attack vs Brute Force Attack 10 pts
✓ Model Answer
Property Dictionary Attack Brute Force Attack
Method used Uses a pre-compiled wordlist of common/leaked passwords (e.g., rockyou.txt). Tries each entry from the list against the target. Tries every possible combination of characters systematically (aaa, aab, aac... zzz). Does not rely on a wordlist.
Speed Fast — limited to the number of entries in the wordlist. A 15-entry list finishes in seconds; even rockyou.txt (14M) is finite. Very slow — the number of combinations grows exponentially with password length and character set. A 6-char lowercase password: 26⁶ = 308 million combinations.
Coverage Partial — only finds passwords that appear in the wordlist. Strong, unique passwords are missed. 100% — guaranteed to find any password eventually, given enough time and computing power.
Which did you use in Lab 7? Dictionary attack. We used a 15-entry custom wordlist loaded into Burp Intruder. This was practical because (1) the wordlist was short enough for Community Edition's 1 req/sec rate limit, (2) DVWA's default password (password) is a common entry found in any dictionary, and (3) it demonstrates the real-world technique where attackers target weak/common passwords first.
Grading Notes (10 pts — distributed across cells)
  • 2 pts — Dictionary "method": wordlist / pre-compiled list of known passwords.
  • 2 pts — Brute force "method": all possible character combinations.
  • 1.5 pts — Dictionary "speed": fast / limited to wordlist size.
  • 1.5 pts — Brute force "speed": very slow / exponential growth.
  • 1 pt — Dictionary "coverage": partial / only common passwords.
  • 1 pt — Brute force "coverage": 100% / guaranteed eventually.
  • 1 pt — Which used in Lab 7: dictionary attack + any valid reason.
Q12 Scenario: A company has no lockout, no rate limiting, and no CAPTCHA. An attacker uses Burp Intruder with rockyou.txt (14M passwords). What three defenses should the company implement? 7 pts
✓ Model Answer

Accept any three of the following defenses. Each defense must include a brief explanation of how it reduces the risk:

Defense How It Reduces the Risk
Account Lockout After N failed attempts (e.g., 5), the account is locked for a period or until admin reset. This directly stops automated bulk guessing — the attacker can only try a handful of passwords before being blocked.
Rate Limiting Enforces a delay between login attempts from the same IP or account (e.g., 2-second wait). Slows Intruder from 1000 req/sec to 0.5 req/sec, making a 14M-entry attack take months instead of hours.
CAPTCHA After failed attempts, a visual or behavioral challenge is presented that automated tools (Intruder) cannot solve. Forces human interaction, stopping bot-driven attacks.
Multi-Factor Authentication (MFA) Even if the correct password is discovered, a second factor (TOTP code, hardware key, SMS) is required. The password alone is insufficient, rendering the entire dictionary attack useless.
Strong Password Policy Enforce minimum length (e.g., 12+ characters), complexity, and check against known breached password lists at registration. If users cannot set "password" or "123456," those entries in rockyou.txt will never match.
CSRF Tokens One-use random token per request prevents automated replay. Intruder must scrape a fresh token before each attempt, significantly complicating the attack.
Grading Notes (7 pts)
  • Award ~2.3 pts per defense (round as needed for 3 defenses = 7 pts).
  • Full credit per defense: names the defense AND explains how it specifically reduces the risk of a dictionary/Intruder attack.
  • Half credit per defense: names the defense but explanation is vague ("it makes it more secure").
  • 0 pts per defense: invented or irrelevant defense (e.g., "install antivirus").
  • Accept any combination from the table above. Also accept: IP blacklisting, geo-blocking, login attempt notifications, anomaly detection.
account lockout rate limiting CAPTCHA MFA strong password policy CSRF token defense in depth
