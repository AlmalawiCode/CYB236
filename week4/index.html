The Hacker's Mindset

Becoming a successful hacker takes more than technical knowledge — it requires a very particular mindset. Understanding how hackers think is essential both for attacking and defending systems.

📖 From the Book — Engineer vs. Hacker

A software engineer measures productivity in value-add: "I added feature X, I improved performance by 10%." Their work is quantifiably measurable.

A hacker measures productivity very differently. Most of hacking is data gathering and analysis — riddled with false positives, it may look like wasted time to an outsider. A hacker might spend weeks or months analyzing a single application before finding a usable entrypoint.

Ahmed: "A hacker is first and foremost a detective. A great hacker is a detective with excellent technical knowledge who is constantly learning and adapting."

Core Traits of a Successful Hacker

🔎
Seeks Entrypoints
Works with existing code to find vulnerabilities, not modify features
📝
Detailed Notes
Carefully logs every attempt and lesson learned
📚
Constantly Learns
Old techniques lose value as the web adapts — must keep updating skills
⏳
Patient
Prepared for weeks without a "big find" — persistence is key

💡 Key Insight

Most hackers don't modify software — they analyze it to find entrypoints. The skills for finding those entrypoints are nearly identical to the recon skills in Part I of the book.

🔗

Applied Recon — Bridging Part I to Part II

Everything learned in Part I — subdomains, API analysis, third-party dependencies, and architecture — is now put to offensive use. Recon and hacking go hand in hand.

📖 How Recon Feeds Offense

Throughout Part II we attack mega-bank.com — the same application profiled in Part I. Every piece of recon data becomes a potential attack surface.

Chapter 10
Cross-Site Scripting (XSS)
🎯

Learning Objectives

1
Explain how XSS attacks work and why web applications that execute dynamic scripts are at risk.
2
Distinguish between Stored, Reflected, DOM-Based, and Mutation-Based XSS.
3
Describe how a Stored XSS attack steals sensitive data from privileged users.
4
Identify DOM sources and sinks that enable DOM-Based XSS without any server interaction.
5
Recognize how mutation-based XSS bypasses sanitization filters using browser DOM behavior.
💡

What Is XSS? — The Core Idea

XSS (Cross-Site Scripting) vulnerabilities are among the most common on the internet. At its core, an XSS attack works by taking advantage of the fact that web applications execute scripts in users' browsers.

If a script being executed can be contaminated or modified by an end user, that web application is at risk. The attacker's goal: inject malicious JavaScript that runs in the victim's browser — stealing data, hijacking sessions, or drawing fake UI.

The Three Major XSS Categories (OWASP)

🗃
Stored XSS
Script is saved in the server database and later served to all visitors
🔁
Reflected XSS
Script is not stored — reflected back from the server in a URL or query
💻
DOM-Based XSS
Script executes entirely in the browser — never touches the server
🧬
Mutation XSS
Payload passes filters safely, then mutates into a valid exploit in the browser
🔍

How XSS Works — The MegaBank Story

Finding XSS starts with testing whether a web application treats user input as text or as DOM markup. If HTML tags entered in a form are rendered by the browser instead of displayed as plain text, an XSS vulnerability exists.

📖 Story — Ahmed Finds XSS at MegaBank Support

Ahmed visits support.mega-bank.com and submits a complaint. He includes a <strong> tag to bold a word. When the page reloads, the text is actually bolded — meaning the browser interpreted his HTML as markup.

Ahmed: "If <strong> works, so will <script>. The server stored my comment and rendered it as HTML without sanitization."

The architectural flaw is simple:

// Vulnerable pattern — user input written directly to DOM
const comment = 'user provided text with <script>...</script>';
const div = document.createElement('div');
div.innerHTML = comment; // ← DANGEROUS: treats content as DOM, not text
wrapper.appendChild(div);

What XSS Attacks Can Do

  • Steal session tokens — hijack the user's authenticated account
  • Exfiltrate PII — collect names, emails, phone numbers visible in the UI and send to attacker's server
  • Run silently — execute in the background with no visible interaction required
  • Draw fake UI — overlay a phishing login form that is indistinguishable from the real site
  • Persist across sessions — in stored XSS, the payload affects every user who views the contaminated content
🗃

Stored XSS — The Database Payload

Stored XSS is the most common type. The malicious script is saved in the server's database, and every user who later views that content has the script executed in their browser.

📖 Story — Ahmed Steals Customer Data

Ahmed submits a support comment that includes a malicious <script> tag instead of normal text. A customer support agent opens the ticket. The script immediately executes in the agent's browser, collects all visible customer records from the DOM, and silently sends them to Ahmed's server.

// What the server stores in the database (looks like a comment)
<script> ... JavaScript that reads the page and sends data to an attacker's server ... </script>

// When the support agent opens the ticket, the browser cannot tell
// the difference between Ahmed's script and the app's own scripts.
// Both execute automatically — the attack is completely invisible.
Ahmed: "The support agent sees nothing unusual — the script tag is invisible. But their browser runs my code and sends every customer record visible on screen to an external server."

🔴 Why Stored XSS is Dangerous

One payload in a shared database can affect every user who views that record. A "featured" video with XSS in its title could attack every visitor of the homepage.

🟢 Easier to Detect

Scripts are stored server-side as plain text. Regularly scanning database entries for <script> tags is a cheap first defense — though advanced payloads may use base64 encoding to evade this.

🔁

Reflected XSS — The Weaponized URL

Reflected XSS is not stored in a database. Instead, the malicious payload is embedded in a URL that the attacker distributes. When a victim clicks the link, the server reads the URL parameter and reflects it back in the HTML response — executing the script.

📖 Story — Ahmed Crafts a Malicious Search Link

Ahmed finds that the support portal's search bar reflects the query in its heading: searching for "open savings account" shows "3 results for open savings account." He tests whether <script> works in the URL parameter:

# Normal search URL
support.mega-bank.com/search?query=open+savings+account

# Reflected XSS payload in the URL
support.mega-bank.com/search?query=<script>alert('XSS');</script>

# Disguised in a legitimate-looking link
<a href="...malicious-url...">Create a New Checking Account</a>
Ahmed: "This link looks legitimate — it's a real domain. But anyone who clicks it has a script execute in their browser. The attack never touched a database, making it very hard to detect."
🔴 Stored XSS
  • Script saved in the database
  • Affects anyone who views the record
  • Easy to detect (scan the database)
  • Hard to distribute — one location
🔴 Reflected XSS
  • Script embedded in a URL
  • Targets specific victims via link
  • Very hard to detect (no DB storage)
  • Easy to distribute via email, ads, social media
💻

DOM-Based XSS — No Server Required

DOM-Based XSS executes entirely in the browser — no server interaction is required. The payload is read from a DOM source (e.g., window.location.hash) and executed via a DOM sink (e.g., document.write()).

📖 Story — The Hash-Based XSS

MegaBank's investment portal reads the URL hash to scroll to a section of funds. A developer also uses document.write() to display the current filter in the heading. Ahmed crafts a URL where the hash contains a script tag:

// Vulnerable client-side code
const hash = document.location.hash; // ← SOURCE
document.write('Showing results for: ' + hash); // ← SINK (dangerous)

// Ahmed's malicious URL
investors.mega-bank.com/listing#<script>alert(document.cookie);</script>
Ahmed: "This attack never reached the server. It's entirely client-side. No database logs, no server logs — nearly impossible to detect with traditional scanning tools."

📋 DOM Sources (Where Data Comes From)

window.location.search, window.location.hash, document.referrer, localStorage, sessionStorage — any DOM object capable of storing user-controlled text.

🎯 DOM Sinks (Where Scripts Execute)

document.write(), element.innerHTML, eval(), setTimeout(string) — any DOM API that interprets a string as executable code or HTML markup.

🧬

Mutation-Based XSS (mXSS) — Bypassing Filters

mXSS is an advanced technique where a payload is crafted to be harmless when scanned by a sanitizer, but mutates into a dangerous script when the browser renders it — exploiting browser-specific DOM optimizations.

📖 Story — Google Search mXSS (2019)

Security researcher Masato Kinugawa found that Google's Closure library used DOMPurify for sanitization. He crafted a payload that DOMPurify considered safe:

// Safe-looking payload (passes DOMPurify)
<noscript><p title="</noscript><img src=x onerror=alert(1)>">

// After browser renders it (mutated — dangerous!)
<noscript><p title="</noscript>
<img src="x" onerror="alert(1)">

Inside the sanitizer's <template> element, scripting is disabled — so the <noscript> tag wraps its children and looks safe. But in the real browser, scripting is enabled, so the <noscript> becomes inert and the img onerror fires.

💡 Why mXSS Is Dangerous

Tools like DOMPurify, OWASP AntiSamy, and Google Caja have all been bypassed by mXSS payloads. Because the payload is valid and safe at sanitization time, it evades all filters. This technique has been used to exploit major email clients and web applications at scale.

✓ Chapter 10 Summary — Cross-Site Scripting (XSS)

  • Stored XSS: payload saved in DB, affects every user who views it — most dangerous, easiest to detect
  • Reflected XSS: payload embedded in URL, distributed via links — stealthy, hard to detect
  • DOM-Based XSS: runs entirely in the browser, requires a source and a sink — never logged server-side
  • Mutation-Based XSS: crafted to pass filters, then mutates in the browser — bypasses even robust sanitizers
  • XSS can steal session tokens, PII, and credentials, and can overlay phishing UIs
  • XSS attacks run behind the scenes — the victim typically sees nothing unusual
  • XSS has been present throughout the web's history and remains rampant due to ever-increasing user interaction
Chapter 11
Cross-Site Request Forgery (CSRF)
🎯

Learning Objectives

1
Explain how CSRF exploits the trust relationship between a browser, a user, and a web server.
2
Explain how CSRF attacks exploit query parameters in HTTP GET requests to trigger unauthorized actions.
3
Identify which HTML tags automatically trigger GET requests without user interaction.
4
Describe how HTML forms can be used to forge POST requests against a privileged endpoint.
🔓

What Is CSRF? — Abusing Browser Trust

CSRF attacks cause a privileged user to perform actions on a server without knowing it. They exploit the fact that browsers automatically attach authentication cookies and credentials to every request — even if the request originated from a different website.

📖 The Trust Problem

When you click a link anywhere on the internet, your browser makes a GET request from your account — with your session cookies attached. The destination server has no way to tell if you clicked it, or if a malicious page tricked you into clicking it.

CSRF exploits this. An attacker crafts a URL that triggers a privileged action (like a bank transfer) and tricks a logged-in user into clicking it.

🏷 Key CSRF Characteristic #1

Privilege escalation: the attacker performs an action using the victim's higher-privilege account (e.g., initiating a bank transfer they couldn't initiate themselves).

🕵 Key CSRF Characteristic #2

Stealth: the user who initiates the request typically doesn't know it happened. Requests occur behind the scenes — no visible browser action.

🔗

Query Parameter Tampering — CSRF via GET

The simplest CSRF attack. A web application that uses HTTP GET with query parameters to perform state changes (e.g., transferring money) is trivially exploitable via a crafted hyperlink.

📖 Story — Ahmed Forges a Bank Transfer

Ahmed discovers MegaBank uses this endpoint to transfer money:

// The vulnerable server route
app.get('/transfer', function(req, res) {
  if (!session.isAuthenticated) { return res.sendStatus(401); }
  // Transfers money based on URL query params — DANGEROUS!
  transferFunds(session.currentUser, req.query.to_user, req.query.amount);
});

Ahmed crafts a malicious link and sends it to a MegaBank customer via email:

<!-- Disguised as a helpful link -->
<a href="https://www.mega-bank.com/transfer?to_user=ahmed&amount=10000">
  Claim Your Promotional Reward
</a>
Ahmed: "The victim is logged in to MegaBank. When they click this link, their browser sends the GET request with their session cookie attached. MegaBank's server sees an authenticated request — and transfers $10,000 to my account."

⚠️ Never Use GET for State Changes

HTTP GET requests should be read-only. Any operation that modifies data (transfers, deletions, updates) must use POST, PUT, or DELETE — which are significantly harder to forge via simple links.

🖼️

Alternate GET Payloads — No Click Required

Many HTML tags automatically trigger GET requests without any user interaction when they load into the browser. This makes them ideal CSRF vectors — no click needed.

<!-- IMG tag: triggers a GET request when it loads into the DOM -->
<img src="https://mega-bank.com/transfer?to_user=ahmed&amount=10000"
     width="0" height="0" border="0">

<!-- VIDEO tag: same behavior — fires a GET to its src on load -->
<video width="1" height="1">
  <source src="https://mega-bank.com/transfer?to_user=ahmed&amount=10000">
</video>

An invisible 0×0 pixel image placed on any webpage will silently trigger the CSRF the moment the page loads. The victim does not need to click anything.

💡 Other Automatic GET Triggers

Any HTML tag with a src attribute can fire a GET request automatically on page load: <img>, <video>, <audio>, <iframe>, <script>, <link>. This is why CSRF vulnerabilities are so broad — even a seemingly innocuous embedded image can silently trigger a state-changing request on behalf of the victim.

📋

CSRF via POST — The Hidden Form Attack

POST-based CSRF is harder to deliver but still possible. HTML forms are the only HTML element that can initiate a POST request without JavaScript — making them the primary vector for POST CSRF.

📖 Story — The Fake Login Form

Ahmed builds a convincing-looking "win a prize" webpage. Hidden inside the HTML is a form that, when submitted, sends a POST request to MegaBank on behalf of the authenticated user:

<!-- Hidden fields carry the CSRF payload -->
<form action="https://www.mega-bank.com/transfer" method="POST">
  <input type="hidden" name="to_user" value="ahmed">
  <input type="hidden" name="amount" value="10000">
  <!-- Visible fields look legitimate to the user -->
  <input type="text" name="username" placeholder="Enter your name">
  <input type="submit" value="Claim Your Prize!">
</form>
Ahmed: "The victim sees a prize form. They fill in their name and click submit. But the form POSTs to MegaBank — not to a prize server — and the hidden fields transfer funds to my account."

💡 Advanced Use Case — Internal Network Access

An attacker who cannot directly access an internal network can use CSRF to have a victim (who is on the internal network) make requests on their behalf. The form's action can target any server the victim has access to — including private intranet servers.

✓ Chapter 11 Summary — Cross-Site Request Forgery (CSRF)

  • CSRF exploits the browser's trust model — cookies are sent with every request, regardless of origin
  • GET-based CSRF: craft a malicious URL and distribute it via email, ads, or social media
  • <img>, <video>, and other tags with src fire GET requests automatically on page load
  • POST-based CSRF requires user interaction (form submission) but can be disguised as a legitimate form
  • CSRF goes completely unnoticed by the victim — one of the stealthiest attack types
  • Never use GET requests for state-changing operations (transfers, deletions, updates)
  • Defense: CSRF tokens, SameSite cookies, Origin/Referer header verification (covered in Part III)
Chapter 12
XML External Entity (XXE)
🎯

Learning Objectives

1
Explain what an XML external entity is and why improperly configured parsers are vulnerable.
2
Describe a Direct XXE attack where the attacker sends a crafted XML payload to read server files.
3
Identify XML-like formats (SVG, HTML, PDF, RTF) that may be parsed by an XML parser.
4
Recognize Indirect XXE where a non-XML API still processes XML internally through a third-party system.
📄

What Is XXE? — A Flaw in the XML Standard

XXE (XML External Entity) attacks are often simple to execute but devastating in effect. The vulnerability lives in the XML specification itself: it includes a feature called external entities that allows an XML file to reference files from the host server's file system.

A poorly configured XML parser that doesn't disable this feature will happily read server files (like /etc/passwd, /etc/shadow, or SSH keys) and return their contents in the response.

📄 XML-Like Formats at Risk

XXE is not limited to .xml files. Any format that shares XML structure may be parsed by an XML parser: SVG images, HTML/DOM, PDF (XFDF), and RTF documents.

🗃 What Attackers Can Steal

/etc/passwd (user accounts), /etc/shadow (hashed passwords), ~/.ssh/id_rsa (private SSH keys), application source code, config files with database credentials.

📷

Direct XXE — The Screenshot Attack

In a Direct XXE attack, the attacker sends a crafted XML payload directly to an API that processes XML. The XML contains an external entity declaration pointing to a file on the server. The parser returns the file's contents.

📖 Story — Ahmed Reads the Password File

MegaBank has a "Send Screenshot to Support" feature that converts the user's DOM (HTML) to XML, then sends it to a server for processing. Ahmed bypasses the client-side code and sends a custom XML payload directly:

<!-- What a normal screenshot request's XML might look like -->
<screenshot><dom> ... page HTML ... </dom></screenshot>

<!-- What Ahmed sends instead — XML with an external entity declaration -->
<!DOCTYPE xxe [<!ENTITY xxe SYSTEM "file:///etc/passwd">]>
<xxe>&xxe;</xxe>

<!-- The parser substitutes &xxe; with the actual file contents
     and returns them embedded in the server's response. -->
Ahmed: "The server's XML parser sees the external entity declaration, opens /etc/passwd, substitutes its contents for &xxe;, and returns it to me in the screenshot response."

💡 How Direct XXE Works — Conceptually

The attacker sends XML to an endpoint that processes it. Inside that XML, they include a standard XML feature called an external entity — a declaration that tells the parser to load a local file and substitute its contents into the XML. If the parser has not disabled this feature (which many do not, by default), it obediently reads the requested file and returns its contents embedded in the response. The attacker receives server file contents without ever touching the server directly.

🔁

Indirect XXE — Hidden XML Processing

Indirect XXE targets APIs that do not visibly accept XML but convert the user's input to XML internally before passing it to another system (like a legacy CRM or enterprise software).

📖 Story — The CRM Integration

MegaBank's modern REST API accepts JSON. But behind the scenes, it converts that JSON to XML to communicate with an old enterprise CRM system. A hacker who knows the CRM is in use (from public records or LinkedIn job postings) can inject an XXE payload into the JSON field — which gets converted to XML and parsed by the vulnerable CRM.

Ahmed: "The public-facing API doesn't accept XML. But if I know the backend uses a CRM that expects XML, I can craft my JSON input so that when converted to XML, it contains the XXE payload."

💡 Why Indirect XXE Is Hard to Spot

Because the public-facing API does not mention XML, the vulnerability is invisible from the outside. Clues that a backend XML system exists include: unusual formatting constraints on input fields (hinting at XML schema requirements), legacy enterprise software in the company's tech stack, and internal API responses that follow XML naming conventions even when the public API uses JSON.

⚠️ The Fix Is One Line

Disabling external entities in an XML parser is usually a single configuration option. But because it is an opt-in security measure, many parsers ship with external entities enabled by default — making XXE one of the easiest vulnerabilities to introduce by accident.

✓ Chapter 12 Summary — XML External Entity (XXE)

  • XXE exploits a feature of the XML standard — external entities that can read local server files
  • Any API accepting XML-like formats (SVG, HTML, RTF, PDF) may be vulnerable
  • Direct XXE: attacker sends a crafted XML payload directly and receives file contents in the response
  • Indirect XXE: input is converted to XML internally — the endpoint may not appear to use XML at all
  • Target files: /etc/passwd, /etc/shadow, SSH keys, config files with credentials
  • The fix is typically a single parser configuration line to disable external entities
  • Because the fix is a single configuration line, XXE is one of the easiest vulnerabilities to introduce by accident and one of the easiest to prevent by design
Chapter 13
Injection
🎯

Learning Objectives

1
Define injection attacks and identify the two core components: an interpreter and an unsanitized user input.
2
Explain how SQL injection works by describing how truthy statements and query chaining escape the intended query.
3
Distinguish between SQL injection, Code injection, and Command injection based on the target interpreter.
4
Explain why command injection is the most dangerous form — granting OS-level access to the server.
🔬

Injection — The Core Concept

An injection attack occurs when user input is passed unsanitized into an interpreter or command-line interface (CLI). The interpreter runs the input as code — executing commands the developer never intended.

📊
SQL Injection
Target: SQL database interpreter
🛠
Code Injection
Target: CLI or application interpreter
🖥
Command Injection
Target: Host operating system shell
📊

SQL Injection — Escaping the Query

SQL injection occurs when user input is concatenated directly into an SQL query string instead of being parameterized. The attacker escapes the intended query and injects their own SQL commands.

📖 Story — Ahmed Empties the Database

A Node.js API endpoint looks up a user by ID using string concatenation:

// Vulnerable query — unsanitized user input concatenated directly
const user_id = req.params.user_id; // comes from HTTP request
const result = await sql.query('SELECT * FROM users WHERE USER = ' + user_id);

// Normal input: SELECT * FROM users WHERE USER = 42
// Truthy attack:
user_id = '1=1' → returns ALL users

// Table drop attack:
user_id = '42; DROP TABLE users;' → deletes the entire user table

// Stealthy credit manipulation:
user_id = '42; UPDATE users SET credits = 10000 WHERE user = 42;'
Ahmed: "The last example is the most dangerous — it modifies data silently without triggering obvious alerts. The developer only notices when account credits don't match billing records."

💡 SQL Injection Trend

SQL injection has declined from ~5% of all vulnerabilities in 2010 to under 1% today, thanks to modern frameworks enforcing parameterized queries. However, it still exists in legacy systems and applications that ignore secure coding practices.

🛠

Code Injection — Beyond SQL

Code injection targets any interpreter or CLI — not just SQL. If an API endpoint invokes a CLI tool (like a video converter, image processor, or PDF generator) and includes user input in the command string without sanitization, the user can inject extra CLI commands.

📖 Story — Ahmed Hijacks the Video Converter

MegaBank's video upload API uses a "converter" CLI tool. The options string is built by concatenating user input:

// Vulnerable: user-provided options concatenated into a CLI call
const options = defaultOptions + req.body.options; // req.body.options not sanitized
exec(`convert -d ${videoData} -n ${videoName} -o ${options}`);

// Normal input: options = '-c h264 -ab 192k'
// Injection input:
options = '-c h264 -ab 192k \\ convert -dir /videos -s 1x1'
// ↑ Adds a second convert command — redirecting video processing

💻 Code Injection

Limited to the target interpreter or CLI's own commands. Cannot access the host OS directly. Example: injecting extra FFmpeg flags into a video processor.

⚠️ Command Injection (More Dangerous)

Escapes the CLI entirely and runs commands against the host operating system. Example: appending && rm -rf / to a Bash command built from user input.

🖥

Command Injection — Full OS Access

Command injection is the most dangerous form of injection. The attacker's input escapes to the host OS shell (usually Bash on Linux servers) and can execute arbitrary OS commands — with the privileges of the web server process.

📖 Story — Ahmed Deletes All Videos

The upload API uses Node.js exec() to remove the raw video file after conversion. The filename comes directly from user input:

// Vulnerable: exec() builds a Bash command from unsanitized filename
exec(`rm /videos/raw/${req.body.name}`);

// Normal: rm /videos/raw/myVideo.mp4
// Injection:
req.body.name = 'myVideo.mp4 && rm -rf /videos/converted/'
// Result: rm /videos/raw/myVideo.mp4 && rm -rf /videos/converted/

What Command Injection Unlocks

  • Steal server data — read any file the web process has access to
  • Rewrite logs — erase evidence of the intrusion from server logs
  • Add backdoor users — create a persistent database or OS user for later access
  • Delete critical files — destroy the application or its data
  • Send spam — use the server's SendGrid or SMTP credentials to send mass email
  • Lock out admins — change passwords and extort the organization
  • Phishing overlay — modify the login form to send credentials to the attacker's server

⚠️ #1 on Every Vulnerability Risk Scale

Command injection sits at the top of every major vulnerability scoring framework. Over 95% of web servers run Unix-based OS (Linux/macOS). If commands run as a superuser, the attacker controls the entire server.

✓ Chapter 13 Summary — Injection

  • Injection attacks have two components: an interpreter and unsanitized user input fed into it
  • SQL Injection: escapes SQL query strings to run arbitrary database commands
  • Code Injection: injects commands into a CLI or application interpreter (FFMPEG, imagemin, etc.)
  • Command Injection: escapes to the host OS shell — the most dangerous, grants full server control
  • The root cause in all injection types: user input is concatenated directly instead of being parameterized — parameterization is the universal defense
  • Application-specific CLIs are less documented than SQL — often have fewer defenses in place
  • Command injection can result in data theft, backdoors, phishing, log tampering, and full server takeover
Chapter 14
Denial of Service (DoS)
🎯

Learning Objectives

1
Explain the difference between DoS and DDoS, and how each affects legitimate users.
2
Describe how a malicious regular expression (evil regex) causes exponential backtracking.
3
Identify resource-intensive server operations that are prime targets for Logical DoS attacks.
4
Distinguish between application-level and network-level DDoS attacks.
⚡

DoS — Consuming Resources to Deny Service

Denial of Service (DoS) attacks aim to consume server or client resources so that legitimate users cannot access the application. Unlike most attacks, DoS rarely causes permanent damage — but it can render a service completely unusable.

👥 Single-Source DoS

One attacker targeting one server. Includes regex DoS and logical DoS. Range: from slight page slowdown to complete crash. Tests are best run in a local dev environment to avoid disrupting real users.

🌐 Distributed DoS (DDoS)

Thousands of coordinated devices (often botnets — hijacked computers) flooding a server. Typically attacks at the network layer (UDP flooding) rather than application layer.

💡 DoS vs Other Attack Types

Unlike data breaches or code injection, DoS attacks typically do not steal data — their goal is pure disruption. However, availability is a core pillar of security (the "A" in the CIA triad — Confidentiality, Integrity, Availability). An application that cannot be reached has failed its users, even if no data was compromised.

🔬

Regex DoS (ReDoS) — The Evil Regex

Regular expressions are used throughout web apps to validate form inputs. While usually fast, certain regex patterns are vulnerable to "evil" or "malicious" inputs that cause the regex engine to perform exponential backtracking — grinding the server or browser to a halt.

📖 The Greedy Regex Trap

Consider the regex: /^((ab)*)+$/. For normal "ab" repeated inputs it runs fast. But adding one extra character triggers catastrophic backtracking:

// Regex: /^((ab)*)+$/

Input: "abababababababababababa" (23 chars)   → 8 ms
Input: "ababababababababababababa" (25 chars)  → 15 ms
Input: "abababababababababababababa" (27 chars) → 31 ms

// Execution time doubles with every 2 characters added!
// At ~50 chars → seconds. At ~80 chars → minutes. Server hangs.

The regex engine exhaustively tries every possible combination of (ab) matches before concluding the string is invalid — an exponential explosion in computation.

💡 Why ReDoS Matters

A malicious regex can sit dormant for years, triggering only with a specific crafted input. It's particularly dangerous in open-source libraries where few developers understand regex internals. Server-side ReDoS can crash the server; client-side ReDoS can freeze the user's browser.

🧠

Logical DoS — Draining Server Resources

Logical DoS exploits expensive server operations that scale with user data. By creating an account with maximum data and repeatedly calling the expensive endpoint, an attacker drains server CPU/memory — degrading performance for all users.

📖 Story — Ahmed Times the API

A photo-sharing app exposes GET /metadata/:userid which aggregates metadata for all a user's albums and photos. Ahmed measures how the response time scales:

// Response times scale with the amount of data in the account
New account (1 album, 1 photo)  → 120 ms
Average user (6 albums, 60 photos)  → 470 ms
Power user (28 albums, 490 photos) → 1,870 ms

// Pattern: the operation is not O(1) — cost grows with data volume.
// An attacker who maximizes data and hammers the endpoint
→ Server resources exhausted. Legitimate users experience timeouts.

What Server Operations to Look For

  • Database writes — especially in bulk or with complex constraints
  • SQL joins — operations that span many tables scale poorly with data volume
  • File backups — copying large files consumes disk I/O and CPU
  • Synchronous operations — blocking calls that hold threads until completion
  • Looping logical operations — server-side loops that grow with user data size
🌐

Distributed DoS (DDoS) — The Botnet Attack

DDoS attacks involve many coordinated devices simultaneously flooding a server. The devices are usually botnets — real computers owned by innocent people, infected with malware that allows remote control by the attacker.

📖 How DDoS Works in Practice

In most real-world DDoS attacks, thousands of bots send UDP packets directly to the server's IP address — not to any specific API endpoint. This saturates the server's available bandwidth, preventing legitimate HTTP requests from reaching the application. This is a network-layer attack, not an application-layer one.

Security Note: "The bots look like real users because they are real computers. Distinguishing bot traffic from legitimate traffic is one of the hardest problems in web security — requiring services like Cloudflare's DDoS protection."

📋 Application-Level DDoS

Bots send requests to expensive API endpoints (like the metadata query above). Each request consumes CPU and database resources. Harder to execute but hard to distinguish from legitimate traffic.

🌐 Network-Level DDoS

Bots flood the server's IP with raw UDP packets — no HTTP headers, no API calls. Saturates bandwidth. Easier to detect (malformed packets) but requires ISP-level mitigation (volume is too high for the server to handle alone).

💡 DoS and Unintended Data Leakage

While DoS typically causes availability issues, some DoS conditions generate error messages that accidentally reveal server internals — stack traces, library versions, and file paths. This is why error handling must be carefully designed: a crashed server can inadvertently disclose more information than a running one.

✓ Chapter 14 Summary — Denial of Service (DoS)

  • DoS: single attacker consumes resources; DDoS: many coordinated devices do the same at scale
  • ReDoS: malicious regex inputs cause exponential backtracking — doubles execution time with each character added
  • Logical DoS: exploit expensive server operations (SQL joins, file backups) that scale poorly with data size
  • DDoS: botnets of hijacked real computers flood a server's IP — typically at the network (UDP) layer
  • DoS vulnerabilities are found by identifying the most resource-intensive operations in the application
  • Unlike most attacks, DoS rarely causes permanent damage — but availability loss can be as costly as a data breach for online businesses
  • Some DoS attempts produce error messages that leak server internals — a secondary vulnerability to capture
