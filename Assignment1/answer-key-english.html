Q3 Why do security professionals use a virtual machine (VM) instead of their physical host machine when performing penetration testing? (at least two reasons) 8 pts
✓ Model Answer

Security professionals prefer VMs for penetration testing for the following reasons:

  • Isolation: The VM is sandboxed from the host OS. Any malware, exploits, or misconfigured tools remain contained inside the VM and cannot damage the host or other systems.
  • Snapshot & rollback: VMs can be snapshotted before risky operations. If a test breaks the environment, the analyst restores the snapshot instead of rebuilding the machine.
  • Portability and reproducibility: A VM image can be shared, exported, and replicated exactly, ensuring all team members test in an identical environment.
  • Legal separation: Keeps the tester's personal/work data separate from test activity logs, reducing legal and privacy exposure.
  • Network control: VM network adapters can be set to host-only or isolated modes, preventing accidental scanning of live production networks.
Grading Notes (8 pts)
  • 4 pts per valid reason (max 2 reasons needed for full credit; a 3rd correct reason can compensate for a weak one).
  • Full credit for a reason: clear identification of the benefit + brief explanation of why it matters in security testing.
  • Partial (2 pts): Correct idea stated but no explanation of the security implication.
  • 0 pts per reason: Vague statements like "it's safer" with no elaboration.
isolation snapshot sandboxing rollback network separation host protection

Q4 What web server software is running behind Juice Shop? How did you identify it? 5 pts
✓ Model Answer

The web server running Juice Shop is Express (Node.js).

It is identified by inspecting the HTTP response headers in the browser's DevTools (F12 → Network tab → select any request → Response Headers):

X-Powered-By: Express

Some installations may also reveal:

Server: (none or custom) X-Powered-By: Express

The X-Powered-By: Express header confirms the backend is built on the Express.js framework running on Node.js.

Q5 What client-side JavaScript framework does Juice Shop use? Name the framework and explain how you identified it. 5 pts
✓ Model Answer

Juice Shop uses Angular (specifically Angular 2+) as its client-side JavaScript framework.

Evidence students can use to identify it:

  • HTML attributes: Custom HTML tags and attributes such as <mat-toolbar>, <app-root>, <router-outlet>, and ng-* directives in the page source (DevTools → Elements tab).

  • Console: Angular apps sometimes print version info in the browser console in development mode.
  • Wappalyzer: Browser extension that fingerprints Angular automatically (if student used it).

Q7 Inspect the cookies set by Juice Shop. List the cookies found and identify missing HttpOnly / Secure flags and what attack each missing flag enables. 12 pts
✓ Model Answer

Juice Shop sets a session cookie (often named token or connect.sid).

Cookie Name HttpOnly Flag Secure Flag
token (JWT) Missing (stored in localStorage or cookie without HttpOnly) Missing (running on HTTP localhost)

Missing HttpOnly flag → enables Cross-Site Scripting (XSS) cookie theft.
Without HttpOnly, JavaScript running in the browser can access the cookie via document.cookie. An attacker who injects a malicious script can steal the session token and impersonate the victim.

Missing Secure flag → enables Man-in-the-Middle (MitM) interception.
Without Secure, the cookie is transmitted over plain HTTP connections. An attacker on the same network (e.g., public Wi-Fi) can intercept the cookie in cleartext using a tool like Wireshark.

Grading Notes (12 pts — 4 pts each criterion)
  • 4 pts — Cookie name(s) listed from the Application/Storage tab.
  • 4 pts — HttpOnly flag status identified + XSS cookie theft explained.
  • 4 pts — Secure flag status identified + MitM/cleartext risk explained.
  • Note: Running locally on HTTP means Secure will always be missing — accept this as correct.
  • Accept token, cookieconsent_status, or any other cookies they observe.
HttpOnly Secure XSS MitM document.cookie session hijacking

Q9 In Reflected XSS, where does the payload travel after you click Submit? Is it stored anywhere permanently? 6 pts
✓ Model Answer

In a Reflected XSS attack, the payload travels through this path:

  • Step 1: The user submits the payload (e.g., <script>alert('XSS')</script>) via the input field.
  • Step 2: The payload is sent to the server as part of the HTTP request (e.g., in the URL query string: ?name=<script>...</script>).
  • Step 3: The server reflects (echoes) the input directly back in the HTTP response HTML without sanitizing it.
  • Step 4: The browser receives the HTML and executes the embedded script.

It is NOT stored permanently. The payload only exists in the URL. If the page is refreshed without the malicious query string, the script does not execute. This is why Reflected XSS attacks are usually delivered via crafted malicious links sent to victims.

Grading Notes (6 pts)
  • 3 pts — Correctly traces the flow: browser → server → reflected in response → executed by browser.
  • 3 pts — Correctly states it is NOT stored permanently and explains the implication.
  • Partial (1–2 pts): Correct direction (client→server→client) but missing explanation of "not stored".
reflected URL query string HTTP response not stored echo sanitization
Q10 In the Stored XSS exercise, you reloaded the guestbook page after submitting the script. What happened, and why did the alert fire again without you resubmitting anything? 8 pts
✓ Model Answer

When the guestbook page was reloaded, the alert box fired again automatically — even though no new input was submitted.

Why this happens:

  • Unlike Reflected XSS, in Stored (Persistent) XSS the server saves the payload to the database when it is first submitted (e.g., as a guestbook entry).
  • Every time any user loads the guestbook page, the server queries the database and includes all saved entries in the HTML response.
  • The malicious script is embedded in the returned HTML and the browser executes it automatically on page load — without any user interaction required.
  • This means every visitor to the page (not just the attacker) will have the script executed in their browser.

This is what makes Stored XSS significantly more dangerous than Reflected XSS: the attack is "baked into" the page and affects all future visitors persistently.

Grading Notes (8 pts)
  • 4 pts — Explains that the payload was saved to the server/database.
  • 4 pts — Explains that every page load retrieves and renders the stored script, causing automatic execution for all visitors.
  • Full credit requires both points. Partial for each.
stored in database persistent page load all visitors affected server-side storage
Q11 Complete the comparison table: Reflected XSS vs Stored XSS 10 pts
✓ Model Answer
Property Reflected XSS Stored XSS
Where is the payload saved? Not saved — it exists only in the URL / HTTP request. The server reflects it back in the response immediately and it disappears after the request. Saved permanently in the server's database (or file storage). Retrieved and served to every visitor.
Who triggers the attack? The victim must click a specially crafted malicious link that contains the payload in the URL. The attacker sends this link (e.g., via phishing email). Any user who simply visits the affected page. No special link required — the payload fires automatically on page load for every visitor.
Which is more dangerous? Stored XSS is more dangerous because: (1) it affects all visitors automatically without any social engineering; (2) the attack is persistent and continues until the payload is removed; (3) a single injection can compromise hundreds or thousands of users.
Grading Notes (10 pts — distributed across 5 cells)
  • 2 pts — Reflected "where saved": not saved / URL only.
  • 2 pts — Stored "where saved": database / server-side storage.
  • 2 pts — Reflected "who triggers": victim clicks malicious link.
  • 2 pts — Stored "who triggers": any visitor / automatic on page load.
  • 2 pts — More dangerous: Stored XSS + valid reason (persistence, all visitors, no link needed).
Q12 A bank's internal support ticket system: attacker submits a stored XSS payload that steals session cookies. Who is at risk, and what can the attacker do with the stolen session cookie? 7 pts
✓ Model Answer

Who is at risk:

  • Bank support agents — Any agent who opens and views the malicious ticket will have the script execute in their browser.
  • Potentially supervisors or managers if they also review the ticket in the same system.
  • Customer data is at risk indirectly if agents have access to it.

What the attacker can do with a stolen session cookie:

  • Session hijacking: The attacker injects the stolen cookie into their own browser (e.g., using DevTools → Application → Cookies), gaining full access to the agent's account without needing a username or password.
  • Access privileged data: With the agent's session, the attacker can view customer account information, transaction histories, or personal data (PII) visible to that agent.
  • Perform unauthorized actions: The attacker can take actions as the agent: approving transactions, changing account settings, downloading reports, or accessing other customers' data.
Grading Notes (7 pts)
  • 3 pts — Correctly identifies bank support agents (not customers) as the primary victims.
  • 4 pts — Explains session hijacking: using the stolen cookie to impersonate the agent and the resulting unauthorized access / data breach impact.
  • Note: Students who say "customers" as primary victims without mentioning agents receive at most 1 pt for the victim question.
session hijacking support agents cookie theft account takeover unauthorized access PII
