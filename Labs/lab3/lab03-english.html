💡 Introduction

In this lab you will observe Cross-Site Scripting (XSS) — one of the most common vulnerabilities on the web. You will use DVWA (Damn Vulnerable Web Application) running locally on your Kali machine to see, first-hand, how an XSS vulnerability allows a browser to execute code that was never part of the legitimate application.

This lab covers two types of XSS: Reflected (the script comes from the URL) and Stored (the script is saved in the database and fires for every visitor).

⚠️

Authorized Use Only

All tasks are performed against DVWA running locally on your own machine. Applying these techniques against any external system without written permission is illegal. Always obtain authorization before testing.

🌐 About DVWA

DVWA (Damn Vulnerable Web Application) is a deliberately insecure PHP/MySQL web application built for security students and professionals to legally practice exploiting common web vulnerabilities in a safe, controlled environment. It is an open-source project maintained by the community and endorsed by OWASP.

🎓

Why Do We Use DVWA?

Real websites cannot be used for attack practice — it is illegal and unethical. DVWA gives you a fully functional web application that is intentionally broken, so you can practise the same techniques used by real attackers without harming any real system.

What Vulnerabilities Does DVWA Cover?

DVWA includes dedicated practice modules for the most critical web vulnerabilities, including:

  • SQL Injection — manipulating database queries through user input
  • Cross-Site Scripting (XSS) — injecting scripts that execute in a victim's browser
  • Cross-Site Request Forgery (CSRF) — tricking a user's browser into making unintended requests
  • Command Injection — executing OS-level commands through a web form
  • File Inclusion / File Upload — abusing file handling to execute malicious code
  • Brute Force — systematically guessing login credentials
  • Weak Session IDs — predicting or hijacking user sessions

Security Levels

DVWA provides three security levels that control how well-protected each module is:

LevelDescriptionPurpose
Low No input validation or filtering at all Learn how the attack works — this is what we use today
Medium Basic protections that can still be bypassed Understand partial defenses and their weaknesses
High Strong but not perfect protections Study how defenses work and where they still fall short
🔗

How We Access DVWA in This Course

DVWA runs inside a Docker container on your Kali Linux virtual machine. Once the container is started, you access it through your browser at http://localhost. Default credentials: admin / password.

⚠️

Important Reminder

DVWA must only run on your local Kali machine, never on a public network. The application is designed to be hacked — exposing it to the internet would allow anyone to compromise your machine.

🎯 Learning Objectives

By the end of this lab, you will be able to:

  • Set up and access DVWA using Docker on Kali Linux
  • Explain how the browser's trust in server responses enables XSS
  • Observe how Reflected XSS executes a script embedded in a URL
  • Observe how Stored XSS persists a script and fires it for every page visitor
  • Articulate why Stored XSS is more dangerous than Reflected XSS

🔁 Task 1: Reflected XSS

💡

Concept Recap — Reflected XSS

In Reflected XSS the server takes a value from the URL and reflects it back in the HTML response without sanitizing it. The browser receives a page that contains the attacker's script and executes it immediately. The payload travels in the URL — it is never stored.

DVWA Module: XSS (Reflected)

Click XSS (Reflected) in the left navigation. You will see a form asking: "What's your name?"

1A: Normal Input

  1. Type your name (e.g., Ahmed) and click Submit
  2. The page responds: "Hello Ahmed"
  3. Look at the browser URL bar — your name appears as a query parameter: ?name=Ahmed
💡

What Is Happening?

The server reads the name value from the URL and inserts it directly into the HTML it sends back. Normally this is fine — but the server does not check whether the input is plain text or an HTML tag.

1B: Test with an HTML Tag

  1. Clear the field and type: <b>Ahmed</b>
  2. Click Submit
  3. The word "Ahmed" now appears bold — the browser rendered your HTML tag, not the server
⚠️

This Is the Flaw

If <b> works, so does <script>. The server treats both identically — it reflects whatever it receives. That is the root of the vulnerability.

1C: Trigger the XSS

  1. Clear the field and type: <script>alert('XSS Works!')</script>
  2. Click Submit
  3. A browser alert dialog appears: "XSS Works!" — click OK
  4. Look at the URL — the entire script tag is embedded as the name parameter

What You Should See:

A browser pop-up alert with the message "XSS Works!". This JavaScript came from the URL — not from the application's own code. The browser could not tell the difference.

1D: What This Means in Practice

The alert() is a harmless demonstration. In a real attack, the script would silently steal session cookies, collect visible page data, or redirect the victim to a phishing site — all without the victim knowing anything happened.

🗃 Task 2: Stored XSS

💡

Concept Recap — Stored XSS

In Stored XSS, the malicious script is saved in the server's database as part of normal content (a comment, message, or review). Every time any user loads that content, the script executes in their browser — no special URL needed, no action required from the attacker.

DVWA Module: XSS (Stored)

Click XSS (Stored) in the left navigation. You will see a simple guestbook with a Name field and a Message field.

2A: Post a Normal Entry

  1. Name: Ahmed  |  Message: Hello, this is my first entry.
  2. Click Sign Guestbook
  3. Your entry appears below the form

2B: Submit a Script as a Message

  1. Name: Tester  |  Message: <script>alert('Stored XSS!')</script>
  2. Click Sign Guestbook
  3. An alert fires immediately — click OK

2C: Reload the Page

  1. Press Ctrl + R to reload
  2. The alert fires again automatically — you submitted nothing new
  3. Click OK
🔴

Why This Is More Dangerous than Reflected XSS

The script is now stored in the database. Every visitor to this guestbook page — a student, an administrator, a customer — will have the script run in their browser. The attacker does not need to be present and does not need to send anyone a special link.

2D: Compare the Two Types

Reflected XSS Stored XSS
Where is the payload? In the URL In the database
What does the victim need to do? Click a malicious link Simply visit the page
Who is affected? Only victims who receive the link Every future visitor
Detectability No server storage — hard to find Stored as text — scannable

🔧 Troubleshooting

❌ DVWA page doesn't load (connection refused)

Solution: Check: sudo docker ps. If dvwa not listed, run sudo docker start dvwa. Go to http://localhost (not port 3000).

❌ Port 80 is already in use

Solution: Stop Apache if running: sudo systemctl stop apache2. Then: sudo docker start dvwa.

❌ DVWA shows a database error after login

Solution: Go to http://localhost/setup.php and click Create / Reset Database. Then log in again.

❌ The script tag does not trigger an alert (browser blocks it)

Solution: Use Firefox (not Chromium). If Firefox still blocks it, open a Private Window (Ctrl+Shift+P) and try again. Make sure the Security Level is Low in DVWA Security.
