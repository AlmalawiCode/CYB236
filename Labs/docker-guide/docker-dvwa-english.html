🐳 What is Docker?

Docker is an open-source platform that allows you to package, distribute, and run applications inside lightweight, isolated environments called containers. Instead of installing software directly on your operating system — which can cause dependency conflicts, break other tools, or leave messy leftovers — Docker wraps each application with everything it needs to run (code, libraries, settings) into one self-contained unit.

Think of Docker as a shipping container in the real world: the same container can be loaded onto any ship, truck, or train without caring what is underneath. In the same way, a Docker container runs the same way on any machine that has Docker installed, regardless of the underlying operating system.

📦 Consistent Environment

DVWA runs exactly the same on every student's Kali machine — no "it works on mine but not yours" problems.

🔒 Isolation

The vulnerable web app runs inside a container, safely separated from your real Kali system.

⚡ Fast Setup

One command starts a fully configured web server with a database, PHP, and DVWA — no manual installation.

🔄 Easy Reset

Remove and recreate the container to get a completely fresh DVWA installation in seconds.

💡

Why Docker in Security Labs?

Running a vulnerable application like DVWA directly on your host system is risky. Docker keeps it isolated: if something goes wrong inside the container, your Kali system is unaffected. You can also destroy and recreate it instantly.

🔑 Docker Key Concepts

Before working with Docker, you need to understand these four core terms:

Term What It Is Real-World Analogy
Image A read-only template — a blueprint for creating containers. It contains the OS, app, and all dependencies, frozen in time. A recipe or a blueprint for a house
Container A live, running instance created from an image. You can start, stop, and restart it. Multiple containers can be made from the same image. The actual house built from the blueprint
Docker Hub An online registry where Docker images are published and shared publicly. Like GitHub, but for Docker images. An app store for pre-built environments
Docker Daemon The background service that manages all Docker operations on your machine. It must be running for any Docker command to work. The factory manager that builds and runs containers
Docker Engine The full Docker software — includes the daemon, CLI, and API. When you run sudo docker ..., you are talking to the engine. The entire factory building

⚖️ Container vs. Virtual Machine

Both containers and virtual machines (VMs) provide isolation, but they work very differently. Understanding this difference helps you know why Docker is used alongside — not instead of — VirtualBox in this course.

Feature 🐳 Docker Container 📦 Virtual Machine (VirtualBox)
What it virtualizes The application and its libraries only An entire computer including the OS
Startup time Seconds Minutes
Disk space Megabytes (DVWA image ≈ 500 MB) Gigabytes (Kali VM ≈ 20+ GB)
RAM usage Very low — shares the host kernel High — needs its own OS memory
Isolation level Process-level (shares host kernel) Full OS-level (completely separate)
Use in this course Run DVWA & Juice Shop quickly Run Kali Linux as your main lab OS

In This Course

You run Kali Linux inside VirtualBox (a VM), and then run DVWA inside Docker (a container) inside that Kali VM. Docker containers are nested inside your virtual machine.

🎯 What is DVWA?

DVWA (Damn Vulnerable Web Application) is a deliberately insecure PHP/MySQL web application designed specifically for security students and professionals to practice web application attacks in a legal, safe, and controlled environment. Every vulnerability in DVWA is intentional — the developers built it broken on purpose.

Property Details
Full Name Damn Vulnerable Web Application
Technology PHP + MySQL (Apache web server)
Purpose Practice common web vulnerabilities legally
Docker Image vulnerables/web-dvwa on Docker Hub
Default Port 80 (HTTP)
Access URL http://localhost or http://127.0.0.1
⚠️

Legal Warning

DVWA must only be used in isolated lab environments on machines you control. Never expose it to the internet or a real network. The vulnerabilities in DVWA are real and can be exploited by others if exposed publicly.

🛡️ DVWA Security Levels

Every module in DVWA can be run at four security levels. This allows you to first understand the vulnerability at its simplest form, then see how real-world defenses make it progressively harder to exploit.

You can change the security level from the DVWA Security menu on the left sidebar after logging in.

Level Description When to Use
Low No security controls at all. Input is passed directly to the backend with zero validation or sanitization. Exploits work with trivial payloads. Learning the attack for the first time. All lab exercises start here.
Medium Basic defenses are applied (e.g., blacklisting some characters, simple input filtering). These defenses are common but flawed — they can be bypassed. Understanding why basic defenses fail and how attackers bypass them.
High Stronger defenses are in place (e.g., whitelisting, parameterized queries, token-based CSRF protection). Exploitation requires more advanced techniques. Seeing what "better" security looks like and how it can still sometimes be defeated.
Impossible The vulnerability is fully patched. The source code represents best-practice secure development. Exploitation is not possible. Studying how to write secure code. Comparing with Low level to understand what was wrong.
⚠️

Always Start at Low

Unless your instructor specifies otherwise, always set DVWA to Low security before starting a lab exercise. Higher levels may prevent the expected attack from working, causing confusion.

📄 Viewing the Source Code

One of DVWA's most powerful teaching features is the "View Source" button at the bottom of each module page. It shows you the actual PHP code running behind the module at each security level — so you can see exactly what makes the code vulnerable or secure.

🔌 Understanding Port Mapping

When you run DVWA with the command docker run -d -p 80:80 ..., the -p 80:80 part tells Docker how to connect your Kali machine to the container. This is called port mapping (or port publishing).

What is a Port?

A port is a numbered communication channel on a computer. When a browser talks to a web server, it uses specific port numbers:

Port Protocol Used For
80 HTTP Standard unencrypted web traffic — used by DVWA and Juice Shop
443 HTTPS Encrypted web traffic (SSL/TLS)
3000 HTTP (custom) Used by some apps like Juice Shop (alternate port)
8080 HTTP (alternate) Common alternative to port 80 for web apps and Burp Suite proxy
3306 MySQL Database connections — used internally by DVWA

How -p 80:80 Works

Your Browser
http://localhost
→ port 80 →
Kali Linux
Host port 80
→ mapped to →
Docker Container
Container port 80

The format is HOST_PORT:CONTAINER_PORT. Both are 80 here, but you could use a different host port to avoid conflicts — for example -p 8081:80 would make DVWA available at http://localhost:8081.

What is localhost?

localhost and 127.0.0.1 are both special addresses that always refer to your own machine. When the browser on your Kali VM requests http://localhost, it connects to port 80 on Kali itself — which Docker then forwards into the DVWA container.

💡

Why Port Conflicts Happen

Only one process can listen on a given host port at a time. If DVWA is already using host port 80 and you try to start Juice Shop on port 80 too, Docker refuses. This is the most common error in this course.

State What It Means Command to Check
No container Container has never been created, or was removed with docker rm. Use docker run to create it. docker ps -a shows nothing
Running Container is active and serving requests. DVWA is accessible in the browser. docker ps → STATUS: Up
Stopped / Exited Container exists on disk but is not running. Use docker start (not docker run) to bring it back. docker ps -a → STATUS: Exited
⚠️

run vs. start — The Most Common Mistake

docker run creates a brand-new container from an image. If a container with that name already exists (even if stopped), run will fail with a "name already in use" error. Use docker start dvwa to restart an existing stopped container.

📋 Listing Containers

Always start here when DVWA is not working. Know what is running before doing anything else.

▶ Show Only Running Containers

sudo docker ps

This shows containers that are currently active. The output columns mean:

ColumnMeaning
CONTAINER IDShort unique ID (e.g. a1b2c3d4e5f6)
IMAGEThe image used (e.g. vulnerables/web-dvwa)
STATUSHow long running (e.g. Up 3 minutes)
PORTSPort mapping (e.g. 0.0.0.0:80->80/tcp)
NAMESContainer name (e.g. dvwa)

📂 Show ALL Containers (Including Stopped)

sudo docker ps -a

The -a flag includes stopped and exited containers. Use this to find a previously created DVWA container that is not currently running.

🔍 Inspect a Specific Container

sudo docker inspect dvwa

Shows detailed JSON information about the container: network settings, mounts, environment variables, and more. Useful for advanced debugging.

📊 Show Container Resource Usage

sudo docker stats dvwa

Shows live CPU and memory usage. Press Ctrl+C to exit.

✅

Reading the Output

If STATUS says Up → DVWA is running. If it says Exited → run docker start dvwa. If there is no row at all → run docker run ... to create it.

🚀 Starting DVWA

Option A — First Time (No Container Exists)

If docker ps -a shows no DVWA container, create and start it:

sudo docker run -d -p 80:80 --name dvwa vulnerables/web-dvwa

If docker ps -a shows STATUS: Exited, restart it without recreating:

If DVWA is running but behaving oddly (e.g., database errors), restart it in place:

🛑 Stopping & Removing Containers

Graceful Stop

Sends a termination signal and waits up to 10 seconds for the container to shut down cleanly:

sudo docker stop dvwa

Force Kill

Immediately terminates the container — no waiting. Use when stop does not respond:

sudo docker kill dvwa

Remove a Stopped Container

Deletes the container from disk. The image is not deleted — you can docker run again to recreate:

sudo docker rm dvwa

Force Stop and Remove in One Command

sudo docker rm -f dvwa

Remove All Stopped Containers at Once

sudo docker container prune
⚠️

stop vs. kill vs. rm

  • stop — politely asks the container to stop (SIGTERM, then SIGKILL after 10 s)
  • kill — immediately kills the container (SIGKILL)
  • rm — deletes the container from the system (container must already be stopped)
  • rm -f — kills and deletes in one step

⚠️ Two Containers Conflict (Port Already in Use)

This is the most common problem in this course. Lab 02 uses Juice Shop on port 80. Labs 03–10 use DVWA on port 80. Only one can own port 80 at a time.

docker: Error response from daemon: driver failed programming external connectivity...
Bind for 0.0.0.0:80 failed: port is already allocated.

Diagnose the Conflict

sudo docker ps

Look at the PORTS column. Whichever container shows 0.0.0.0:80->80/tcp is occupying port 80.

Fix It: Stop the Conflicting Container, Then Start DVWA

sudo docker stop juice-shop
sudo docker start dvwa

Switching Between Applications

Switch to Juice Shopsudo docker stop dvwa && sudo docker start juice-shop
Stop everythingsudo docker stop $(sudo docker ps -q)

🔧 Common Problems & Solutions

❌ "port is already allocated"

Cause: Another container (Juice Shop or something else) is bound to port 80.

sudo docker ps
sudo docker stop juice-shop
sudo docker start dvwa
❌ "The container name /dvwa is already in use"

Cause: A stopped container named "dvwa" already exists. You tried to docker run again instead of docker start.

Fix — just start the existing one:

sudo docker start dvwa

Fix — or remove it and recreate:

sudo docker rm dvwa
sudo docker run -d -p 80:80 --name dvwa vulnerables/web-dvwa
❌ DVWA shows "Cannot connect to the database"

Cause: MySQL inside the container has not finished initializing (takes ~10–15 seconds after start).

Fix: Wait 15 seconds, refresh the browser. If it persists:

sudo docker restart dvwa

Then go to http://localhost/setup.php → click Create / Reset Database.

❌ Browser shows "This site can't be reached" or Connection Refused

Cause: DVWA container is not running, or wrong URL.

sudo docker ps          # Check if dvwa is Up
sudo docker start dvwa  # If not running

Make sure you use http://localhost (not https://).

❌ Container is stuck — docker stop does nothing

Cause: The container is frozen and ignoring the SIGTERM signal.

sudo docker kill dvwa
sudo docker start dvwa
❌ "Cannot connect to the Docker daemon" or "docker: command not found"

Cause: Docker is not installed, or the Docker daemon is not running.

sudo apt update && sudo apt install -y docker.io
sudo systemctl start docker
sudo systemctl enable docker
sudo systemctl status docker
❌ "Permission denied" when running docker commands

Cause: You forgot sudo, or your user is not in the docker group.

Fix — always use sudo in Kali:

sudo docker ps

Or add your user to the docker group (permanent fix):

sudo usermod -aG docker $USER
newgrp docker
❌ DVWA login fails with correct credentials

Cause: The database was not initialized after container start.

Fix: Visit http://localhost/setup.php and click "Create / Reset Database". Then log in with admin / password.

❌ Burp Suite cannot intercept DVWA traffic when using localhost

Cause: Browsers automatically bypass the proxy for localhost and 127.0.0.1. This is a built-in browser security rule — loopback addresses are never sent through a proxy, so Burp Suite never sees the requests even when FoxyProxy is enabled.

Fix — use the machine's real IP address instead of localhost:

Step 1: Find your Kali machine's actual IP address:

hostname -i

This returns an IP such as 10.0.2.15 or 192.168.x.x — the real network address of your Kali VM.

Step 2: Open DVWA in the browser using that IP instead of localhost:

http://10.0.2.15

Replace 10.0.2.15 with whatever hostname -i returned on your machine.

Step 3: Make sure Burp Suite proxy is configured to listen on 127.0.0.1:8080 (default), and FoxyProxy is set to route traffic through 127.0.0.1 port 8080.

Step 4: Now browse to DVWA using the real IP. Burp Suite will intercept the requests normally.

💡

Why does this work?

When you use the real IP (10.x.x.x), the browser treats the request as a normal network request and sends it through the configured proxy (Burp Suite). When you use localhost or 127.0.0.1, the browser short-circuits directly to the loopback interface — skipping the proxy entirely.

⚠️

The IP may change between sessions

VirtualBox assigns the IP dynamically. Always run hostname -i at the start of each lab session to confirm the current IP before using it in the browser.
