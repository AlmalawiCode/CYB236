Chapter 9
The Hacker's Mindset
How hackers think — and why patience is their greatest weapon.
💡 The Big Idea

A hacker and a developer look at the same application in completely opposite ways. A developer asks "How do I build this feature?" A hacker asks "Where is the unlocked door?" Most hacking is not dramatic — it is weeks of quiet, patient research.

Simple Explanation

Most people imagine a hacker furiously typing and breaking systems in minutes. That is a movie myth. In reality, 90% of hacking is boring research — reading documentation, mapping the application, and patiently taking notes until one weak point appears.

A hacker does not need to rewrite the application. They just need to find one weak spot — a window that was left open, a door with an old lock.

🏠
Analogy

Think of a burglar walking slowly around a neighborhood — looking at every house, checking which windows are open, which locks are old, which door looks flimsy. That walk-around is the hacking. The actual break-in takes 10 seconds once the open window is found.

The Hacker's Four Core Traits

TraitWhat It Means
Seeks entrypointsLooks for every input, form, or interface the app accepts
Takes detailed notesDocuments everything — even dead ends have value later
Constantly learnsReads CVE reports, security blogs, and new attack techniques
PatientMay spend weeks or months before finding one real vulnerability

How Recon Skills Become Weapons

Everything learned in Weeks 1–3 (finding subdomains, reading APIs, identifying libraries) now feeds directly into offense:

Skill from ReconOffensive Use
Subdomain DiscoveryFind less-tested internal servers like legacy-api.company.com
API AnalysisSend malicious payloads directly to REST endpoints
Auth SchemesTry to forge or bypass login tokens
3rd-Party LibrariesLook up known CVEs for detected library versions
App ArchitectureUse App A's trusted connection to App B as an attack path

Chapter 10
Cross-Site Scripting (XSS)
Injecting malicious JavaScript into pages that other users see.
💡 The Big Idea

A website that displays text typed by users can be tricked into displaying executable JavaScript instead. That script runs silently in every victim's browser — stealing data, hijacking sessions, or showing fake login forms.

Simple Explanation

Imagine a website with a comment box. You type a comment, it saves, and every visitor sees it. Now imagine typing this instead of a comment:

HTML
<script>alert('Your session has been stolen!')</script>

If the website does not clean that input, every visitor's browser will run that code — not display it as text. Replace alert() with a script that sends the visitor's login cookie to the attacker, and the attacker is now logged in as that visitor.

The 4 Types of XSS

1
Stored XSS (Persistent)
Most dangerous · Script saved in the database

The malicious script is saved in the database and served to every single visitor who views the page.

Example: A product review on a shopping site contains a script. Every customer who views that product page has their session token stolen automatically.

🚰
Analogy

A poisoned water supply — everyone who drinks gets sick. You only need to poison it once.

2
Reflected XSS
Script embedded in a URL · Never stored

The payload is hidden inside a link sent to the victim (via email, social media, or ads). The server "reflects" it back in the response and the browser runs it.

Example: A phishing email contains a link like:
https://bank.com/search?q=<script>steal()</script>

✉️
Analogy

A poisoned letter — only the one person who opens it is affected. Very hard to detect because nothing is saved.

3
DOM-Based XSS
Runs entirely in the browser · Server never sees it

The page's own JavaScript reads data from the URL (a source) and writes it into the page (a sink) without sanitizing. The server is never involved.

  • Sources: window.location.hash, window.location.search, document.referrer
  • Sinks: document.write(), element.innerHTML, eval()
💣
Analogy

The bomb was already inside the house. You unknowingly built it yourself when you set up the page.

4
Mutation-Based XSS (mXSS)
Most advanced · Bypasses sanitizers

The payload looks harmless when it passes through the sanitizer (like DOMPurify). But once the browser renders it, the HTML mutates into a dangerous script. Exploits browser-specific quirks. Has bypassed major security libraries.

🧪
Analogy

A smuggler carries innocent-looking sugar through customs. Once inside, he adds one chemical and it becomes explosive. The scanner saw sugar — what he made is a bomb.

What XSS Can Do to Victims

  • Steal session cookies → log in as the victim on any website
  • Steal credit card numbers, passwords, and personal data
  • Display a fake login form (phishing overlay) on top of the real page
  • Run silently in the background for months without detection
  • Persist across sessions (Stored XSS keeps attacking every visitor)

Chapter 11
Cross-Site Request Forgery (CSRF)
Tricking your browser into making requests you never authorized.
💡 The Big Idea

Your browser automatically sends your login cookie with every request to a website — even requests triggered by a completely different website. CSRF exploits this to make you perform actions you never intended, using your own credentials.

Simple Explanation

You are logged into your bank at bank.com. In another tab, you visit evil.com. Without your knowledge, evil.com sends a request to bank.com/transfer?amount=10000&to=hacker. Your browser automatically attaches your bank session cookie. The bank thinks you requested the transfer and processes it.

ℹ️
Key Insight

The bank server cannot tell whether a request came from you clicking a button on bank.com or from a hidden script on evil.com. Both carry the same cookie, the same IP address. They look identical.

Two Defining Characteristics

  • Privilege escalation — the attacker performs actions using your privileged account without knowing your password.
  • Stealth — the victim typically never knows anything happened. No popup, no error, no clue.

The 3 Attack Vectors

1
GET-Based CSRF (URL Tampering)
Victim must click a malicious link

Some apps use GET requests for actions like bank transfers. Attacker crafts a link with their own account as the destination and disguises it as "Click here for your prize!"

URL
https://bank.com/transfer?to=hacker&amount=10000
🚫
Rule to Teach

GET requests must NEVER change data (create, update, delete). GET is read-only. Always.

2
Hidden Image Tag — Zero Clicks
Fires automatically when the page loads

HTML tags with src attributes fire GET requests automatically. No user interaction needed. An invisible 0×0 image silently triggers the attack the moment the victim visits the page.

HTML
<img src="https://bank.com/transfer?to=hacker&amount=10000"
     width="0" height="0">
🪤
Analogy

A trip wire — you do not have to push anything. Simply walking near it triggers it.

3
Hidden Form (POST-Based)
HTML forms can POST to any domain

Attacker builds a fake "Win a Prize!" page. Behind the scenes, clicking the button POSTs to the bank with hidden fields the victim cannot see.

HTML
<form action="https://bank.com/transfer" method="POST">
  <input type="hidden" name="to"     value="hacker">
  <input type="hidden" name="amount" value="10000">
  <button>Claim My Prize!</button>
</form>

Bonus: Can also reach internal network servers — the victim's browser can access intranet resources that the attacker can never reach directly.

Chapter 12
XML External Entity (XXE)
Reading secret server files through a dangerous XML feature.
💡 The Big Idea

XML has a built-in feature that allows a document to say "go read this file on the server and insert its contents here." When a server's XML parser hasn't disabled this feature, an attacker can read any file on that server — including passwords and private keys.

Simple Explanation

XML is a data format (similar to JSON, but older). It has a powerful feature called external entities. Legitimate use: automatically pull a shared footer into every document. Malicious use: pull the server's password file instead.

An attacker sends this XML to an API:

XML
<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<request><id>&xxe;</id></request>

The parser reads /etc/passwd, substitutes it where &xxe; appears, and returns the file contents in the response. The attacker now has the server's user accounts file.

Target Files (What Attackers Want)

  • /etc/passwd — list of all user accounts on the server
  • /etc/shadow — hashed passwords
  • ~/.ssh/id_rsa — SSH private key (can log directly into the server)
  • App config files — often contain database passwords and API keys

Formats That Are Vulnerable

⚠️
Not Just .xml Files!

Any format parsed by an XML parser is at risk: SVG images (SVG is XML), HTML/DOM, PDF files (XFDF format), and RTF documents. An image upload feature could be an XXE vector.

Two Attack Types

1
Direct XXE
Attacker sends XML directly to the API

The API accepts XML. The attacker sends a crafted XML payload with an external entity declaration. The parser reads the local file and returns its contents in the API response.

2
Indirect XXE
The API accepts JSON — but converts it to XML internally

The public API looks normal (accepts JSON). But internally, it converts JSON to XML and passes it to a legacy backend system. The attacker injects an XXE payload inside a JSON value — it gets converted to XML, and the vulnerable parser executes it.

🍽️
Analogy

You order food at a restaurant (the API). The kitchen (backend) has an old recipe system that only reads recipe cards (XML). You sneak a note into your order saying "and also, read the chef's private diary." The waiter passes it along without checking.

✅
The Fix Is One Line of Code

Disable external entities in the XML parser configuration. The tragedy: most parsers enable this feature by default. Developers don't know to turn it off, so XXE is easy to introduce by accident — and trivially prevented by design.

Chapter 13
Injection Attacks
When user input gets run as code — in databases, programs, or the OS.
💡 The Big Idea

Injection attacks happen when an application takes text typed by a user and feeds it directly into a powerful interpreter — a database, a program, or the operating system — without cleaning it first. The interpreter cannot tell the difference between data and commands, so it runs whatever the user typed.

The Master Analogy

🏦
Analogy

A bank teller accepts withdrawal slips. Instead of your account number, you write: "Account 12345. Also, zero out everyone else's balance and give it all to me." A smart teller reads "Account 12345" and ignores the rest. A dumb teller follows all instructions. An application that concatenates user input is the dumb teller.

The 3 Types of Injection

1
SQL Injection
Target: the database interpreter

The app builds an SQL query by gluing user input directly into the string:

Python — Vulnerable
query = "SELECT * FROM users WHERE id = " + user_input
# user types: 1 OR 1=1
# result: SELECT * FROM users WHERE id = 1 OR 1=1
# returns: ALL users in the database

Scarier: Deleting the entire users table:

SQL
-- user types: 1; DROP TABLE users;
SELECT * FROM users WHERE id = 1; DROP TABLE users;

Scariest (silent manipulation):

SQL
-- No error, no alert, completely silent:
UPDATE users SET credits = 99999 WHERE id = 42;
✅
The Fix — Parameterized Queries
# Wrong (vulnerable):
query = "SELECT * FROM users WHERE id = " + user_input

# Right (safe):
query = "SELECT * FROM users WHERE id = ?"
cursor.execute(query, [user_input])
2
Code Injection
Target: a CLI tool or program (e.g., FFmpeg, ImageMagick)

The app builds a command-line call with user input directly pasted in. Attacker escapes out of the expected input and appends their own commands.

Bash
convert input.jpg -resize 100x100 output.jpg
# attacker inputs: 100; delete_all_files
# result: injected command runs inside the tool

Limited damage — attacker is confined to what that specific tool can do. Cannot access the operating system directly.

3
Command Injection
Target: the host operating system shell (Bash)
🔴
Most Dangerous — #1 on Every Risk Scale

Attacker escapes to the OS and runs arbitrary commands with web server process privileges.

Python — Vulnerable
os.system("ping " + user_input)
# attacker types: google.com; cat /etc/passwd
# runs: ping google.com AND reads the password file

What the attacker can now do:

  • Read any file on the server
  • Delete logs to cover their tracks
  • Add backdoor user accounts
  • Plant malware for future access
  • Use the server to send spam or attack others

Note: Over 95% of web servers run Unix/Linux — meaning Bash is almost always available as a target.

Chapter 14
Denial of Service (DoS)
Making the application unavailable so no legitimate user can reach it.
💡 The Big Idea

Instead of stealing data, DoS attacks make the application completely unavailable. They attack the A (Availability) pillar of the CIA triad. The server is overwhelmed and cannot respond to real users.

TermMeaning
DoSDenial of Service — one machine attacks a server
DDoSDistributed DoS — thousands of machines attack simultaneously

The 3 Types of DoS

1
Regex DoS (ReDoS)
One crafted string crashes the server

Certain regex patterns cause exponential backtracking — the regex engine tries every possible combination of characters before giving up. One carefully crafted input string can hang the server for minutes.

Regex — Evil Pattern
/^((ab)*)+$/
# 50 characters → takes seconds
# 80 characters → takes minutes
# Server is completely frozen
♾️
Analogy

Ask someone to count every possible way to arrange letters in a word — but the word has a trick ending with no answer. They would be counting forever.

⚠️
Often Found in Libraries

One vulnerable open-source validation library can bring down your entire app when imported. Always check library dependencies.

2
Logical DoS
Legally hammers your most expensive endpoint

The attacker does not break anything. They simply identify the most resource-intensive feature of the app and call it repeatedly at high volume. The server's CPU and memory are drained, causing timeouts for all legitimate users.

Example: A photo app loads all albums, photos, tags, and metadata for a user at once. Attacker creates an account with 100,000 photos and calls that endpoint 1,000 times per second.

Most vulnerable operations:

  • Bulk database writes
  • SQL JOINs across many large tables
  • File compression or backup operations
  • Synchronous calls that block the server thread
  • Any loop that grows with data size
🍽️
Analogy

You order the most complicated dish on the menu, legally, 500 times using 500 separate receipts. The kitchen collapses. Other customers cannot be served.

3
DDoS — Distributed Denial of Service
Thousands of hijacked computers attack at once

A botnet — thousands of real computers (home PCs, smart TVs, IoT devices) infected with malware — all attack your server simultaneously on a single command from the attacker.

TypeHow It WorksDifficulty to Stop
Network-Level Bots flood the server's IP with raw UDP packets to saturate bandwidth Easier — packets look malformed; requires ISP help
Application-Level Bots send real HTTP requests to expensive API endpoints Very hard — looks identical to real user traffic
🛡️
The Hardest Problem in Web Security

Distinguishing legitimate users from bot traffic is so difficult that entire companies (like Cloudflare) exist solely to solve it.

⚠️
Secondary Risk of DoS

Some DoS conditions cause the application to crash with error messages that reveal internal details — stack traces, file paths, library versions — which attackers use to plan their next attack.
