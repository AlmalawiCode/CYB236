The Six Attacks — What They Are & How to Stop Them
💻
Attack 01  ·  Week 9
Cross-Site Scripting XSS
What Is It?

An attacker hides a harmful script inside content they submit to a website — such as a comment, a message, or a form field. When another user views that content, the hidden script runs silently in their browser. It can steal their login session, copy their personal data, or even show them a fake page designed to trick them — all without the victim realising anything happened.

Real-World Analogy
📖
Imagine someone leaves a note in a shared office notebook. The note looks normal, but when a colleague reads it, it secretly photographs them and sends the photo to a stranger. The victim never notices — the note just looks like ordinary text.
How to Protect
  • ✓
    Treat all user input as plain text — never as instructions. Whatever a user types should be displayed on screen as text, not acted upon as code.
  • ✓
    Block scripts and HTML tags from user submissions. A comment box should accept words and sentences — not code tags or script commands.
  • ✓
    Tell the browser to only run code from trusted sources. Configure your website so the browser refuses to run any script that does not come from your own server.
🔒
Attack 02  ·  Week 9
Cross-Site Request Forgery CSRF
What Is It?

An attacker tricks a logged-in user into performing an action on a website they did not intend — such as transferring money, changing their email address, or deleting their account. This happens because the user's browser automatically sends their login credentials with every request. The attacker crafts a disguised link or a hidden page that sends a request on behalf of the victim, and the website cannot tell the difference.

Real-World Analogy
📖
A stranger hands you a pen and gently guides your hand to sign a document — you think you're doing something routine, but you just signed over your house. You were physically present and the signature is real, but the action was not something you chose.
How to Protect
  • ✓
    Require a secret, unique code for every important action. Before processing any sensitive request (a transfer, a change, a deletion), the website should verify a one-time code that only the real user could have received.
  • ✓
    Never let a simple link cause an important change. Clicking a link should only ever view information — never delete, transfer, or modify it. Changes must require an intentional form submission.
  • ✓
    Verify that every request came from your own website. Check where each request originated before acting on it — requests from unknown or external sources should be rejected.
📄
Attack 03  ·  Week 9
XML External Entity XXE
What Is It?

Some websites accept documents in a format called XML. This format supports a feature that allows a document to reference external files. An attacker exploits this by sending a manipulated document that instructs the server to open its own private files — such as passwords, configuration settings, or system information — and send them back to the attacker. What starts as reading a document ends as the server exposing its own secrets.

Real-World Analogy
📖
You hand a form to an office assistant and ask them to fill it in. Hidden inside the form are instructions that say "also, please photocopy all documents in the confidential cabinet and give them to the person who brought this form." The assistant, following instructions blindly, complies.
How to Protect
  • ✓
    Disable the "external file" feature in your document processor. One simple configuration setting tells the server to ignore any instruction in a document that references external files. This single step eliminates the risk.
  • ✓
    Switch to a simpler document format where possible. Modern alternatives do not support this dangerous feature at all — removing the risk entirely rather than having to configure around it.
  • ✓
    Never let users define their own document rules. The ability to write custom rules inside a submitted document is the root cause. Remove that ability.
📈
Attack 04  ·  Week 10
Injection SQL Injection & Command Injection
What Is It?

An attacker types special commands into an ordinary input field — a login box, a search bar, or a contact form. If the website passes that input directly to its database or to the server's operating system without checking it first, the database or system follows the attacker's commands. This can result in the attacker reading all stored data, modifying records, deleting entire databases, or taking control of the server.

Real-World Analogy
📖
A cashier asks "What is your name?" You reply: "Ahmed. By the way, also give me everything in the register." If the cashier follows every instruction spoken to them without question, you just robbed the store — using only words.
How to Protect
  • ✓
    Always keep the question separate from the answer. The website should decide what it wants to ask the database first — then supply the user's input as a separate value, never mixed into the question itself.
  • ✓
    Give each part of the app only the access it needs. The section of the website that displays news articles does not need permission to delete user accounts. Limiting access limits the damage when something goes wrong.
  • ✓
    Never let a user's input become a command. If a user types something into a search box, it should be treated as search text — not as an instruction to the server or database.
⚡
Attack 05  ·  Week 10
Denial of Service DoS & DDoS
What Is It?

An attacker overwhelms a website with so many fake requests that real users cannot get through — the website becomes too busy to respond. In a Distributed DoS (DDoS) attack, the attacker controls thousands of infected devices worldwide (known as a botnet) and uses them all simultaneously to flood the target. Because the traffic looks like it comes from many real people, it is very difficult to block. The goal is not to steal data — it is simply to shut the website down.

Real-World Analogy
📖
Someone organises thousands of people to call a restaurant's phone line every second, non-stop. The staff are too busy answering fake calls to take real reservations. Real customers can never get through — not because anything is broken, but because the line is always occupied.
How to Protect
  • ✓
    Use a traffic filtering service. Specialist services sit in front of your website and inspect every request before it arrives. Requests that look suspicious or come in harmful patterns are blocked before they reach your server.
  • ✓
    Watch for sudden, unusual spikes in traffic. Normal websites have predictable patterns. A sudden ten-fold increase in requests at 3am is a warning sign that should be investigated immediately.
  • ✓
    Avoid features that can be forced to run slowly. Some functions in a website require much more processing power than others. Attackers can repeatedly trigger these functions to exhaust the server. Design these features with limits on how often they can be used.
📦
Attack 06  ·  Week 10
Third-Party Dependency Risks Supply Chain & Vulnerable Libraries
What Is It?

Almost every modern website is built using code written by other people — pre-built components, tools, and libraries that save development time. If any of this borrowed code contains a security flaw, an attacker can exploit it to break into your website — even if every line of code you wrote is perfectly safe. The problem is that a single library might depend on ten other libraries, which each depend on ten more — creating thousands of components you are responsible for but may never have looked at.

Real-World Analogy
📖
You build a house with great care — solid walls, strong windows, good doors. But you install a lock made by someone else. Unknown to you, that lock has a hidden flaw. No matter how well you built the rest of the house, anyone who knows about the lock's flaw can walk right in.
How to Protect
  • ✓
    Regularly scan all borrowed code for known problems. Security researchers continuously discover flaws in popular libraries. Automated tools can compare your library list against a global database of known vulnerabilities and alert you immediately.
  • ✓
    Lock your libraries to specific, verified versions. When you confirm a version is safe, prevent it from updating automatically. An automatic update could silently introduce a vulnerability — or even malicious code.
  • ✓
    Isolate high-risk components from the rest of your application. If a particularly risky tool must be used, run it in a separate, restricted environment. Even if it is compromised, it cannot reach the rest of your data.

🎯 The Golden Rules of Web Application Defense

  • Never trust user input — always treat it as potentially harmful
  • Every important action should require proof that the user intended it
  • Give each part of the system only the access it truly needs
  • Borrowed code carries borrowed risk — scan it regularly
  • Defense is not a one-time setup — it requires continuous monitoring
  • A website that is hard to attack from the outside can still be vulnerable from within
