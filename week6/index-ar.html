الفصل 14
حجب الخدمة (DoS)
🎯

أهداف التعلم

1
شرح الفئات الثلاث الرئيسية لهجمات حجب الخدمة — Regex DoS والحجب المنطقي والحجب الموزّع — وكيف تستنزف موارد الخادم أو العميل.
2
وصف كيف تتسبّب أنماط regex الخبيثة في تراجع تكاسلي أسّي، مع توضيح سبب تضاعف وقت التنفيذ مع كل زوج من الأحرف المضافة.
3
تحديد العمليات المُكلفة في واجهات API — كتابة قاعدة البيانات، عمليات SQL join، النسخ الاحتياطي — التي يمكن إساءة استخدامها لتنفيذ الحجب المنطقي.
4
فهم آلية عمل شبكات Botnet في هجمات DDoS — أجهزة مخترقة تُهاجم هدفاً واحداً بشكل جماعي — وسبب صعوبة الدفاع عنها مقارنةً بهجمات المهاجم المنفرد.
🚫

ما هو هجوم حجب الخدمة؟

يُعدّ هجوم حجب الخدمة (DoS) من أكثر فئات الهجمات شهرةً وانتشاراً — وهو مجموعة من الهجمات التي تهدف إلى جعل النظام غير متاح أو متدهور الأداء للمستخدمين الشرعيين عبر استنزاف موارد الخادم أو العميل.

📖 من الكتاب — طيف هجمات DoS

تأتي هجمات DoS في أشكال متعددة: من النسخة الموزّعة المعروفة التي تشمل آلاف الأجهزة المنسّقة، إلى حجب الخدمة على مستوى الكود الذي يُصيب مستخدماً واحداً بسبب تعبير نمطي (regex) خاطئ يُطيل وقت التحقق من النصوص.

تتفاوت خطورة هذه الهجمات بشكل كبير — من تعطيل خادم نشط بالكامل، إلى إبطاء تحميل صفحة الويب أو توقّف الفيديو في منتصف التشغيل.

نظراً لهذا التنوع، يصعب جداً اختبار هجمات حجب الخدمة. تحظر معظم برامج مكافآت الأخطاء تقديم ثغرات حجب الخدمة تجنّباً للتأثير على المستخدمين الحقيقيين.

في الغالب، لا تُسبّب هجمات DoS ضرراً دائماً للتطبيق، لكنها تُعطّل تجربة المستخدمين الشرعيين. ثلاثة أهداف رئيسية تقف وراء كل هجوم:

💻
استنزاف موارد الخادم
إرهاق المعالج والذاكرة وعرض النطاق الترددي أو اتصالات قاعدة البيانات
📱
استنزاف موارد العميل
تشغيل عمليات ثقيلة في متصفح المستخدم أو جهازه المحمول حتى يتجمّد
🚫
طلب موارد غير متاحة
إجبار الخادم على البحث عن موارد مقفلة أو مفقودة أو في حالة توقف

⚠️ أفضل ممارسة للاختبار

نظراً لتأثير هجمات حجب الخدمة على المستخدمين الحقيقيين، يجب دائماً اختبار هذه الثغرات في بيئة تطوير معزولة — وليس على أنظمة الإنتاج الحيّة.

🔍

حجب خدمة Regex (ReDoS)

تُعدّ ثغرات حجب الخدمة المبنية على التعبيرات النمطية (ReDoS) من أكثر أشكال DoS شيوعاً في تطبيقات الويب اليوم. تُستخدم التعبيرات النمطية للتحقق من صحة حقول النماذج — كالتأكد من أن كلمة المرور تحتوي فقط على أحرف مقبولة.

📖 من الكتاب — Regex الخبيث

التعبيرات النمطية سريعة بطبيعتها، لكن يمكن تصميمها خصيصاً لتعمل ببطء شديد — وتُعرف هذه بـالتعبيرات الخبيثة أو الشريرة. تعتمد معظمها على المعامل + الذي يحوّل البحث إلى "جشع" — يبحث عن أكثر من تطابق بدلاً من التوقف عند الأول.

يُسبّب هذا تراجعاً أسّياً عند فشل المطابقة، إذ يُجرّب المحرّك كل تركيبة ممكنة قبل إعلان الفشل.

آلية التراجع — التعبير النمطي الشرير

خذ مثلاً هذا التعبير: /^((ab)*)+$/

  • ^ — يطابق من بداية السطر
  • (ab)* — يطابق صفراً أو أكثر من تركيبات "ab"
  • + — يجد كل تطابق ممكن (عملية جشعة)
  • $ — يطابق حتى نهاية السطر

اختبار النص ababababababab يعمل بسرعة. لكن إضافة حرف واحد — abababababababa — يُجبر المحرّك على تجربة كل تركيبة ممكنة قبل إعلان عدم وجود تطابق:

// JavaScript — نمط Regex خبيث (شرير) const evil = /^((ab)*)+$/; // مدخلات آمنة — يتوقف المحرّك مبكراً evil.test('ababababababab'); // أقل من 1ms — تطابق نظيف // مدخلات خبيثة — حرف 'a' الزائد يُفجّر التراجع evil.test('abababababababa'); // بطيء — يجرّب كل التركيبات evil.test('abababababababababababababa'); // ~31ms عند 27 حرفاً evil.test('ababababababababababababababa'); // ~62ms عند 29 حرفاً (يتضاعف!) // التركيبات التي يجرّبها المحرّك قبل الفشل: // (abababababababa) → غير صالح // (ababababababa)(ba) → غير صالح // (abababababa)(baba) → غير صالح // ... تكرارات كثيرة ... // (ab)(ab)(ab)(ab)(ab)(ab)(a) → غير صالح → FAIL

وقت التنفيذ يتضاعف مع كل زوج من الأحرف

المدخل الخبيث الطول وقت التنفيذ
abababababababababababa 23 حرفاً ~8 ms
ababababababababababababa 25 حرفاً ~15 ms
abababababababababababababa 27 حرفاً ~31 ms
ababababababababababababababa 29 حرفاً ~62 ms (تضاعف)

💡 القنبلة الموقوتة المخفية

قد يظل التعبير النمطي الخبيث كامناً لسنوات حتى يعثر أحد المهاجمين على مدخل يُفجّر التراجع. مكتبات البرامج مفتوحة المصدر أكثر عرضةً للخطر — إذ نادراً ما يُدرَّب المساهمون على اكتشاف الأنماط الخبيثة.

⚙️

ثغرات الحجب المنطقي

في الحجب المنطقي، يستنزف المهاجم موارد الخادم باستغلال منطق التطبيق نفسه — ليس خللاً في regex، بل نقطة نهاية (API endpoint) مُكلفة يمكن إساءة استخدامها. نتيجةً لذلك، يعاني المستخدمون الشرعيون من تدهور في الأداء أو انقطاع في الخدمة.

للبحث عن ثغرات الحجب المنطقي، ابحث عن العمليات الأكثر استهلاكاً للموارد:

🗃
عمليات كتابة قاعدة البيانات
كل عملية كتابة تُقفل صفوفاً وتستهلك موارد القرص
💾
كتابة الملفات والسجلات
عمليات متزامنة تحجب الخيط البرمجي حتى تنتهي
📋
عمليات SQL Join
ربط جداول كبيرة يزداد تعقيداً مع حجم البيانات
🔄
النسخ الاحتياطي للملفات
مهام النسخ المُشغَّلة عبر API تُشبع موارد القرص
⏱️
العمليات المتزامنة
تحجب الخيط البرمجي للخادم كلياً حتى تكتمل
🔢
الاستعلامات التكرارية
حلقات تتوسّع مع حجم بيانات المستخدم بلا حدود
📖 دراسة حالة — API تطبيق مشاركة الصور

يكشف تطبيق لمشاركة الصور عن نقطة نهاية GET /metadata/:userid لاسترجاع بيانات وصفية لجميع الصور. داخلياً، تُنفّذ عملية SQL join عبر جداول: مستخدمون ← ألبومات ← صور ← بيانات وصفية. يتوسّع وقت التنفيذ بشكل كبير مع بيانات المستخدم:

نوع الحسابالألبومات / الصورزمن الاستجابة
حساب جديدألبوم واحد، صورة واحدة120 ms
مستخدم عادي6 ألبومات، 60 صورة470 ms
مستخدم مكثّف28 ألبوماً، 490 صورة1,870 ms

يُنشئ المهاجم حساباً، يرفع آلاف الصور في مئات الألبومات، ثم يُرسل طلبات متكررة إلى نقطة النهاية — مما يُرهق الخادم ويُدهور الأداء لجميع المستخدمين الآخرين.

💡 كيف تكتشف ثغرات الحجب المنطقي

قِس وقت استجابة API باستخدام أدوات المطوّر في المتصفح. كرّر نفس الطلب ~100 مرة واحسب المتوسط لاستبعاد الارتفاعات العشوائية. ابحث عن نقاط النهاية التي تزداد بطئاً مع تزايد بيانات المستخدم — فهي مرشّحة الحجب المنطقي.

🌐

الحجب الموزّع (DDoS)

في الحجب الموزّع للخدمة (DDoS)، لا يُنفّذ الهجوم مهاجم واحد بل شبكة ضخمة من الأجهزة غير الشرعية ترسل طلبات بشكل متزامن. على نطاق واسع، يُغرق هذا موارد الخادم ويحجب المستخدمين الشرعيين كلياً.

🚫 حجب الخدمة المنفرد (DoS)

مهاجم واحد يستغل ثغرة محددة في الكود — regex أو منطقية أو غيرها. يستهدف نقاط نهاية معينة. أسهل في التخفيف بعد اكتشاف الثغرة.

🚫🚫 الحجب الموزّع (DDoS)

آلاف الأجهزة تُهاجم هدفاً واحداً في وقت واحد. يستهدف عادةً طبقة الشبكة (فيضان UDP)، لا منطق التطبيق. أصعب بكثير في الدفاع لأن الحركة تبدو شرعية.

كيف تعمل شبكات الروبوتات (Botnet)

  • 1
    توزيع البرامج الخبيثة
    يُوزّع المهاجم برامج ضارة عبر تنزيلات تبدو مشروعة أو مرفقات بريد إلكتروني أو مواقع مخترقة. تُثبَّت البرامج على أجهزة الضحايا دون علمهم.
  • 2
    تجميع شبكة الروبوتات
    تصبح الأجهزة المصابة — حواسيب، هواتف ذكية، أجهزة IoT (أجهزة توجيه، أجهزة منزلية ذكية) — "روبوتات" يتحكم فيها المهاجم عن بُعد. مجتمعةً تُشكّل شبكة الروبوتات (Botnet).
  • 3
    الهجوم المنسّق
    يأمر المهاجم الشبكة بإغراق عنوان IP الهدف بحزم UDP في وقت واحد. كل حزمة تبدو صادرة من جهاز مختلف، مما يجعل الفلترة صعبة للغاية.
  • 4
    انقطاع الخدمة
    يُستهلك عرض النطاق الترددي للخادم بالكامل من قِبل الحركة غير الشرعية. لا يستطيع المستخدمون الشرعيون الاتصال — تبدو الخدمة كأنها معطّلة تماماً.

⚠️ أجهزة إنترنت الأشياء هدف سهل

يمكن اختراق أجهزة IoT (أجهزة التوجيه، نقاط الاتصال، الأجهزة الذكية) وإضافتها إلى شبكات الروبوتات بسهولة أكبر من أجهزة الحاسوب بسبب ضعف إعدادات الأمان الافتراضية وندرة تحديثات البرامج الثابتة.

✓ ملخص الفصل 14 — حجب الخدمة (DoS)

  • تهدف هجمات DoS إلى استنزاف موارد الخادم أو العميل، أو طلب موارد غير متاحة
  • ReDoS يستغل أنماط regex خبيثة تُسبّب تراجعاً أسّياً — وقت التنفيذ يتضاعف مع كل زوج من الأحرف في المدخل الخبيث
  • الحجب المنطقي يُسيء استخدام نقاط نهاية API المُكلفة (عمليات SQL join، النسخ الاحتياطي، كتابة قاعدة البيانات) التي تتوسّع مع بيانات المستخدم
  • DDoS يستخدم شبكات الروبوتات — أجهزة مخترقة — لإغراق الهدف بحركة تفوق طاقته
  • معظم هجمات DDoS تستهدف طبقة الشبكة (فيضان UDP)، لا منطق التطبيق
  • اختبر ثغرات حجب الخدمة دائماً في بيئات تطوير معزولة — لا على أنظمة الإنتاج الحيّة
الفصل 15
استغلال تبعيات الجهات الخارجية
🎯

أهداف التعلم

1
شرح سبب تشكيل البرمجيات مفتوحة المصدر (البرمجيات مفتوحة المصدر) مخاطر أمنية كبيرة — لا سيما صعوبة مراجعة كود طرف ثالث يتطوّر باستمرار.
2
وصف طرق التكامل الأربع مع البرمجيات مفتوحة المصدر — الفروع/الشوكات، التثبيت المستضاف ذاتياً، الكود المصدري المباشر، ومديرو الحزم — والمخاطر الأمنية الخاصة بكل منها.
3
تحديد الهجمات الحقيقية على npm — left-pad وeslint-scope وevent-stream — وآلية كل اختراق.
4
استخدام قواعد بيانات CVE (NVD وMitre) للبحث عن ثغرات معروفة في تبعيات الجهات الخارجية، وفهم كيف تُمكّن بيانات الاستطلاع من استغلالها.
🔗

تبعيات البرمجيات مفتوحة المصدر — سطح الهجوم المخفي

لا سرّ في أن البرمجيات الحديثة مبنية على البرمجيات مفتوحة المصدر (البرمجيات مفتوحة المصدر). حتى أضخم المنتجات التجارية تعتمد بشكل كبير على حزم طرف ثالث يُساهم فيها مطوّرون من حول العالم.

📖 من الكتاب — من يعتمد على البرمجيات مفتوحة المصدر؟

بعض أبرز المنتجات المبنية على البرمجيات مفتوحة المصدر:

  • Reddit — BackBoneJS, Bootstrap
  • Twitch — Webpack, Nginx
  • YouTube — Polymer
  • LinkedIn — EmberJS
  • Microsoft Office Web — AngularJS
  • Amazon DocumentDB — MongoDB
السلسلة لا تكون أقوى من أضعف حلقاتها — والأضعف في الغالب هي تلك التي خضعت لأدنى مستوى من ضمان الجودة.

الاعتماد على البرمجيات مفتوحة المصدر يعني الاعتماد على قاعدة كود لم تُراجَع بنفس صرامة الكود الأول. التقييم الأمني النقطي لقاعدة كود البرمجيات مفتوحة المصدر ضخمة مكلف — ويصبح قديماً بمجرد أن تتلقّى المكتبة تحديثاً جديداً.

✅ لماذا يستخدم المطوّرون البرمجيات مفتوحة المصدر؟

السرعة، توفير التكاليف، الوظائف المُجرَّبة، دعم المجتمع. أمر واحد في مدير الحزم يُثبّت في ثوانٍ ما قد يستغرق أسابيع لبنائه. ضغط السوق يجعل البرمجيات مفتوحة المصدر الخيار الافتراضي.

🚫 لماذا يستهدف المهاجمون البرمجيات مفتوحة المصدر؟

واسع الانتشار، كثيراً ما يكون قديماً، وندراً ما يُراجَع بالكامل. ثغرة واحدة في حزمة شائعة قد تُصيب ملايين التطبيقات في وقت واحد — نسبة هجوم/تأثير عالية جداً.

⚙️

طرق تكامل البرمجيات مفتوحة المصدر

فهم كيفية تكامل تطبيق الويب مع البرمجيات مفتوحة المصدر أمر حاسم — طريقة التكامل تُحدّد ما يتدفق بينهما من بيانات، وكيف يتحرّك، وما مستوى الصلاحيات الممنوح لكود الطرف الثالث.

  • 1
    الفروع والشوكات (Git)
    يُنشئ المطوّر فرعاً أو شوكة من مستودع البرمجيات مفتوحة المصدر ويُشغّل نسخته منه. الخطر: سهولة السحب العرضي لكود غير مُراجَع من الفرع الرئيسي إلى بيئة الإنتاج. الشوكات توفّر عزلاً أفضل لكنها تُعقّد عمليات الدمج المستقبلية.
  • 2
    التطبيقات المستضافة ذاتياً
    بعض البرامج (مثل WordPress) تأتي مع سكريبت تثبيت بنقرة واحدة. الخطر: هذه السكريبتات تعمل بصلاحيات مديرية على الخادم — سكريبت مخترق قد يُثبّت باباً خلفياً مع تنفيذ كود عن بُعد (RCE) كاملاً.
  • 3
    تكامل الكود المصدري المباشر (نسخ/لصق)
    بسيط للأدوات الصغيرة (50-100 سطر)، معقّد للمكتبات الكبيرة ذات التبعيات. الخطر: إصلاح ثغرة أمنية في المصدر الأصلي لا يصلك تلقائياً — النسخة الضعيفة تبقى في كودك إلى أجل غير مسمى.
  • 4
    مديرو الحزم (npm وMaven وغيرها)
    الأسلوب الأكثر شيوعاً اليوم. يسحب مدير الحزم التبعيات — وتبعياتها الفرعية بشكل تعاودي — عند البناء. الخطر: حزمة واحدة خبيثة أو مخترقة في السجل العام قد تنتشر تلقائياً إلى ملايين التطبيقات.

⚠️ برامج التثبيت المستضافة ذاتياً خطر بالغ

تطلب برامج التثبيت ذات النقرة الواحدة صلاحيات مرتفعة. سكريبت تثبيت مخترق يعمل بصلاحيات مدير النظام يمكنه تثبيت أبواب خلفية واستخراج بيانات الاعتماد وتمكين تنفيذ الكود عن بُعد (RCE) بشكل مستمر.

📦

مديرو الحزم كمتجهات هجوم

في عالم اليوم، تتم معظم تكاملات البرمجيات مفتوحة المصدر عبر مديري الحزم — تطبيقات تُنزّل التبعيات الصحيحة وتُعدّها تلقائياً. npm يُغذّي منظومة JavaScript؛ وMaven يُغذّي منظومة Java.

📖 من الكتاب — هجمات npm الحقيقية

left-pad (2016): مكتبة أداة بسيطة من سطر واحد يُشرف عليها شخص واحد سُحبت من npm. أوقف هذا الحدث أنابيب البناء لملايين التطبيقات حول العالم. استجابةً لذلك، لم يعد npm يسمح بحذف الحزم بعد مرور وقت معيّن.

eslint-scope (2018): اختُرقت بيانات اعتماد مالك الحزمة. نشر المهاجم نسخة جديدة تسرق بيانات الاعتماد المحلية من كل جهاز تُثبَّت عليه.

event-stream (2018): أُضيفت تبعية فرعية خبيثة (flatmap-stream) تحتوي على كود يسرق محافظ Bitcoin من أجهزة المستخدمين.

npm (JavaScript / Node.js)

أكبر سجل حزم في الوجود. يُعرَّف بملفَّي package.json وpackage-lock.json. يحلّ التبعيات من المستوى الأعلى وجميع التبعيات الفرعية التعاودية. آليات أمان تاريخياً مرنة جعلته هدفاً متكرراً.

Maven (Java)

أقدم وأشهر مدير حزم لـJava، تدعمه مؤسسة Apache. يسبق Git في نشأته. تعرّض هو الآخر لهجمات، وإن حظيت بتغطية إعلامية أقل من npm.

💡 مخاطرة التبعيات الفرعية التعاودية

يحلّ npm ليس فقط التبعيات المباشرة بل أيضاً جميع التبعيات الفرعية التعاودية. ثغرة مدفونة في عمق أربع طبقات من التبعيات قد تخترق تطبيقك — حتى لو لم تُضِف تلك الحزمة يداً بيد قط.

📋

قواعد بيانات CVE — البحث عن الثغرات المعروفة

أسرع طريقة عملية لاستغلال تبعيات طرف ثالث هي البحث عن ثغرات موثّقة وغير مُصلَّحة في الحزم التي يستخدمها التطبيق الهدف — ثغرات أُصلحت في الإصدارات الأحدث لكن لم تُحدَّث في التطبيق بعد.

📖 من الكتاب — قواعد بيانات CVE

يُفصح علناً عن الثغرات في الحزم الشهيرة ويُوثَّق في قواعد بيانات مثل:

  • NVD (قاعدة البيانات الوطنية للثغرات) — تُديرها وزارة التجارة الأمريكية؛ تُقيّم الثغرات بدرجة خطورة CVSS
  • Mitre CVE (نقاط الضعف والتعرّض الشائعة) — برعاية وزارة الأمن الداخلي الأمريكي؛ المعيار العالمي لمعرّفات الثغرات
تُسهّل قواعد بيانات CVE إيجاد الثغرات المعروفة واستغلالها — لكن لا بد أولاً من تحديد الإصدارات المستخدمة بالضبط في التطبيق الهدف عبر الاستطلاع.

مكتبة JQuery — المستخدمة في أكثر من 10 ملايين موقع — مثال مثالي. يكشف بحث في NVD عن عشرات الثغرات الموثّقة على مدار السنوات، مع خطوات إعادة الإنتاج وتقييمات الخطورة.

❌ بدون وعي بـ CVE
  • تشغيل JQuery الإصدار 1.x (ثغرات XSS معروفة)
  • لا فحص آلي لشجرة التبعيات
  • لا تحديث للحزم بعد الإطلاق الأول
  • لا متابعة للإفصاحات الأمنية
  • ثغرات معروفة تبقى دون إصلاح لأشهر
✓ مع الوعي بـ CVE
  • مراجعة دورية لـNVD/Mitre لجميع التبعيات
  • تشغيل Snyk وnpm audit تلقائياً في خطّ CI/CD
  • تحديث أو استبدال الحزم الضعيفة فوراً
  • تتبّع إصدارات التبعيات عبر الشجرة كاملة
  • الاشتراك في النشرات الأمنية للحزم الحيوية

✓ ملخص الفصل 15 — استغلال تبعيات الجهات الخارجية

  • تطبيقات الويب الحديثة تعتمد على البرمجيات مفتوحة المصدر اعتماداً كبيراً — كل تبعية متجه هجوم محتمل لم يُراجَع بصرامة الكود الأول
  • أربع طرق للتكامل: الفروع/الشوكات، التثبيت المستضاف، الكود المباشر، مديرو الحزم — لكل منها مخاطر مختلفة
  • مديرو الحزم (npm، Maven) أهداف عالية القيمة — حزمة واحدة مخترقة قد تنتشر إلى ملايين التطبيقات في وقت واحد
  • الهجمات الحقيقية (left-pad، eslint-scope، event-stream) تُثبت أن اختراق بيانات اعتماد مُشرف الحزمة وسيلة هجوم فعّالة ومتكررة
  • قواعد بيانات CVE (NVD، Mitre) توثّق الثغرات المعروفة — ادمجها مع الاستطلاع لتحديد الإصدارات القابلة للاستغلال
  • التبعيات الفرعية التعاودية تعني أن تطبيقك يُشغّل آلاف الحزم لم تختَرها يداً بيد
الفصل 26
الدفاع ضد هجمات حجب الخدمة
🎯

أهداف التعلم

1
بناء أنظمة تسجيل شاملة تُسجّل توقيت كل طلب وأداء المهام غير المتزامنة، لتمكين الكشف والتحقيق في محاولات حجب الخدمة السابقة.
2
استخدام أدوات التحليل الساكن (محقّقات Linter) ومختبرات أداء regex للكشف عن الأنماط الخبيثة ومنعها من دخول قاعدة الكود.
3
تصنيف نقاط نهاية API حسب درجة خطر الحجب (عالية/متوسطة/منخفضة)، وتصميم استراتيجيات تخفيف مناسبة لكل مستوى.
4
تقييم استراتيجيات التخفيف من DDoS — خدمات إدارة عرض النطاق والثقب الأسود — وفهم المقايضة بين حجب الحركة الخبيثة والشرعية.
📄

التسجيل — خط الدفاع الأول

عادةً ما تستهلك هجمات حجب الخدمة موارد النظام، مما يجعل اكتشافها صعباً بدون تسجيل شامل. هجوم مرّ عبر قنوات API مشروعة قد يكون من المستحيل تشخيصه بعد وقوعه دون سجلات مناسبة.

📖 من الكتاب — استراتيجية التسجيل

يجب أن يشمل نظام التسجيل الشامل تسجيل:

  • كل طلب مع وقت الاستجابة المقابل له
  • أداء المهام غير المتزامنة (مثل النسخ الاحتياطي المُشغَّل عبر API الذي يعمل في الخلفية ولا يُعيد استجابة)
  • مقاييس استهلاك الموارد (المعالج، الذاكرة، تجمّع اتصالات قاعدة البيانات) لكل نقطة نهاية
يُمكّنك هذا من اكتشاف أي محاولات — عرضية أو خبيثة — لاستغلال ثغرة حجب خدمة كانت ستُصعَّب تشخيصها بدون هذه السجلات.

💡 مخاطر DoS ليست ثنائية — بل طيف

بخلاف XSS (إما ثغرة موجودة أو لا)، يُقاس خطر DoS على طيف: عالٍ / متوسط / منخفض. حتى الكود المكتوب بجودة عالية قد يقع ضحية الحجب المنطقي إن امتلك المهاجم موارد كافية. صنّف نقاط النهاية حسب مستوى الخطر وحدّد الدفاعات بناءً على ذلك.

🔍

الحماية من حجب خدمة Regex

يُعدّ Regex DoS أسهل أشكال الحجب في الدفاع عنه، لكنه يتطلّب معرفة مسبقة بكيفية بناء الأنماط الخبيثة. خطوتان دفاعيتان رئيسيتان:

  • 1
    فحص قاعدة الكود بحثاً عن regex الخبيث
    ابحث عن تعبيرات نمطية تحتوي معامل الجشع + مطبّقاً على مجموعات متكررة — أنماط مشابهة لـ(a[ab]*)+. استخدم أداة Linter مفتوحة المصدر (مثل safe-regex أو vuln-regex-detector) لوسم الأنماط الخبيثة تلقائياً في خطّ CI/CD.
  • 2
    حظر التعبيرات النمطية المقدَّمة من المستخدم
    إذا كانت أي ميزة تسمح للمستخدمين بتقديم تعبيراتهم النمطية الخاصة (مثل فلاتر البحث)، فهذا خطر غير مقبول. السماح بـregex من المستخدمين يُشبه "السير في حقل ألغام" — يتطلب جهداً منسّقاً هائلاً للصيانة الآمنة.
// خطر — regex مقدَّم من المستخدم (لا تسمح بهذا أبداً) app.post('/search', (req, res) => { const pattern = new RegExp(req.body.userPattern); // ✗ خطر ReDoS! const results = items.filter(i => pattern.test(i)); // ✗ الخادم عرضة للحجب }); // آمن — أنماط محددة مسبقاً من جانب الخادم فقط const SAFE_PATTERNS = { email: /^[^\s@]+@[^\s@]+\.[^\s@]+$/, username: /^[a-zA-Z0-9_]{3,20}$/, phone: /^\+?[0-9]{7,15}$/ }; // ✓ تحقّق من اختيار المستخدم مقابل الخريطة الآمنة — لا تُقيّم المدخل الخام

🛠 افحص تبعياتك أيضاً

تأكّد من أن أياً من الحزم الخارجية التي تُدمجها لا تستخدم regex من المستخدم أو تعبيرات نمطية سيئة الكتابة. النمط الخبيث في مكتبة npm تعتمد عليها خطير بنفس قدر ذلك الموجود في كودك.

⚙️

الحماية من الحجب المنطقي

الحجب المنطقي أصعب بكثير في الكشف عنه ومنعه مقارنةً بـRegex DoS — فهو مُضمَّن في منطق الأعمال، لا في نمط يمكن فحصه ساكناً. الاستراتيجية الرئيسية هي تصنيف نقاط نهاية API حسب درجة خطر الحجب، ثم تطبيق التخفيفات المناسبة.

📖 من الكتاب — طيف المخاطر

يجب قياس خطر الحجب على ثلاثة مستويات: عالٍ / متوسط / منخفض — لا كتقييم ثنائي آمن/ضعيف. مستخدم على حاسوب قوي قد لا يلاحظ دالة قابلة للاستغلال من جانب العميل، بينما مستخدم على هاتف قديم سيُعاني. مهاجم بموارد ضخمة قد يُرهق حتى الكود الفعّال.

يجب أن نُفكّر في الوظائف المكشوفة من حيث خطر الحجب: عالٍ/متوسط/منخفض. هذا أكثر منطقية من حكم ثنائي آمن/ضعيف، إذ يعتمد الحجب على استهلاك موارد يصعب تصنيفه بشكل مطلق.

استراتيجيات التخفيف لنقاط النهاية عالية الخطر

تحديد عدد طلبات المستخدم في الدقيقة لنقاط النهاية المُكلفة
📋

الحماية من هجمات DDoS

لا يمكن منع هجمات DDoS — بل يمكن التخفيف منها فقط. نظراً لاستهداف هذه الهجمات عرض النطاق الترددي بحركة مرور تبدو مشروعة، لا تنجح استراتيجيات الترقيع التقليدية.

  • 1
    خدمات إدارة عرض النطاق الترددي
    خدمات خارجية تعترض كل الحركة الواردة وتُحلّل كل حزمة قبل إعادة توجيهها إلى خادمك. الحزم المصنّفة خبيثة (بناءً على النمط أو المصدر أو التكرار) تُسقط قبل وصولها لبنيتك التحتية. هذه الخدمات فعّالة لامتلاكها طاقة استيعاب تفوق خوادمك بكثير.
  • 2
    الثقب الأسود (Blackholing)
    أنشئ خوادم "طعم" إضافية إلى جانب خادمك الرئيسي. يُوجَّه الحركة المشبوهة أو المتكررة إلى خادم الثقب الأسود — يبدو وكأنه يعمل لكنه لا يُنفّذ أي عمليات حقيقية. الحركة الشرعية تستمر لخادمك الأصلي. فعّال ضد الهجمات الصغيرة لكن قد يُخطئ في توجيه حركة شرعية.
  • 3
    رسم خط أساسي للحركة الطبيعية
    قبل نشر فلاتر DDoS المتشدّدة، ارسم مقاييس دقيقة لأنماط حركة المستخدمين الشرعيين. الفلاتر الشديدة الحساسية ستحجب مستخدمين حقيقيين إذا لم تعرف ما يبدو عليه "الطبيعي". قِس أولاً، ثم اضبط.

خدمة إدارة عرض النطاق

خدمة خارجية أمام بنيتك التحتية. تكلفة أعلى لكن فعّالية عالية. تمتص الهجمات الضخمة بفضل طاقتها الاستيعابية الكبيرة. الخيار الأفضل للتطبيقات الإنتاجية المعرّضة لخطر DDoS حقيقي.

الثقب الأسود (Blackholing)

خوادم طعم مخصّصة. تكلفة أقل لكن يحتاج منطق توجيه مخصّصاً. فعّال ضد الهجمات الصغيرة. قد يُوجَّه الحركة الشرعية بالخطأ. مناسب كطبقة دفاعية تكميلية.

⚠️ الفلترة المفرطة خطر حقيقي

كل استراتيجية تخفيف من DDoS تنطوي على خطر حجب المستخدمين الشرعيين إذا كانت الفلاتر متشدّدة جداً. لهذا يُعدّ رسم خط أساسي للحركة الطبيعية — معرفة ما يبدو عليه النمط "العادي" — أمراً أساسياً قبل نشر أي تدابير مضادة.

✓ ملخص الفصل 26 — الدفاع ضد هجمات حجب الخدمة

  • التسجيل الشامل هو الأساس — سجّل كل طلب مع توقيته لتتمكّن من التحقيق في محاولات الحجب بأثر رجعي
  • Regex DoS: افحص قاعدة الكود بأدوات Linter للأنماط الخبيثة؛ لا تسمح أبداً بـregex من المستخدم في أي ميزة
  • الحجب المنطقي: صنّف نقاط النهاية (عالٍ/متوسط/منخفض)؛ طبّق تحديد المعدل، الحصص، الترقيم، المهلة، التخزين المؤقت، وطوابير المهام
  • DDoS لا يمكن منعه — خفّف تأثيره بخدمات إدارة عرض النطاق وتقنية الثقب الأسود كطبقات دفاعية متداخلة
  • ارسم خطاً أساسياً لحركة المستخدمين الشرعيين قبل نشر فلاتر مضادة لـDDoS، لتجنّب حجب المستخدمين الحقيقيين
  • حتى لو لم تتوقّع أن تكون هدفاً اليوم، طبّق تدابير مضادة لحجب الخدمة استباقياً قبل أن تحتاجها

تأمين تبعيات الجهات الخارجية
🎯

أهداف التعلم

1
نمذجة شجرة التبعيات الكاملة للتطبيق — بما فيها التبعيات المتعدية (تبعيات الدرجة الرابعة) — باستخدام npm ls والأوامر المشابهة.
2
استخدام أدوات الفحص التلقائي (Snyk، npm audit، NIST NVD) لمقارنة شجرة التبعيات بقواعد بيانات CVE وتحديد الحزم الضعيفة.
3
تطبيق مبدأ فصل المسؤوليات لعزل الكود الخارجي عالي الخطر على خوادم منفصلة والتواصل معه عبر JSON على HTTP.
4
تنفيذ إدارة الحزم الآمنة باستخدام تثبيت الإصدار وأمر npm shrinkwrap والإشارة إلى Git SHA لمنع التحديثات غير المصرّح بها.
🍁

فهم أشجار التبعيات

من أهم المفاهيم في أمان الجهات الخارجية شجرة التبعيات — الرسم البياني الهرمي لكل حزمة يعتمد عليها تطبيقك، بما في ذلك الحزم التي تعتمد عليها تبعياتك (تبعيات الدرجة الرابعة)، وهكذا بشكل تعاودي.

📖 من الكتاب — تعارضات الإصدارات في الواقع

في المثالي، كل مكوّن يعتمد على JQuery يستخدم نفس الإصدار. في الواقع:

// شجرة تبعيات حقيقية — تعارضات الإصدارات شائعة Primary App v1.6 → JQuery 3.4.0 // تبعية مباشرة: مُصلَّحة Primary App v1.6 → SPA Framework v1.3.2 → JQuery 2.2.1 // تبعية فرعية: ضعيفة! الفحص التلقائي للتبعيات

مراجعة شجرة تبعيات تحتوي آلاف الحزم يدوياً شبه مستحيل. الفحص التلقائي هو الأسلوب الوحيد القابل للتوسّع لتقييم أمان الشجرة الكاملة لتطبيق كبير.

  • 1
    استخراج الشجرة الكاملة
    استخدم npm ls --depth=[depth] أو npm list لإنتاج شجرة التبعيات الكاملة كقائمة منظّمة. هذه القائمة هي مدخل فحص الثغرات. في المنظومات الأخرى، يوفّر Maven وGradle وpip أوامر مكافئة.
  • 2
    المقارنة بقواعد بيانات CVE
    قارن كل حزمة وإصدارها في الشجرة بـNIST NVD أو قاعدة بيانات استشارات npm. أدوات مثل Snyk أو npm audit تُؤتمت هذه العملية وتُقدّم اقتراحات للإصلاح.
  • 3
    الفحص عند إضافة أي تبعية جديدة
    عند إضافة أي تبعية خارجية جديدة، يجب فحص شجرة تبعياتها الفرعية تلقائياً — ويُفضَّل تضمين هذا الفحص كبوّابة في خطّ CI/CD تمنع النشر عند اكتشاف ثغرات عالية الخطورة.
  • 4
    تفضيل NIST للمشاريع طويلة الأمد
    قاعدة بيانات NIST NVD ممولة من الحكومة الأمريكية وتتمتع باستمرارية عالية. للتطبيقات التي يجب أن تعمل لسنوات، تُعدّ NIST مرجعاً أكثر استقراراً من قواعد بيانات الاستشارات الخاصة بكل مورّد.
// npm — فحص الثغرات المدمج $ npm audit # مثال على المخرجات: found 3 vulnerabilities (1 moderate, 2 high) high Prototype Pollution in lodash <4.17.21 high Remote Code Execution in serialize-javascript moderate ReDoS in acorn // إصلاح تلقائي حيثما أمكن $ npm audit fix // Snyk — فحص متقدّم مع توجيهات للإصلاح $ snyk test // فحص مرة واحدة $ snyk monitor // مراقبة مستمرة في خطّ CI/CD

تقنيات التكامل الآمن

كيفية تصميم التكامل مع حزمة طرف ثالث تؤثر مباشرةً في نطاق الضرر عند وقوع حادثة أمنية. المبدأ الرئيسي هو فصل المسؤوليات — عزل الكود المحفوف بالمخاطر بحيث لا ينتشر الاختراق إلى بقية التطبيق.

📖 من الكتاب — فصل المسؤوليات

أحد أكبر مخاطر تشغيل كود طرف ثالث على خادمك الرئيسي أنه قد يمتلك آثاراً جانبية، أو — إذا اختُرق — أن يستحوذ على موارد النظام إذا لم يُطبَّق مبدأ أدنى الصلاحيات بشكل صحيح.

الحل: شغّل التكامل مع الطرف الثالث على خادمه الخاص. أرسل إليه واستقبل منه بيانات JSON عبر HTTP. صيغة JSON تضمن أن تنفيذ السكريبتات على خادمك الرئيسي غير ممكن دون ثغرات إضافية مُتسلسلة.

❌ تكامل مترابط بإحكام

كود الطرف الثالث يعمل مباشرةً داخل عملية التطبيق الرئيسية. الحزمة المخترقة تصل إلى كل ذاكرة الخادم، متغيرات البيئة، بيانات الاعتماد، ونظام الملفات. اختراق كامل للتطبيق من استغلال حزمة واحدة.

✓ تكامل معزول (Microservice)

كود الطرف الثالث يعمل على خادمه المعزول. التواصل عبر HTTP/JSON. الحزمة المخترقة محاصرة داخل حدود خادمها — لا تصل مباشرةً إلى بيانات الاعتماد أو بيانات التطبيق الرئيسي. نطاق ضرر محدود.

💡 مقايضات العزل

رغم أن العزل يُقلّل الخطر على الخادم الرئيسي، إلا أن البيانات السرية المُرسَلة لخادم التبعية قد تتعرّض للكشف إذا اختُرق ذلك الخادم. كما يُضيف هذا الأسلوب تأخيراً طفيفاً في الأداء بسبب رحلات HTTP ذهاباً وإياباً. العزل مثالي للحزم عالية الخطورة لكن قد لا يكون ممكناً دائماً.

📦

إدارة الحزم الآمنة

عند استخدام مديري الحزم كـnpm، ثمة مخاطر مقبولة مع كل حزمة مُثبَّتة. تثبيت الإصدار والتجميد (Shrinkwrap) تقنيتان تُقلّصان بشكل كبير خطر تحديثات صامتة غير مرخّصة تُدخل ثغرات جديدة.

📖 من الكتاب — مخاطر الإصدار الدلالي

يستخدم الإصدار الدلالي ثلاثة أرقام: رئيسي.ثانوي.تصحيح (مثلاً 1.0.24). افتراضياً، يضع npm علامة الإقواس (^) قبل كل إصدار، مما يعني الترقية التلقائية لآخر تصحيح. هذا يعني أن myLib@^1.0.23 قد تصبح myLib@1.0.24 دون علمك، وإن كان التصحيح الجديد مخترقاً فتطبيقك يتأثر تلقائياً.

  • 1
    إزالة علامة ^ — تثبيت الإصدار الدقيق
    في ملف package.json، أزِل البادئة ^ من إصدارات التبعيات. هذا يُجبر npm على استخدام الإصدار المُراجَع بالضبط — لا ترقيات تلقائية، فقط ترقيات يدوية ومتعمّدة.
  • 2
    تجميد شجرة التبعيات الكاملة (Shrinkwrap)
    شغّل npm shrinkwrap لإنشاء ملف npm-shrinkwrap.json. يُثبّت هذا الإصدار الدقيق لكل تبعية وكل تبعية فرعية في الشجرة بالكامل — لا فقط الحزم المباشرة. أضف هذا الملف إلى مستودعك.
  • 3
    الإشارة إلى Git SHA للحد الأقصى من الأمان
    عدّل ملف shrinkwrap للإشارة إلى هاشات Git SHA بدلاً من أرقام الإصدارات وحدها. يُزيل هذا الخطر النادر لكن الحقيقي المتمثّل في قيام مُشرف الحزمة بنشر كود جديد تحت رقم إصدار موجود.
  • 4
    نشر مرآة npm الخاصة بك
    للأمان الأعلى، استضِف مرآة npm خاصة تحتوي فقط على الإصدارات المُعتمَدة والمُراجَعة من كل تبعية. يسحب خطّ CI/CD من مرآتك، لا من السجل العام.
// package.json — مقارنة تثبيت الإصدار // غير آمن — ^ يسمح بالترقية الصامتة التلقائية { "dependencies": { "express": "^4.18.2", // ✗ قد تُصبح 4.18.3 بصمت "lodash": "^4.17.21" // ✗ نفس الخطر } } // آمن — إصدار مثبّت بالضبط، بلا ^ { "dependencies": { "express": "4.18.2", // ✓ هذا الإصدار المُراجَع بالضبط "lodash": "4.17.21" // ✓ هذا الإصدار المُراجَع بالضبط } } // إنشاء ملف التجميد — يُثبّت شجرة التبعيات الكاملة $ npm shrinkwrap // ينشئ npm-shrinkwrap.json — أضفه إلى مستودعك!

🛠 قيود Shrinkwrap

يُثبّت التجميد الإصدارات لكنه لا يحمي بالكامل من قيام مُشرف الحزمة بإعادة استخدام رقم إصدار موجود لكود جديد — وهو أمر ممكن تقنياً في npm. للقضاء على هذا الخطر، عدّل ملف التجميد للإشارة إلى Git SHA أو استضِف مرآة npm خاصة.

✓ ملخص الفصل 27 — تأمين تبعيات الجهات الخارجية

  • كل تكامل مع طرف ثالث يحمل مخاطر مقبولة — الهدف تقليصها والتحكّم فيها بضوابط متداخلة، لا القضاء على التبعيات كلياً
  • نمذجة الشجرة الكاملة (باستخدام npm ls) — التطبيقات الكبيرة قد تمتلك آلاف التبعيات الفرعية المتعدية التي لم تختَرها مباشرةً
  • استخدم الفحص التلقائي لـCVE (Snyk، npm audit، NIST NVD) في خطّ CI/CD — المراجعة اليدوية لا تتوسّع مع الشجرات الكبيرة
  • عزل التبعيات عالية الخطورة على خوادم منفصلة؛ تواصل معها عبر JSON/HTTP لتحديد نطاق الضرر عند الاختراق
  • ثبّت إصدارات التبعيات: أزِل ^ من package.json، شغّل npm shrinkwrap، وفكّر في الإشارة إلى Git SHA للحزم الحيوية
  • تبعيات الجهات الخارجية ستحمل دائماً بعض المخاطر — الدفاع المتعمّق يقبل هذا ويُضيف طبقات ضوابط لتقليل التعرّض
