نظرة عامة — إكمال صورة الدفاع

يختتم الأسبوع 10 الجزء الثالث من المقرر — النصف الدفاعي من الكتاب. بعد أن تناولنا الدفاع ضد XSS وCSRF وXXE في الأسبوع 9، ننتقل الآن إلى فئات الهجوم الثلاث المتبقية: الحقن (Injection)، وحجب الخدمة (DoS)، واستغلال تبعيات الجهات الخارجية. يقدّم كل فصل تقنيات ملموسة وأنماط برمجية وقرارات معمارية تُقلّل من مخاطر هذه الفئات أو تُزيلها.

📈
ف.25 — الدفاع ضد الحقن
اكتشاف مخاطر حقن SQL، واستخدام العبارات المُجهَّزة، وتطبيق مبدأ الصلاحية الدنيا، وقائمة الأوامر البيضاء.
⚡
ف.26 — الدفاع ضد DoS
منع DoS عبر التعابير النمطية، وتصنيف مخاطر DoS المنطقي، والتخفيف من DDoS عبر خدمات النطاق الترددي والتوجيه للثقب الأسود.
📦
ف.27 — أمن الجهات الخارجية
تقييم أشجار التبعيات، وفحص قواعد بيانات CVE تلقائيًا، وعزل الحزم الخطرة، وتثبيت الإصدارات بالتغليف المُحكم.

📋 الدفاع المتعدد الطبقات عبر جميع فئات الهجوم

لا يكفي ضبط واحد وحده. التطبيقات الأكثر صمودًا تجمع بين ممارسات البرمجة الآمنة، والقيود المعمارية (كمبدأ الصلاحية الدنيا)، والفحص التلقائي، والسجلات المراقَبة — بحيث إن أخفقت طبقة واحدة، تبقى الطبقات الأخرى في مكانها.

🎯

الأهداف التعليمية

1
اكتشاف مخاطر حقن SQL في قاعدة الكود الحالية من خلال تحديد استيراد محوّلات قواعد البيانات، وبناء الاستعلامات الخام، وأنماط التسلسل النصي غير الآمن.
2
تطبيق العبارات المُجهَّزة لتجميع استعلامات SQL مسبقًا وشرح سبب عدم إمكانية تغيير نية الاستعلام بيانات المستخدم عبر التحديد المسبق للمعاملات.
3
تطبيق مبدأ الصلاحية الدنيا عبر وحدات التطبيق للحدّ من الضرر في حال استغلال ثغرة حقن، وإنشاء قائمة بيضاء بالأوامر التي يمكن للمستخدم الوصول إليها من جانب الخادم.
4
تصنيف مخاطر DoS باعتبارها طيفًا لا ثنائيًا، والتمييز بين Regex DoS والـ DoS المنطقي، وشرح سبب احتمال حجب تخفيفات DDoS لحركة المرور الشرعية.
5
تأمين تكاملات الجهات الخارجية من خلال تقييم أشجار التبعيات الكاملة، وتشغيل فحوصات CVE التلقائية، وعزل الحزم على خوادم منفصلة، وتثبيت الإصدارات بالتغليف المُحكم.
📈

ف.25 — اكتشاف حقن SQL في قاعدة الكود

قبل الدفاع ضد حقن SQL، يجب أولًا تحديد أماكن بناء استعلامات SQL في قاعدة الكود. في معظم تطبيقات الويب الحديثة، تجري عمليات SQL من جانب الخادم، بعد طبقة التوجيه — لذا يمكن تجاهل مجلد العميل عادةً. غير أن مسارات التحليل وخدمات الخلفية يجب تضمينها لأنها تتفاعل مع قواعد البيانات بشكل متكرر.

البحث عن استدعاءات SQL عبر محوّلات متعددة

كثيرًا ما تستخدم التطبيقات أكثر من محوّل SQL واحد. أسهل طريقة للبحث عن كل استدعاءات SQL في مشروع Node.js هي البحث عن عبارات استيراد المحوّل:

// ابحث عن هذه الاستيرادات لتحديد جميع عمليات SQL في قاعدة الكود const sql = require('mssql'); // SQL Server عبر محوّل NodeMSSQL const mysql = require('mysql'); // MySQL عبر محوّل mysql // يُنهي كلا المحوّلَين الاستعلامات عادةً بـ .query(x) // بعض المحوّلات تستخدم صياغة أكثر وضوحًا — ابحث عنها أيضًا: const getUserByUsername = function(username) { const q = new sql(); q.select('*'); q.from('users'); q.where(`username = ${username}`); // ✗ غير آمن — تضمين مباشر };

⚠️ محوّلات DSL قد تُخفي استدعاءات SQL

تستخدم بعض البرامج الخادمية لغة مخصصة (DSL) تُولّد SQL نيابةً عنك. قد لا تبدو هذه الاستدعاءات كـ SQL خام، لكنها لا تزال تُنفّذ استعلامات تحت الغطاء. تحقق دائمًا من وثائق ORM ومكتبة التحليل لديك لفهم طريقة تفاعلها مع قاعدة البيانات.

🔑 ابدأ ببحث شامل

أنشئ قائمة بكل محوّل SQL وORM وDSL مستخدم في تطبيقك — بما في ذلك تلك المستخدمة من قِبَل التبعيات الخارجية. أي موقع تلمس فيه بيانات المستخدم استعلامًا يُعدّ نقطة حقن محتملة.

🔒

العبارات المُجهَّزة — خط الدفاع الأول

العبارات المُجهَّزة هي الدفاع الأكثر شيوعًا وفاعلية ضد حقن SQL. تعمل عن طريق تجميع الاستعلام أولًا باستخدام متغيرات وهمية، ثم تمرير القيم الفعلية بشكل منفصل. لأن نية الاستعلام تُثبَّت قبل معالجة أي بيانات مستخدم، يصبح من المستحيل على المدخلات تغيير ما يفعله الاستعلام.

كيف تعمل العبارات المُجهَّزة

  1. 1
    تجميع الاستعلام بعناصر نائبة
    تستقبل قاعدة البيانات بنية الاستعلام وتُجمّعها — مثلًا: SELECT name FROM products WHERE price <= ?. يُعدّ ? متغيرًا وهميًا. في هذه المرحلة، تُثبَّت نية الاستعلام بشكل نهائي.
  2. 2
    تمرير قيم المتغيرات بشكل منفصل
    تُمرَّر بيانات المستخدم إلى الاستعلام المُجمَّع مسبقًا. تعامل قاعدة البيانات هذه البيانات كقيمة عادية — ليس كصياغة SQL. لا يمكن لأي إدخال مُصمَّم تحويل SELECT إلى DELETE.
  3. 3
    التنفيذ وإلغاء التخصيص
    يُنفَّذ الاستعلام بالقيم المرتبطة، ثم يُلغى تخصيص العبارة المُجمَّعة لتحرير مساحة ذاكرتها.

مثال على عبارة مُجهَّزة في MySQL

-- ✓ آمن: نية الاستعلام تُجمَّع قبل تطبيق بيانات المستخدم PREPARE q FROM 'SELECT name, barCode FROM products WHERE price <= ?'; SET @price = 12; -- قيمة يوفرها المستخدم EXECUTE q USING @price; -- تملأ القيمة العنصر النائب ? بعد التجميع DEALLOCATE PREPARE q; -- حتى لو كانت @price = "5; UPDATE users SET balance=10000" -- لن يُنفَّذ UPDATE — فهو لم يُجمَّع قط

❌ الاستعلام التقليدي (غير آمن)

تُرسَل بيانات المستخدم وسلسلة الاستعلام معًا كسلسلة واحدة. يمكن لمدخل مُصمَّم الهروب من الاستعلام وحقن أوامر SQL جديدة.

'SELECT name FROM products WHERE price <= ' + price + ';'

✅ العبارة المُجهَّزة (آمن)

يُجمَّع الاستعلام أولًا. تملأ بيانات المستخدم عنصرًا نائبًا بعد التجميع. لا يمكن لأي إدخال مُصمَّم تغيير بنية الاستعلام.

🔑 مدعوم في كل مكان

العبارات المُجهَّزة مدعومة في كل قاعدة بيانات SQL كبرى تقريبًا: MySQL وOracle وPostgreSQL وMicrosoft SQL Server وغيرها. التكلفة الوحيدة هي ذهاب وإياب إضافي إلى قاعدة البيانات للتجميع — وهو أمر لا يكاد يُذكر في معظم التطبيقات.

🗃

الدفاعات الخاصة بقواعد البيانات

إضافةً إلى العبارات المُجهَّزة، توفر قواعد بيانات SQL الكبرى دوالّ ترميز وإفلات مدمجة تُطهّر مجموعات الأحرف الخطرة. يجب استخدامها عندما يتعذّر تحديد معاملات الاستعلام بالكامل — لكن يجب التعامل معها كـ تخفيف، لا كدفاع شامل.

قاعدة البيانات الدالة / الأسلوب ما تُفلته
Oracle (Java) ESAPI.encoder().encodeForSQL(new OracleCodec(), str) يُرمّز الأحرف الخطرة في سياقات Oracle SQL
MySQL SELECT QUOTE('test''case') يُفلت الشرطات المائلة العكسية والاقتباسات المفردة وNULL — يعيد سلسلة مُقتبَسة بشكل صحيح
MySQL mysql_real_escape_string() يُفلت الشرطات المائلة العكسية والاقتباسات المفردة/المزدوجة و\n و\r

⚠️ ليست دفاعًا شاملًا

تُقلّل أدوات التطهير الخاصة بقاعدة البيانات من مخاطر الحقن لكن لا ينبغي أن تكون الدفاع الوحيد. لا تحمي من جميع أنواع الحقن، وتعتمد على تحديد المطوّر لكل مدخل خطر وإفلاته بشكل صحيح. افضّل دائمًا العبارات المُجهَّزة كضبط أساسي.

🔍

أهداف الحقن العامة

الحقن لا يقتصر على SQL. أي مكوّن يقبل مدخلات نصية ويمرّرها إلى مترجم أو مُجمِّع أو أداة سطر أوامر يُعدّ هدفًا محتملًا للحقن. عند مراجعة قاعدة الكود، انتبه بشكل خاص للأهداف عالية المخاطر التالية:

  • قواعد بيانات SQL — الهدف الأكثر شيوعًا؛ استخدم العبارات المُجهَّزة والمُرمِّزات الخاصة بقاعدة البيانات
  • جدولة المهام — الوظائف المجدولة التي تقبل معاملات ديناميكية يمكن التلاعب بها لتشغيل أوامر غير مقصودة
  • مكتبات الضغط / التحسين — أدوات CLI كـ FFMPEG التي تقبل مسارات الملفات أو خياراتها من مدخلات المستخدم
  • سكريبتات النسخ الاحتياطي عن بُعد — سكريبتات تستدعي نظام التشغيل المُضيف بحجج يتحكم فيها المستخدم
  • أنظمة السجلات — حقن السجلات يمكنه إتلاف ملفات السجل أو استغلال مُحلِّلات السجل المنبثقة
  • أي استدعاء لنظام التشغيل — أوامر الصدفة المُنشأة ببيانات يتحكم فيها المستخدم هي نواقل حقن أوامر
  • أي مترجم أو مُجمِّع — دوالّ نمط eval التي تُنفّذ سلاسل يوفرها المستخدم
  • التبعيات الخارجية — التبعيات الفرعية قد تتفاعل هي نفسها مع CLIs أو قواعد البيانات بطرق خطرة
📖 من الكتاب — الحقن خارج قواعد البيانات

قد يستخدم تطبيق لاستضافة الفيديو FFMPEG لضغط الفيديو، يُستدعى من نقطة نهاية Node.js API. إذا جاءت أسماء الملفات أو خيارات الضغط المُمرَّرة إلى FFMPEG مباشرةً من مدخلات المستخدم دون تطهير، يمكن للمهاجم حقن أوامر صدفة تُنفَّذ بصلاحيات عملية خادم الويب.

🔓

مبدأ الصلاحية الدنيا (PoLA)

ينصّ مبدأ الصلاحية الدنيا (المعروف أيضًا بمبدأ الامتياز الأدنى) على أن كل وحدة في النظام يجب أن تصل فقط إلى البيانات والوظائف اللازمة لتشغيل تلك الوحدة بشكل صحيح — لا أكثر. تطبيق هذا المبدأ لا يمنع الحقن، لكنه يُقلّص بشكل كبير الضرر الذي يمكن للمهاجم إحداثه إذا اكتشف ثغرة حقن واستغلّها.

📖 من الكتاب — مثال النسخ الاحتياطي عبر CLI

تخيّل أداة CLI تُنشئ نسخًا احتياطية تلقائية لصور ملفات تعريف المستخدمين. إذا كانت تعمل بـ صلاحيات المدير، فإن هجوم حقن ناجح ضدها قد يُفضح الخادم بأكمله. بدلًا من ذلك، إذا عملت الأداة فقط بأدنى صلاحيات قراءة الملفات اللازمة لمهمة النسخ الاحتياطي، فإن حجم الضرر عند الاختراق يكون أصغر بكثير.

تطبيق PoLA في الممارسة العملية

❌ بدون PoLA

مستخدم قاعدة بيانات واحد بصلاحيات مدير كاملة تستخدمه جميع الوحدات. الحقن في أي وحدة يمنح المهاجم تحكمًا كاملًا في قاعدة البيانات — قراءةً وكتابةً وحذفًا وتعديلًا للمخطط.

✅ مع PoLA

كل وحدة لها دور قاعدة بيانات خاص مقتصر على العمليات التي تحتاجها فعلًا. الحقن في وحدة البحث يُتيح فقط SELECT — لا UPDATE أو DROP.

🔑 PoLA يزداد أهميةً مع نموّ التطبيقات

كلما ازداد تعقيد التطبيق، تعددت تفاعلاته بين الوحدات، وزادت أهمية PoLA. قد تبدو إدارة حسابات ومفاتيح إضافية عبئًا على المطوّرين، لكنها ضرورية لاحتواء الاختراقات في الأنظمة الإنتاجية.

✅

القائمة البيضاء للأوامر

السماح للعميل بإرسال أوامر عشوائية لتنفيذها على الخادم هو نمط معماري عالي المخاطر ينبغي تجنّبه كليًا. عندما يكون تنفيذ الأوامر من جانب الخادم حتميًا، يجب تعريف قائمة بيضاء بالأوامر المسموح بها وتطبيقها — ولا يجوز استخدام قائمة سوداء قط.

❌ بلا قائمة بيضاء (مُعرَّض للخطر)

أي أمر تدعمه مكتبة CLI قابل للتنفيذ من قِبَل العميل. يستطيع المهاجم استدعاء أوامر لم تُعدّ للمستخدمين أبدًا، أو تسلسل أوامر لإنشاء سلوك غير مقصود.

✅ نهج القائمة البيضاء (آمن)

تُسمح فقط الأوامر الواردة في مصفوفة مُعرَّفة مسبقًا. يُرفض فورًا أي أمر غير موجود في القائمة بحالة 400 قبل التنفيذ.

const cli = require('../util/cli'); const commands = ['print', 'cut', 'copy', 'paste', 'refresh']; // ✓ آمن — تُمرَّر فقط الأوامر المُدرجة في القائمة البيضاء إلى CLI const postCommands = function(req, res) { const userCommands = req.body.commands; userCommands.forEach((c) => { if (!commands.includes(c)) { return res.sendStatus(400); } }); cli.run(req.body.commands); };

💡 القائمة البيضاء مقابل القائمة السوداء

نهج القائمة السوداء خطير لأن التطبيقات تتطوّر — أمر جديد يُضاف لاحقًا قد يُتيح للمستخدم وصولًا غير مقصود، ولن تحمي القائمة السوداء منه ما لم تُحدَّث يدويًا. القائمة البيضاء بالمقابل تُطبّق سياسة الرفض الافتراضي: الأوامر الجديدة تُحظر حتى تُوافَق عليها صراحةً.

🔍

ف.26 — الحماية من Regex DoS

Regex DoS هو أسهل أشكال DoS منعًا، لكنه يستلزم معرفة ما تبحث عنه. يستهدف الهجوم التعابير النمطية ذات التراجع الأسّي — الأنماط التي، عند تغذيتها بمدخل مُصمَّم بعناية، تجعل محرك التعابير النمطية يقضي وقتًا طويلًا جدًا في محاولة كل تركيبة مطابقة ممكنة.

ما الذي يجعل التعبير النمطي "شريرًا"

تتضمن أنماط التعابير النمطية الخطرة عادةً مجموعة متكررة مع مُحدِّد جشع:

// نمط يمكنه التسبب في تراجع أسّي: (a[ab]*)+ // + يعني: جد جميع المطابقات المحتملة قبل الإرجاع (جشع) // * يعني: طابق التعبير الفرعي أكبر عدد ممكن من المرات // معًا: يجرّب المحرك عددًا أسّيًا من المسارات

ثلاث خطوات لمنع Regex DoS

  1. 1
    استخدم أداة تحليل ثابت أو مُختبِر أداء للتعابير النمطية
    يمكن لأدوات المصدر المفتوح فحص تعابيرك النمطية بحثًا عن الأنماط "الشريرة" وتنبيهك إليها قبل وصولها إلى الإنتاج. الاختبار اليدوي للأداء بمدخلات معاكِسة فعّال أيضًا في رصد التعابير الإشكالية.
  2. 2
    لا تسمح أبدًا بتعابير نمطية يُوفّرها المستخدم في تطبيقك
    السماح للمستخدمين برفع أو إنشاء تعابير نمطية خطر بالغ. لا توجد طريقة موثوقة لتقييم التعابير النمطية العشوائية التي يوفّرها المستخدمون بأمان دون جهد كبير — تجنّب هذا النمط كليًا.
  3. 3
    راجع تكاملات الجهات الخارجية
    تأكد أن أي تطبيق تتكامل معه لا يستخدم تعابير نمطية يوفّرها المستخدم أو تعابير نمطية مكتوبة بشكل رديء داخليًا. Regex DoS في تبعية يُؤثّر على تطبيقك تمامًا مثل ذلك في كودك الخاص.
⚡

الحماية من DoS المنطقي

DoS المنطقي أصعب اكتشافًا ومنعًا من Regex DoS. يستهدف منطق التطبيق الذي يسمح عن غير قصد للمهاجم (أو حتى المستخدم الشرعي) باستهلاك قدر غير متناسب من موارد الخادم أو العميل — كنقطة نهاية API مُكلِفة حسابيًا يمكن استدعاؤها بشكل متكرر.

مخاطر DoS طيف لا ثنائية

على خلاف XSS — حيث تكون الثغرة إما موجودة أو غير موجودة — تقع مخاطر DoS على مقياس متدرّج:

خطر منخفض ("آمن")

استغلال هذا الكود يتطلب مهاجمًا بموارد ضخمة. حتى لو استُهدف، يكون تأثيره ضئيلًا. يُعدّ آمنًا للأغراض العملية.

خطر متوسط

يمكن إساءة استخدام المنطق في ظروف معينة — مثلًا، ملحوظ فقط على الأجهزة ضعيفة الطاقة، أو بجهد منسّق فقط. ينبغي إعطاؤه أولوية للمعالجة.

💡 أخطئ نحو الحذر

عند تصنيف مخاطر DoS، الأسلم معاملة الكود متوسط الخطر على أنه مُعرَّض للخطر. دالة بطيئة على الأجهزة المحمولة اليوم قد تصبح هدف DDoS غدًا إذا نما التطبيق في الحجم أو الشهرة.

تحديد المجالات عالية المخاطر

للحماية من DoS المنطقي، حدّد الأجزاء من قاعدة الكود التي تتفاعل مع موارد النظام الحيوية:

  • العمليات الثقيلة على المعالج التي تُشغَّل بواسطة نقاط نهاية API (الحسابات المُكثِّفة، معالجة الصور)
  • استعلامات قاعدة البيانات بلا حدود للنتائج — مثلًا، استعلام يمكنه إرجاع ملايين الصفوف
  • الدوالّ العودية بلا حدود صارمة للعمق
  • الوظائف غير المتزامنة في الخلفية (النسخ الاحتياطية، تحويل الملفات) القابلة لتكرار التشغيل عبر API
  • حلقات العرض من جانب العميل التي تنمو مع حجم مجموعة بيانات يوفّرها المستخدم

هجمات DDoS والشبكات الروبوتية

هجمات الحرمان الموزّع من الخدمة (DDoS) أصعب بكثير في الدفاع مقارنةً بهجمات DoS من مهاجم واحد، لأنها تنشأ من مصادر متعددة في وقت واحد، يتحكم فيها مهاجم مركزي. بدلًا من استهداف ثغرة في الكود، تُغرق معظم هجمات DDoS الهدف عبر حجم ضخم من الحركة التي تبدو شرعية.

📖 من الكتاب — كيف تعمل الشبكات الروبوتية

يُوزّع مهاجم DDoS برمجيات خبيثة عبر قنوات شرعية (مُرفقة ببرامج، روابط تصيّد، إلخ). تعمل هذه البرمجيات الخبيثة بصمت على الأجهزة المصابة — أجهزة كمبيوتر شخصية وهواتف محمولة وأجهزة IoT كأجهزة توجيه وأجهزة ذكية — وتفتح بابًا خلفيًا. يستطيع المهاجم بعد ذلك أن يأمر آلاف هذه الأجهزة "الزومبي" بإغراق خادم الهدف في وقت واحد. تُسمى هذه الشبكة من الأجهزة المخترقة بوتنت (robot + network).

⚠️ لا يمكن منع DDoS كليًا

لأن حركة DDoS تُحاكي الحركة الشرعية بأساليب عديدة، لا يوجد فلتر مثالي. الهدف هو التخفيف — تقليل الأثر على المستخدمين الشرعيين مع حجب أكبر قدر ممكن من الحركة الخبيثة. أي تخفيف مُهاجِم يحجب ما يكفي من الحركة الخبيثة سيُخاطر أيضًا بحجب بعض المستخدمين الشرعيين.

🛡

تقنيات التخفيف من DDoS

رغم عدم إمكانية منع DDoS كليًا، تُقلّل عدة تقنيات تأثيره بشكل ملحوظ:

التقنية كيف تعمل المقايضة
خدمات إدارة النطاق الترددي خدمات خارجية تعترض جميع الحركة قبل وصولها إلى خادمك. يُحلَّل كل حزمة في الوقت الفعلي وتُحجب إذا طابقت أنماطًا خبيثة. فعّالة جدًا للهجمات عالية الحجم. التكلفة؛ قد تحجب المستخدمين الشرعيين بسبب إيجابيات كاذبة
التوجيه للثقب الأسود تُعاد توجيه الحركة المشبوهة أو المتكررة إلى خادم وهمي ("ثقب أسود") يبدو نشطًا لكنه لا ينجز عمليات حقيقية. تستمر الحركة الشرعية إلى الخادم الفعلي. قد تُوجَّه الحركة الشرعية إلى الثقب الأسود إذا كانت دقة الاستهداف غير كافية
تحديد معدل الطلبات تحديد عدد الطلبات التي يمكن لعنوان IP واحد أو مستخدم تقديمها خلال نافذة زمنية. سهل التطبيق وفعّال ضد الهجمات منخفضة التعقيد. عناوين IP المشتركة (NAT، البروكسيات) قد تتسبب في حجب مستخدمين أبرياء
السجلات الشاملة تسجيل جميع الطلبات مع الطوابع الزمنية وأوقات الاستجابة. تسجيل الوظائف غير المتزامنة في الخلفية يدويًا. يُتيح الكشف عن أنماط DoS التي كانت ستمرّ دون أن تُلاحَظ. يتطلب بنية تحتية للتخزين والمراقبة

🔑 اعرف خط الأساس قبل نشر التخفيفات

فلاتر DDoS شديدة العدوانية ستحجب المستخدمين الشرعيين. قبل تطبيق إجراءات التخفيف، اجمع مقاييس تفصيلية عن سلوك المستخدم الطبيعي — معدلات الطلبات، والأنماط الجغرافية، وأنواع الأجهزة. يُتيح هذا الخط الأساسي ضبط الفلاتر لتقليل الإيجابيات الكاذبة مع زيادة الحماية.

ف.27 — تقييم أشجار التبعيات

نادرًا ما تستخدم تطبيقات الويب الحديثة تبعياتها الخارجية المباشرة فحسب. لكل تبعية في الغالب تبعياتها الخاصة — تُسمى تبعيات الجهة الرابعة — والتي قد تملك هي الأخرى تبعياتها، وهكذا. تُسمى السلسلة الكاملة شجرة التبعيات.

📖 من الكتاب — مشكلة التبعيات في العالم الحقيقي

تخيّل تطبيقًا يستخدم jQuery مباشرةً، لكنه يستخدم أيضًا إطار SPA ومكتبة مكوّنات UI — وكلاهما يتطلب jQuery أيضًا، لكن بإصدار أقدم. في تطبيق كبير بـ 100 تبعية خارجية، يمكن أن يُنتج هذا النمط شجرة تبعيات تضم آلاف أو عشرات آلاف الحزم والإصدارات الفريدة.

# استخدم npm ls لعرض شجرة التبعيات الكاملة npm ls # مثال على الإخراج — كل عنصر بادئ هو تبعية فرعية: @fortawesome/ember-fontawesome@0.1.14 @fortawesome/fontawesome-svg-core@1.2.19 @fortawesome/fontawesome-common-types@0.2.19 deduped broccoli-file-creator@1.2.0 broccoli-plugin@1.3.1 deduped mkdirp@0.5.1 deduped broccoli-merge-trees@2.0.1 merge-trees@1.0.1

لماذا تهمّ فوارق الإصدارات

الخطر في العالم الحقيقي

تطبيقك يستخدم jQuery 3.4.0 (آمن)، لكن إطار SPA الخاص بك يستخدم jQuery 2.2.1 (مُعرَّض للثغرات). حتى لو كان الاستيراد من المستوى الأعلى آمنًا، إصدار التبعية الفرعية لا يزال مُحمَّلًا وقابلًا للاستغلال.

الهدف

كل تبعية فريدة بكل إصدار فريد يجب تقييمها بشكل مستقل. الثغرة في إصدار تبعية فرعية خطيرة بالقدر ذاته كثغرة في تبعية مباشرة.

التقييم التلقائي للتبعيات وفحص CVE

مراجعة آلاف التبعيات يدويًا أمر غير عملي. الأدوات التلقائية ضرورة لا غنى عنها. النهج الأكثر فاعلية هو نمذجة شجرة التبعيات كبنية بيانات ثم مقارنتها بقاعدة بيانات CVE (الثغرات والتعرضات الشائعة) المعروفة.

الأدوات والأوامر

# عرض شجرة التبعيات الكاملة حتى عمق محدد npm list --depth=5 # تشغيل تدقيق تلقائي يقارن شجرتك بـ CVEs المعروفة npm audit # Snyk — ماسح ثغرات خارجي شهير snyk test
المورد ما يوفّره
قاعدة البيانات الوطنية للثغرات (NIST) قاعدة بيانات CVE ممولة حكوميًا وطويلة الأمد. موثوقة لمقارنة إصدارات التبعيات بالثغرات المعروفة.
Snyk ماسح خارجي يتكامل مع npm وMaven وpip ومديري الحزم الآخرين. يوفر توجيه المعالجة إلى جانب تقارير الثغرات.
npm audit أمر npm مدمج يفحص حزمك المثبتة في مقابل قاعدة بيانات تنبيهات npm ويُبلّغ عن الثغرات المعروفة.

🔑 أتمتة الفحص عند كل إضافة تبعية

لا ينبغي أن يكون فحص CVE نشاطًا يُجرى مرة واحدة. في كل مرة تُضاف أو تُحدَّث تبعية، يجب إعادة فحص الشجرة بالكامل. ادمج فحص CVE في خط أنابيب CI/CD حتى تجري فحوصات الثغرات تلقائيًا قبل نشر أي كود.

التكامل الآمن — فصل الاهتمامات

إحدى أكثر التقنيات المعمارية فاعلية للحدّ من مخاطر تبعية خارجية مخترقة هي تشغيلها على خادم معزول خاص بها. يتواصل تطبيقك الرئيسي معها عبر HTTP، مُرسِلًا ومستقبِلًا حمولات JSON.

  1. 1
    نشر التبعية الخطرة على خادمها الخاص
    تعمل التبعية في بيئة لا تملك وصولًا إلى قاعدة بيانات تطبيقك أو نظام ملفاته أو شبكته الداخلية. إذا اختُرقت الحزمة، يكون المهاجم محصورًا في ذلك الخادم.
  2. 2
    التواصل عبر HTTP + JSON فقط
    تضمن حمولات JSON عدم إمكانية تنفيذ سكريبتات على خادم التطبيق الرئيسي من خلال قناة الاتصال وحدها. تتصرف التبعية كدالة عديمة الحالة — مدخل يدخل، ومخرج يخرج.
  3. 3
    تهيئة قواعد جدار الحماية لخادم التبعية
    قيّد ما يمكن لخادم التبعية الاتصال به. بدون جدار حماية مُهيَّأ بشكل صحيح، يمكن لتبعية مخترقة الوصول إلى قاعدة بياناتك أو خدماتك الداخلية، مما يُبطل العزل.

⚠️ للعزل تكاليفه

تشغيل التبعيات على خوادم منفصلة يُدخل زمن تأخير (ذهاب وإياب HTTP إضافي) وتعقيدًا تشغيليًا (المزيد من الخوادم لإدارتها). علاوةً على ذلك، أي بيانات سرية تُرسَل إلى خادم التبعية يمكن اعتراضها أو تسجيلها إذا اختُرقت الحزمة. العزل يُقلّل المخاطر — لا يُزيلها.

🔒

إدارة الحزم الآمنة — تثبيت الإصدار والتغليف المُحكم

عندما تُراجع تبعية وتتحقق من أنها آمنة بإصدار محدد، تريد منع تغيير ذلك الإصدار دون علمك. يوفر npm ومديرو الحزم المماثلون أدوات لهذا الغرض.

الإصدار الدلالي وخطر علامة الإقحام

يستخدم npm الإصدار الدلالي بثلاثة أرقام: major.minor.patch. افتراضيًا، يُضيف npm علامة إقحام (^) قبل كل إصدار تبعية، مما يُوجّه npm لتثبيت دائم لأحدث إصلاح:

// package.json — السلوك الافتراضي مع علامة الإقحام "dependencies": { "myLib": "^1.0.23" // قد يُرقَّى تلقائيًا إلى 1.0.24 أو 1.0.25 إلخ. } // أزِل علامة الإقحام لتثبيت الإصدار المُراجَع تمامًا: "dependencies": { "myLib": "1.0.23" // يستخدم دائمًا 1.0.23 تحديدًا }

التغليف المُحكم — تثبيت الشجرة بأكملها

تثبيت الإصدار على المستوى الأعلى يتحكم فقط في تبعياتك المباشرة. التبعيات الفرعية قد تتغير. التغليف المُحكم يُثبّت شجرة التبعيات بأكملها:

# أنشئ ملف تغليف مُحكم يُثبّت جميع الإصدارات في الشجرة npm shrinkwrap # ينشئ هذا الأمر npm-shrinkwrap.json # من هذه النقطة، كل تبعية وتبعية فرعية # مُثبَّتة على الإصدار الدقيق الذي تمت مراجعته
التقنية ما تُثبّته القيد
إزالة علامة الإقحام (^) التبعيات المباشرة من المستوى الأعلى فقط لا تُثبّت إصدارات التبعيات الفرعية
npm shrinkwrap شجرة التبعيات بأكملها (جميع المستويات) قد يُعيد الصانع النشر باستخدام رقم إصدار موجود — لا يستطيع shrinkwrap اكتشاف ذلك
مراجع Git SHA عمليات إيداع محددة في مستودع Git أكثر تعقيدًا في الإدارة؛ يتطلب الوصول إلى مستودعات المصدر
مرآة npm خاصة نسختك الخاصة من كل إصدار حزمة مُراجَع أعلى أمانًا، لكن أعلى تكلفةً تشغيليةً

🔑 الدفاع المتعمق للتبعيات

النهج الأكثر أمانًا يجمع عدة تقنيات: غلّف شجرة التبعيات بـ shrinkwrap، واستخدم مراجع Git SHA للحزم الأكثر أهمية، وفكّر في تشغيل مرآة npm خاصة للبيئات عالية الأمان. لا تُزيل أي تقنية وحدها خطر تبعية مخترقة — يجب تطبيقها بشكل متراكب.

🛡 ملخص الفصول 25–27 — الدفاع ضد الحقن وDoS والتبعيات الخارجية

  • اكتشف مخاطر حقن SQL بإيجاد جميع استيرادات محوّلات قواعد البيانات وبناء الاستعلامات الخام في قاعدة الكود — بما فيها تلك الموجودة في مسارات التحليل وخدمات الخلفية
  • العبارات المُجهَّزة تُجمِّع الاستعلامات قبل تطبيق بيانات المستخدم — لا يمكن لأي مدخل مستخدم تغيير نية الاستعلام، مما يُزيل معظم مخاطر حقن SQL
  • المُرمِّزات الخاصة بقواعد البيانات (Oracle ESAPI وMySQL QUOTE و mysql_real_escape_string) تُقلّل من مخاطر الحقن لكنها تخفيفات لا دفاعًا شاملًا
  • أهداف الحقن تتجاوز SQL — جدولة المهام وCLIs والسجلات واستدعاءات نظام التشغيل وأي مترجم كلها أسطح حقن محتملة
  • مبدأ الصلاحية الدنيا يُحدّد ضرر الاختراق بضمان وصول كل وحدة فقط إلى البيانات والوظائف التي تحتاجها فعلًا
  • عندما يجب ربط مدخلات المستخدم بأوامر جانب الخادم، طبّق قائمة بيضاء — لا قائمة سوداء أبدًا — بالعمليات المسموح بها
  • يُمنع Regex DoS بفحص التعابير النمطية "الشريرة" (المجموعات المتكررة مع المُحدِّدات الجشعة) وعدم السماح أبدًا بتعابير نمطية يوفّرها المستخدم
  • مخاطر DoS طيف — صنّف الكود كخطر مرتفع/متوسط/منخفض بدلًا من ثنائية آمن/مُعرَّض؛ أخطئ نحو الحذر
  • لا يمكن منع DDoS كليًا — خفّفه بخدمات إدارة النطاق الترددي والتوجيه للثقب الأسود وتحديد معدل الطلبات، مستخدمًا خط أساس حركة المرور الشرعية لضبط الفلاتر
  • أشجار التبعيات قد تضم آلاف الحزم — الفحص التلقائي لـ CVE (npm audit وSnyk وNIST NVD) ضروري ويجب تشغيله عند كل تغيير في التبعيات
  • عزل الحزم الخارجية الخطرة على خادمها الخاص يتواصل عبر HTTP + JSON لاحتواء أثر تبعية مخترقة
  • ثبّت شجرة تبعياتك بأكملها مع npm shrinkwrap؛ استخدم مراجع Git SHA أو مرآة npm خاصة للبيئات عالية الأمان
