📋 Layered Defense Across All Attack Classes

Ch.25 — Detecting SQL Injection in Your Codebase

Before defending against SQL injection, you must first locate where SQL queries are constructed in your codebase. In most modern web applications, SQL operations occur server-side, past the routing layer — so the client folder can usually be skipped. However, analytics routes and background services must be included because they frequently interact with databases.

Finding SQL Calls Across Multiple Adapters

Applications often use more than one SQL adapter. The easiest way to find all SQL calls in a Node.js project is to search for the adapter import statements:

// Search for these imports to locate all SQL operations in the codebase const sql = require('mssql'); // SQL Server via NodeMSSQL adapter const mysql = require('mysql'); // MySQL via mysql adapter // Both adapters typically conclude queries with a .query(x) call // Some adapters use a more literal syntax — look for these too: const getUserByUsername = function(username) { const q = new sql(); q.select('*'); q.from('users'); q.where(`username = ${username}`); // ✗ UNSAFE — direct interpolation };

⚠️ DSL Adapters Can Hide SQL Calls

Some server software uses a Domain-Specific Language (DSL) that generates SQL on your behalf. These calls may not look like raw SQL, but they still execute queries under the hood. Always check your ORM and analytics library documentation to understand how they interact with the database.

🔑 Start With a Comprehensive Search

Compile a list of every SQL adapter, ORM, and DSL used in your application — including those used by third-party dependencies. Any location where user-supplied data touches a query is a potential injection point.

🔒

Prepared Statements — The First Line of Defense

Prepared statements are the most widely recommended and effective defense against SQL injection. They work by compiling the query first with placeholder variables, then supplying the actual values separately. Because the query's intention is fixed before any user data is considered, it is impossible for user input to alter what the query does.

How Prepared Statements Work

  1. 1
    Compile the query with placeholders
    The database receives and compiles the query structure — e.g., SELECT name FROM products WHERE price <= ?. The ? is a bind variable (placeholder). At this point, the query's intention is locked in stone.
  2. 2
    Supply the variable values separately
    User-supplied data is passed to the already-compiled query. The database treats this data as a plain value — never as SQL syntax. A SELECT cannot become a DELETE no matter what the user submits.
  3. 3
    Execute and deallocate
    The query executes with the bound values, then the compiled statement is deallocated to free its namespace in memory.

MySQL Prepared Statement Example

-- ✓ SAFE: Query intention is compiled BEFORE user data is applied PREPARE q FROM 'SELECT name, barCode FROM products WHERE price <= ?'; SET @price = 12; -- user-supplied value EXECUTE q USING @price; -- value fills the ? placeholder after compilation DEALLOCATE PREPARE q; -- Even if @price = "5; UPDATE users SET balance=10000" -- the extra UPDATE never fires — it was never compiled

❌ Traditional Query (Unsafe)

User data and query string are sent together as a single string. A crafted input can escape the query and inject new SQL commands.

'SELECT name FROM products WHERE price <= ' + price + ';'

✅ Prepared Statement (Safe)

Query is compiled first. User data fills a placeholder after compilation. No amount of crafted input can change the query's structure.

🔑 Supported Everywhere

Prepared statements are supported by virtually every major SQL database: MySQL, Oracle, PostgreSQL, Microsoft SQL Server, and more. The only trade-off is a minor performance cost — one extra round-trip to the database for compilation — which is negligible in most applications.

🗃

Database-Specific Defenses

In addition to prepared statements, major SQL databases provide their own built-in encoding and escaping functions that sanitize risky character sets. These should be used whenever a query cannot be fully parameterized — but they should be treated as a mitigation, not a complete defense.

Database Function / Method What It Escapes
Oracle (Java) ESAPI.encoder().encodeForSQL(new OracleCodec(), str) Encodes characters that would be dangerous in Oracle SQL contexts
MySQL SELECT QUOTE('test''case') Escapes backslashes, single quotes, and NULL — returns a properly single-quoted string
MySQL mysql_real_escape_string() Escapes backslashes, single/double quotes, \n, and \r

⚠️ Not a Comprehensive Defense

Database-specific sanitizers reduce injection risk but should never be the only defense. They do not protect against all injection variants, and they depend on the developer correctly identifying and escaping every risky input. Always prefer prepared statements as the primary control.

🔍

Generic Injection Targets

Injection is not limited to SQL. Any component that accepts text input and passes it to an interpreter, compiler, or command-line utility is a potential injection target. When auditing your codebase, pay close attention to the following high-risk targets:

  • SQL databases — the most common target; use prepared statements and database-specific encoders
  • Task schedulers — scheduled jobs that accept dynamic parameters can be manipulated to run unintended commands
  • Compression / optimization libraries — CLI tools like FFMPEG that accept file paths or options from user input
  • Remote backup scripts — scripts that call the host OS with user-controlled arguments
  • Loggers — log injection can corrupt log files or exploit downstream log parsers
  • Any call to the host OS — shell commands constructed with user data are command injection vectors
  • Any interpreter or compiler — eval-style functions that execute user-supplied strings
  • Third-party dependencies — subdependencies may themselves interact with CLIs or databases in risky ways
📖 From the Book — Injection Beyond Databases

A video-hosting application might use FFMPEG for video compression, called from a Node.js API endpoint. If the filename or compression options passed to FFMPEG come directly from user input without sanitization, an attacker can inject shell commands that execute with the permissions of the web server process.

🔓

Principle of Least Authority (PoLA)

The Principle of Least Authority (also called Principle of Least Privilege) states that every module in a system should only have access to the data and functionality required for that module to operate correctly — nothing more. Applying this principle does not prevent injection, but it dramatically limits the damage an attacker can cause if an injection vulnerability is found and exploited.

📖 From the Book — CLI Backup Example

Imagine a CLI tool that automatically backs up user profile photos. If this CLI runs with admin privileges, a successful injection attack against it could expose the entire application server. If instead the CLI runs with only the minimal file-read permissions it needs for its backup task, the blast radius of a breach is dramatically smaller.

Applying PoLA in Practice

❌ Without PoLA

A single database user with full admin access is used by all modules. An injection in any module gives the attacker full database control — read, write, delete, and schema modification.

✅ With PoLA

Each module has its own database role scoped to exactly the operations it needs. An injection in the search module can only perform SELECT — not UPDATE or DROP.

🔑 PoLA Becomes More Important as Apps Scale

The more complex an application becomes, the more inter-module interactions it has, and the more important PoLA is. Managing additional accounts and keys may seem like overhead for developers, but it is essential for containing breaches in production systems.

✅

Whitelisting Commands

Allowing a client to send arbitrary commands to be executed on the server is a high-risk architectural pattern that should be avoided entirely. When server-side command execution is unavoidable, a whitelist of permitted commands must be defined and enforced — never a blacklist.

❌ No Whitelist (Vulnerable)

Any command supported by the CLI library is executable by the client. An attacker can invoke commands never intended for end-user use, or chain commands to create unintended behavior.

✅ Whitelist Approach (Safe)

Only commands that appear in a pre-defined whitelist array are permitted. Any command not on the list is immediately rejected with a 400 status before execution.

const cli = require('../util/cli'); const commands = ['print', 'cut', 'copy', 'paste', 'refresh']; // ✓ SAFE — only whitelisted commands are forwarded to the CLI const postCommands = function(req, res) { const userCommands = req.body.commands; userCommands.forEach((c) => { if (!commands.includes(c)) { return res.sendStatus(400); } }); cli.run(req.body.commands); };

💡 Whitelist vs. Blacklist

A blacklist approach is dangerous because applications evolve — a new command added later may provide the user with unintended access, and the blacklist will not protect against it unless manually updated. A whitelist, by contrast, defaults to deny: new commands are blocked until explicitly approved.

Ch.26 — Protecting Against Regex DoS

Regex DoS is the easiest form of DoS to prevent, but it requires knowing what to look for. The attack targets regular expressions with exponential backtracking — patterns that, when fed a carefully crafted input, cause the regex engine to spend an extremely long time trying every possible match combination.

What Makes a Regex "Evil"

Dangerous regex patterns typically involve a repeated group with a greedy quantifier:

// Pattern that can cause exponential backtracking: (a[ab]*)+ // The + means: find all potential matches before returning (greedy) // The * means: match the subexpression as many times as possible // Combined: the engine tries an exponential number of paths

Three Steps to Prevent Regex DoS

  1. 1
    Use a static analysis tool or regex performance tester
    OSS tools can scan your regular expressions for "evil" patterns and flag them before they reach production. Manual performance testing with adversarial inputs is also effective for catching problematic regex.
  2. 2
    Never allow user-supplied regex in your application
    Allowing users to upload or compose regular expressions is an extreme risk. There is no reliable way to safely evaluate arbitrary user-provided regex without significant effort — avoid this pattern entirely.
  3. 3
    Audit third-party integrations
    Ensure that no application you integrate with uses user-supplied regex or poorly written regular expressions internally. A regex DoS in a dependency affects your application just as much as one in your own code.
⚡

Protecting Against Logical DoS

Logical DoS is harder to detect and prevent than Regex DoS. It targets application logic that accidentally allows an attacker (or even a legitimate user) to consume a disproportionate amount of server or client resources — such as a computationally expensive API endpoint that can be called repeatedly.

DoS Risk Is a Spectrum, Not Binary

Unlike XSS — where a vulnerability either exists or it doesn't — DoS risk exists on a scale:

Low Risk ("Safe")

Exploiting this code would require an attacker with massive resources. Even if targeted, the impact is minimal. Considered safe for practical purposes.

Medium Risk

The logic can be abused under certain conditions — e.g., only noticeable on low-powered devices, or only with a coordinated effort. Should be prioritized for remediation.

💡 Err on the Side of Caution

When classifying DoS risk, it is safer to treat medium-risk code as vulnerable. A function that is slow on mobile devices today may become a DDoS target tomorrow if the application grows in scale or visibility.

Identifying High-Risk Areas

To protect against logical DoS, identify the parts of your codebase that interact with critical system resources:

  • CPU-intensive operations triggered by API endpoints (heavy computation, image processing)
  • Database queries without result limits — e.g., a query that can return millions of rows
  • Recursive functions without strict depth limits
  • Async background jobs (backups, file conversion) that can be triggered repeatedly via API
  • Client-side rendering loops that grow with the size of a user-supplied dataset
🌐

DDoS Attacks & Botnets

Distributed Denial of Service (DDoS) attacks are much harder to defend against than single-attacker DoS because they originate from many sources simultaneously, controlled by a centralized attacker. Rather than targeting a code bug, most DDoS attacks overwhelm the target through sheer volume of legitimate-looking traffic.

📖 From the Book — How Botnets Work

A DDoS attacker distributes malware through legitimate channels (bundled with software, phishing links, etc.). This malware runs silently on infected devices — PCs, mobile phones, IoT devices like routers and smart appliances — and opens a backdoor. The attacker can then remotely command thousands of these "zombie" devices to simultaneously flood a target server. This network of compromised devices is called a botnet (robot + network).

⚠️ DDoS Cannot Be Fully Prevented

Because DDoS traffic mimics legitimate traffic in many ways, there is no perfect filter. The goal is mitigation — reducing the impact on legitimate users while blocking as much malicious traffic as possible. Any aggressive mitigation that blocks enough malicious traffic will also risk blocking some legitimate users.

🛡

DDoS Mitigation Techniques

While DDoS cannot be fully prevented, several techniques significantly reduce its impact:

Technique How It Works Trade-Off
Bandwidth Management Services Third-party services intercept all traffic before it reaches your server. Each packet is analyzed in real time and blocked if it matches malicious patterns. Highly effective for high-volume attacks. Cost; may have false positives blocking legitimate users
Blackholing Suspicious or repeated traffic is redirected to a decoy ("blackhole") server that appears functional but performs no real operations. Legitimate traffic continues to the real server. Can accidentally route legitimate traffic to the blackhole if targeting accuracy is insufficient
Rate Limiting Limit the number of requests a single IP address or user can make per time window. Simple to implement and effective against low-sophistication attacks. Shared IPs (NAT, proxies) may cause innocent users to be blocked
Comprehensive Logging Log all requests with timestamps and response times. Log async background jobs manually. Enables detection of DoS patterns that would otherwise go unnoticed. Requires storage and monitoring infrastructure

🔑 Know Your Baseline Before Deploying Mitigations

Overly aggressive DDoS filters will block legitimate users. Before implementing mitigation measures, collect detailed metrics on normal user behavior — request rates, geographic patterns, device types. This baseline allows you to tune filters to minimize false positives while maximizing protection.

Ch.27 — Evaluating Dependency Trees

Modern web applications rarely use only their direct third-party dependencies. Each dependency often has its own dependencies — called fourth-party dependencies — which may have their own, and so on. The full chain is called a dependency tree.

📖 From the Book — The Real-World Dependency Problem

Consider an application that uses jQuery directly, but also uses an SPA Framework and a UI Component Library — both of which also require jQuery, but at an older version. In a large app with 100 third-party dependencies, this pattern can result in a dependency tree spanning thousands or tens of thousands of unique packages and versions.

# Use npm ls to view your full dependency tree npm ls # Example output — every indented item is a subdependency: @fortawesome/ember-fontawesome@0.1.14 @fortawesome/fontawesome-svg-core@1.2.19 @fortawesome/fontawesome-common-types@0.2.19 deduped broccoli-file-creator@1.2.0 broccoli-plugin@1.3.1 deduped mkdirp@0.5.1 deduped broccoli-merge-trees@2.0.1 merge-trees@1.0.1

Why Version Differences Matter

Real-World Risk

Your app uses jQuery 3.4.0 (safe), but your SPA Framework uses jQuery 2.2.1 (vulnerable). Even though your top-level import is safe, the subdependency version is still loaded and exploitable.

The Goal

Each unique dependency at each unique version must be independently evaluated. A vulnerability in a subdependency version is just as dangerous as one in a direct dependency.

📊

Automated Dependency Evaluation & CVE Scanning

Manually reviewing thousands of dependencies is impractical. Automated tooling is essential. The most effective approach is to model the dependency tree as a data structure and then compare it against a known CVE (Common Vulnerabilities and Exposures) database.

Tools and Commands

# List the full dependency tree to a specified depth npm list --depth=5 # Run an automated audit comparing your tree against known CVEs npm audit # Snyk — a popular third-party vulnerability scanner snyk test
Resource What It Provides
NIST National Vulnerability Database Government-funded, long-lived CVE database. Reliable for comparing dependency versions against known vulnerabilities.
Snyk Third-party scanner that integrates with npm, Maven, pip, and other package managers. Provides remediation guidance alongside vulnerability reports.
npm audit Built-in npm command that checks your installed packages against the npm advisory database and reports known vulnerabilities.

🔑 Automate on Every Dependency Addition

CVE scanning should not be a one-time activity. Every time a new dependency is added or updated, the full tree should be re-scanned. Integrate CVE scanning into your CI/CD pipeline so that vulnerability checks happen automatically before any code is deployed.

🛠

Secure Integration — Separation of Concerns

One of the most effective architectural techniques for reducing the risk of a compromised third-party dependency is to run it on its own isolated server. Your main application communicates with it via HTTP, sending and receiving JSON payloads.

  1. 1
    Deploy the risky dependency on its own server
    The dependency runs in an environment with no access to your application's database, file system, or internal network. If the package is compromised, the attacker is isolated to that server.
  2. 2
    Communicate via HTTP + JSON only
    JSON payloads ensure that script execution on the main application server is not possible through the communication channel alone. The dependency behaves like a stateless function — input in, output out.
  3. 3
    Configure firewall rules for the dependency server
    Restrict what the dependency server can connect to. Without a properly configured firewall, a compromised dependency could still reach your database or internal services, defeating the isolation.

⚠️ Isolation Has Costs

Running dependencies on separate servers introduces latency (extra HTTP round-trips) and operational complexity (more servers to maintain). Additionally, any confidential data sent to the dependency server could still be intercepted or logged if the package is compromised. Isolation reduces risk — it does not eliminate it.

🔒

Secure Package Management — Version Locking & Shrinkwrapping

When you audit a dependency and determine it is safe at a specific version, you want to prevent that version from changing without your knowledge. npm and similar package managers offer tools for this.

Semantic Versioning & the Caret Risk

npm uses semantic versioning with three numbers: major.minor.patch. By default, npm adds a caret (^) before each dependency version, which instructs npm to always install the latest patch:

// package.json — default behavior with caret "dependencies": { "myLib": "^1.0.23" // may auto-upgrade to 1.0.24, 1.0.25, etc. } // Remove the caret to pin to the exact audited version: "dependencies": { "myLib": "1.0.23" // always uses exactly 1.0.23 }

Shrinkwrapping — Locking the Entire Tree

Pinning the top-level version only controls your direct dependencies. Subdependencies can still change. Shrinkwrapping locks the entire dependency tree:

# Generate a shrinkwrap file that locks ALL versions in the tree npm shrinkwrap # This creates npm-shrinkwrap.json # From this point, every dependency AND subdependency # is pinned to the exact version that was audited
Technique What It Locks Limitation
Remove caret (^) Top-level direct dependencies only Does not lock subdependency versions
npm shrinkwrap Entire dependency tree (all levels) A maintainer could redeploy using an existing version number — shrinkwrap cannot detect this
Git SHA references Specific commits in a Git repository More complex to manage; requires access to source repositories
Private npm mirror Your own copy of each audited package version Highest security, but highest operational overhead

🔑 Defense in Depth for Dependencies

The most secure approach combines multiple techniques: shrinkwrap the dependency tree, reference Git SHAs for the most critical packages, and consider running a private npm mirror for high-security environments. No single technique fully eliminates the risk of a compromised dependency — they must be layered.

🛡 Chapters 25–27 Summary — Injection, DoS & Third-Party Defense

  • Detect SQL injection risk by finding all database adapter imports and raw query constructions in the codebase — including those in analytics routes and background services
  • Prepared statements pre-compile queries before user data is applied — the query's intention cannot be changed by any user input, eliminating most SQL injection risk
  • Database-specific encoders (Oracle ESAPI, MySQL QUOTE / mysql_real_escape_string) reduce injection risk but are mitigations, not a comprehensive defense
  • Injection targets extend beyond SQL — task schedulers, CLIs, loggers, OS calls, and any interpreter are all potential injection surfaces
  • The Principle of Least Authority limits breach damage by ensuring each module only has access to the data and functionality it strictly needs
  • When user input must map to server-side commands, enforce a whitelist — never a blacklist — of permitted operations
  • Regex DoS is prevented by scanning for evil regex patterns (repeated groups with greedy quantifiers) and never allowing user-supplied regex
  • DoS risk exists on a spectrum — classify code as high/medium/low risk rather than binary safe/vulnerable; err on the side of caution
  • DDoS cannot be fully prevented — mitigate with bandwidth management services, blackholing, and rate limiting, using your legitimate traffic baseline to tune filters
  • Dependency trees can span thousands of packages — automated CVE scanning (npm audit, Snyk, NIST NVD) is essential and should run on every dependency change
  • Isolate risky third-party packages on their own server communicating via HTTP + JSON to contain the blast radius of a compromised dependency
  • Lock your entire dependency tree with npm shrinkwrap; use Git SHA references or a private npm mirror for the highest-security environments
