🔍 What is a Vulnerability Assessment?

In Labs 01–07 you learned individual attack techniques: SQL injection, XSS, CSRF, command injection, and password attacks using Burp Suite. This capstone lab brings everything together in a structured Vulnerability Assessment (VA) — the systematic process of discovering, classifying, and reporting security weaknesses in a target system.

A Vulnerability Assessment is not the same as penetration testing. It focuses on finding and documenting vulnerabilities rather than fully exploiting them to gain maximum access. The deliverable is a professional report that a development team or management can use to prioritize remediation efforts.

💡

What This Lab Simulates

You are acting as a security consultant performing a vulnerability assessment against a client's web application (simulated by DVWA). You will test manually and automatically, then score each finding using CVSS v3.1 to produce a standardized severity baseline.

CVSS Severity Ratings

🔴 Critical 9.0–10.0

Remotely exploitable, no authentication needed, full system compromise. Requires immediate remediation.

🟠 High 7.0–8.9

Significant impact, relatively easy to exploit. Remediate within 30 days.

🟡 Medium 4.0–6.9

Moderate impact or requires specific conditions. Remediate within 90 days.

🟢 Low 0.1–3.9

Limited impact, often needs physical access or user interaction. Address in next dev cycle.

CVSS v3.1 — The Scoring Standard

CVSS (Common Vulnerability Scoring System) v3.1 is the industry-standard framework for assessing the severity of security vulnerabilities. It produces a numeric score from 0.0 to 10.0. Every finding in a professional VA report must include a CVSS base score with its full vector string so the score is reproducible by anyone.

CVSS v3.1 Base Metrics

Metric GroupMetricValues
ExploitabilityAttack Vector (AV)Network (N), Adjacent (A), Local (L), Physical (P)
Attack Complexity (AC)Low (L), High (H)
Privileges Required (PR)None (N), Low (L), High (H)
User Interaction (UI)None (N), Required (R)
ScopeScope (S)Unchanged (U), Changed (C)
ImpactConfidentiality (C)None (N), Low (L), High (H)
Integrity (I)None (N), Low (L), High (H)
Availability (A)None (N), Low (L), High (H)
💡

CVSS Vector String Notation

Scores are written as a vector: CVSS:3.1/AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H. This encodes all metric values and lets anyone reproduce the score. Use the online CVSS v3.1 calculator at nvd.nist.gov/vuln-metrics/cvss/v3-calculator for all scoring in this lab.

⚙️ Setup: Start DVWA & Tools

Step 1: Start DVWA

# Start the DVWA Docker container
sudo docker start dvwa

# Confirm it is running
sudo docker ps

# Get your Kali IP address
hostname -I

Note your IP address (e.g., 10.0.2.15). Use this IP in all URLs throughout the lab — never use localhost.

Step 2: Set DVWA to Low Security

  1. Open Firefox and navigate to http://<your-ip>
  2. Log in: admin / password
  3. Click DVWA Security in the left menu
  4. Select Low from the dropdown and click Submit

Step 3: Start Burp Suite

  1. Open a terminal and run: burpsuite
  2. Select Temporary project → Start Burp
  3. Confirm the proxy listener is running on 127.0.0.1:8080
  4. In Firefox, activate the Burp Proxy FoxyProxy profile
  5. Set Intercept to OFF for now

Step 4: Verify Nikto and OWASP ZAP

Nikto is an open-source command-line web server scanner. It automatically checks a target web server for thousands of known vulnerabilities, outdated software versions, dangerous files, and misconfigurations (such as missing security headers or exposed admin interfaces). Nikto is fast and noisy — it does not try to hide its activity — making it ideal for authorized assessments where speed matters more than stealth.

OWASP ZAP (Zed Attack Proxy) is a free, open-source web application security scanner maintained by OWASP. Unlike Nikto, ZAP works as an intercepting proxy and includes both passive scanning (observing traffic) and active scanning (sending attack payloads). It can find a broader range of application-layer vulnerabilities such as XSS, SQL injection, and broken authentication. ZAP also has a graphical interface, making it approachable for beginners while still being powerful enough for professional use.

# Verify Nikto is installed
nikto -Version

# If Nikto is not installed, install it
sudo apt-get update && sudo apt-get install -y nikto

# Verify OWASP ZAP is available
which zaproxy || which owasp-zap

# If OWASP ZAP is not installed, install it
sudo apt-get update && sudo apt-get install -y zaproxy

🔬 Task 1: Manual Testing with Burp Suite

💡

Concept — Manual Testing Builds Understanding

Automated scanners miss business-logic flaws and context-dependent vulnerabilities. Manual testing — using Burp Suite's Proxy and Repeater — ensures you understand why each vulnerability exists and can provide meaningful, specific remediation advice. In professional engagements, manual findings often carry more weight than automated ones.

DVWA: In-Scope Modules • Security: Low

2A: SQL Injection

Navigate to SQL Injection in the DVWA left menu.

  1. Enter 1 in the User ID field and click Submit — observe the normal output
  2. Enter ' OR '1'='1 and click Submit — observe: all user records are returned (WHERE clause bypassed)
  3. Enter 1' UNION SELECT user, password FROM users-- - and click Submit
  4. Observe: usernames and MD5 password hashes are returned in the First name / Surname fields

✅ Expected Output — UNION-Based SQL Injection:

ID: 1' UNION SELECT user, password FROM users-- -
First name: admin
Surname: 5f4dcc3b5aa765d61d8327deb882cf99

First name: gordonb
Surname: e99a18c428cb38d5f260853678922e03

The “Surname” field contains MD5 password hashes. 5f4dcc3b5aa765d61d8327deb882cf99 is the MD5 hash of password — crackable instantly with any rainbow table.

2B: Command Injection

Navigate to Command Injection in the DVWA left menu.

  1. Enter 127.0.0.1 and click Submit — observe normal ping output
  2. Enter 127.0.0.1; whoami and click Submit — observe the OS user is appended to the output
  3. Enter 127.0.0.1; cat /etc/passwd and click Submit
  4. Observe: the full contents of /etc/passwd are returned alongside the ping

✅ Expected Output — Command Injection:

PING 127.0.0.1 (127.0.0.1) 56(84) bytes of data.
...
www-data

root:x:0:0:root:/root:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
...

The semicolon (;) chains a second OS command after the ping. User input reaches the shell directly. The web server process runs as www-data.

2C: XSS (Reflected)

Navigate to XSS (Reflected) in the DVWA left menu.

  1. In the “What's your name?” field, enter your name normally — observe it reflected in the response
  2. Enter <script>alert('XSS')</script> and click Submit — an alert box appears
  3. Enter <script>alert(document.cookie)</script> and click Submit
  4. Observe: the alert displays the current session cookie value

✅ Expected Output — Reflected XSS Cookie Theft:

An alert dialog displays: PHPSESSID=abc123def456; security=low

A real attacker crafts a malicious URL containing this payload. Any victim who clicks the link sends their session cookie to the attacker, enabling session hijacking without knowing the victim's password.

2D: XSS (Stored)

Navigate to XSS (Stored) in the DVWA left menu.

  1. In the Message field, enter: <script>alert('Stored XSS')</script>
  2. In the Name field, enter any name and click Sign Guestbook
  3. The page reloads — the alert fires automatically on page load
  4. Refresh the page — the alert fires again (payload persists in the database)
  5. Open a private/incognito tab, navigate to the Stored XSS page — the alert fires for any visitor
⚠️

Stored XSS Affects Every User Who Visits the Page

Unlike Reflected XSS (which requires tricking a specific victim into clicking a link), Stored XSS fires automatically for every user who loads the page. In a real application, a payload in an admin panel would steal the session cookie of every administrator who visits — without any attacker interaction after the initial submission.

2E: CSRF

Navigate to CSRF in the DVWA left menu.

  1. Turn Burp Intercept ON, enter any new password in both fields and click Change
  2. Observe the captured request: GET /vulnerabilities/csrf/?password_new=test&password_conf=test&Change=Change
  3. Forward the request and turn Intercept OFF
  4. Note that the entire password change is a simple GET request — all parameters are in the URL

✅ What a CSRF Attack Looks Like:

<!-- Attacker embeds this on any webpage -->
<img src="http://<your-ip>/vulnerabilities/csrf/?password_new=hacked&password_conf=hacked&Change=Change">

When a logged-in DVWA user visits any page containing this <img> tag, the browser automatically sends the GET request — silently changing their password to hacked with no user interaction required beyond visiting the page.

✍️ Task 1 — Questions

🔑 Answer Key — Task 1

  1. Q1: All records: ' OR '1'='1. Hashes: 1' UNION SELECT user, password FROM users-- -. MD5 is fast, unsalted, and has precomputed rainbow tables. 5f4dcc3b5aa765d61d8327deb882cf99 = password in any rainbow table lookup.
  2. Q2: Semicolon ;. Others: && (AND chaining), | (pipe), || (OR), $(cmd) (substitution).
  3. Q3: Reflected: targeted, requires victim to click a crafted link. Stored: passive, fires for every user who loads the page. Stored XSS is rated more severe — wider impact with no victim interaction needed beyond normal browsing.
  4. Q4: GET parameters are in the URL, embeddable in <img src>. Browsers auto-fetch images — no user interaction needed. POST requires a form submit, which is harder to trigger silently and is not auto-sent for resources like images.

📊 Task 2: Automated Scanning & CVSS Scoring

💡

Concept — Automated Tools Complement Manual Testing

Automated scanners are fast and consistent but prone to false positives and blind to logic flaws. Manual testing is thorough but slow. A professional VA uses both: manual testing for depth and understanding, automated tools for breadth and confirmation. You will then score every finding with CVSS to give stakeholders a standardized severity baseline.

Tools: Nikto • OWASP ZAP • CVSS Calculator

3A: Run Nikto Against DVWA

Nikto is a fast web server scanner that identifies server misconfigurations, missing security headers, default credentials, and version disclosure.

# Run Nikto against DVWA (replace with your IP)
nikto -h http://<your-ip> -o nikto_dvwa.txt

# View the saved results
cat nikto_dvwa.txt

Nikto finds server-level issues — missing security headers, default credentials, version disclosure. These complement the application-level vulnerabilities found in Task 2.

3B: Run OWASP ZAP Against DVWA

OWASP ZAP is a full-featured web application scanner. The quickest approach is its built-in automated scan.

  1. Launch ZAP: Applications → Web Application Analysis → OWASP ZAP, or run zaproxy in a terminal
  2. In ZAP, click Automated Scan
  3. Enter the target URL: http://<your-ip>
  4. Click Attack and wait for the scan to complete (5–10 minutes)
  5. Review the Alerts tab — findings are sorted by risk level (High, Medium, Low, Informational)
⚠️

ZAP and Authentication

DVWA requires login for most vulnerability pages. For a deeper ZAP scan, browse DVWA through ZAP's proxy while logged in first (so ZAP captures the session cookie), then run the scan. For this lab, reviewing ZAP's findings on accessible pages is sufficient. Note in your report that a deeper scan with authenticated session would yield additional findings.

💡

Why Command Injection (9.9) Scores Higher Than SQL Injection (8.8)

The key difference is the Scope (S) metric. Command Injection has S:C (Changed) because exploiting the web application directly impacts the underlying operating system — a completely different security component. SQL Injection has S:U (Unchanged) because its impact stays within the application's data boundary. Scope:Changed applies a significant score multiplier in the CVSS formula.

✍️ Task 2 — Questions

🔑 Answer Key — Task 2

  1. Q1: Nikto-only: missing X-Frame-Options, X-Content-Type-Options, server version disclosure (Apache/PHP), directory indexing on /config/. Nikto specializes in server-level configuration — headers, metadata, outdated software, default creds. Application-logic flaws require manual or ZAP-level testing.
  2. Q2: ZAP confirms SQLi and Reflected XSS but typically misses CSRF (needs session context) and Stored XSS (needs authenticated browsing). Manual testing provides full coverage. ZAP adds header-level findings. Some ZAP alerts are false positives requiring manual verification.
  3. Q3: The Scope (S) metric. Command Injection: S:C (Changed) — impacts the OS beyond the web app. SQL Injection: S:U (Unchanged) — impact stays within the database. CVSS applies a formula multiplier for Changed scope because it crosses security boundaries.
  4. Q4: Score ≈ 9.8. Removing the authentication requirement makes it exploitable by any unauthenticated internet user, not just logged-in accounts. PR:N vs PR:L dramatically expands the potential attacker population and urgency of remediation.

🔧 Troubleshooting

❌ DVWA is not accessible — browser shows connection refused

Solution: The Docker container may have stopped. Run: sudo docker ps. If DVWA is not listed, start it: sudo docker start dvwa. Confirm your IP with hostname -I and use that IP (not localhost) in all URLs.

❌ SQL Injection payload returns no results or a database error

Solution: Confirm DVWA security is set to Low. Medium and High use different query structures requiring different payloads. Also verify you are using straight apostrophes ('), not curly quotes. Copy payloads directly from the lab rather than typing them by hand.

❌ XSS alert box does not appear — the script shows as plain text

Solution: DVWA security is above Low. At Medium and High, DVWA strips <script> tags. Set security back to Low. If the alert still does not fire, try an event-based payload as a fallback: <img src=x onerror=alert(document.cookie)>.

❌ Stored XSS payload does not persist — alert fires once then stops

Solution: The DVWA guestbook entry may have been cleared (DVWA resets its database periodically) or the payload was truncated by the HTML maxlength attribute on the message field. To bypass maxlength: turn Burp Intercept ON, submit the form, and modify the mtxMessage parameter directly in Burp before forwarding.

❌ Nikto scan completes instantly with very few results

Solution: Confirm DVWA is reachable: curl http://<your-ip>. Specify the port explicitly: nikto -h http://<your-ip> -p 80. If DVWA runs on a non-standard port (e.g., 8080), replace -p 80 with the correct port. You should see at least the Apache/PHP version and missing header findings.

❌ OWASP ZAP finds only 1–2 alerts

Solution: ZAP's automated scan of DVWA is limited without authentication context. To improve results: (1) Log into DVWA through ZAP's proxy (activate ZAP as a proxy in Firefox, browse DVWA while logged in, then run the scan — ZAP will carry the session cookie). (2) Even with limited results, ZAP still finds server-header issues. Document all findings and note the authentication limitation in your report.

❌ Command Injection shows ping output but the chained command does not execute

Solution: Confirm you are on the Command Injection module specifically (not another module). Security must be Low — Medium/High sanitize the semicolon. First test with 127.0.0.1 alone to confirm the normal ping works, then append ; whoami. If the semicolon is blocked, try the pipe: 127.0.0.1 | whoami.

❌ My CVSS score does not match the reference table

Solution: CVSS metric selection has nuance. Common errors: (1) Confusing PR:N (no login needed) with PR:L (any logged-in user) — DVWA Low requires a login, so most findings use PR:L. (2) Scope: Command Injection is S:C (Changed) because it impacts the OS; SQL Injection is S:U (Unchanged) because it stays within the database. The CVSS calculator updates the score in real time as you select each metric.
