The Castle Metaphor

Defending a web application is somewhat akin to defending a medieval castle. This metaphor from the book gives us a practical way to understand how security resources must be prioritized.

📖 From the Book — The Castle Defender

A castle consists of a number of buildings and walls, which represent the core application code. Outside the castle are buildings that integrate with and support the castle's owner — these represent the application's dependencies and integrations.

In wartime, due to the large surface area, it is essential for defenses to be prioritized. It would be infeasible to maximize defensive fortifications at every potential entrance point.

In web application security, such prioritization and vulnerability management is often the job of security engineers in large corporations — or more generalized software engineers in smaller companies. These professionals take on the role of master defender.
🏛
Castle Walls
Core application code — the primary perimeter to defend
🏠
Outer Buildings
Third-party dependencies and integrations — wider attack surface
🛡
Gatekeepers
Authentication & authorization — who gets in and what they can do
📍
Weak Points
Unpatched vulnerabilities — entry points a hacker will seek out first

💡 Key Insight — Why Offense Comes First

The book deliberately covers recon and offense before defense. By understanding how hackers map and break into web applications, you can derive the best ways to prioritize your defenses and camouflage your application architecture from malicious eyes.

🏙

The Five Pillars of Defense

The book organizes the entire defense strategy into five sequential stages. Each stage builds on the previous one, forming a complete security lifecycle.

  • 1
    Defensive Software Architecture
    Security begins before a single line of code is written. Evaluate data flow, storage, and integration risks at the architecture phase. Fixing a security flaw here costs 30–60× less than fixing it in production (NIST).
  • 2
    Comprehensive Code Reviews
    Every commit should be reviewed not just for quality, but for security. Key questions: How is data transmitted? How is it stored? How is it presented to users? How is it processed on the server?
  • 3
    Vulnerability Discovery
    Proactively find vulnerabilities before customers or the public do. Methods include bug bounty programs, internal red/blue teams, third-party penetration testers, and corporate incentives for engineers to log known issues.
  • 4
    Vulnerability Management
    Not all vulnerabilities carry equal risk. Triage and prioritize based on: financial risk, difficulty of exploitation, type of data compromised, contractual agreements, and existing mitigations. Track fixes through to completion.
  • 5
    Regression Testing
    After deploying a fix, write a regression test to assert the vulnerability no longer exists. Approximately 25% of security vulnerabilities are regressions — bugs previously closed that were re-opened.

📊 The Cost of Finding Bugs Late

NIST research shows: the cost of removing an application security vulnerability during the design phase is 30–60× less than if removed during production. This is the single strongest argument for security-first architecture.

Vulnerability Discovery Methods

Method Who Finds It Risk to Business
Customer notification End users High — reputation already damaged
Public disclosure External researchers / hackers High — worst case; mass exploitation possible
Bug bounty programs External security researchers Medium — controlled, incentivized disclosure
Red / Blue teams Internal attackers & defenders Low — found internally before production
Penetration testing Third-party testers Low — structured, authorized testing

⚠️ The Logging Trap

Lack of logging for known vulnerabilities has led to the demise of several companies. If a vulnerability is known but not yet patched, additional logging must be put in place to detect any exploitation attempts while the fix is being developed.

🛠

Mitigation Strategies

An overall best practice for any security-conscious organization is to actively mitigate the risk of vulnerabilities occurring in the codebase. This is not a single-phase activity — it happens from architecture all the way through regression testing.

📖 From the Book — Defense in Depth

Mitigation strategies should be widespread, like a net trying to catch as many fish as possible. In crucial areas of an application, mitigation should also run deep.

All the skills and techniques covered in recon, offense, and defense are synergistic. Improving your mastery of any one discipline will result in more efficient use of your time across all three.

❌ Penetrate-and-Patch

React only when vulnerabilities are discovered or exploited. No proactive strategy. High cost, high risk.

✓ Holistic Defense

Security embedded at every stage — architecture, code review, discovery, management, and regression testing — from day one.

✓ Chapter 17 Summary — Securing Modern Web Applications

  • Defense is the natural conclusion of recon and offense — you must understand attacks to build defenses
  • The castle metaphor: prioritize defenses across a large surface area, just like a medieval fortress
  • Security engineering spans five stages: architecture → code review → discovery → management → regression testing
  • NIST: fixing a flaw in architecture costs 30–60× less than fixing it in production
  • Approximately 25% of vulnerabilities are regressions — write tests after every fix
  • Proactive discovery (bug bounties, red teams, pen testing) saves money and protects reputation

Analyzing Feature Requirements

The first step in securing any web application is the architecture phase. Before writing code, a cross-functional team evaluates business requirements and derives security risks from them.

📖 Case Study — MegaMerch

MegaBank decides to launch a merchandise store: MegaMerch. The product team defines five business requirements:

  • Users can create accounts and sign in
  • User accounts contain full name, address, and date of birth
  • Users can browse the store and search for items
  • Users can save credit cards and bank accounts for later use
  • Authenticated users have different privileges than guests

Business Requirements → Security Risks

Requirement Detected Security Risk Derived Priority
Storing credentials How do we handle sessions, logins, and cookies? Critical
Storing PII (name, address, DOB) Are laws applicable? Is data encrypted at rest? Critical
Elevated user privileges vs. guests Authorization model — how are permission levels enforced? Critical
Search engine functionality Separate search DB vs. primary DB — permission sync risks Medium
Storing financial data (credit cards) PCI-DSS compliance; consider outsourcing to specialists Critical

💡 Architecture-Phase Rule

Any organization with separate security and R&D teams must ensure communication pathways between the two are built into the development process. Features cannot be properly analyzed in a security silo.

🔑

Authentication & Authorization

Because MegaMerch stores credentials and supports different user tiers, it has both an authentication system and an authorization system. These are distinct concepts that work together.

🔐 Authentication

"Who are you?" — Verifying the identity of a user. Involves login flows, credentials sent over the network, and sessions/cookies issued after a successful login.

🛡 Authorization

"What can you do?" — Enforcing what actions an authenticated user is allowed to take. Differentiates between guests, registered users, and administrators.

Three Core Questions for Any Auth System

  • 1
    How do we handle data in transit?
    Credentials sent over the network must be encrypted. An unencrypted login form is vulnerable to man-in-the-middle interception.
  • 2
    How do we handle the storage of credentials?
    Passwords must never be stored in plain text. Even database administrators should not be able to read user passwords.
  • 3
    How do we handle various authorization levels?
    The application must enforce what each tier of user (guest, customer, admin) is allowed to access and modify — at every API endpoint.

⚠️ Access Control Failure

Failing to enforce authorization on every endpoint — even internal or "hidden" endpoints — is a critical vulnerability. A guest who discovers an admin API URL must be rejected at the server side, never just hidden from the UI.

🔒

SSL / TLS — Protecting Data in Transit

One of the most important architectural decisions is how to handle data in transit. All data sent over the network — especially credentials — must be encrypted to prevent man-in-the-middle attacks.

📖 From the Book — SSL vs. TLS

SSL (Secure Sockets Layer) was designed by Netscape in the mid-1990s. Multiple versions have been released, but several known architectural vulnerabilities have been found in SSL.

TLS (Transport Layer Security) was defined by RFC 2246 in 1999, offering upgraded security in response to SSL's architectural issues. TLS cannot interoperate with older SSL due to the amount of architectural differences.

TLS offers the most rigid security, while SSL has higher historical adoption but multiple vulnerabilities that reduce its integrity as a cryptographic protocol.

SSL — Secure Sockets Layer

Designed 1995 by Netscape. Wider adoption historically. Multiple known vulnerabilities (POODLE, BEAST). Not recommended for new applications.

TLS — Transport Layer Security

RFC 2246, 1999. Upgraded SSL architecture. Current standard: TLS 1.3 (2018). Used by HTTPS. Recommended for all modern web applications.

🔒
HTTPS
HTTP Secure — URI scheme requiring TLS/SSL before any data can flow
🔐
Lock Icon
All major browsers display a lock when TLS is active and valid
🛑
Browser Warning
Browsers warn users if a TLS/SSL connection is compromised on an HTTPS request
📄
Certificate
TLS requires a certificate from a Certificate Authority (e.g., Let's Encrypt) to authenticate the server

💡 MegaMerch Decision

For MegaMerch, all data must be encrypted via TLS prior to being sent over the network. This is the first and most fundamental architectural requirement — it protects credentials, PII, and financial data from interception.

💻

Secure Credentials — Password Policy

Most developers think password security is about length and special characters. The book teaches something more fundamental: security comes from entropy — the amount of randomness and unpredictability in a password.

📖 From the Book — Dictionary Attacks

When a hacker attempts to brute-force logins, the easiest route is a dictionary attack: use a list of the top most common passwords, then try combinations of common password structures.

It is not so much the length of the password that will protect you, but instead the lack of observable patterns and avoidance of common words and phrases.

Three Attack Types on Passwords

📕
Dictionary Attack
Try top 1,000–1,000,000 most common passwords from a wordlist
🔨
Hybrid Attack
Common passwords + variations + common structures (leet-speak, suffixes)
🔢
Brute Force
Iterate through all possible combinations — only viable on short passwords

Enforcing High-Entropy Passwords at MegaMerch

❌ Weak Password Policy
  • Minimum 8 characters only
  • No restriction on common passwords
  • User's name allowed in password
  • Date of birth allowed in password
  • No check against known breach lists
✓ Strong Password Policy
  • Reject any password in top 1,000 common list
  • Prevent first name, last name in password
  • Prevent birthdate & address fragments
  • No forced expiry (promotes reuse)
  • Offer a password strength indicator

💡 Key Principle — Entropy Over Complexity Rules

A 4-word passphrase like "correct horse battery staple" has higher entropy than "P@ssw0rd!". Requiring special characters without restricting common patterns gives a false sense of security.

🔙

Password Hashing — BCrypt & PBKDF2

Passwords must never be stored in plain text. When storing a password for the first time, it must be hashed — a one-way, irreversible transformation — before being written to the database.

📖 From the Book — Why Hash?

Hashing algorithms differ from encryption in a key way: they are not reversible. This means even your own staff cannot steal user passwords — protecting users from rogue employees and database breaches alike.

When a user logs in, you hash their submitted password and compare it to the stored hash. No decryption is ever needed.

The Three Cases: What Happens After a Database Breach?

🔒
Plain Text Storage
Result: All passwords compromised immediately
⚠️
MD5 Hashing
Result: Hacker cracks many passwords via rainbow tables (precomputed hash→password tables)
🔒
BCrypt Hashing
Result: Unlikely any passwords will be cracked — years of compute time per password on modern hardware

BCrypt vs. PBKDF2

BCrypt

Based on Blowfish Cipher (1993) + Unix Crypt. Becomes slower on faster hardware — the more powerful the attacker's hardware, the more operations are required. Makes brute-forcing computationally infeasible.

PBKDF2

Based on key stretching. First hash is fast; each additional attempt becomes progressively slower. Configure with maximum iterations your hardware can handle. Good alternative when BCrypt is unavailable.

// Example: BCrypt password hashing (Node.js) const bcrypt = require('bcrypt'); const saltRounds = 12; // Higher = slower to crack // At registration — hash before storing const hash = await bcrypt.hash(plainTextPassword, saltRounds); // ✓ Store hash in database, never the plain text // At login — compare submitted password to stored hash const match = await bcrypt.compare(submittedPassword, storedHash); // ✓ Returns true/false — no decryption ever needed // NEVER do this: // ✗ db.store({ password: plainTextPassword }) // ✗ db.store({ password: md5(plainTextPassword) })

💡 The Core Rule

Hashing algorithms for passwords should be intentionally slow on modern hardware. This limits the number of guesses an attacker can make per second, making brute force impractical for any well-hashed password.

📱

Two-Factor Authentication (2FA)

In addition to requiring strong, hashed passwords over TLS, MegaMerch should offer two-factor authentication (2FA) to users who want to maximize their account security.

📖 From the Book — The 2FA Principle

2FA operates on a simple but powerful principle: require two independent secrets to log in. Even if a hacker obtains a user's password (via phishing, a data breach, or brute force), they still cannot access the account without the second factor.

In the absence of any vulnerabilities in the 2FA app or messaging protocol, 2FA eliminates all remote logins that were not initiated by the owner of the account.

2FA Types — Comparing the Options

2FA Method How It Works Security Level Best For
SMS / Text Message One-time code sent to registered phone number Moderate Consumer web apps (easy UX)
Authenticator App (e.g., Google Authenticator) Time-based one-time passwords (TOTP) generated on device Strong Consumer & business web apps
Hardware Token (USB key) Physical device plugs in and generates a one-time token Strongest Enterprise employees, high-value accounts

💡 MegaMerch Recommendation

Distributing physical hardware tokens to all MegaMerch customers would be impractical. The recommended approach is to offer an authenticator app / SMS 2FA option that users can opt into. Even SMS-based 2FA is "an order of magnitude safer than application use without 2FA" (Hoffman).

🔐
Something You Know
Password — factor 1, potentially stolen remotely
📱
Something You Have
Phone / hardware token — factor 2, requires physical access to steal
🚫
Attack Blocked
Remote attacker with only the password cannot complete login
✅
Legitimate User
Knows password AND has physical device — login succeeds
📄

PII & Financial Data

When storing Personally Identifiable Information (PII) — such as full name, address, date of birth — a web application must ensure storage is legal in the countries it operates in and that the data is protected against breach.

📖 From the Book — PII Obligations

Beyond legality, you want to ensure that in the case of a database breach or server compromise, PII is not exposed in a format that makes it easily abusable.

Similar rules apply to financial data, such as credit card numbers (also included under PII laws in some countries, e.g., GDPR in the EU, PCI-DSS for payment card data).

Store PII In-House

Full control, but requires compliance expertise (GDPR, PDPA, PCI-DSS). Deep re-architecture required if compliance gaps found post-launch. High liability.

Outsource PII / Financial Storage

Smaller companies may find it more effective to outsource to a compliant specialist (e.g., Stripe for payments). Reduces liability and engineering burden significantly.

Data Type Example Key Requirement
Credentials Passwords, tokens Hash (BCrypt/PBKDF2), never store plain text
PII Name, address, DOB, email Encrypt at rest, GDPR/PDPA compliance, data minimization
Financial Credit card numbers PCI-DSS compliance; consider outsourcing to payment processor
Session Tokens Cookies, JWT tokens Rotate regularly, expire on logout, use HttpOnly & Secure flags

⚠️ Session Management

Session tokens issued after authentication must be treated like credentials. They must be transmitted over TLS, stored with HttpOnly and Secure cookie flags, invalidated on logout, and rotated after privilege changes (e.g., after login or 2FA completion) to prevent session fixation attacks.

✓ Chapter 18 Summary — Secure Application Architecture

  • Architecture phase is the cheapest and most effective place to catch security flaws — before code is written
  • Business requirements must be analyzed for security risks: authentication, authorization, PII, search, financial data
  • Authentication = "who are you?"; Authorization = "what can you do?" — both must be enforced server-side
  • TLS (HTTPS) is mandatory for all data in transit — especially credentials and financial data
  • Passwords must be hashed with BCrypt or PBKDF2 — never stored in plain text or with reversible encryption
  • 2FA eliminates remote account takeover; authenticator app or SMS are practical for consumer applications
  • PII and financial data require legal compliance (GDPR, PCI-DSS); outsourcing is often the safest option for smaller teams
  • Session tokens must use HttpOnly + Secure flags, rotate after login, and expire on logout
