Securing Modern Web Applications

Learning Objectives

🏝

The Defense Mindset

Modern web applications are far more complex and distributed than their predecessors. This expanded complexity increases the attack surface — meaning more entry points that a hacker can try to exploit. Defending a web application starts not at the keyboard, but in the mind of the developer.

📖 From the Book — The Castle Analogy

Defending a web application is somewhat akin to defending a medieval castle. A castle consists of buildings and walls — representing the core application code. Outside are buildings that integrate with and support the castle, just like an application's dependencies and integrations.

In wartime, it is impossible to fortify every entrance equally. Smart defenders prioritize. The same is true in web application security — defenses must be prioritized based on risk, not applied uniformly at maximum strength everywhere.

These professionals take on the role of master defender, using software engineering skills combined with recon and hacking skills to reduce the probability of a successful attack, mitigate potential damages, and manage active or past damages.

Penetrate-and-Patch vs. Holistic Approach

The old-fashioned approach to security — "penetrate-and-patch" — waits until a vulnerability is discovered (by a hacker, a customer, or a security scanner) and only then fixes it. This reactive approach is expensive, slow, and dangerous.

✗ Penetrate-and-Patch (Reactive)
  • Security reviewed only after code ships
  • Vulnerabilities discovered by attackers first
  • Re-architecture is costly once users depend on it
  • Each bug is fixed in isolation — root cause ignored
  • No proactive culture of security in engineering
✓ Holistic Approach (Proactive)
  • Security designed before writing a single line of code
  • Architecture reviewed for risk at the design stage
  • Code reviewed for security on every merge request
  • Automated testing catches regressions before release
  • Security culture is embedded in the development process

🔑 Key Insight

All skills — recon, offense, and defense — are synergistic. Understanding how a hacker would map and exploit your application gives you insight into how to prioritize defenses and hide architectural details from malicious eyes.

🏛

Three Pillars of Secure Development

A holistic approach to web application security follows three sequential, interdependent pillars. Each builds on the previous one — skipping or weakening any pillar increases overall risk.

🧬
Pillar 1: Architecture
Design security into the system before writing code. Evaluate data flow, trust boundaries, and threat models.
🔎
Pillar 2: Code Review
Review every commit for common vulnerabilities and logic bugs. Cross-team review reduces conflict of interest.
📊
Pillar 3: Vuln Discovery
Use automation, penetration testing, and bug bounties to catch what slipped through architecture and review.
📖 From the Book — Code Review Checklist

Catching security holes at the code review level on a per-commit basis is easier than one would think. The major questions to ask are:

  • How is data being transmitted from point A to point B (format, encryption)?
  • How is data being stored (plain text, hashed, encrypted)?
  • When data reaches the client, how is it presented to the user (DOM injection risks)?
  • When data reaches the server, what operations occur on it and how is it persisted?
📋

Vulnerability Management & Regression Testing

Not all vulnerabilities carry equal risk. The priority of a fix must be derived from a structured risk assessment before any engineering effort begins.

Risk Assessment Factors

  • Financial risk — what is the potential monetary loss if exploited?
  • Difficulty of exploitation — how easy is it for an attacker to trigger?
  • Type of data compromised — PII, financial data, credentials, or public data?
  • Existing contractual obligations — does the vulnerability violate customer SLAs?
  • Mitigation measures already in place — are there compensating controls reducing risk?

📊 Regression Statistics

A security engineer from a large software company (10,000+ employees) reported that approximately 25% of their security vulnerabilities were regressions — previously fixed bugs that reopened. Regression tests are the most effective and cheapest defense against this pattern.

After patching a vulnerability, always write a regression test — a test that will fail if the vulnerability is ever reintroduced. This small upfront investment can prevent a vulnerability from silently returning in a future refactor or rollback.

🔑 Chapter 17 Summary

  • Modern web apps have large attack surfaces requiring prioritized, not exhaustive, defenses
  • The holistic approach (architecture → code review → discovery) outperforms penetrate-and-patch
  • Risk assessment determines the order and urgency of vulnerability fixes
  • Regression testing prevents fixed vulnerabilities from silently reopening
  • Recon and offensive knowledge make defenders significantly more effective

Analyzing Feature Requirements for Security Risk

The first step in securing any application is the architecture phase — before a single line of code is written. This phase is a cross-functional exercise where engineers, product managers, and security architects work together to evaluate data flow, trust boundaries, and potential attack surfaces.

📖 From the Book — The MegaMerch Case Study

Consider MegaBank, which is launching a new e-commerce platform called MegaMerch. The business requirements seem simple at first glance:

  • Users can create accounts and sign in
  • User accounts contain full name, address, and date of birth
  • Users can browse and search for items
  • Users can save credit cards and bank accounts for later use

But a high-level security analysis immediately reveals significant risk areas hidden inside these simple requirements:

  • We are storing credentials — authentication system needed
  • We are storing personally identifiable information (PII) — legal and privacy obligations
  • Users have elevated privileges compared to guests — authorization model needed
  • Users can search — a secondary database may be required (sync complexity)
  • We are storing financial data — highest risk category, strict compliance rules

💡 Architecture-First Principle

Any organization with separate security and R&D teams must build communication pathways between them into the development process. Features cannot be properly analyzed in a silo — stakeholders from both engineering and product development must be involved.

📊 The NIST Finding

Based on a study of popular web applications, NIST concluded: "The cost of removing an application security vulnerability during the design phase ranges from 30–60 times less than if removed during production." Upfront security architecture is not overhead — it is an investment with massive returns.

🔒

SSL/TLS & Secure Data in Transit

One of the most critical architectural decisions is how to handle data moving across the network. All data transmitted over the internet must be encrypted to prevent man-in-the-middle (MitM) attacks — where an attacker intercepts communication between the client and server to steal credentials, session tokens, or financial data.

SSL — Secure Sockets Layer

Designed by Netscape in the mid-1990s. Multiple versions released since. Has several known architectural vulnerabilities (e.g., POODLE, BEAST). Widely adopted but considered deprecated for security use.

TLS — Transport Layer Security

Defined by RFC 2246 in 1999. Offers significantly improved security over SSL. Cannot interoperate with older SSL versions due to architectural differences. Recommended for all modern web applications.

The HTTP specification provides HTTPS ("HTTP Secure") — a URI scheme that requires TLS/SSL before any data is transmitted. Browsers display a lock icon when HTTPS is active and warn users when a connection is compromised.

⚠️ Production Requirement

All data sent over the network — including login forms, API payloads, and cookies — must travel over HTTPS/TLS. Any plaintext HTTP endpoint in a production application is an immediate security risk.

🔐

Secure Credentials & Password Hashing

Storing credentials securely is one of the most impactful architectural decisions. The wrong approach — storing passwords in plain text — can cause total credential compromise in any database breach. The right approach uses one-way hashing algorithms designed specifically to resist brute-force attacks.

Why Hashing Works

Hashing algorithms are irreversible — you cannot recover the original password from a hash. When a user logs in, their input is hashed and compared to the stored hash. Even MegaMerch staff cannot recover user passwords.

Scenario Storage Method Result After Breach Risk
Case #1 Plain text All passwords immediately compromised Critical
Case #2 MD5 hash Many passwords cracked via rainbow tables High
Case #3 BCrypt hash Unlikely any passwords are cracked Low

Recommended Algorithms

BCrypt

Derived from Blowfish Cipher + Unix Crypt. Uniquely designed to become slower as hardware improves — making brute-force attacks computationally infeasible on modern hardware. The gold standard for password hashing.

PBKDF2

Based on key stretching — each additional hashing attempt becomes slower. Takes a configurable minimum iteration count — always set to your hardware's maximum. A strong alternative when BCrypt is unavailable.

💡 The Slow Hashing Principle

When a hacker brute-forces passwords, they automate millions of guesses per second. A slow hashing algorithm like BCrypt can make cracking a single complex password take years on modern hardware — making it practically unbreakable.

Two-Factor Authentication (2FA)

Beyond strong hashing, well-architected applications should offer 2FA to users who want an extra layer of protection. Phone app/SMS-based 2FA may not be as secure as hardware tokens, but it eliminates the vast majority of remote account compromise attempts — a hacker would need both the account password and the user's physical device.

🔒 Chapter 18 Summary

  • Security architecture must happen before coding — retrofitting security into production is 30–60× more expensive
  • Business requirements always contain hidden security risks — analyze them proactively
  • All data in transit must be encrypted via TLS; HTTP in production is unacceptable
  • Passwords must be hashed with slow algorithms (BCrypt, PBKDF2) — never stored as plain text
  • Offering 2FA significantly reduces account compromise risk for users who opt in

How to Start a Security Code Review

A security code review follows the same starting workflow as a functional code review — the differences emerge in what you look for, not how you begin. The review should always take place after the architecture has been reviewed and approved.

📖 From the Book — The Standard Git Review Flow

A common local review flow that can be performed from the terminal:

  1. 1
    Check out master
    git checkout master — ensure you are starting from the stable base branch.
  2. 2
    Pull the latest changes
    git pull origin master — fetch and merge the most recent production code.
  3. 3
    Check out the feature branch
    git checkout <username>/feature — switch to the branch under review.
  4. 4
    Run a diff against master
    git diff origin/master... — see every changed file and every line that changed.

Archetypical vs. Logic Vulnerabilities

Archetypical Vulnerabilities

Well-known categories: XSS, CSRF, SQL injection, XXE. Can be found with automated tools and security scanners. Covered by OWASP Top 10 checklists.

Custom Logic Vulnerabilities

Require deep understanding of the application's business logic, user roles, and intended behavior. Cannot be found by automated scanners. Example: an API endpoint that lets a regular user set isMember: true to bypass the moderator approval process.

⚠️ Logic Bugs Are Invisible to Scanners

Automated security tools cannot detect logic vulnerabilities. They require a human reviewer who understands the users, functionality, and business impact of the feature being reviewed. This is why human code review cannot be replaced by automation alone.

📍

Where to Start: The 5-Step Review Framework

When reviewing an unfamiliar codebase, the order in which you read the code determines how quickly you can prioritize risk. A logical entry path is to start where users interact with the application and trace inward.

  1. 1
    Review client-side code first
    Gain an understanding of the business logic and user-facing functionality. This tells you what users can do and what data flows to the server.
  2. 2
    Evaluate the API layer
    Using knowledge from the client review, audit the specific API endpoints users call. Understand what dependencies the API layer relies on.
  3. 3
    Trace dependencies in the API layer
    Carefully review databases, helper libraries, logging functions, and file operations. This covers the majority of user-facing functionality.
  4. 4
    Look for unintentionally exposed APIs
    Find public-facing API endpoints that may be accidentally exposed or intended for future features but accessible now.
  5. 5
    Review the remainder of the codebase
    By now you understand the application architecture organically. Cover remaining code in descending order of risk.
🚫

Secure-Coding Anti-Patterns

Security anti-patterns are solutions that look correct but are fundamentally incomplete or unsafe. Recognizing them quickly accelerates the review process and prevents dangerous bugs from reaching production.

Anti-Pattern 1: Blacklists

A blacklist blocks a specific set of known-bad inputs. The problem: it can never be complete. New bad inputs will always emerge that were not anticipated when the list was created.

// ✗ INSECURE — Blacklist approach const blacklist = ['http://www.evil.com', 'http://www.badguys.net']; const isDomainAccepted = function(domain) { return !blacklist.includes(domain); // attacker just buys a new domain }; // ✓ SECURE — Whitelist approach const whitelist = ['https://happy-site.com', 'https://www.my-friends.com']; const isDomainAccepted = function(domain) { return whitelist.includes(domain); // only known-good domains pass };

🔑 Whitelists Over Blacklists — Always

A whitelist only allows explicitly approved inputs. An attacker cannot bypass a whitelist by registering a new domain or crafting a new payload — they must be on the approved list. Whitelists require more maintenance but offer fundamentally stronger security guarantees.

Anti-Pattern 2: Boilerplate Code in Production

Frameworks and libraries ship with default configurations designed for ease of development, not security. Launching default/boilerplate code into production without evaluation exposes the application to known misconfigurations.

📖 From the Book — The MongoDB Incident

A configuration mistake in older versions of MongoDB made databases accessible over the internet by default when installed on a web server — with no mandatory authentication. The result: tens of thousands of MongoDB databases were hijacked by automated scripts demanding Bitcoin ransoms.

Two configuration lines — preventing external accessibility — would have prevented the entire incident.

Anti-Pattern 3: Trust-By-Default

Many applications run all server-side modules under a single user account with broad permissions. If one module is compromised, the attacker inherits permissions for all other modules — logging, file system, database, and more.

✗ Trust-By-Default (Insecure)
  • One server user account with all permissions
  • SQL injection in one module = full server access
  • Log files, disk, and database all exposed
  • No isolation between modules
✓ Least-Privilege (Secure)
  • Each module runs under its own user account
  • Permissions scoped to only what the module needs
  • SQL injection in one module = only DB access compromised
  • Failure in one module does not cascade to others

Anti-Pattern 4: Client/Server Coupling

When client and server code are tightly bound — for example, PHP templates embedding authentication logic — the attack surface multiplies. The server must handle and sanitize HTML structures, form data, and authentication payloads simultaneously, making security mechanisms overlap and become difficult to manage.

💡 Separation of Concerns = Security

A properly separated client/server application communicates over a predefined, typed data format (e.g., JSON over HTTPS). The server only accepts structured payloads — not raw HTML. This eliminates an entire class of injection vulnerabilities and makes security mechanisms simpler and more consistent.

🔎 Chapter 19 Summary

  • Security code review must follow architecture review — never precede it
  • Human review is irreplaceable for detecting custom logic vulnerabilities
  • Start reviews at the client and trace inward through APIs, dependencies, and remaining code
  • Whitelists always outperform blacklists — incomplete security is dangerous security
  • Avoid boilerplate in production, trust-by-default permissions, and tight client/server coupling

Why Vulnerability Discovery Is Still Needed

Even with secure architecture and thorough code review, vulnerabilities still slip through. They may emerge from unexpected behavior in different environments, third-party library updates, or edge cases no reviewer anticipated. Vulnerability discovery processes target production code and catch what the earlier phases missed.

🔍
Static Analysis
Scans source code without executing it. Fast and cheap. Best for common vulnerabilities in statically typed languages.
⚙️
Dynamic Analysis
Executes the application and analyzes outputs at runtime. Slower but confirms actual vulnerabilities rather than hypothetical ones.
📋
Regression Testing
Tests previously fixed vulnerabilities to ensure they never reopen. Simple, cheap, and extremely effective at blocking repeat bugs.
🔍

Static Analysis

Static analysis tools read source code — without executing it — and evaluate it for syntax errors, common misconfigurations, and known vulnerability patterns. They run locally during development (as linters) or on every push to the repository in a CI/CD pipeline.

Common Static Analysis Tools

Tool Languages Cost Best For
Checkmarx Most major languages Paid Enterprise-grade, full OWASP Top 10 coverage
PMD Java Free Java-specific code quality and security checks
Bandit Python Free Python security issues — hardcoded secrets, injection risks
Brakeman Ruby Free Ruby on Rails security scanning

What Static Analysis Can Detect

  • General XSS — look for DOM manipulation via innerHTML
  • Reflected XSS — look for variables pulled directly from URL parameters
  • DOM XSS — look for dangerous sinks like setInterval() receiving user input
  • SQL Injection — look for user-provided strings concatenated into queries
  • CSRF — look for state-changing HTTP GET requests
  • ReDoS — look for improperly written regular expressions

⚠️ Static Analysis Limitation — Dynamic Languages

Static analysis works best on statically typed languages (Java, C#) where variable types are known at parse time. In dynamically typed languages like JavaScript, variables are mutable and change type at runtime — making it very difficult for a static tool to determine what data flows where, leading to many false positives and missed vulnerabilities.

⚙️

Dynamic Analysis

Dynamic analysis executes the application in a production-like environment and analyzes outputs at runtime. Unlike static analysis, dynamic analysis can confirm actual vulnerabilities rather than hypothetical ones — because it observes what the application actually does, not what the code suggests it might do.

IBM AppScan (Paid)

Enterprise-grade dynamic analysis tool with broad framework support. High accuracy with deep configuration.

Veracode (Paid)

Cloud-based dynamic and static analysis. Integrates into CI/CD pipelines. Known for low false-positive rate.

📖 From the Book — Static vs Dynamic Trade-offs

Dynamic analysis is fantastic at picking up actual vulnerabilities, whereas static analysis picks up many potential vulnerabilities but has limited ways of confirming them.

Dynamic tools can detect vulnerabilities that only manifest at runtime — for example, sensitive data improperly stored in memory, or side-channel attacks that depend on real execution timing. These are completely invisible to static analysis.

The trade-off is in maintenance, cost, and performance: dynamic analysis requires a full production-like environment and is significantly slower than static analysis. But its results are more accurate and actionable.
📋

Vulnerability Regression Testing

Regression tests are written after a vulnerability is fixed to ensure it never returns. They are simple to write, easy to integrate into existing test suites, and offer long-lasting protection against repeat vulnerabilities.

📖 From the Book — Steve and Jed's CSRF Story

Software engineer Steve builds a subscription management endpoint that accepts HTTP GET requests:

// ✗ VULNERABLE — State-changing GET endpoint (CSRF risk) app.get('/changeSubscriptionTier', function(req, res) { if (!currentUser.isAuthenticated) { return res.sendStatus(401); } if (!req.params.newTier) { return res.sendStatus(400); } modifySubscription(currentUser, req.params.newTier) .then(() => res.sendStatus(200)) .catch(() => res.sendStatus(400)); });

Steve's friend Jed notices that a link like GET /api/changeSubscriptionTier?newTier=corporation can be sent to Steve — and when Steve clicks it, his subscription is silently changed on his behalf. This is a CSRF vulnerability. The fix: switch to HTTP POST. Then Steve writes a regression test:

// ✓ Regression test — ensures the endpoint only accepts POST const testTierChange = function() { requester.options('http://app.com/api/changeSubscriptionTier') .on('response', function(res) { const verbs = res.headers['Allow'].split(','); if (verbs.length > 1) { return tester.fail(); } // only one verb allowed if (verbs[0] !== 'POST') { return tester.fail(); } // must be POST }); };

If a future change reintroduces a GET version of this endpoint — even accidentally — the regression test will immediately fail, blocking the change from being merged.

🔑 When to Run Regression Tests

Vulnerability regression tests should be run on commit or push hooks — rejecting the commit or push if any test fails. Regularly scheduled daily runs are a strong secondary option for complex version control environments.

💰

Responsible Disclosure & Bug Bounty Programs

Even with strong internal automation, some vulnerabilities are only found by the application's own users. A well-defined responsible disclosure program gives tech-savvy users a safe, legal path to report what they find — instead of going public or staying silent.

Responsible Disclosure

A good responsible disclosure program includes:

  • A clear list of how users can test the application without legal risk
  • A defined submission template for vulnerability reports
  • A time window (weeks or months) where the reporter agrees not to publish while the fix is being developed
  • A commitment from the organization to investigate and respond within a defined time frame

Bug Bounty Programs

Bug bounty programs go further — they pay cash rewards for properly documented, verified vulnerability reports. Platforms like HackerOne and BugCrowd provide customizable legal templates, submission interfaces, and triage workflows.

Third-Party Pen Testers

Can be assigned specific application areas to test in depth. Can be provided with source code under legal agreement for more accurate results. Best used on high-risk or newly written areas before release to production.

Bug Bounty Hunters

Freelance security researchers who test when and where they choose. Provide broad, unsystematic coverage across the application. Top hunters are highly skilled and motivated by financial reward.

📊 The SSDL Outcome

By combining architecture review, security code review, static analysis, dynamic analysis, regression testing, responsible disclosure, and bug bounty programs into a Secure Software Development Life Cycle (SSDL), an organization can release production web applications with high confidence and minimal fear of serious security holes being discovered or exploited.

📊 Chapter 20 Summary

  • Secure architecture and code review still leave gaps — production-targeting discovery processes are essential
  • Static analysis is fast and cheap but produces false positives and misses runtime and logic vulnerabilities
  • Dynamic analysis confirms real vulnerabilities at runtime — more accurate but slower and costlier
  • Vulnerability regression tests are simple, cheap, and prevent fixed bugs from silently returning
  • Responsible disclosure programs and bug bounties extend discovery coverage beyond internal teams
  • Combining all techniques into an SSDL gives organizations the highest level of security assurance
