🔐 What is Burp Intruder?

In Labs 05 and 06 you learned Burp Suite's Proxy, Repeater, and Site Map tools. In this lab you will work with the most powerful automated testing tool in Burp's arsenal: Intruder.

Burp Intruder is an automated request-fuzzing engine. It takes any captured HTTP request, lets you mark one or more parameters as payload positions, then sends that request repeatedly — substituting a different value on each attempt from a wordlist or other payload source. This automates what would otherwise take hours of manual work in Repeater.

💡

What We Use Intruder For in This Lab

We will use Intruder to perform a dictionary attack against DVWA's Brute Force login module — automatically testing a list of common passwords to find the correct one by comparing server response lengths.

The Four Intruder Attack Modes

Intruder supports four attack types. The right choice depends on how many payload positions you have and how you want payloads combined.

🎯 Sniper Used in this lab

One payload list. Tests each payload in one position at a time. Best for testing a single parameter — like a password field — with a wordlist.

🎲 Battering Ram

One payload list. Inserts the same payload into all positions simultaneously. Useful when the same value must appear in multiple fields at once.

⚒️ Pitchfork

Multiple payload lists — one per position — iterated in parallel. Best for matched username+password pairs (user1:pass1, user2:pass2).

💣 Cluster Bomb

Multiple payload lists, all combinations tested. Every username is tried against every password. True credential stuffing, but generates many requests.

Intruder vs Repeater: What Is the Difference?

FeatureRepeater (Lab 06)Intruder (This Lab)
Number of requestsOne at a time — you click Send manuallyHundreds or thousands — automated
Payload variationYou type each change by handLoaded from a wordlist, iterated automatically
Best forManual exploitation, precise one-off testsFuzzing, password attacks, parameter discovery
Results viewOne response at a timeTable of all responses — sortable by length or status
💡

Community Edition Rate Limit

Burp Suite Community Edition (free) throttles Intruder to approximately 1 request per second. Professional Edition has no limit. For this lab you will use a short 15-password wordlist, so the throttle is not a problem — the attack completes in about 15 seconds. You will observe the throttling indicator and understand why the Professional edition matters in real engagements. In v2025, the rate-limit notice appears as a banner in the attack results window or as a status message in the lower toolbar of the results view.

📖 Dictionary vs Brute Force Attacks

Pure Brute Force

A pure brute force attack tries every possible combination of characters — aaa, aab, aac… up to zzzzzzz. For a 6-character lowercase password, that is 26&sup6; = 308 million combinations. It is guaranteed to find the password eventually, but can take an impractical amount of time.

Dictionary Attacks

A dictionary attack uses a wordlist — a pre-compiled file of common passwords, leaked passwords, and likely guesses. Most real-world passwords appear in these lists because people reuse predictable choices. A dictionary attack trades completeness for speed.

Attack TypeMethodSpeedCoverage
Brute ForceEvery possible character combinationVery slow100% — finds any password eventually
DictionaryCommon passwords from a wordlistFastPartial — only finds weak or common passwords
HybridDictionary + mutations (add 123, !, capitalise)MediumGood — catches “password123”, “Admin!” etc.

What Is rockyou.txt?

rockyou.txt is the most well-known password wordlist in security testing. It contains over 14 million real passwords leaked in a 2009 breach of the RockYou social gaming platform. Because these are actual passwords real users chose, it is extremely effective against weak credentials.

📷 Task 1: Capture the Login Request

💡

Concept — Why We Capture First

Intruder cannot build a request from scratch — it needs a real HTTP request as a template. The workflow is always: browse normally → Burp captures the request → send to Intruder → mark positions → run attack. Capturing first ensures Intruder has the correct URL, headers, cookies, and parameter structure for the target.

DVWA: Brute Force Module

1A: Navigate to the DVWA Brute Force Module

  1. In Firefox, confirm FoxyProxy is active (Burp Proxy profile selected)
  2. Turn Burp Intercept ON: Proxy → Intercept → click until it reads "Intercept is on"
  3. In Firefox, navigate to http://<your-ip>/vulnerabilities/brute/
  4. Firefox may freeze on intermediate requests — click Forward in Burp for each one until the Brute Force page loads in Firefox
  5. You should see a small form with Username and Password fields and a Login button

1B: Submit a Test Login and Capture It

  1. In the Brute Force form, enter:
    • Username: admin
    • Password: test (deliberately wrong — we just need the request structure)
  2. Click the Login button
  3. Burp catches the request — Firefox freezes
  4. Switch to Burp Suite → Proxy → Intercept tab

✅ What the Captured Request Looks Like:

GET /vulnerabilities/brute/?username=admin&password=test&Login=Login HTTP/1.1
Host: 10.0.2.15
User-Agent: Mozilla/5.0 ...
Cookie: PHPSESSID=abc123def456; security=low
Referer: http://10.0.2.15/vulnerabilities/brute/

DVWA's Brute Force module uses a GET request — credentials appear in the URL as query parameters. The password=test parameter is the one we will turn into a payload position in Task 2.

1C: Send to Intruder

  1. In Burp's Intercept panel, right-click anywhere on the request text
  2. Select "Send to Intruder"
  3. Click Forward to release the request (Firefox unfreezes and shows a failed login — expected)
  4. Turn Intercept OFF
  5. Click the Intruder tab at the top of Burp Suite — in v2025 the request appears in a numbered attack sub-tab (e.g., Attack 1) inside the Intruder view; click that sub-tab to proceed
💡

v2025 Shortcut: Send Directly from HTTP History

In Burp Suite v2025 you can also send a request to Intruder directly from Proxy → HTTP history: find the brute-force GET request in the history list, right-click it, and select "Send to Intruder". This skips the Intercept step. Either method works — the Intercept method above is more instructive for understanding the full workflow.

✅

Checkpoint

The Intruder tab should now contain the captured request. You may see a small orange dot or asterisk on the Intruder tab indicating a request is waiting. Proceed to Task 2 to configure the attack.

✍️ Task 1 — Questions

🔑 Answer Key — Task 1

  1. Q1: GET method. Credentials appear in the URL as query parameters: ?username=admin&password=test&Login=Login.
  2. Q2: GET credentials are visible in the URL, stored in browser history, and written to server access logs. Any system or person with log access can read the password in plaintext. POST sends credentials in the body, excluded from standard URL logs.
  3. Q3: PHPSESSID=<value>; security=low. The session cookie proves the requests come from an authenticated session. Intruder automatically carries this cookie (taken from the original captured request) in every attack request, so each attempt is processed by the server as coming from a valid logged-in user.

⚙️ Task 2: Configure Intruder

💡

Concept — Payload Positions

A payload position is a location in the request that Intruder will replace with each wordlist entry. Positions are marked with the § symbol (section sign). Burp auto-detects all parameter values and wraps them in § markers. You will clear those and mark only the password value manually — keeping the username fixed as admin.

Burp Suite: Intruder Tab

2A: Check the Positions Sub-Tab

  1. Click the Intruder tab at the top of Burp Suite, then click the numbered attack sub-tab (e.g., Attack 1) that was created when you sent the request to Intruder
  2. The Positions view is shown by default — this contains the request editor with the Attack type dropdown at the top and the Add §, Clear §, and Auto § buttons on the right side
  3. The captured request is displayed with several values highlighted in green — these are auto-detected payload positions
  4. The URL looks something like: ...username=§admin§&password=§test§&Login=§Login§

2B: Clear Auto-Detected Positions and Set the Password Position

  1. Click the "Clear §" button on the right — removes all auto-detected positions. The green highlights disappear. (Note: v2025 also shows an "Auto §" button — do not click it; Auto § would re-detect all parameters automatically, which is not what we want here.)
  2. In the request text, locate password=test
  3. Click and drag to select only the word test (the value, not the parameter name)
  4. Click the "Add §" button — it becomes password=§test§ highlighted in green
  5. The URL should now read: ...username=admin&password=§test§&Login=Login

✅ Correct Positions Configuration:

GET /vulnerabilities/brute/?username=admin&password=§test§&Login=Login HTTP/1.1
Host: 10.0.2.15
Cookie: PHPSESSID=abc123def456; security=low

Only the password value is highlighted in green. The username stays fixed as admin for every request. Intruder will substitute each wordlist entry in place of test.

2C: Set the Attack Type

  1. At the top of the Positions view (above the request editor), find the Attack type dropdown — in v2025 this is prominently displayed at the top of the attack configuration area
  2. Select Sniper (may already be the default)
  3. Sniper is correct: one wordlist, one position (the password field)

2D: Load the Payload List

  1. Click the Payloads tab to open the Payloads side panel — in v2025 this panel appears on the right side of the Intruder attack view and can be toggled open or closed
  2. Under Payload type, confirm Simple list is selected
  3. Under Payload configuration (labeled Payload settings in older Burp versions), click "Load"
  4. Navigate to your home directory (/home/<your-username>/) and select dvwa_wordlist.txt
  5. All 15 passwords should now appear in the payload list box
💡

Alternative: Paste Passwords Directly

If the file browser does not work as expected, you can type or paste passwords one at a time into the "Enter a new item" field and click Add for each one. The result is identical.

2E: Disable URL Encoding (Recommended)

  1. Scroll down in the Payloads panel to find the Payload encoding section (at the bottom of the panel, below the payload list)
  2. Uncheck "URL-encode these characters"
  3. This prevents Burp from encoding characters in passwords (e.g., turning ! into %21). Not critical for this wordlist, but correct practice.
✅

Configuration Summary

Attack type: Sniper  •  Payload position: §test§ in the password field  •  Payload list: 15 common passwords loaded from dvwa_wordlist.txt via the Payloads side panel → Payload configuration → Load. You are ready to start the attack.

✍️ Task 2 — Questions

🔑 Answer Key — Task 2

  1. Q1: Sniper. One payload list, one position. We know the username is admin so we only need to cycle through passwords. Pitchfork and Cluster Bomb are for multiple positions with multiple lists, which is unnecessary here.
  2. Q2: Symbol: § (section sign). After setting position: password=§test§
  3. Q3: Auto-detection marked username, password, and Login values as positions. With Sniper and three positions, Burp would waste requests cycling through the username and the Login button text. We want to fuzz only the password field with the username fixed as admin, so all other positions must be removed.

📈 Task 3: Run the Attack & Analyze Results

💡

Concept — How to Identify the Correct Password

The server returns HTTP 200 OK whether the login succeeds or fails — DVWA does not redirect on success at Low security level. You cannot use the status code to find the answer. Instead, look at the response length: a failed login returns a short error message, while a successful login returns a page with a welcome message. The correct password will produce a noticeably different response length.

3A: Start the Attack

  1. Click the "Start attack" button (top-right area of the Intruder attack configuration)
  2. A new window (or separate tab, depending on your Burp version) opens — the Intruder attack results table
  3. Burp begins sending requests automatically at approximately 1 per second (Community Edition rate limit)
  4. A rate-limiting notice will appear — in v2025 this is a banner or status message such as "Burp Intruder is running in Community Edition mode with throttled attack speed" or similar; the exact wording may vary by version but the meaning is the same — this is expected
  5. Watch the table fill in row by row as each request completes
  6. Wait for all 15 requests to complete

3B: Find the Correct Password by Response Length

  1. Once all 15 requests finish, look at the Length column in the results table
  2. Click the Length column header to sort responses by size
  3. One row will have a noticeably different (larger) length compared to all others
  4. The Payload column for that row shows the correct password

Attack Complete

The correct password is password. Burp Intruder found it by sending 15 automated requests and identifying the one with a different response length. This is the exact technique security professionals use to test how well a login form resists automated password attacks.

✍️ Task 3 — Questions

🔑 Answer Key — Task 3

  1. Q1: Correct password: password. Total requests: 15 (one per wordlist entry in Sniper mode).
  2. Q2: DVWA returns HTTP 200 for both success and failure at Low security — no redirect, no different code. The distinguishing factor is response length: the success page is longer because it contains the welcome message that the failure page does not.
  3. Q3: “Welcome to the password protected area admin”
  4. Q4: 14,000,000 ÷ 1 req/sec ≈ 162 days at Community Edition speed. Burp Professional at 100 req/sec: ~1.6 days. Dedicated hardware at 1,000 req/sec: ~4 hours. Key insight: even weak throttling makes large wordlist attacks slow, but common passwords like “password” appear very early in any wordlist and are found almost immediately.

🛡️ Task 4: Countermeasures & Defense

💡

Concept — Why the Attack Worked

The attack succeeded because DVWA at Low security has zero protections: no delay between attempts, no lockout after failures, no CAPTCHA, no CSRF token. Real-world applications must implement layered defenses. DVWA's higher security levels let you observe what these defenses look like from an attacker's perspective.

4A: Switch to Medium Security — Observe the Rate Delay

  1. In DVWA, click DVWA Security in the left menu
  2. Change the security level to Medium and click Submit
  3. In Burp Intruder, click "Start attack" again (same configuration)
  4. Observe: each row takes noticeably longer to appear than before
  5. The attack still completes — but much more slowly

✅ What DVWA Medium Security Does:

DVWA Medium adds a 2-second server-side sleep before responding to each brute force attempt. Every request takes 2+ seconds instead of milliseconds. For 15 passwords the attack now takes ~30 seconds. For rockyou.txt at 2 seconds per guess: over 324 days — making automated attacks with large wordlists impractical under time constraints.

4B: Switch to High Security — Observe the CSRF Token

  1. Change DVWA Security to High and click Submit
  2. Return to the Brute Force page in Firefox (with Intercept ON)
  3. Submit the login form again and capture the new request in Burp
  4. Look at the URL in the captured request — notice an extra parameter
  5. Try running Intruder with the same old configuration against High security — all attempts will fail even if the wordlist contains the correct password

✅ What the High Security Request Contains:

GET /vulnerabilities/brute/?username=admin&password=test
    &Login=Login&user_token=a3f9b2c1d4e5f608 HTTP/1.1
Host: 10.0.2.15
Cookie: PHPSESSID=abc123; security=high

High security adds a CSRF (Cross-Site Request Forgery) token — a random value generated fresh for each page load and valid for only one request. Intruder reuses the same token for all 15 attempts. The server accepts the first request, then rejects all subsequent ones because the token has already been consumed. The correct password cannot be identified even when it is in the wordlist.

4C: Real-World Defenses Against Password Attacks

DefenseHow It WorksEffectiveness
Account Lockout After N failed attempts, the account is locked for a time or until manually reset by an admin High — directly stops automated bulk guessing
Rate Limiting Server enforces a delay between login attempts from the same IP or account Medium — slows attacks significantly; can be bypassed with distributed attacks
CAPTCHA Visual or behavioral challenge after failed attempts that humans can solve but bots cannot Medium — stops dumb automation; advanced bots can sometimes bypass
CSRF Tokens One-use random token per request prevents replay and automated submissions without prior page load High for simple automation — attackers must also scrape tokens, requiring more complex tools
Strong Password Policy Enforce minimum length and complexity so user passwords don't appear in common wordlists High — the first and most fundamental defense
Multi-Factor Authentication Even if the password is discovered, a second factor (TOTP app, hardware key) blocks access Very High — renders password-only attacks useless

✍️ Task 4 — Questions

🔑 Answer Key — Task 4

  1. Q1: 2-second server-side sleep per attempt. For 15 passwords: ~30 seconds. For 14 million passwords at 2 sec/guess: ~324 days. Rate limiting does not prevent attacks but makes large-scale dictionary attacks impractical within any reasonable timeframe, especially combined with other defenses.
  2. Q2: user_token=<random_value> — a CSRF token. It is single-use. The Intruder attack reuses the token from the originally captured request. The server accepts only the first request (token consumed), then rejects all 14 remaining requests because their token is invalid. The correct password is present in the list but cannot be identified because all responses look like failures.
  3. Q3: Accept any three from the table. Key reasoning: MFA makes password knowledge useless; account lockout terminates automated attempts after few tries; strong password policy ensures the password is not in any wordlist. CSRF tokens are also highly effective against simple automation.
  4. Q4: Ideal combination: rate limiting + account lockout (5–10 failures) + MFA + strong password requirements + CSRF tokens. This is defense-in-depth: each layer compensates for the others' weaknesses. MFA is the most critical because it protects even after a password breach.

🔧 Troubleshooting

❌ Intruder tab does not contain the request from Task 1

Solution: The request was not sent to Intruder. Return to Burp Proxy → HTTP History, find the Brute Force GET request (URL contains /vulnerabilities/brute/), right-click it, and select "Send to Intruder". Alternatively, repeat Task 1B with Intercept ON and right-click the intercepted request.

❌ After clicking "Add §" nothing is highlighted in green

Solution: You must select text first by clicking and dragging to highlight the word test in the request, then click Add §. If no text is selected when you click Add §, nothing happens. Try clicking at the start of test, holding the mouse button, and dragging to the end of test before releasing.

❌ All 15 results show the same response length — no password stands out

Solution: The session cookie has likely expired. Log back into DVWA (admin / password), then navigate to the Brute Force page, capture a fresh request with Intercept ON, and send the new request to Intruder. Also confirm DVWA security is set to Low — Medium/High change response behavior.

❌ Intruder attack shows "Connection refused" for all requests

Solution: DVWA stopped running. Check: sudo docker ps. If DVWA is not listed: sudo docker start dvwa. Also confirm the Host header in the Intruder request shows your machine IP (from hostname -I), not localhost.

❌ Cannot load the wordlist file in the Intruder Payloads tab

Solution: Confirm the file exists: cat ~/dvwa_wordlist.txt. If it is missing, re-run the cat > ~/dvwa_wordlist.txt << 'EOF' command from the Setup section. Alternatively, in the Payloads tab, type each password into the "Enter a new item" field and click Add for each one.

❌ DVWA Brute Force page does not appear — redirected to login

Solution: Your DVWA session has expired. Log back into DVWA (admin / password), then navigate to the Brute Force module again. Make sure FoxyProxy is still active so the new session cookie is captured by Burp.

❌ After the lab, Firefox has no internet access

Solution: FoxyProxy is still active and routing traffic to Burp, which is now closed. Click the FoxyProxy icon in the Firefox toolbar and select "Disable FoxyProxy". Do this at the end of every lab session.

❌ Intruder interface looks different from the lab description (v2025 UI)

Solution: Burp Suite v2025 features a redesigned Intruder interface. Key differences from older versions: (1) After sending a request to Intruder, it appears in a numbered attack sub-tab (e.g., Attack 1) inside the Intruder view — click that sub-tab to see the Positions editor. (2) The Payloads configuration is now in a collapsible side panel — click the Payloads tab/button on the right side to expand it. (3) A new "Auto §" button appears alongside "Add §" and "Clear §" — do not use Auto § for this lab (it would mark all parameters automatically). The core workflow (positions → payloads → start attack) is unchanged.

❌ Cannot find "Payload settings" or "Payload encoding" in the Payloads panel (v2025)

Solution: In Burp Suite v2025, the section previously called "Payload settings" is now labeled "Payload configuration". The "Payload encoding" section with the "URL-encode these characters" checkbox is still present — scroll down to the bottom of the Payloads side panel to find it. If the panel appears too narrow, drag its left border to resize it.

❌ Cannot find Proxy listener settings (v2025)

Solution: In Burp Suite v2025 the proxy listener settings can be reached two ways: (1) Proxy tab → click the "Proxy settings" button near the top of the tab. (2) Click the Settings gear icon (top-right corner of Burp) → Tools → Proxy. The listener should show 127.0.0.1:8080 as Running. If it shows Not running, click the toggle next to it to start it.
