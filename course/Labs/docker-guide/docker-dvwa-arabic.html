🐳 ما هو Docker؟

Docker منصة مفتوحة المصدر تتيح لك تغليف التطبيقات وتوزيعها وتشغيلها داخل بيئات معزولة وخفيفة الوزن تُعرف بـ الحاويات (Containers). بدلاً من تثبيت التطبيق مباشرةً على نظام التشغيل — مما قد يُسبب تعارضات في المكتبات أو يُعطِّل أدوات أخرى — يُغلِّف Docker كل تطبيق مع كل ما يحتاجه (الشيفرة، المكتبات، الإعدادات) داخل وحدة مكتفية بذاتها.

فكِّر في Docker كـ حاوية شحن في العالم الحقيقي: يمكن تحميل الحاوية ذاتها على أي سفينة أو شاحنة أو قطار دون أن تُعنى بما يوجد تحتها. وبالمثل، تعمل حاوية Docker بالطريقة نفسها على أي جهاز مثبَّت عليه Docker بغض النظر عن نظام التشغيل.

📦 بيئة موحَّدة

يعمل DVWA بالطريقة ذاتها على جهاز كل طالب — لا مشكلات "يعمل عندي ولا يعمل عندك".

🔒 عزل آمن

يعمل التطبيق المعرَّض للثغرات داخل حاوية، معزولاً بأمان عن نظام Kali الحقيقي.

⚡ إعداد سريع

أمر واحد يُشغِّل خادم ويب متكامل مع قاعدة بيانات وPHP وDVWA — دون أي تثبيت يدوي.

🔄 إعادة ضبط سهلة

احذف الحاوية وأعِد إنشاءها للحصول على نسخة DVWA جديدة تماماً في ثوانٍ.

💡

لماذا Docker في مختبرات الأمن؟

تشغيل تطبيق معرَّض للثغرات مثل DVWA مباشرةً على النظام الأصلي محفوف بالمخاطر. يبقيه Docker معزولاً: إذا حدث خطأ داخل الحاوية، يبقى نظام Kali سليماً. كما يمكنك تدمير الحاوية وإعادة إنشائها فورياً.

🔑 المفاهيم الأساسية في Docker

قبل العمل مع Docker، عليك فهم هذه المصطلحات الأربعة الجوهرية:

المصطلح ما هو؟ التشبيه الواقعي
Image (صورة) قالب للقراءة فقط — مخطط لإنشاء الحاويات. يحتوي على نظام التشغيل والتطبيق وجميع الاعتمادات، محجوزاً في الوقت. وصفة طبخ أو مخطط بناء منزل
Container (حاوية) نسخة حية جارية تُنشأ من صورة. يمكن تشغيلها وإيقافها وإعادة تشغيلها. يمكن إنشاء حاويات متعددة من الصورة ذاتها. المنزل الفعلي المبني من المخطط
Docker Hub سجل إلكتروني حيث تُنشر صور Docker ويتم تشاركها علناً. مثل GitHub لكن لصور Docker. متجر تطبيقات للبيئات الجاهزة
Docker Daemon الخدمة التي تعمل في الخلفية وتُدير جميع عمليات Docker على جهازك. يجب تشغيلها لأي أمر Docker. مدير المصنع الذي يبني الحاويات ويشغِّلها
Docker Engine برنامج Docker الكامل — يشمل الخدمة وواجهة الأوامر والAPI. عند كتابة sudo docker ... تتواصل مع المحرك. مبنى المصنع بأكمله

في هذا المقرر

أنت تُشغِّل Kali Linux داخل VirtualBox (جهاز افتراضي)، ثم تُشغِّل DVWA داخل Docker (حاوية) داخل Kali. حاويات Docker متداخلة داخل جهازك الافتراضي.

🎯 ما هو DVWA؟

DVWA (Damn Vulnerable Web Application) هو تطبيق ويب PHP/MySQL معرَّض للثغرات عمداً، مُصمَّم خصيصاً لطلاب وممارسي أمن المعلومات لتدريب هجمات تطبيقات الويب في بيئة قانونية وآمنة ومضبوطة. كل ثغرة في DVWA مقصودة — طوَّره المطوِّرون معطوباً عن سبق إصرار وترصُّد.

الخاصيةالتفاصيل
الاسم الكاملDamn Vulnerable Web Application
التقنيةPHP + MySQL (خادم Apache)
الغرضممارسة ثغرات الويب الشائعة بشكل قانوني
صورة Dockervulnerables/web-dvwa على Docker Hub
المنفذ الافتراضي80 (HTTP)
عنوان الوصولhttp://localhost أو http://127.0.0.1
⚠️

تحذير قانوني

يجب استخدام DVWA فقط في بيئات مختبرية معزولة على أجهزة تتحكم فيها. لا تعرِّضه للإنترنت أو شبكة حقيقية. الثغرات في DVWA حقيقية ويمكن استغلالها من قِبَل آخرين إذا نُشر علناً.

🧩 وحدات الثغرات في DVWA

يتضمن DVWA مجموعة من وحدات الثغرات، تُظهر كل منها نوعاً مختلفاً من هجمات الويب. ستستخدم معظمها في CYB236:

الوحدة نوع الثغرة ما ستتعلمه
🔐 Brute Force ضعف في المصادقة تخمين كلمات المرور آلياً باستخدام قوائم الكلمات (المختبر 07 مع Burp Intruder)
💉 Command Injection حقن أوامر نظام التشغيل حقن أوامر النظام عبر نموذج ويب لتنفيذها على الخادم
🔗 CSRF تزوير الطلبات عبر المواقع خداع متصفح مستخدم مُسجَّل دخوله لإرسال طلبات غير مصرَّح بها
📂 File Inclusion تضمين الملفات المحلية/البعيدة تضمين ملفات غير مقصودة (مثل /etc/passwd) عبر معاملات URL
📤 File Upload رفع ملفات غير مقيَّد رفع ملفات خبيثة (كـ PHP shell) للوصول إلى الخادم
🤖 Insecure CAPTCHA ضعف في التحقق تجاوز التحقق بـ CAPTCHA بسبب ثغرات منطق الخادم
💾 SQL Injection حقن SQL التلاعب باستعلامات قاعدة البيانات عبر حقول الإدخال لاستخراج البيانات
🔍 SQL Injection (Blind) حقن SQL الأعمى استخراج بيانات قاعدة البيانات حين لا يُظهر التطبيق أخطاء مباشرة
🪪 Weak Session IDs رموز جلسة متوقَّعة فهم كيفية تخمين رموز الجلسة الضعيفة واختطافها
📜 XSS (DOM) XSS على مستوى DOM حقن سكريبتات عبر DOM المتصفح دون الوصول إلى الخادم
📢 XSS (Reflected) XSS المنعكس حقن سكريبتات تُعاد فوراً في الاستجابة
💬 XSS (Stored) XSS المخزَّن حقن سكريبتات تُخزَّن دائماً وتُقدَّم لمستخدمين آخرين
🔎 JavaScript إساءة استخدام المنطق من جانب العميل التلاعب بـ JavaScript من جانب العميل لتجاوز التحقق

🛡️ مستويات الأمان في DVWA

يمكن تشغيل كل وحدة في DVWA على أربعة مستويات أمان. يتيح ذلك لك فهم الثغرة بأبسط أشكالها أولاً، ثم رؤية كيف تجعل الدفاعات الحقيقية استغلالها أصعب تدريجياً.

يمكنك تغيير مستوى الأمان من قائمة "DVWA Security" في الشريط الجانبي الأيسر بعد تسجيل الدخول.

المستوىالوصفمتى تستخدمه
Low لا ضوابط أمان على الإطلاق. يُمرَّر الإدخال مباشرةً إلى الخلفية دون أي تحقق أو تطهير. تعمل الاستغلالات بحمولات بسيطة. تعلُّم الهجوم للمرة الأولى. جميع تمارين المختبر تبدأ من هنا.
Medium تُطبَّق دفاعات أساسية (مثل القائمة السوداء لبعض الأحرف، فلترة مدخلات بسيطة). هذه الدفاعات شائعة لكنها معيبة — يمكن تجاوزها. فهم سبب فشل الدفاعات الأساسية وكيف يتحايل عليها المهاجمون.
High دفاعات أقوى (مثل القائمة البيضاء، الاستعلامات المُعلَّمة، حماية CSRF بالرموز). الاستغلال يتطلب أساليب أكثر تقدماً. رؤية ما يبدو عليه أمن أفضل، وكيف يمكن أحياناً هزيمته.
Impossible الثغرة مُصلَحة بالكامل. تمثِّل الشيفرة المصدرية أفضل ممارسات التطوير الآمن. الاستغلال مستحيل. دراسة كيفية كتابة شيفرة آمنة. المقارنة مع مستوى Low لفهم الخطأ.
⚠️

ابدأ دائماً من Low

ما لم يحدد المدرب خلاف ذلك، اضبط DVWA دائماً على مستوى Low قبل بدء أي تمرين. المستويات الأعلى قد تمنع الهجوم المتوقَّع من العمل مما يُسبب ارتباكاً.

📄 عرض الشيفرة المصدرية

من أقوى ميزات DVWA التعليمية زرّ "View Source" في أسفل صفحة كل وحدة. يعرض الشيفرة PHP الفعلية التي تعمل خلف الوحدة عند كل مستوى أمان — حتى ترى بالضبط ما يجعل الشيفرة معرَّضة للثغرات أو آمنة.

🔌 فهم تمرير المنافذ (Port Mapping)

عند تشغيل DVWA بالأمر docker run -d -p 80:80 ...، يُخبر الجزء -p 80:80 Docker بكيفية ربط جهاز Kali بالحاوية. يُسمى هذا تمرير المنافذ (Port Mapping).

ما هو المنفذ؟

المنفذ (Port) هو قناة اتصال مُرقَّمة على الجهاز. عند تواصل المتصفح مع خادم الويب، يستخدم أرقام منافذ محددة:

المنفذالبروتوكولالاستخدام
80HTTPحركة الويب غير المشفَّرة — يستخدمه DVWA وJuice Shop
443HTTPSحركة الويب المشفَّرة (SSL/TLS)
3000HTTP (مخصص)يستخدمه بعض التطبيقات مثل Juice Shop كمنفذ بديل
8080HTTP (بديل)بديل شائع للمنفذ 80، يستخدمه Burp Suite كـ Proxy
3306MySQLاتصالات قاعدة البيانات — يستخدمه DVWA داخلياً

كيف يعمل -p 80:80؟

متصفحك
http://localhost
→ منفذ 80 →
Kali Linux
منفذ المضيف 80
→ يُمرَّر إلى →
حاوية Docker
منفذ الحاوية 80

الصيغة هي منفذ_المضيف:منفذ_الحاوية. كلاهما 80 هنا، لكن يمكنك استخدام منفذ مضيف مختلف لتجنب التعارضات — مثلاً -p 8081:80 يجعل DVWA متاحاً على http://localhost:8081.

ما هو localhost؟

localhost و127.0.0.1 كلاهما عناوان خاصة تُشير دائماً إلى جهازك الخاص. عند طلب المتصفح في Kali الخاص بك http://localhost، يتصل بالمنفذ 80 على Kali نفسه — والذي يُعيد Docker توجيهه إلى حاوية DVWA.

💡

لماذا تحدث تعارضات المنافذ؟

يمكن لعملية واحدة فقط الاستماع على منفذ مضيف معين في آنٍ واحد. إذا كان DVWA يستخدم المنفذ 80 وحاولت تشغيل Juice Shop على المنفذ 80 أيضاً، يرفض Docker ذلك. هذا أكثر الأخطاء شيوعاً في هذا المقرر.

المرة الأولى: تهيئة قاعدة البيانات

في المرة الأولى (أو بعد إعادة إنشاء الحاوية)، ستصل إلى صفحة الإعداد بدلاً من صفحة الدخول. انتقل إلى أسفل الصفحة واضغط "Create / Reset Database". سيُعيد DVWA توجيهك تلقائياً إلى صفحة الدخول.

  4. اضغط "Submit"
  5. أنت الآن جاهز لاستخدام أي وحدة ثغرات

🔄 دورة حياة حاوية Docker

تمرُّ حاوية Docker بحالات متميزة. فهم هذه الحالات يُفسِّر لك سبب استخدام start أحياناً بدلاً من run، وسبب حدوث تعارض في اسم الحاوية.

لا شيء موجود
← docker run ←
جارية
docker ps يُظهرها
← stop/kill ←
متوقفة (Exited)
docker ps -a يُظهرها
← docker rm ←
لا حاوية
محذوفة نهائياً
الحالةما تعنيهأمر التحقق
لا حاوية لم تُنشأ الحاوية قط، أو حُذفت بـ docker rm. استخدم docker run لإنشائها. docker ps -a لا يُظهر شيئاً
جارية الحاوية نشطة وتخدم الطلبات. DVWA متاح في المتصفح. docker ps ← STATUS: Up
متوقفة / Exited الحاوية موجودة على القرص لكنها لا تعمل. استخدم docker start (وليس docker run) لإعادة تشغيلها. docker ps -a ← STATUS: Exited
⚠️

run مقابل start — الخطأ الأكثر شيوعاً

docker run ينشئ حاوية جديدة كلياً من صورة. إذا كانت حاوية بهذا الاسم موجودة (حتى لو متوقفة)، سيفشل run بخطأ "الاسم مستخدَم". استخدم docker start dvwa لإعادة تشغيل حاوية متوقفة موجودة.

📋 عرض الحاويات

ابدأ دائماً من هنا عند عدم عمل DVWA. اعرف ما يعمل قبل فعل أي شيء آخر.

▶ عرض الحاويات الجارية فقط

sudo docker ps

يعرض الحاويات النشطة حالياً. أعمدة الناتج تعني:

العمودالمعنى
CONTAINER IDمعرِّف مختصر (مثل a1b2c3d4e5f6)
IMAGEالصورة المستخدمة (مثل vulnerables/web-dvwa)
STATUSمدة التشغيل (مثل Up 3 minutes)
PORTSتمرير المنافذ (مثل 0.0.0.0:80->80/tcp)
NAMESاسم الحاوية (مثل dvwa)

📂 عرض جميع الحاويات (بما فيها المتوقفة)

sudo docker ps -a

الخيار -a يشمل الحاويات المتوقفة والخارجة. استخدمه للعثور على حاوية DVWA من جلسات سابقة.

🔍 فحص حاوية بعينها

sudo docker inspect dvwa

يعرض معلومات JSON مفصَّلة عن الحاوية: إعدادات الشبكة، التحميلات، متغيرات البيئة. مفيد لتصحيح الأخطاء المتقدم.

📊 عرض استهلاك الموارد مباشرةً

sudo docker stats dvwa

يعرض استهلاك CPU والذاكرة في الوقت الفعلي. اضغط Ctrl+C للخروج.

✅

قراءة الناتج

إذا قال STATUS كلمة Up → DVWA يعمل. إذا قال Exited → نفِّذ docker start dvwa. إذا لم يكن هناك صف → نفِّذ docker run ... لإنشائه.

🚀 تشغيل DVWA

الخيار أ — المرة الأولى (لا حاوية موجودة)

إذا لم يُظهر docker ps -a أي حاوية DVWA، أنشئها وشغِّلها:

sudo docker run -d -p 80:80 --name dvwa vulnerables/web-dvwa
الخيارالمعنى
runأنشئ حاوية جديدة من صورة وشغِّلها
-dمنفصل: شغِّل في الخلفية وأعِد الطرفية إليك
-p 80:80مرِّر منفذ المضيف 80 → منفذ الحاوية 80
--name dvwaسمِّ الحاوية "dvwa" للرجوع إليها بسهولة
vulnerables/web-dvwaالصورة المستخدمة من Docker Hub

الخيار ب — الحاوية موجودة لكنها متوقفة

إذا أظهر docker ps -a الحالة Exited، أعِد تشغيلها دون إعادة إنشاء:

sudo docker start dvwa

الخيار ج — إعادة تشغيل حاوية جارية

إذا كانت DVWA تعمل لكنها تتصرف بشكل غريب (مثل أخطاء قاعدة البيانات):

sudo docker restart dvwa

التحقق من إمكانية الوصول

  1. نفِّذ sudo docker ps — تأكد من ظهور dvwa بحالة Up
  2. افتح Firefox في Kali
  3. انتقل إلى http://localhost
  4. إذا ظهرت صفحة الإعداد، اضغط "Create / Reset Database"
  5. سجِّل الدخول بـ admin / password

🛑 إيقاف الحاويات وحذفها

الإيقاف الآمن

يُرسل إشارة إنهاء وينتظر حتى 10 ثوانٍ لإغلاق الحاوية بأمان:

sudo docker stop dvwa

الإيقاف القسري

يُنهي الحاوية فوراً — بلا انتظار. استخدمه حين لا يستجيب stop:

sudo docker kill dvwa

حذف حاوية متوقفة

يحذف الحاوية من القرص. لا تُحذف الصورة — يمكنك docker run مجدداً لإعادة الإنشاء:

sudo docker rm dvwa

إيقاف وحذف بأمر واحد

sudo docker rm -f dvwa

حذف جميع الحاويات المتوقفة دفعةً واحدة

sudo docker container prune
⚠️

stop مقابل kill مقابل rm

  • stop — يطلب بأدب من الحاوية التوقف (SIGTERM، ثم SIGKILL بعد 10 ثوانٍ)
  • kill — يُنهي الحاوية فوراً (SIGKILL)
  • rm — يحذف الحاوية من النظام (يجب أن تكون متوقفة)
  • rm -f — يقتل ويحذف في خطوة واحدة

⚠️ تعارض الحاويتين (المنفذ مستخدَم)

هذه هي المشكلة الأكثر شيوعاً في المقرر. المختبر 02 يستخدم Juice Shop على المنفذ 80. المختبرات 03–10 تستخدم DVWA على المنفذ 80. يمكن لحاوية واحدة فقط امتلاك المنفذ 80 في آنٍ واحد.

docker: Error response from daemon: driver failed programming external connectivity...
Bind for 0.0.0.0:80 failed: port is already allocated.

تشخيص التعارض

sudo docker ps

انظر إلى عمود PORTS. أيُّ حاوية تُظهر 0.0.0.0:80->80/tcp هي التي تشغل المنفذ 80.

الإصلاح: أوقِف الحاوية المتعارضة ثم شغِّل DVWA

sudo docker stop juice-shop
sudo docker start dvwa

التبديل بين التطبيقين

التبديل إلى Juice Shopsudo docker stop dvwa && sudo docker start juice-shop
إيقاف كل شيءsudo docker stop $(sudo docker ps -q)

🔧 المشكلات الشائعة وحلولها

❌ خطأ "port is already allocated"

السبب: حاوية أخرى (Juice Shop أو غيرها) مرتبطة بالمنفذ 80.

sudo docker ps
sudo docker stop juice-shop
sudo docker start dvwa
❌ خطأ "The container name /dvwa is already in use"

السبب: حاوية متوقفة باسم "dvwa" موجودة. حاولت docker run بدلاً من docker start.

الإصلاح — شغِّل الموجودة فقط:

sudo docker start dvwa

أو احذفها وأعِد الإنشاء:

sudo docker rm dvwa
sudo docker run -d -p 80:80 --name dvwa vulnerables/web-dvwa
❌ DVWA تعرض "Cannot connect to the database"

السبب: MySQL داخل الحاوية لم ينتهِ من التهيئة بعد (يستغرق 10–15 ثانية).

الإصلاح: انتظر 15 ثانية ثم حدِّث المتصفح. إذا استمرت المشكلة:

sudo docker restart dvwa

ثم انتقل إلى http://localhost/setup.php → اضغط Create / Reset Database.

❌ المتصفح يعرض "This site can't be reached" أو رفض الاتصال

السبب: حاوية DVWA لا تعمل، أو عنوان خاطئ.

sudo docker ps          # تحقق إذا كانت dvwa تعمل
sudo docker start dvwa  # إذا لم تكن تعمل

تأكد من استخدام http://localhost (وليس https://).

❌ الحاوية متجمِّدة — docker stop لا يفعل شيئاً

السبب: الحاوية مُعلَّقة وتتجاهل إشارة SIGTERM.

sudo docker kill dvwa
sudo docker start dvwa
❌ "Cannot connect to the Docker daemon" أو "docker: command not found"

السبب: Docker غير مثبَّت أو الخدمة لا تعمل.

sudo apt update && sudo apt install -y docker.io
sudo systemctl start docker
sudo systemctl enable docker
sudo systemctl status docker
❌ خطأ "Permission denied" عند تشغيل أوامر docker

السبب: نسيت sudo، أو المستخدم غير مضاف إلى مجموعة docker.

الإصلاح — استخدم sudo دائماً في Kali:

sudo docker ps

أو أضِف المستخدم إلى مجموعة docker (حل دائم):

sudo usermod -aG docker $USER
newgrp docker
❌ فشل تسجيل الدخول إلى DVWA رغم صحة البيانات

السبب: لم تُهيَّأ قاعدة البيانات بعد بدء تشغيل الحاوية.

الإصلاح: انتقل إلى http://localhost/setup.php واضغط "Create / Reset Database". ثم سجِّل الدخول بـ admin / password.

❌ Burp Suite لا يعترض حركة DVWA عند استخدام localhost

السبب: تتجاوز المتصفحات البروكسي تلقائياً عند استخدام localhost أو 127.0.0.1. هذه قاعدة أمان مدمجة في المتصفح — عناوين الـ Loopback لا تُرسَل أبداً عبر البروكسي، لذا لن يرى Burp Suite الطلبات حتى لو كان FoxyProxy مفعَّلاً.

الإصلاح — استخدم عنوان IP الحقيقي للجهاز بدلاً من localhost:

الخطوة 1: اكتشف عنوان IP الحقيقي لجهاز Kali:

hostname -i

يعيد هذا الأمر عنواناً مثل 10.0.2.15 أو 192.168.x.x — وهو العنوان الشبكي الفعلي لجهازك الافتراضي.

الخطوة 2: افتح DVWA في المتصفح باستخدام ذلك العنوان بدلاً من localhost:

http://10.0.2.15

استبدل 10.0.2.15 بالعنوان الذي أعاده hostname -i على جهازك.

الخطوة 3: تأكد من إعداد بروكسي Burp Suite للاستماع على 127.0.0.1:8080 (الافتراضي)، ومن إعداد FoxyProxy لتوجيه الحركة عبر 127.0.0.1 المنفذ 8080.

الخطوة 4: تصفَّح DVWA باستخدام عنوان IP الحقيقي. سيعترض Burp Suite الطلبات بشكل طبيعي.

💡

لماذا يعمل هذا؟

عند استخدام عنوان IP الحقيقي (10.x.x.x)، يتعامل المتصفح مع الطلب كطلب شبكة عادي ويُرسله عبر البروكسي المُعدَّ (Burp Suite). أما عند استخدام localhost أو 127.0.0.1، فيختصر المتصفح الطريق مباشرةً إلى واجهة Loopback — متجاوزاً البروكسي كلياً.

⚠️

قد يتغير عنوان IP بين الجلسات

يُخصِّص VirtualBox العنوان ديناميكياً. نفِّذ دائماً hostname -i في بداية كل جلسة مختبر لتأكيد العنوان الحالي قبل استخدامه في المتصفح.
