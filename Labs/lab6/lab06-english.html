💡 Introduction

In Lab 05 you learned what Burp Suite is, how a proxy works, and how to configure FoxyProxy so your browser traffic flows through Burp. You intercepted your first HTTP request and verified the setup is working.

In this lab you will apply that knowledge against a real vulnerable web application: DVWA (Damn Vulnerable Web Application). You will use Burp Suite to capture login traffic, analyze raw HTTP requests and responses, use the Repeater tool to manually test a parameter, and build an attack surface map using the Site Map.

⚠️

Authorized Use Only

All tasks are performed against DVWA running locally on your own machine. Using Burp Suite against any system you do not own or have written permission to test is illegal.

💻

What is DVWA?

DVWA (Damn Vulnerable Web Application) is a PHP/MySQL application intentionally built with security vulnerabilities. It is used by security students and professionals to practice web exploitation techniques in a safe, legal environment. It includes modules for SQL Injection, XSS, CSRF, File Upload, Brute Force, and more.

📷 Task 1: Capture HTTP Traffic

💡

Concept — HTTP History

Every action you take in the browser generates HTTP requests. Burp's HTTP History records all of them with full detail: the request line, headers, body, and the server's complete response. This gives you far more information than the browser's DevTools, and each entry can be saved, filtered, and replayed.

1A: Generate Traffic by Browsing DVWA

In Firefox (FoxyProxy active, Intercept OFF), perform these actions one at a time:

  1. Visit http://<your-ip> — the DVWA login page
  2. Log in with admin / password
  3. Click SQL Injection in the left menu, type 1 in the User ID field, and click Submit
  4. Click XSS (Reflected), type your name, and click Submit
  5. Click Brute Force in the left menu

1B: Explore the HTTP History

  1. Switch to Burp Suite → Proxy → HTTP history
  2. You will see a list of captured requests. Each row shows:
    • # — request number (order captured)
    • Host — the server address
    • Method — GET or POST
    • URL — the path requested
    • Status — HTTP response code (200, 302, 404…)
    • Length — size of the response in bytes
  3. Click on the login POST request (look for POST /login.php)
  4. In the bottom panel, read both the Request tab and the Response tab

✅ What You Should See in the Login Request:

The Request tab will show something like:

POST /login.php HTTP/1.1
Host: 10.0.2.15
Content-Type: application/x-www-form-urlencoded
Cookie: PHPSESSID=abc123def456; security=low

username=admin&password=password&Login=Login

The Response tab will show a 302 Found redirect to /index.php — meaning the login was successful.

💡

What is the PHPSESSID Cookie?

After a successful login, the server sets a PHPSESSID cookie. This is the session token — it proves to the server who you are on every subsequent request. Any attacker who steals this value can impersonate you without knowing your password. This is called session hijacking.

✍️ Task 1 — Fill In Your Answers

🔑 Answer Key — Task 1

  1. Q1: Typically 15–30+ requests. Each page load triggers multiple requests: the HTML page, CSS files, JavaScript, and images.
  2. Q2: POST. POST sends credentials in the request body (not the URL). GET would expose the password in the address bar, browser history, server access logs, and any intermediate proxies.
  3. Q3: PHPSESSID — a 32-character hexadecimal string generated by PHP to uniquely identify the user's session.
  4. Q4: 302 Found — an HTTP redirect. After verifying credentials, the server tells the browser to navigate to the dashboard. The browser follows this redirect automatically.

📊 Task 2: Analyze Requests & Responses

💡

Concept — Reading Raw HTTP

Every HTTP request has three parts: a request line (method + path + version), headers (metadata like cookies, content type, browser info), and optionally a body (data sent with POST requests). Burp shows you all of this in raw text, exactly as the browser sends it — more detail than any browser DevTools panel.

2A: Find and Inspect the SQL Injection Request

  1. In HTTP history, look for a GET request with a URL containing /vulnerabilities/sqli/
  2. Click on it to select it
  3. In the Request tab (bottom panel), read the full raw request
  4. Notice the id=1 parameter in the URL — this is the SQL injection entry point

✅ What the SQL Injection Request Looks Like:

GET /vulnerabilities/sqli/?id=1&Submit=Submit HTTP/1.1
Host: 10.0.2.15
Cookie: PHPSESSID=abc123; security=low
Referer: http://10.0.2.15/vulnerabilities/sqli/
User-Agent: Mozilla/5.0 ...

The id=1 query parameter is sent directly to a SQL query on the server. This is the parameter a tester would manipulate to perform SQL injection.

2B: Compare GET and POST Requests

  1. Find the XSS (Reflected) request — a GET to /vulnerabilities/xss_r/
  2. Find the login request — a POST to /login.php
  3. Compare where the user's input appears: in the URL vs. in the body

2C: Check for Missing Security Headers

  1. Click any response from DVWA in HTTP history
  2. Switch to the Response tab
  3. Look at the response headers — search for X-Frame-Options, Content-Security-Policy, and X-XSS-Protection

✅ What You Should (Not) See:

DVWA is intentionally insecure. Its responses will be missing most security headers. A properly hardened application would include:

X-Frame-Options: DENY
Content-Security-Policy: default-src 'self'
X-Content-Type-Options: nosniff
Strict-Transport-Security: max-age=31536000

The absence of these headers means the application is vulnerable to clickjacking, XSS, and MIME-sniffing attacks.

✍️ Task 2 — Fill In Your Answers

🔑 Answer Key — Task 2

  1. Q1: In the URL as a query parameter: ?id=1&Submit=Submit. Method: GET.
  2. Q2: In the request body: username=admin&password=password&Login=Login. Method: POST.
  3. Q3: Content-Security-Policy (prevents XSS), X-Frame-Options (prevents clickjacking), X-Content-Type-Options (prevents MIME sniffing), Strict-Transport-Security (forces HTTPS).
  4. Q4: PHPSESSID=<value>; security=low. PHPSESSID authenticates the session. If stolen, an attacker can hijack the session without knowing the password.

⚙️ Task 3: Modify Requests with Repeater

💡

Concept — The Repeater Tool

Burp's Repeater lets you take any captured request, edit it, send it, and see the response — as many times as you like. This is essential for manually testing vulnerabilities: you can tweak a parameter, add an injection payload, change a header, or remove a cookie to see exactly how the server reacts, without touching the browser.

Burp Suite: Repeater Tab

3A: Send the SQL Injection Request to Repeater

  1. In HTTP history, right-click on the GET /vulnerabilities/sqli/?id=1 request
  2. Select Send to Repeater
  3. Click the Repeater tab at the top of Burp Suite
  4. You will see the raw request in the left panel

3B: Send the Original Request

  1. Click Send (the orange button)
  2. The server's response appears in the right panel
  3. In the Response panel, click the Render tab to see the HTML output visually

✅ Expected Response for id=1:

First name: admin
Surname: admin

The server looked up user ID 1 in the database and returned the name. This confirms the request is working correctly through Repeater.

3C: Modify the Parameter and Resend

  1. In the Repeater request panel, find id=1 in the URL
  2. Change it to id=2 and click Send — observe the different user returned
  3. Change it to id=5 and click Send again
  4. Now try id=1' (add a single quote) and click Send

✅ What Happens with id=1' (a single quote):

You have an error in your SQL syntax; check the manual that
corresponds to your MySQL server version for the right syntax
to use near ''1''' at line 1

A SQL error message is returned — this confirms the parameter is vulnerable to SQL injection. The database is leaking internal error details, which is a serious security misconfiguration.

⚠️

Why Repeater is So Powerful

With Repeater you can test dozens of payloads in seconds without touching the browser. You have full control over every byte in the request — headers, cookies, method, and body. Professional testers use Repeater to confirm vulnerabilities, test bypass techniques, and understand exactly how the server processes input.

3D: Intercept and Modify a Live Request

  1. In Burp → Proxy → Intercept, click the button to turn Intercept ON
  2. In Firefox, submit the XSS Reflected form with the name Test
  3. Firefox freezes — Burp has caught the request mid-flight
  4. In Burp's Intercept panel, change name=Test to name=Burp
  5. Click Forward — Firefox receives the modified response
  6. The page now says "Hello Burp" even though you typed "Test"
  7. Turn Intercept OFF again when done

✍️ Task 3 — Fill In Your Answers

🔑 Answer Key — Task 3

  1. Q1: First name: Gordon — Surname: Brown (DVWA user ID 2 by default).
  2. Q2: MySQL syntax error: "You have an error in your SQL syntax; check the manual..." — confirms the input is injected directly into a SQL query.
  3. Q3: SQL errors leak internal information: the database engine (MySQL), query structure, and sometimes table/column names. This helps an attacker craft a working injection payload and confirms no input validation is in place.
  4. Q4: Repeater allows editing any part of the request including headers and POST body, not just URL parameters. It sends requests instantly without page reloads, preserves cookies automatically, and shows the raw response — all in one view.

🗺️ Task 4: Map the Attack Surface with Site Map

💡

Concept — Attack Surface Mapping

Before testing an application, security professionals build a map of all reachable URLs, parameters, and endpoints. Burp's Site Map automatically organizes every URL your browser visited into a tree view, grouped by host and path. This is your application's attack surface — every entry is a potential test target.

4A: Scope the Target

  1. In Burp, click the Target tab
  2. Click the Site map sub-tab
  3. In the left panel, right-click on your DVWA host entry (e.g., http://10.0.2.15)
  4. Select "Add to scope"
  5. If Burp asks whether to stop logging out-of-scope traffic, click Yes
💡

Why Set a Scope?

Setting a scope tells Burp to focus only on your target. It prevents accidental testing of external sites (Google, CDN providers, etc.) that your browser loads in the background, and keeps the Site Map clean and focused on DVWA.

4B: Browse All DVWA Modules

In Firefox, click through every module in DVWA's left menu: Brute Force, Command Injection, CSRF, File Inclusion, File Upload, Insecure CAPTCHA, SQL Injection, SQL Injection (Blind), Weak Session IDs, XSS (DOM), XSS (Reflected), XSS (Stored).

  1. After visiting all modules, return to Burp → Target → Site map
  2. Expand the host tree (e.g., http://10.0.2.15)
  3. Expand the /vulnerabilities/ folder
  4. You should see a separate entry for each DVWA module you visited

🔧 Troubleshooting

❌ DVWA doesn't load in Firefox

Solution: Check that the Docker container is running: sudo docker ps. If DVWA is not listed, start it: sudo docker start dvwa. Make sure you are using http://<your-ip> (from hostname -I), not localhost.

❌ No requests appearing in Burp's HTTP History

Solution: Confirm FoxyProxy is enabled (Burp Proxy profile active). Confirm you are navigating to http://<your-ip> — not localhost or 127.0.0.1. Confirm Burp's proxy listener is 127.0.0.1:8080 and status shows Running. Use http:// not https://.

❌ Firefox is frozen and DVWA won't load

Solution: Burp's Intercept is ON. Go to Burp → Proxy → Intercept and click Forward (repeat for each resource) or click the button to turn Intercept OFF. For Tasks 1 and 2 you want Intercept OFF so traffic flows freely.

❌ Repeater shows "Connection refused"

Solution: DVWA stopped running. Check: sudo docker ps. If not listed: sudo docker start dvwa. Also confirm the Host header in Repeater matches your machine IP (hostname -I) — not localhost.

❌ Site Map is empty or shows only one entry

Solution: Browse DVWA more — click every module in the left menu. Ensure the host is in scope (right-click host → Add to scope). If the filter is set to "Show only in-scope items", make sure your host IP is added to scope first.

❌ After the lab, Firefox has no internet access

Solution: FoxyProxy is still active and routing traffic to Burp, which is no longer running. Click the FoxyProxy icon and select "Disable FoxyProxy". Always do this at the end of every lab session.
