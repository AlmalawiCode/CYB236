Introduction

In this lab you will take your first steps into web application reconnaissance — the information-gathering phase that happens before any security testing. You will explore a target web application using only your web browser and a few simple terminal commands.

The target application is OWASP Juice Shop, a deliberately vulnerable online juice store. It is built with the modern technologies you studied in Chapter 3.

üí°

What is OWASP Juice Shop?

Juice Shop is a modern web application intentionally built for security education. It uses:

  • Angular — a Single-Page Application (SPA) framework
  • Node.js / Express — a JavaScript web server
  • REST API with JSON data format
  • SQLite database

All of these technologies were covered in Chapter 3 of your textbook.

⚠️

Important Legal Reminder

Reconnaissance should ONLY be performed against applications you own or have written permission to test. In this lab, Juice Shop runs locally on your own machine, so you are fully authorized.

Learning Objectives

By the end of this lab, you will be able to:

  • Set up and run OWASP Juice Shop using Docker on Kali Linux
  • Identify the technology stack of a web application using simple tools
  • Use browser Developer Tools to observe REST API requests
  • Read and understand JSON data sent between client and server
  • Inspect client-side storage (cookies, localStorage, sessionStorage)
  • Write a short recon summary documenting your findings

Part 1: Setting Up OWASP Juice Shop

We will run Juice Shop using Docker inside your Kali VM.

Step 1: Install Docker

Open a terminal in your Kali VM and run these commands one by one:

# Update packages and install Docker
sudo apt update
sudo apt install -y docker.io

# Start Docker
sudo systemctl start docker
sudo systemctl enable docker

# Verify Docker is working
sudo docker --version

Expected Output:

You should see: Docker version 24.x.x, build xxxxxxx

Step 2: Download and Run Juice Shop

# Download Juice Shop (this may take a few minutes)
sudo docker pull bkimminich/juice-shop

# Start Juice Shop on port 3000
sudo docker run -d -p 3000:3000 --name juice-shop bkimminich/juice-shop

Step 3: Open It in Firefox

  1. Open Firefox in your Kali VM
  2. Go to http://localhost:3000
  3. You should see the Juice Shop homepage — an online store selling fruit juices. A welcome banner will appear — click Dismiss to close it. You may also see a cookie consent banner at the bottom — click to dismiss that as well.
  4. The homepage displays a grid of product cards with images and prices. You should see products such as Apple Juice, Apple Pomace, Banana Juice, Carrot Juice, Eggfruit Juice, Green Smoothie, Lemon Juice, and others. The Juice Shop logo appears in the top navigation bar.
✅

Checkpoint

If you can see the Juice Shop homepage with product images (like Apple Juice, Banana Juice, etc.) and a navigation bar with the Juice Shop logo, you are ready! Make sure you dismiss the welcome banner and cookie consent banner before continuing.

Task 1: Identify the Technology Stack

Your first job is to figure out what technologies this application is built with. As you learned in Chapter 3, modern web apps use a combination of web servers, frameworks, and databases.

1A: Use whatweb from the Terminal

Open a new terminal tab (keep Firefox open) and run:

whatweb http://localhost:3000

What to Look For:

The output will list technologies it detected, such as the web server name, scripting language, or framework. Write down everything you see.

1B: Check Response Headers in Firefox

⚠️

Important: F12, Not F11!

Press F12 to open Developer Tools. Do NOT press F11 — that makes Firefox go full-screen, which is not what we want.

  1. In Firefox, make sure you are on http://localhost:3000
  2. Press F12 on your keyboard — a panel will appear at the bottom of the browser window. This is Developer Tools.
  3. At the top of that panel, you will see a row of tabs: Inspector, Console, Debugger, Network, Style Editor, ... — Click on Network
  4. The Network tab will be empty at first — this is normal! It only records requests that happen after you open it. Press Ctrl + R to reload the page. You should now see a list of requests appear.
  5. Look at the list of requests. The first row should show a request to the URL localhost:3000 (it may show as / or GET localhost or GET /). Click on that first row.
  6. A details panel appears on the right side. At the top of this panel, you will see sub-tabs: Headers, Cookies, Request, Response, Timings. Click on Headers if it is not already selected.
  7. Inside the Headers panel, scroll down. You will see two sections: Request Headers (what your browser sent) and Response Headers (what the server sent back). Look at the Response Headers section.
üí°

Key Headers to Look For in Response Headers

Scroll through the Response Headers and look for these specific header names:

  • Content-Type — tells you the data format (you should see text/html; charset=UTF-8 for the main page)
  • X-Content-Type-Options — you should see nosniff (a security header that prevents MIME-type sniffing)
  • X-Frame-Options — you should see SAMEORIGIN (a security header that prevents clickjacking)
  • Feature-Policy — you should see payment 'self' (restricts which features the page can use)
  • Access-Control-Allow-Origin — you should see * (allows any website to make requests to this server — a potential security concern!)
  • X-Recruiting — you should see /#/jobs (this is a Juice Shop easter egg — a hidden path in the application!)

Note: You might expect to see X-Powered-By: Express (which would reveal the server framework), but Juice Shop disables this header. This is actually a good security practice — hiding the server technology makes it harder for attackers to find known vulnerabilities. We will confirm the framework is Express using other methods (like whatweb from Step 1A).

1C: Check the Page Source for the Client Framework

  1. Close Developer Tools for now (press F12 again to toggle it off)
  2. Right-click anywhere on the Juice Shop page
  3. Select View Page Source from the menu (or press Ctrl + U) — a new tab will open showing the raw HTML code
  4. Scroll all the way to the bottom of the page source
  5. Near the bottom, look at the <script> tags. You should see exactly these four JavaScript files: runtime.js, polyfills.js, vendor.js, and main.js
  6. Also scroll back up and look for a tag called <app-root> in the <body> section
üí°

Hint

The files runtime.js, polyfills.js, vendor.js, and main.js are the signature build output of Angular (an SPA framework from Chapter 3). The <app-root> tag is also Angular-specific — it is the root component where Angular renders the entire application.

Task 2: Explore REST API Requests in the Browser

As you learned in Chapter 3, modern web applications communicate through REST APIs. The browser sends HTTP requests (GET, POST, etc.) and the server responds with data. Let's watch this happen in real time — you will be surprised how much API traffic happens just from loading the homepage.

2A: Open the Network Tab

  1. In Firefox, press F12 to open Developer Tools (the panel at the bottom)
  2. Click the Network tab at the top of the Developer Tools panel
  3. You will see a row of small filter buttons below the tab bar: All, HTML, CSS, JS, XHR, Fonts, Images, Media, WS. Click the XHR button — this filters the list to show only API calls (XHR stands for XMLHttpRequest, which is how the browser makes API calls in the background)
  4. Press Ctrl + R to reload the page so the Network tab starts recording fresh requests
⚠️

Network Tab Looks Empty?

If you see no requests in the list, make sure you pressed Ctrl + R to reload the page after opening the Network tab. The Network tab only captures requests that happen while it is open.

2B: Observe the API Calls That Happen Automatically

Without clicking anything — just by loading the homepage — you should already see a burst of API requests in the Network tab. The Angular frontend automatically calls several REST endpoints to build the page. Look at the XHR filter results and you should see requests to these endpoints:

# HTTP Method URL Path What It Does
1 GET /rest/admin/application-version Gets the Juice Shop version (returns {"version":"19.1.1"})
2 GET /rest/admin/application-configuration Gets the full application configuration (theme, features, settings)
3 GET /api/Challenges/?name=Score%20Board Checks if the Score Board challenge exists
4 GET /rest/languages Gets the list of supported languages (40 languages)
5 GET /api/Quantitys/ Gets product inventory/stock quantities
6 GET /rest/products/search?q= Searches all products (empty query = return all 36 products)
7 GET /rest/user/whoami Checks if a user is logged in (returns {"user":{}} when nobody is logged in)
üí°

How to Inspect Each Request

When new requests appear in the Network tab list, you can read three things directly from the list columns:

  • Method column — shows GET, POST, PUT, or DELETE
  • URL / File column — shows the API endpoint path (like /rest/products/search?q=)
  • Status column — shows a number: 200 = success, 304 = not modified (cached)

To see more details, click on any request row. A details panel opens on the right where you can see Headers, Request body, and Response body.

This is exactly how REST APIs work as described in Chapter 3!

2C: Click on a Product and Watch for More Requests

Now click on any product card on the homepage. Watch the Network tab — you should see new requests appear:

  1. Click on any product (for example, a juice product). A product detail popup will appear.
  2. In the Network tab, you should see new requests to /rest/products/{id}/reviews where {id} is the product number. For example, if you click product 30 you will see /rest/products/30/reviews.
  3. Click on one of these review requests. In the Response sub-tab, you should see JSON data like this:
{
  "status": "success",
  "data": [
    {
      "message": "I bought it, would buy again. 5/7",
      "author": "admin@juice-sh.op",
      "product": 3,
      "likesCount": 0,
      "likedBy": [],
      "_id": "b7N67f6mBnPBL2xhc"
    }
  ]
}
⚠️

Security Observation

Notice the review endpoints: /rest/products/30/reviews and /rest/products/3/reviews. The reviews contain email addresses of users (like admin@juice-sh.op and uvogin@juice-sh.op). This is sensitive data exposure — user email addresses are being leaked to anyone who views product reviews!

Task 3: Read JSON Data

As you learned in Chapter 3, REST APIs use JSON to send and receive data. Let's examine the actual JSON payloads the Juice Shop sends back.

3A: View the Application Version JSON

  1. Make sure Developer Tools is still open (F12) and you are on the Network tab with the XHR filter selected. If the list is empty, press Ctrl + R to reload.
  2. In the list of requests, find the one that shows application-version in the URL column.
  3. Click on that request row. A details panel will open on the right side.
  4. In the details panel, you will see sub-tabs at the top: Headers, Cookies, Request, Response, Timings
  5. Click the Response sub-tab — this shows the JSON data the server sent back.

Expected JSON Response:

{"version":"19.1.1"}

This tells us the exact version of Juice Shop running on the server. An attacker could use this to look up known vulnerabilities for this specific version.

3B: View the Product Search JSON

  1. In the Network tab, find the request to /rest/products/search?q= and click on it.
  2. Click the Response sub-tab. You will see a large JSON response containing all 36 products in the shop.
  3. Look at the structure of the first product in the JSON:

What a Single Product Object Looks Like:

{
  "id": 1,
  "name": "Apple Juice (1000ml)",
  "description": "The all-time classic.",
  "price": 1.99,
  "deluxePrice": 0.99,
  "image": "apple_juice.jpg",
  "createdAt": "2026-02-17 11:33:04.274 +00:00",
  "updatedAt": "2026-02-17 11:33:04.274 +00:00",
  "deletedAt": null
}
üí°

Understanding the JSON Structure

Notice the JSON fields: id, name, description, price, deluxePrice, image. These are the data fields stored in the database for each product. Also notice the deletedAt: null field — this suggests the database uses soft deletes (marking items as deleted rather than removing them), which means deleted products might still be accessible through the API.

3C: View the whoami JSON

  1. Find the request to /rest/user/whoami and click on it.
  2. Click the Response sub-tab. You should see:

Expected JSON Response:

{"user":{}}

The empty user object means nobody is logged in. This endpoint lets the Angular frontend check the login status. After logging in, this response would contain user details.

3D: Use curl to Get JSON from the Terminal

You can also request the same JSON data directly from the terminal. Open a terminal and try these commands:

# Get the application version
curl "http://localhost:3000/rest/admin/application-version" | python3 -m json.tool

# Search for all products (returns all 36 products)
curl "http://localhost:3000/rest/products/search?q=" | python3 -m json.tool

# Check who is logged in (nobody, since curl has no session)
curl "http://localhost:3000/rest/user/whoami" | python3 -m json.tool
üí°

What does python3 -m json.tool do?

It formats the raw JSON into a readable, indented structure. Without it, the JSON would appear as one long, hard-to-read line. This is the kind of practical skill mentioned in Chapter 3.

⚠️

Security Observations

Think about what you just discovered:

  • The admin endpoints (/rest/admin/application-version and /rest/admin/application-configuration) are accessible without any authentication. Anyone can see the server version and full configuration!
  • The connection uses HTTP (not HTTPS), meaning all data travels in plain text over the network.
  • The API naming convention /api/Quantitys/ exposes internal database model names (notice the incorrect pluralization — this reveals the ORM layer).

Task 4: Inspect Client-Side Storage

Chapter 3 taught you that browsers can store data locally using cookies, localStorage, and sessionStorage. Let's see what Juice Shop stores on your browser just from visiting the homepage.

4A: Open the Storage Inspector

  1. Make sure you are on the Juice Shop homepage (http://localhost:3000) and have dismissed the welcome banner and cookie consent banner
  2. Press F12 to open Developer Tools (the panel at the bottom of the browser)
  3. Look at the tab bar at the top of the Developer Tools panel. You will see tabs like: Inspector, Console, Debugger, Network, ... Look for a tab called Storage.

    Cannot find the Storage tab? It is often hidden because there are too many tabs to fit on screen. Look for a double arrow icon (») at the far right end of the tab bar. Click on it — a dropdown menu will appear listing the hidden tabs. Click Storage from that menu.
  4. Once you are on the Storage tab, you will see a left sidebar with a tree structure. It lists three storage types:
    • Cookies
    • Local Storage
    • Session Storage
  5. Click the small triangle (▶) next to Cookies to expand it, then click on http://localhost:3000 underneath. The right panel will show a table of all cookies stored by Juice Shop.
  6. Do the same for Local Storage → http://localhost:3000
  7. Do the same for Session Storage → http://localhost:3000
üí°

What You Should See

Under Cookies, you should see exactly these 3 entries:

  • language — value: en (your language preference)
  • welcomebanner_status — value: dismiss (remembers you closed the welcome popup)
  • cookieconsent_status — value: dismiss (remembers you accepted the cookie consent)

Notice that these cookies store user preferences, not authentication data. Check the columns in the cookie table — do any of these cookies have the HttpOnly flag set? Do any have the Secure flag?

Under Local Storage and Session Storage, check if there are any entries. Since you are not logged in, you should see either no entries or only non-sensitive data like UI state.

⚠️

Security Observations About the Cookies

Look carefully at the cookie properties:

  • None of the cookies have the Secure flag — this means they will be sent over unencrypted HTTP connections
  • The cookies are set without HttpOnly — this means JavaScript code running on the page can read and modify them
  • The language cookie tells an attacker what language the user speaks — useful for crafting targeted phishing attacks

Troubleshooting

❌ Docker command not found

Solution: Run sudo apt update && sudo apt install -y docker.io then sudo systemctl start docker.

❌ "Cannot connect to the Docker daemon"

Solution: Run sudo systemctl start docker. If still failing, try sudo dockerd & and wait a few seconds.

❌ Juice Shop page doesn't load

Solution: Check if the container is running: sudo docker ps. If not listed, start it: sudo docker start juice-shop. Make sure the URL is http://localhost:3000 (not https).

❌ Port 3000 is already in use

Solution: Run sudo docker stop juice-shop && sudo docker rm juice-shop, then re-run the docker run command from Part 1.
