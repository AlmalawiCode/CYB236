💡 مقدمة

في المختبر 05 تعلّمت ما هو Burp Suite، وكيف يعمل البروكسي، وكيف تُهيئ FoxyProxy لتمر حركة مرور المتصفح عبر Burp. اعترضت أول طلب HTTP وتأكدت من أن الإعداد يعمل بشكل صحيح.

في هذا المختبر ستُطبّق تلك المعرفة على تطبيق ويب ضعيف حقيقي: DVWA (Damn Vulnerable Web Application). ستستخدم Burp Suite لالتقاط حركة تسجيل الدخول، وتحليل طلبات واستجابات HTTP الخام، واستخدام أداة Repeater لاختبار معامل يدوياً، وبناء خريطة سطح هجوم باستخدام Site Map.

⚠️

للاستخدام المصرّح به فقط

تُنفَّذ جميع المهام ضد DVWA العامل محلياً على جهازك الخاص. استخدام Burp Suite ضد أي نظام لا تملكه أو لا تحمل إذناً كتابياً باختباره هو أمر غير قانوني.

💻

ما هو DVWA؟

DVWA (Damn Vulnerable Web Application) هو تطبيق PHP/MySQL مبني عن قصد بثغرات أمنية. يستخدمه طلاب الأمن والمحترفون لممارسة تقنيات استغلال الويب في بيئة آمنة وقانونية. يتضمن وحدات لـ SQL Injection وXSS وCSRF ورفع الملفات والقوة الغاشمة والمزيد.

أدوات Burp Suite المستخدمة في هذا المختبر

⚙️ الإعداد: تشغيل DVWA والتأكد من Burp

الخطوة 1: تشغيل DVWA

# تشغيل حاوية DVWA Docker (إن كانت مُنشأة مسبقاً)
sudo docker start dvwa

# إن لم تكن مُنشأة بعد، نفّذ هذا مرة واحدة:
sudo docker run -d -p 80:80 --name dvwa vulnerables/web-dvwa
# احصل على عنوان IP لجهاز Kali
hostname -I

دوّن العنوان الظاهر (مثلاً: 10.0.2.15). ستستخدم هذا العنوان للوصول إلى DVWA في المتصفح — وليس localhost.

الخطوة 2: التأكد من تشغيل Burp Suite

  • إن لم يكن Burp مفتوحاً، شغّله: burpsuite → Temporary project → Start Burp
  • تأكد من أن مستمع البروكسي على 127.0.0.1:8080 ويُظهر Running (Proxy → Proxy settings)

الخطوة 3: تفعيل FoxyProxy

  • انقر أيقونة FoxyProxy في Firefox وفعّل ملف تعريف Burp Proxy
  • إن لم تكن قد هيّأت FoxyProxy بعد، ارجع إلى المختبر 05 وأكمل الجزأين 3 و4 أولاً

الخطوة 4: إيقاف الاعتراض وفتح DVWA

  1. في Burp، انتقل إلى Proxy → Intercept وتأكد من أنه يقرأ "Intercept is off" (يجب أن تتدفق حركة المرور بحرية)
  2. في Firefox، انتقل إلى http://<عنوان-ip-جهازك> (مثلاً: http://10.0.2.15)
  3. يجب أن تظهر صفحة تسجيل دخول DVWA
  4. سجّل الدخول: اسم المستخدم admin / كلمة المرور password
  5. إن طُلب منك ذلك، انقر Setup / Reset DB، ثم سجّل الدخول مجدداً
  6. انتقل إلى DVWA Security واضبط مستوى الأمان على Low
✅

نقطة تحقق

إن تحمَّل DVWA في Firefox وظهرت الطلبات في تبويب HTTP History الخاص بـ Burp، فإعداد البروكسي يعمل. أنت جاهز لبدء المهام.

📷 المهمة 1: التقاط حركة HTTP

💡

المفهوم — سجل HTTP

كل إجراء تتخذه في المتصفح يُولّد طلبات HTTP. يسجّل سجل HTTP History الخاص بـ Burp جميعها بتفاصيل كاملة: سطر الطلب، والترويسات، والجسم، والاستجابة الكاملة للخادم. يمنحك هذا معلومات أكثر بكثير مما توفره أدوات DevTools في المتصفح، ويمكن حفظ كل إدخال وتصفيته وإعادة تشغيله.

1أ: توليد حركة المرور بالتصفح في DVWA

في Firefox (FoxyProxy مفعّل، الاعتراض متوقف)، نفّذ هذه الإجراءات واحداً تلو الآخر:

  1. زيارة http://<عنوان-ip-جهازك> — صفحة تسجيل دخول DVWA
  2. تسجيل الدخول بـ admin / password
  3. النقر على SQL Injection في القائمة اليسرى، كتابة 1 في حقل User ID والنقر على Submit
  4. النقر على XSS (Reflected)، كتابة اسمك، والنقر على Submit
  5. النقر على Brute Force في القائمة اليسرى

1ب: استكشاف سجل HTTP

  1. انتقل إلى Burp Suite → Proxy → HTTP history
  2. ستجد قائمة بالطلبات الملتقطة. كل صف يُظهر:
    • # — رقم الطلب (ترتيب الالتقاط)
    • Host — عنوان الخادم
    • Method — GET أو POST
    • URL — المسار المطلوب
    • Status — رمز استجابة HTTP (200، 302، 404…)
    • Length — حجم الاستجابة بالبايت
  3. انقر على طلب POST الخاص بتسجيل الدخول (ابحث عن POST /login.php)
  4. في اللوحة السفلية، اقرأ تبويبَي Request وResponse

✅ ما يجب أن تراه في طلب تسجيل الدخول:

سيُظهر تبويب Request شيئاً مثل:

POST /login.php HTTP/1.1
Host: 10.0.2.15
Content-Type: application/x-www-form-urlencoded
Cookie: PHPSESSID=abc123def456; security=low

username=admin&password=password&Login=Login

سيُظهر تبويب Response إعادة توجيه 302 Found إلى /index.php — مما يعني نجاح تسجيل الدخول.

💡

ما هي كوكي PHPSESSID؟

بعد تسجيل الدخول بنجاح، يضع الخادم كوكي PHPSESSID. هذا هو رمز الجلسة (Session Token) — يُثبت للخادم من أنت في كل طلب لاحق. أي مهاجم يسرق هذه القيمة يمكنه انتحال هويتك دون معرفة كلمة مرورك. يُسمى هذا اختطاف الجلسة (Session Hijacking).

✍️ المهمة 1 — أجب على الأسئلة التالية

🔑 مفتاح الإجابات — المهمة 1

  1. س1: عادةً 15–30+ طلباً. كل تحميل صفحة يُطلق طلبات متعددة: صفحة HTML، وملفات CSS، وJavaScript، والصور.
  2. س2: POST. يرسل POST بيانات الاعتماد في جسم الطلب (وليس في الرابط). كان GET سيُظهر كلمة المرور في شريط العنوان وسجل المتصفح ولوجات الخادم وأي بروكسيات وسيطة.
  3. س3: PHPSESSID — سلسلة سداسية عشرية من 32 حرفاً تُولّدها PHP لتعريف جلسة المستخدم بشكل فريد.
  4. س4: 302 Found — إعادة توجيه HTTP. بعد التحقق من بيانات الاعتماد، يُخبر الخادم المتصفح بالانتقال إلى لوحة التحكم. يتبع المتصفح هذه الإعادة تلقائياً.

📊 المهمة 2: تحليل الطلبات والاستجابات

💡

المفهوم — قراءة HTTP الخام

كل طلب HTTP له ثلاثة أجزاء: سطر الطلب (الطريقة + المسار + الإصدار)، والترويسات (البيانات الوصفية كالكوكيز ونوع المحتوى ومعلومات المتصفح)، وبشكل اختياري الجسم (البيانات المرسلة مع طلبات POST). يُظهر Burp كل هذا كنص خام، تماماً كما يرسله المتصفح — بمزيد من التفاصيل مما توفره أي لوحة DevTools.

2أ: إيجاد وفحص طلب SQL Injection

  1. في سجل HTTP History، ابحث عن طلب GET يحتوي رابطه على /vulnerabilities/sqli/
  2. انقر عليه لتحديده
  3. في تبويب Request (اللوحة السفلية)، اقرأ الطلب الخام كاملاً
  4. لاحظ المعامل id=1 في الرابط — هذه هي نقطة دخول حقن SQL

✅ شكل طلب SQL Injection:

GET /vulnerabilities/sqli/?id=1&Submit=Submit HTTP/1.1
Host: 10.0.2.15
Cookie: PHPSESSID=abc123; security=low
Referer: http://10.0.2.15/vulnerabilities/sqli/
User-Agent: Mozilla/5.0 ...

يُرسَل معامل id=1 مباشرةً إلى استعلام SQL على الخادم. هذا هو المعامل الذي يُعدّله المختبر لتنفيذ حقن SQL.

2ب: مقارنة طلبات GET و POST

  1. ابحث عن طلب XSS (Reflected) — طلب GET إلى /vulnerabilities/xss_r/
  2. ابحث عن طلب تسجيل الدخول — طلب POST إلى /login.php
  3. قارن أين يظهر مدخل المستخدم: في الرابط مقابل جسم الطلب

2ج: التحقق من الترويسات الأمنية المفقودة

  1. انقر على أي استجابة من DVWA في سجل HTTP History
  2. انتقل إلى تبويب Response
  3. تفحّص ترويسات الاستجابة — ابحث عن X-Frame-Options وContent-Security-Policy وX-XSS-Protection

✅ ما يجب (ألا) تراه:

DVWA غير آمن عن قصد. ستكون استجاباته خالية من معظم الترويسات الأمنية. التطبيق المحمي بشكل صحيح سيتضمن:

X-Frame-Options: DENY
Content-Security-Policy: default-src 'self'
X-Content-Type-Options: nosniff
Strict-Transport-Security: max-age=31536000

غياب هذه الترويسات يعني أن التطبيق عرضة لهجمات Clickjacking وXSS وانتحال نوع MIME.

✍️ المهمة 2 — أجب على الأسئلة التالية

🔑 مفتاح الإجابات — المهمة 2

  1. س1: في الرابط كمعامل استعلام: ?id=1&Submit=Submit. الطريقة: GET.
  2. س2: في جسم الطلب: username=admin&password=password&Login=Login. الطريقة: POST.
  3. س3: Content-Security-Policy (تمنع XSS)، X-Frame-Options (تمنع Clickjacking)، X-Content-Type-Options (تمنع انتحال نوع MIME)، Strict-Transport-Security (تُجبر على HTTPS).
  4. س4: PHPSESSID=<قيمة>; security=low. تُصادق PHPSESSID على الجلسة. إن سُرقت، يمكن للمهاجم اختطاف الجلسة دون معرفة كلمة المرور.

⚙️ المهمة 3: تعديل الطلبات باستخدام Repeater

💡

المفهوم — أداة Repeater

تتيح لك أداة Repeater في Burp أخذ أي طلب ملتقط وتعديله وإرساله ورؤية الاستجابة — بالعدد الذي تريد. هذا ضروري لاختبار الثغرات يدوياً: يمكنك تغيير معامل، أو إضافة حمولة حقن، أو تغيير ترويسة، أو حذف كوكي لترى بالضبط كيف يتفاعل الخادم، دون لمس المتصفح.

Burp Suite: تبويب Repeater

3أ: إرسال طلب SQL Injection إلى Repeater

  1. في سجل HTTP History، انقر بزر الفأرة الأيمن على طلب GET /vulnerabilities/sqli/?id=1
  2. اختر Send to Repeater
  3. انقر تبويب Repeater في أعلى Burp Suite
  4. ستجد الطلب الخام في اللوحة اليسرى

3ب: إرسال الطلب الأصلي

  1. انقر Send (الزر البرتقالي)
  2. تظهر استجابة الخادم في اللوحة اليمنى
  3. في لوحة Response، انقر تبويب Render لرؤية مخرجات HTML بصرياً

✅ الاستجابة المتوقعة لـ id=1:

First name: admin
Surname: admin

بحث الخادم عن المستخدم ذي المعرّف 1 في قاعدة البيانات وأعاد الاسم. هذا يؤكد عمل الطلب بشكل صحيح عبر Repeater.

3ج: تعديل المعامل وإعادة الإرسال

  1. في لوحة طلب Repeater، ابحث عن id=1 في الرابط
  2. غيّره إلى id=2 وانقر Send — لاحظ المستخدم المختلف المُعاد
  3. غيّره إلى id=5 وانقر Send مجدداً
  4. الآن جرّب id=1' (أضف علامة اقتباس مفردة) وانقر Send

✅ ما يحدث مع id=1' (علامة اقتباس مفردة):

You have an error in your SQL syntax; check the manual that
corresponds to your MySQL server version for the right syntax
to use near ''1''' at line 1

تُعاد رسالة خطأ SQL — هذا يؤكد أن المعامل عرضة لحقن SQL. قاعدة البيانات تُسرّب تفاصيل الأخطاء الداخلية، وهو خطأ أمني خطير.

⚠️

لماذا Repeater قوي جداً

مع Repeater يمكنك اختبار عشرات الحمولات في ثوانٍ دون لمس المتصفح. لديك تحكم كامل في كل بايت في الطلب — الترويسات والكوكيز والطريقة والجسم. يستخدم المختبرون المحترفون Repeater لتأكيد الثغرات واختبار تقنيات التجاوز وفهم كيفية معالجة الخادم للمدخلات.

3د: اعتراض وتعديل طلب حي

  1. في Burp → Proxy → Intercept، انقر الزر لتشغيل Intercept ON
  2. في Firefox، أرسل نموذج XSS Reflected باسم Test
  3. يتجمد Firefox — اصطاد Burp الطلب في منتصف طريقه
  4. في لوحة Intercept الخاصة بـ Burp، غيّر name=Test إلى name=Burp
  5. انقر Forward — يتلقى Firefox الاستجابة المعدّلة
  6. تقول الصفحة الآن "Hello Burp" رغم أنك كتبت "Test"
  7. أوقف الاعتراض OFF مجدداً عند الانتهاء

✍️ المهمة 3 — أجب على الأسئلة التالية

🔑 مفتاح الإجابات — المهمة 3

  1. س1: الاسم الأول: Gordon — اسم العائلة: Brown (المستخدم ذو المعرّف 2 في DVWA افتراضياً).
  2. س2: خطأ في صياغة MySQL: "You have an error in your SQL syntax; check the manual..." — يؤكد أن المدخل يُحقن مباشرةً في استعلام SQL.
  3. س3: تُسرّب أخطاء SQL معلومات داخلية: محرك قاعدة البيانات (MySQL)، وبنية الاستعلام، وأحياناً أسماء الجداول/الأعمدة. يساعد هذا المهاجم على صياغة حمولة حقن تعمل ويؤكد عدم وجود تحقق من المدخلات.
  4. س4: يسمح Repeater بتعديل أي جزء من الطلب بما فيه الترويسات وجسم POST، وليس معاملات الرابط فقط. يرسل الطلبات فورياً دون إعادة تحميل الصفحة، ويحفظ الكوكيز تلقائياً، ويُظهر الاستجابة الخام — كل ذلك في عرض واحد.

🗺️ المهمة 4: رسم خريطة سطح الهجوم باستخدام Site Map

💡

المفهوم — رسم خريطة سطح الهجوم

قبل اختبار أي تطبيق، يبني خبراء الأمن خريطة بجميع الروابط والمعاملات والنقاط النهائية القابلة للوصول. تُنظّم Site Map الخاصة بـ Burp تلقائياً كل رابط زاره متصفحك في عرض شجري، مُجمَّعاً حسب المضيف والمسار. هذا هو سطح الهجوم للتطبيق — كل إدخال هو هدف اختبار محتمل.

4أ: تحديد النطاق

  1. في Burp، انقر تبويب Target
  2. انقر التبويب الفرعي Site map
  3. في اللوحة اليسرى، انقر بزر الأيمن على إدخال مضيف DVWA (مثلاً: http://10.0.2.15)
  4. اختر "Add to scope"
  5. إن سألك Burp عن إيقاف تسجيل حركة المرور خارج النطاق، انقر Yes
💡

لماذا تحدد النطاق؟

تحديد النطاق يُخبر Burp بالتركيز فقط على هدفك. يمنع الاختبار العرضي لمواقع خارجية (Google ومزودي CDN وغيرها) التي يُحمّلها متصفحك في الخلفية، ويُبقي Site Map نظيفة ومركّزة على DVWA.

4ب: تصفح جميع وحدات DVWA

في Firefox، انقر على كل وحدة في القائمة اليسرى لـ DVWA: Brute Force, Command Injection, CSRF, File Inclusion, File Upload, Insecure CAPTCHA, SQL Injection, SQL Injection (Blind), Weak Session IDs, XSS (DOM), XSS (Reflected), XSS (Stored).

  1. بعد زيارة جميع الوحدات، ارجع إلى Burp → Target → Site map
  2. وسّع شجرة المضيف (مثلاً: http://10.0.2.15)
  3. وسّع مجلد /vulnerabilities/
  4. يجب أن ترى إدخالاً منفصلاً لكل وحدة DVWA زرتها

✅ ما يجب أن تُظهره خريطة Site Map:

http://10.0.2.15
  ├── /
  ├── /index.php
  ├── /login.php
  ├── /logout.php
  ├── /dvwa/
  └── /vulnerabilities/
        ├── /vulnerabilities/brute/
        ├── /vulnerabilities/csrf/
        ├── /vulnerabilities/exec/
        ├── /vulnerabilities/fi/
        ├── /vulnerabilities/sqli/
        ├── /vulnerabilities/sqli_blind/
        ├── /vulnerabilities/upload/
        ├── /vulnerabilities/weak_id/
        ├── /vulnerabilities/xss_d/
        ├── /vulnerabilities/xss_r/
        └── /vulnerabilities/xss_s/

✍️ المهمة 4 — أجب على الأسئلة التالية

🔑 مفتاح الإجابات — المهمة 4

  1. س1: 12 وحدة ثغرات: /brute/، /captcha/، /csrf/، /exec/، /fi/، /sqli/، /sqli_blind/، /upload/، /weak_id/، /xss_d/، /xss_r/، /xss_s/.
  2. س2: يقيّد النطاق نشاط Burp على الهدف (10.0.2.15)، ويمنع الاختبار العرضي للموارد الخارجية ويُبقي الخريطة نظيفة ومركّزة.
  3. س3: توفر Site Map قائمة هيكلية كاملة بجميع نقاط دخول التطبيق المكتشفة. يستخدم المختبر هذا لتخطيط التغطية الشاملة وضمان عدم تفويت أي نقطة نهائية.
  4. س4: تستمر Site Map الخاصة بـ Burp طوال الجلسة كاملة، وتتيح تعديل الطلبات وإعادة إرسالها عبر Repeater، وتدعم تحديد النطاق والتصفية، وتتكامل مع أدوات الهجوم. يُظهر DevTools طلبات الصفحة الحالية فقط، ويُمسح عند التحديث، ولا يمكنه تعديل الطلبات أو إعادة إرسالها.

🔧 استكشاف الأخطاء

❌ DVWA لا يُحمَّل في Firefox

الحل: تحقق من تشغيل حاوية Docker: sudo docker ps. إن لم يكن DVWA مدرجاً، شغّله: sudo docker start dvwa. تأكد من استخدامك http://<عنوان-ip-جهازك> (من hostname -I) وليس localhost.

❌ لا تظهر طلبات في سجل HTTP الخاص بـ Burp

الحل: تأكد من تفعيل FoxyProxy (ملف تعريف Burp Proxy نشط). تأكد من تصفحك إلى http://<عنوان-ip-جهازك> — وليس localhost أو 127.0.0.1. تأكد من أن مستمع بروكسي Burp على 127.0.0.1:8080 ويُظهر Running. استخدم http:// وليس https://.

❌ Firefox متجمد وDVWA لا يُحمَّل

الحل: الاعتراض في Burp مفعّل. انتقل إلى Burp → Proxy → Intercept وانقر Forward (كرر لكل مورد) أو انقر الزر لإيقاف الاعتراض OFF. في المهمتين 1 و2 تريد الاعتراض متوقفاً لتتدفق حركة المرور بحرية.

❌ Repeater يُظهر "Connection refused"

الحل: توقف DVWA عن العمل. تحقق: sudo docker ps. إن لم يكن مدرجاً: sudo docker start dvwa. تأكد أيضاً من أن ترويسة Host في Repeater تطابق عنوان IP جهازك (من hostname -I) — وليس localhost.

❌ خريطة Site Map فارغة أو تُظهر إدخالاً واحداً فقط

الحل: تصفح DVWA أكثر — انقر على كل وحدة في القائمة اليسرى. تأكد من أن المضيف ضمن النطاق (انقر بزر الأيمن على المضيف → Add to scope). إن كان الفلتر مضبوطاً على "Show only in-scope items"، تأكد من إضافة عنوان IP المضيف إلى النطاق أولاً.

❌ بعد المختبر، Firefox لا يملك وصولاً للإنترنت

الحل: لا يزال FoxyProxy يُوجِّه الحركة إلى Burp الذي لم يعد يعمل. انقر أيقونة FoxyProxy واختر "Disable FoxyProxy". افعل هذا دائماً في نهاية كل جلسة مختبر.
