The Vulnerability Management Pipeline

Part of any good Secure Software Development Life Cycle (SSDL) is a well-defined pipeline for obtaining, triaging, and resolving vulnerabilities found in a web application. Vulnerabilities appear at every phase — architecture, development, and production — and each one needs a structured, repeatable response.

The vulnerability management pipeline connects discovery (Chapter 20) with resolution, ensuring that nothing falls through the cracks between finding a bug and permanently fixing it.

🔎
1. Reproduce
Confirm the vulnerability exists in a controlled staging environment before taking any action.
📊
2. Score & Rank
Use CVSS or a similar system to quantify severity and prioritize fixes against other work.
🔧
3. Fix
Implement a permanent, application-wide solution — or a tracked temporary fix with a new issue opened.
✅
4. Regression Test
Write a test that fails if this vulnerability is ever reintroduced — and ship it with the fix.

📋 Where This Fits in the SSDL

Chapter 21 follows directly from the vulnerability discovery techniques of Chapter 20. Once a vulnerability is found — whether through static analysis, dynamic analysis, a bug bounty submission, or a user report — vulnerability management is what happens next to ensure it is resolved properly and permanently.

Step 1 — Reproducing Vulnerabilities

After receiving a vulnerability report, the first step is always reproduction — confirming the vulnerability actually exists and behaves as described — before any engineering effort is invested in a fix.

Why Reproduce First?

📖 From the Book — The False Positive Problem

Sometimes a user-defined configuration error looks exactly like a vulnerability. For example, a user "accidentally" makes an image on a photo-hosting app "public" when they usually set their photos to "private." Without reproducing the issue, an engineer could waste hours investigating a non-issue — or pay a bug bounty reward that was never earned.

  • Confirms the report is valid — avoids wasting engineering hours on false positives
  • Reveals deeper impact — reproduction often uncovers that the vulnerability's scope is wider than originally reported
  • Protects bug bounty programs — a bounty should never be paid for a false positive; reproduction provides documented proof
  • Creates understanding — seeing the vulnerability execute gives engineers insight into root cause and fix complexity
  • Establishes the baseline — the reproduction log is the foundation for writing the regression test later

🔑 Always Reproduce and Log

Reproduce right away and document the results. Logging the exact steps, environment state, and observed behavior gives every engineer involved a shared baseline — and becomes essential when verifying the fix later.

🖥️

The Staging Environment

Reproduction requires a staging environment — a separate, controlled environment that mirrors production as closely as possible, but contains no real users or live data.

Requirements for a Good Staging Environment

Requirement Why It Matters
Mirrors production closely Vulnerabilities that only appear in specific configurations won't reproduce in a simplified setup
Fully automated setup Manual setup is slow and error-prone — automation ensures a reliable, repeatable environment every time
Contains mock users & objects Staging needs realistic data shapes to logically and visually represent production behavior
Internal access only Staging must not be accessible from the public internet — only via the internal network or encrypted login
New features available before release Staging lets teams test security of new features before they reach real users

⚠️ Never Reproduce on Production

Attempting to reproduce a vulnerability against live production systems risks affecting real users, corrupting real data, or — in the case of a DoS vulnerability — taking the service offline. Always use an isolated staging environment.

📊

Step 2 — Ranking Vulnerability Severity

After reproducing a vulnerability, the next step is to rank it by severity so it can be prioritized against other engineering work. Not all vulnerabilities are equally urgent — a scoring system makes prioritization objective, consistent, and defensible.

A good scoring system must be:

Robust Enough

It must be able to compare any two vulnerabilities accurately — even ones that affect different parts of the application or involve different attack methods.

Flexible Enough

It must apply to uncommon and unusual forms of vulnerability — not only the well-known OWASP Top 10 types. Custom logic bugs and IoT-connected systems need scoring too.

The most widely used method is the Common Vulnerability Scoring System (CVSS).

🖌️

The Common Vulnerability Scoring System (CVSS)

CVSS is a freely published, industry-standard framework for scoring vulnerabilities based on how easy they are to exploit and what damage a successful exploitation can cause. It is a strong starting point for any organization — especially those with limited security budgets or small dedicated security teams.

CVSS v3.1 breaks scoring into three independent levels:

🎯
Base Score
Scores the vulnerability itself, in isolation. Independent of environment or any mitigations in place. The most commonly used score.
🕐
Temporal Score
Scores how well-equipped your organization is to respond given the current state of the vulnerability — exploitability, fix availability, report quality.
🌏
Environmental Score
Scores the vulnerability relative to your specific application — weighting by how critical confidentiality, integrity, and availability are to your business.

Score Ranges

0.1 – 4.0
🟢 Low
4.1 – 6.9
🟡 Medium
7.0 – 8.9
🔴 High
9.0 – 10.0
⚫ Critical

📊 CVSS Limitations

CVSS is a general-purpose system and is sometimes criticized for not accurately scoring rare, unique, or chained vulnerabilities (where multiple smaller bugs combine into a more severe exploit). For standard OWASP Top 10 vulnerabilities, it provides a reliable, well-documented baseline. Organizations with specialized systems may extend it with their own custom scoring layers.

🎯

CVSS Base Score — All 8 Input Factors

The CVSS v3.1 base score is calculated from 8 input factors. Each input describes a specific dimension of how the vulnerability works and what it can impact. The algorithm combines all 8 to produce a final score between 0 and 10.

AV
Attack Vector
Options: Network · Adjacent · Local · Physical Network = attacker can exploit remotely over the internet (most severe). Physical = attacker needs physical access to the device (least severe).
AC
Attack Complexity
Options: Low · High Low = attack is repeatable with no setup. High = requires specific conditions (e.g., a particular user logged in at a specific time and page).
PR
Privileges Required
Options: None · Low · High None = guest user can attack. Low = regular user. High = admin account required to exploit.
UI
User Interaction
Options: None · Required None = attack succeeds automatically. Required = victim must take an action (e.g., click a malicious link).
S
Scope
Options: Unchanged · Changed Unchanged = impact stays within the exploited component. Changed = attack spreads beyond the original system (e.g., DB exploit affects the OS).
C
Confidentiality Impact
Options: None · Low · High How much sensitive data is disclosed? High = complete exposure of all data. Severity depends on your application's data type (health records vs. public posts).
I
Integrity Impact
Options: None · Low · High None = attacker cannot change application state. High = attacker can modify all or most stored data, server state, or session data.
A
Availability Impact
Options: None · Low · High Can the attacker deny access to legitimate users? High = complete service disruption (critical for DoS attacks against 24/7 applications).

How to Use the Online CVSS Calculator

You do not need to compute the score manually. The official CVSS v3.1 calculator is freely available at nvd.nist.gov/vuln-metrics/cvss/v3-calculator ↗. Follow these steps to score any vulnerability:

  1. 1
    Open the calculator
    Click nvd.nist.gov/vuln-metrics/cvss/v3-calculator ↗ to open it in your browser. You will see the Base Score section with all 8 inputs displayed as clickable button groups.
  2. 2
    Select a value for each of the 8 inputs
    Work through each input group — Attack Vector, Attack Complexity, Privileges Required, User Interaction, Scope, Confidentiality, Integrity, Availability — and click the option that best describes the vulnerability. Each click updates the score in real time.
  3. 3
    Read the Base Score and severity label
    Once all 8 inputs are selected, the calculator displays the final numerical score (0–10) and the severity label: Low, Medium, High, or Critical.
  4. 4
    Copy the Vector String
    Below the score, the calculator generates a compact vector string such as:
    CVSS:3.1/AV:N/AC:L/PR:N/UI:R/S:U/C:N/I:L/A:N
    Copy it into your vulnerability ticket — it encodes all 8 choices so any engineer can see exactly how the score was derived.
  5. 5
    (Optional) Add Temporal and Environmental inputs
    Scroll down to the Temporal and Environmental sections for a more contextual score. Most teams use Base Score alone for standard triage.

The numerical score (e.g., 4.3) tells you the severity. The vector string tells you why. Always copy both into your vulnerability ticket so that any engineer reviewing the report months later can see exactly which factors drove the score.

Base Score Example — Scoring a CSRF Vulnerability

Let's apply the 8-factor base scoring model to a real vulnerability: a CSRF attack on a subscription management endpoint that accepts HTTP GET requests (covered in Chapter 20's regression test example).

Input Chosen Value Reasoning
Attack Vector (AV) Network The attacker sends a malicious link via email — fully remote, over the internet
Attack Complexity (AC) Low Crafting the link requires no special setup — just a URL with parameters
Privileges Required (PR) None Any unauthenticated attacker can create and send the malicious link
User Interaction (UI) Required The victim must click the link — the attack does not execute automatically
Scope (S) Unchanged Impact is limited to the victim's subscription data — does not spread further
Confidentiality (C) None No data is read or disclosed — the attack only modifies state
Integrity (I) Low The victim's subscription tier is changed without consent — limited state change
Availability (A) None The application remains fully available — no disruption to other users
📊 Result

Running these 8 values through the CVSS v3.1 calculator produces a base score of approximately 4.3 — placing this vulnerability in the Medium severity range.

This tells us: the vulnerability is real and should be fixed, but it is not an emergency. It should be queued in the next sprint, not deployed as a hotfix today. A critical (9.0+) vulnerability — such as an unauthenticated remote code execution — would require immediate action regardless of sprint priorities.

🕐

CVSS Temporal Scoring

The temporal score answers a different question than the base score. Instead of asking "how severe is this vulnerability?", it asks: "how well-equipped is your organization to deal with this vulnerability right now?"

The temporal score uses three factors, each reflecting the current state of the vulnerability at the time of reporting:

Factor What It Measures Range
Exploitability Is this vulnerability theoretical, a proof-of-concept, or a fully working exploit that can be deployed as-is? Unproven → High
Remediation Level Is a tested, official fix already available? Or is no solution currently known? Official Fix (O) → Unavailable (U)
Report Confidence How reliable and detailed is the vulnerability report? Does it include reproduction steps? Unknown → Confirmed
📖 From the Book — Temporal in Practice

A vulnerability with a high exploitability score (working exploit already circulating online), no remediation available, and a confirmed report with full reproduction is a situation requiring immediate action — even if the base score was only "High" rather than "Critical."

The temporal score captures this urgency. It does not change the intrinsic severity of the vulnerability itself — only how urgent the response must be given the current state of knowledge and available defenses.

📊 Temporal Score Range

The temporal score uses the same 0–10 scale as the base score. However, temporal scores can only be equal to or lower than the base score — mitigations in place reduce the effective score, but cannot raise it above the intrinsic severity of the vulnerability itself.

🌏

CVSS Environmental Scoring

The environmental score takes all of the base score inputs and adds three new requirements that describe how important confidentiality, integrity, and availability are to your specific application. The same vulnerability scored in a hospital's patient record system should be treated very differently than the same vulnerability in a test sandbox.

Confidentiality Requirement

How sensitive is the data your application stores? A healthcare application with patient records scores High. A public news aggregator with no user accounts scores Low.

Integrity Requirement

How damaging is unauthorized state change in your application? A corporate tax record system scores High. A temporary test sandbox that is thrown away after use scores Low.

Availability Requirement

How critical is uptime? A payment processor with a 99.99% SLA scores High. An internal developer tool with no uptime promises scores Low.

How It Works

The environmental score reweights the base score inputs based on these three requirements — raising or lowering the effective score based on how much the exploit could actually hurt your organization specifically.

💡 Base Score = Objective · Environmental Score = Contextual

The base score measures the vulnerability in a vacuum. The environmental score measures its real impact on your organization. A base score of 5.5 (Medium) could become a 9.2 (Critical) environmental score for a healthcare application storing protected health information. Always compute both when the stakes are high.

🧰

Advanced & Custom Vulnerability Scoring

CVSS is a general-purpose starting point, but organizations with specialized architectures may need to build their own scoring adjustments on top of CVSS to account for risks that the standard system does not model well.

📖 From the Book — IoT and Physical Systems

Consider a web application that controls a network of security cameras. The CVSS Attack Vector options (Network, Adjacent, Local, Physical) do not fully capture the additional implications of a compromise in this scenario:

  • A camera compromise could leak sensitive video footage of tenants — potentially illegal under privacy law
  • The attack surface includes not just the web application but also the physical devices connected to it
  • Loss of camera functionality could have physical security implications beyond data risk

For such applications, CVSS should be used as a foundation, but custom scoring dimensions should be added to reflect these unique risks.

📊 Evaluating Your Scoring System Over Time

Any scoring system — whether pure CVSS or a custom extension — should be regularly evaluated based on its ability to accurately predict damage caused by vulnerabilities that were not fixed in time. If a low-severity rated vulnerability turns out to cause significant harm, the scoring model should be updated to weight similar vulnerabilities higher in the future.

🔧

Step 3 — Fixing Vulnerabilities

After reproduction and scoring, the vulnerability must be fixed correctly. The CVSS score determines priority and urgency, but other business factors also influence when a fix ships — including customer contracts, SLA obligations, and whether the vulnerability is actively being exploited.

📖 From the Book — Scoring Is Not the Only Factor

A vulnerability's CVSS score tells you how severe it is in isolation. But business priorities can override the queue. For example, a Medium severity vulnerability may need to ship before a High one if a major enterprise customer has a contractual requirement that the medium issue be resolved before their next audit date. Scoring informs priority — it does not fully determine it.

Rules for Fixing Correctly

  1. 1
    Prefer permanent, application-wide solutions
    Whenever possible, fix the class of vulnerability — not just the single reported instance. An injection fix in one endpoint should trigger a review and fix of all endpoints that use the same pattern.
  2. 2
    If only a partial fix is possible — open a new bug immediately
    Never close a bug for a partial fix without opening a new tracked issue for the remaining vulnerable surface area. The new issue must include its own CVSS score and be added to the backlog.
  3. 3
    Ship a regression test with every fix
    Every closed security bug must have a regression test delivered alongside it. This is non-negotiable. The test ensures the vulnerability cannot silently reopen in a future refactor or rollback.

⚠️ Never Close a Bug Without a New One

Closing a vulnerability bug for a partial fix without a follow-up issue loses all accumulated context — the reproduction steps, the root cause analysis, and the technical understanding of the scope. Engineers who come across the code months later will have no record that a vulnerability was only partially fixed. Vulnerabilities can also grow in risk as the application's feature set expands, even if the original code containing them has not changed.

✅

Step 4 — Vulnerability Regression Tests

Regression tests are written after a vulnerability is fixed to ensure it never comes back. They are simple to write, easy to integrate into existing test suites, and provide permanent protection against the specific vulnerability type they cover.

Why Regression Tests Grow in Value Over Time

As a codebase grows in size and feature set, the number of opportunities for a previously fixed vulnerability to accidentally reopen increases. A regression test written today protects against every future:

  • Refactor — code restructuring that moves or renames the vulnerable function
  • Dependency update — a library upgrade that changes underlying behavior
  • New feature — a new endpoint built on a similar pattern to the old one
  • Rollback — reverting to an older version that predates the fix
📖 From the Book — Steve, Jed, and the CSRF Regression

Engineer Steve builds a subscription management endpoint using a GET request — a CSRF vulnerability. His friend Jed notices that sending a crafted link like GET /api/changeSubscriptionTier?newTier=corporation silently upgrades Steve's subscription. Steve fixes the endpoint to use POST, then writes a regression test:

// ✓ Regression test — ensures the CSRF fix can never be silently reverted const testTierChangeOnlyAcceptsPOST = function() { requester .options('http://app.com/api/changeSubscriptionTier') .on('response', function(res) { const verbs = res.headers['Allow'].split(','); // Rule 1: only one HTTP verb should be allowed on this endpoint if (verbs.length > 1) { return tester.fail('Multiple verbs exposed'); } // Rule 2: that one verb must be POST — never GET if (verbs[0] !== 'POST') { return tester.fail('GET still accessible'); } tester.pass(); }); };

If a future code change reintroduces a GET version of this endpoint — even accidentally — this test will immediately fail and block the change from being merged.

When to Run Regression Tests

✅ On Commit / Push Hooks (Recommended)

Reject the commit or push if any regression test fails. This is the earliest possible point of failure — developers get immediate feedback before the code ever enters the main branch.

🕐 Scheduled Daily Runs (Secondary)

A strong secondary option for complex multi-branch environments. Catches regressions introduced through indirect changes that weren't caught by commit hooks.

🔑 Every Closed Security Bug = One New Regression Test

This rule is simple and absolute. Closing a security bug without a regression test means the vulnerability is only one future refactor away from coming back — silently. Regression tests are the most cost-effective long-term security investment in the entire SSDL.

📋 Chapter 21 Summary — Vulnerability Management

  • Vulnerability management is a 4-step pipeline: Reproduce → Score → Fix → Regression Test
  • Reproduction in a staging environment confirms validity, prevents wasted effort, and protects bug bounty payments
  • CVSS v3.1 uses 8 base inputs (AV, AC, PR, UI, S, C, I, A) to generate an objective 0–10 severity score
  • Temporal scoring measures how ready your team is to respond; environmental scoring contextualizes risk to your specific business
  • Fixes must be permanent and application-wide — partial fixes must always spawn a new tracked issue with its own score
  • Every closed security bug must ship with a regression test — no exceptions
  • Regression tests grow in value exponentially as the codebase grows, permanently blocking previously fixed vulnerabilities from returning
