Lab 10: Implementing Injection Defenses

Topic 10 — Secure Development Methodologies IV

DVWA Source Code + Parameterized Queries + Command Sanitization

Lab Purpose

This is the final lab of the course and focuses on the developer's perspective. Students examine the vulnerable PHP source code behind DVWA's injection modules and understand exactly why the low-security code is exploitable. They then study how DVWA's high-security version fixes the code, connecting theory (Topic 10: Defending Against Injection, DoS, Third-Party) to actual implementation.

Tools Required

DVWA (Source Code View) Text Editor

Planned Tasks

  • Task 1 — SQL Injection: Vulnerable vs Secure Code: Open DVWA SQL Injection at LOW security and view the source code. Identify the string concatenation vulnerability. Then switch to HIGH and view the parameterized query. Write a side-by-side comparison explaining what changed and why it is now safe.
  • Task 2 — Command Injection: Vulnerable vs Secure Code: Open DVWA Command Injection at LOW security and view the source code. Identify the unsanitized shell call. Switch to HIGH and observe the whitelisting/sanitization applied. Explain the defense mechanism.
  • Task 3 — DoS Awareness: Discuss (no hands-on) two types of DoS relevant to web apps: resource exhaustion (too many requests) and application-level DoS (causing server errors through input). Identify one DVWA vulnerability that could lead to application-level DoS if exploited at scale.
  • Task 4 — Third-Party Dependencies: List three hypothetical third-party libraries a web app might use. For each, explain what would happen if that library had an unpatched vulnerability and how you would detect and manage it (e.g., dependency scanners, version pinning).

Learning Outcomes

  • Read and understand vulnerable vs secure code for SQL and Command injection
  • Explain parameterized queries and input whitelisting as concrete defense techniques
  • Understand DoS from an application security perspective
  • Describe a strategy for managing third-party dependency risks
Instructor Note: This lab is primarily a code-reading and analysis exercise. No exploitation needed — students have already done that. The focus is on understanding the FIX, not the attack. Good lab for a written deliverable (short report or worksheet).
